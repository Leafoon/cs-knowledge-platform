# 第26章：评测与基准 — Agent 能力度量

评测是 Agent 开发的关键环节。没有评测，我们就无法知道 Agent 的能力边界，也无法有效地改进它。本章系统讲解 Agent 评测体系的构建。

---

## 26.1 评测维度

### 26.1.1 核心评估指标

| 维度 | 指标 | 说明 | 重要性 |
|:---|:---|:---|:---|
| 任务完成 | Success Rate / Pass Rate | 任务成功完成的百分比 | ★★★★★ |
| 推理质量 | Reasoning Accuracy | 推理步骤的正确性 | ★★★★ |
| 工具使用 | Tool Call Accuracy | 工具选择和参数正确率 | ★★★★★ |
| 效率 | Steps / Tokens / Cost | 完成任务的资源消耗 | ★★★★ |
| 安全性 | Safety Score | 安全违规的比例 | ★★★★★ |
| 延迟 | Latency (P50/P95/P99) | 任务完成的时间分布 | ★★★ |
| 可靠性 | Consistency | 多次执行的一致性 | ★★★★ |
| 用户体验 | User Satisfaction | 用户满意度评分 | ★★★★ |

### 26.1.2 评估指标详解

**任务完成率（Success Rate）**：最直接的指标，衡量 Agent 成功完成任务的比例。但任务完成的定义需要谨慎——什么算"成功完成"？对于代码修复任务，成功意味着修复后的代码通过所有测试。对于 Web 操作任务，成功意味着到达目标页面并完成指定操作。

**推理准确性（Reasoning Accuracy）**：评估 Agent 的推理过程是否正确。这不仅包括最终答案的正确性，还包括中间推理步骤的合理性。Agent 可能通过错误的推理过程碰巧得到了正确的答案——这种情况应该被判定为推理不正确。

**工具使用效率（Tool Call Accuracy）**：评估 Agent 选择和调用工具的能力。可以进一步分解为：工具选择准确率（Agent 是否选择了正确的工具）、参数生成准确率（Agent 生成的参数是否正确）和工具调用成功率（工具调用是否成功执行）。

**Token 效率（Tokens per Task）**：衡量完成任务消耗的 Token 数。Token 效率直接影响 Agent 的成本和延迟。提升 Token 效率的方法包括：压缩 Prompt、减少中间推理步骤、使用更小的模型处理简单任务。

**延迟（Latency）**：衡量任务完成的时间分布。延迟通常用百分位数来表示：P50（中位数延迟）、P95（95% 的请求延迟低于此值）和 P99。对于实时交互场景，P95 延迟应该控制在 10 秒以内。

**可靠性（Consistency）**：衡量 Agent 多次执行同一任务的一致性。高可靠性的 Agent 在多次执行同一任务时，应该得到相同或相似的结果。提升可靠性的方法包括：使用低 temperature、增加 Self-Consistency 采样次数、添加结果验证步骤。

## 26.3 LLM-as-Judge

### 26.3.1 基本实现

```python
def llm_as_judge(question, answer, reference, llm):
    prompt = f"""评估以下回答的质量。

问题：{question}
参考答案：{reference}
待评答案：{answer}

请从以下维度评分（1-5分）：
1. 准确性：答案是否正确
2. 完整性：是否涵盖了所有要点
3. 相关性：是否切题

输出 JSON 格式："""

    response = llm.invoke([HumanMessage(content=prompt)])
    import json
    try:
        return json.loads(response.content)
    except:
        return {"score": 3, "feedback": response.content}
```

### 26.3.2 多维度评估

```python
def multi_dimension_judge(question, answer, reference, llm, criteria=None):
    if criteria is None:
        criteria = ["准确性", "完整性", "相关性", "可读性", "创造性"]

    prompt = f"""评估以下回答。

问题：{question}
参考：{reference}
回答：{answer}

评估维度：{', '.join(criteria)}

对每个维度评分（1-5），并给出总体评价。"""

    response = llm.invoke([HumanMessage(content=prompt)])
    return {"feedback": response.content}
```

## 26.4 评测管线

```python
class AgentEvaluator:
    def __init__(self, agent, test_cases):
        self.agent = agent
        self.test_cases = test_cases
        self.results = []

    def evaluate(self):
        import time
        for case in self.test_cases:
            start = time.time()
            result = self.agent.run(case["input"])
            latency = time.time() - start
            score = 1.0 if case.get("expected", "") in result else 0.0
            self.results.append({
                "input": case["input"],
                "expected": case.get("expected", ""),
                "actual": result,
                "score": score,
                "latency": latency,
            })
        return self._summarize()

    def _summarize(self):
        scores = [r["score"] for r in self.results]
        latencies = [r["latency"] for r in self.results]
        return {
            "total_cases": len(self.results),
            "success_rate": sum(1 for s in scores if s >= 0.8) / len(scores),
            "avg_score": sum(scores) / len(scores),
            "avg_latency": sum(latencies) / len(latencies),
            "p95_latency": sorted(latencies)[int(len(latencies) * 0.95)],
        }
```

## 26.5 评估方法论

### 26.5.1 评估维度的全面性

一个 Agent 可能在某个维度上表现优秀，但在其他维度上表现不佳。评估时需要综合考虑多个维度，根据应用场景确定各维度的权重。

### 26.5.2 评估数据集的代表性

Benchmark 的任务分布应该与实际应用场景匹配。如果使用与实际场景不匹配的 Benchmark 进行评估，评估结果可能没有实际参考价值。

### 26.5.3 评估的可重复性

Agent 的行为具有一定的随机性，多次执行同一任务可能得到不同的结果。评估时需要多次运行取平均值，并报告方差或置信区间。

### 26.5.4 评估的人类参与

对于复杂任务，自动评估可能不够准确。评估 Agent 生成的代码质量，不仅要看测试是否通过，还要看代码的可读性、可维护性和安全性。

## 26.6 评测深度解析

### 26.6.1 评估维度的权衡

不同评估维度之间存在权衡关系。提高推理准确性通常会增加 Token 消耗（因为需要更详细的推理过程），从而降低 Token 效率。提高安全性可能增加延迟（因为需要额外的检查步骤）。在实际评估中，需要根据应用场景的优先级来权衡这些维度。

例如，对于金融场景的 Agent，安全性和准确性是最重要的，可以容忍较高的成本和延迟。对于客服场景的 Agent，延迟和用户体验是最重要的，需要在质量和速度之间找到平衡。

### 26.6.2 评估数据集的设计

一个好的评估数据集应该具有以下特征：

**代表性**：数据集的任务分布应该与实际应用场景匹配。如果你的 Agent 主要用于代码修复，那么评估数据集应该以代码修复任务为主。

**多样性**：数据集应该覆盖各种难度级别和任务类型。简单任务可以验证 Agent 的基本能力，复杂任务可以测试 Agent 的极限。

**可验证性**：每个任务都应该有明确的成功标准。对于代码修复任务，成功标准是测试通过；对于问答任务，成功标准是答案正确。

**时效性**：数据集应该定期更新，以反映最新的技术发展和应用场景。

### 26.6.3 评估结果的解读

评估结果需要谨慎解读。高评估分数不一定意味着 Agent 在实际应用中表现良好，因为评估数据集可能与实际场景存在差异。低评估分数也不一定意味着 Agent 没有价值，因为评估数据集可能过于严格。

一个更好的评估方法是：在评估数据集上建立基线，然后在实际用户场景中进行 A/B 测试，比较 Agent 与现有解决方案的差异。

### 26.7.4 评估的最佳实践

1. **建立基线**：在评估开始前，建立一个简单的基线方案，作为比较的基准。
2. **多次运行**：由于 Agent 的行为具有随机性，每个测试用例应该运行多次，取平均值。

4. **分析失败案例**：重点关注失败的测试用例，分析失败的原因，找出改进方向。
5. **持续评估**：评估不是一次性的，而是持续的过程。每次模型更新或 Prompt 修改后，都应该重新评估。
