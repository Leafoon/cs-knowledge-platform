## 1.1 现实类比：一家餐厅

<div data-component="ProcessVsThreadVsCoroutine"></div>

在深入技术细节之前，我们先用一个生活中的例子来建立直觉。想象一下"并发"这个概念在餐厅中的体现。

### 进程 = 不同的餐厅

```
┌─────────────────────────────────────────────────────────────────────┐
│                         一条美食街                                   │
│                                                                     │
│   ┌──────────────┐    ┌──────────────┐    ┌──────────────┐         │
│   │  中餐厅 A     │    │  西餐厅 B     │    │  日料店 C     │         │
│   │              │    │              │    │              │         │
│   │  自己的厨房   │    │  自己的厨房   │    │  自己的厨房   │         │
│   │  自己的食材   │    │  自己的食材   │    │  自己的食材   │         │
│   │  自己的厨师   │    │  自己的厨师   │    │  自己的厨师   │         │
│   │  自己的账本   │    │  自己的账本   │    │  自己的账本   │         │
│   └──────────────┘    └──────────────┘    └──────────────┘         │
│                                                                     │
│   各自独立运作，互不影响。一家餐厅关门不会影响隔壁。                     │
│   但要交换食材或共享资源，需要专门的物流通道（进程间通信）。              │
└─────────────────────────────────────────────────────────────────────┘
```

**进程的关键特点：**
- 每家餐厅有**完全独立**的厨房、食材库、账本
- 一家餐厅着火了，隔壁餐厅不受影响（**隔离性强**）
- 如果中餐厅想借西餐厅的奶油，需要走专门的物流通道（**IPC - 进程间通信**）
- 开一家新餐厅需要大量投资：装修、采购、招人（**创建成本高**）

### 线程 = 同一家餐厅里的多个厨师

```
┌─────────────────────────────────────────────────────────────────────┐
│                        一家中餐厅的厨房                              │
│                                                                     │
│   ┌─────────────┐  ┌─────────────┐  ┌─────────────┐               │
│   │   厨师 1     │  │   厨师 2     │  │   厨师 3     │               │
│   │  (线程 1)    │  │  (线程 2)    │  │  (线程 3)    │               │
│   └──────┬──────┘  └──────┬──────┘  └──────┬──────┘               │
│          │                │                │                       │
│          └────────────────┼────────────────┘                       │
│                           │                                        │
│                    ┌──────▼──────┐                                 │
│                    │  共享厨房    │  ← 共享的厨具、食材、灶台          │
│                    │  (进程内存)  │                                 │
│                    └─────────────┘                                 │
│                                                                     │
│   厨师们共享同一套厨具和食材库。交换数据方便，                           │
│   但也容易出问题：两个人同时抢同一把刀（竞态条件）。                     │
└─────────────────────────────────────────────────────────────────────┘
```

**线程的关键特点：**
- 多个厨师**共享**同一个厨房、同一批食材、同一套厨具
- 传递菜品只需要放在传菜台上就行（**共享内存，通信成本低**）
- 但两个厨师可能同时去拿同一把刀（**竞态条件**）
- 如果老板只允许一个厨师用灶台（**GIL**），那其他人只能等着

### 协程 = 一个厨师同时照看多口锅

```
┌─────────────────────────────────────────────────────────────────────┐
│                    一个厨师的时间线                                   │
│                                                                     │
│  时间 ──────────────────────────────────────────────────────────►   │
│                                                                     │
│  ┌─────┐         ┌─────┐         ┌─────┐                          │
│  │切菜A│ ──等待── │切菜B│ ──等待── │翻炒A│ ──等待── │调味B│          │
│  │     │  锅烧水  │     │  油热了  │     │  收汁   │     │          │
│  └─────┘         └─────┘         └─────┘         └─────┘          │
│     │               │               │               │              │
│     ▼               ▼               ▼               ▼              │
│  ┌─────┐        ┌─────┐        ┌─────┐        ┌─────┐            │
│  │锅 A │        │锅 B │        │锅 A │        │锅 B │             │
│  │烧水中│       │热油中│       │翻炒中│       │调味中│             │
│  └─────┘        └─────┘        └─────┘        └─────┘            │
│                                                                     │
│  核心：一个厨师（线程），多口锅（协程）。                               │
│  一口锅在等待时，厨师转去处理另一口锅。                                 │
│  任何时候只有一个锅在被操作，但所有锅都在"推进"。                       │
└─────────────────────────────────────────────────────────────────────┘
```

**协程的关键特点：**
- 始终只有**一个厨师**（单线程），但同时照看**多口锅**（多个协程）
- 锅里的水在烧的时候，厨师不需要站在旁边等（**I/O 等待时让出执行权**）
- 厨师自己决定什么时候切换到下一口锅（**程序主动调度，不是操作系统**）
- 几乎没有切换成本，只需要转身看另一口锅（**用户态切换，极轻量**）

### 三者的类比总结

```
┌──────────────┬──────────────────┬──────────────────┬──────────────────┐
│   类比维度    │      进程        │      线程        │      协程        │
├──────────────┼──────────────────┼──────────────────┼──────────────────┤
│ 餐厅类比     │ 不同餐厅         │ 同一厨房的厨师   │ 一个厨师多口锅   │
│ 执行单位     │ 独立程序实例     │ 进程内的执行流   │ 协作式任务       │
│ 资源共享     │ 不共享           │ 共享进程内存     │ 共享线程内存     │
│ 调度方式     │ 操作系统         │ 操作系统         │ 程序自己         │
│ 切换成本     │ 高（换餐厅）     │ 中（换厨师）     │ 低（转个身）     │
│ 隔离性       │ 强               │ 弱               │ 弱               │
│ 通信方式     │ IPC（物流通道）  │ 共享变量（传菜台）│ await（交接锅）  │
└──────────────┴──────────────────┴──────────────────┴──────────────────┘
```

## 1.2 进程是什么

进程（Process）是操作系统**分配资源的基本单位**。每一个正在运行的程序，至少对应一个进程。

### 进程的定义

当你在终端输入 `python server.py` 并回车时，操作系统会：

1. 创建一个新的进程
2. 为它分配一块独立的内存空间
3. 加载 Python 解释器的代码到内存
4. 开始执行 `server.py` 的字节码

```
┌─────────────────────────────────────────────────────────────────┐
│                        操作系统                                  │
│                                                                 │
│  ┌───────────────────────┐    ┌───────────────────────┐        │
│  │      进程 A            │    │      进程 B            │        │
│  │    (python server.py)  │    │    (python worker.py)  │        │
│  │                        │    │                        │        │
│  │  ┌──────────────────┐ │    │  ┌──────────────────┐ │        │
│  │  │   代码段          │ │    │  │   代码段          │ │        │
│  │  │   (Python 字节码) │ │    │  │   (Python 字节码) │ │        │
│  │  └──────────────────┘ │    │  └──────────────────┘ │        │
│  │  ┌──────────────────┐ │    │  ┌──────────────────┐ │        │
│  │  │   数据段          │ │    │  │   数据段          │ │        │
│  │  │   (全局变量等)    │ │    │  │   (全局变量等)    │ │        │
│  │  └──────────────────┘ │    │  └──────────────────┘ │        │
│  │  ┌──────────────────┐ │    │  ┌──────────────────┐ │        │
│  │  │   堆             │ │    │  │   堆             │ │        │
│  │  │   (动态分配内存)  │ │    │  │   (动态分配内存)  │ │        │
│  │  └──────────────────┘ │    │  └──────────────────┘ │        │
│  │  ┌──────────────────┐ │    │  ┌──────────────────┐ │        │
│  │  │   栈             │ │    │  │   栈             │ │        │
│  │  │   (函数调用信息)  │ │    │  │   (函数调用信息)  │ │        │
│  │  └──────────────────┘ │    │  └──────────────────┘ │        │
│  │                        │    │                        │        │
│  │  文件描述符表          │    │  文件描述符表          │        │
│  │  PID: 1234            │    │  PID: 5678            │        │
│  └───────────────────────┘    └───────────────────────┘        │
│                                                                 │
│        两块内存完全独立，互不干扰                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 进程的内存布局

每个进程都有自己独立的虚拟地址空间，通常包含以下区域：

| 内存区域 | 存放内容 | 特点 |
|---------|---------|------|
| 代码段（Text） | 程序的机器指令 | 只读，共享 |
| 数据段（Data） | 全局变量、静态变量 | 可读写 |
| 堆（Heap） | 动态分配的内存（`malloc`/`new`） | 向高地址增长 |
| 栈（Stack） | 函数调用帧、局部变量 | 向低地址增长 |
| 文件描述符表 | 打开的文件、网络连接 | 进程级别 |

### 进程之间为什么互相隔离

操作系统的虚拟内存机制保证了进程之间的隔离：

```python
# process_a.py
import os

# 进程 A 设置一个全局变量
shared_value = 100
print(f"进程 A (PID={os.getpid()}): shared_value = {shared_value}")

# 进程 A 的内存空间与进程 B 完全独立
# 修改进程 A 的变量不会影响进程 B
```

```python
# process_b.py
import os

# 进程 B 有自己独立的 shared_value
shared_value = 200
print(f"进程 B (PID={os.getpid()}): shared_value = {shared_value}")
```

```bash
# 同时运行两个进程
$ python process_a.py
进程 A (PID=1234): shared_value = 100

$ python process_b.py
进程 B (PID=5678): shared_value = 200
```

即使两个进程运行相同的代码、使用相同的变量名，它们的内存也是完全独立的。进程 A 修改 `shared_value` 不会影响进程 B。

### 使用 multiprocessing 创建进程

```python
import multiprocessing
import os
import time


def worker(name, duration):
    """工作进程函数"""
    pid = os.getpid()
    print(f"  [进程 {pid}] {name}: 开始工作，预计耗时 {duration} 秒")
    time.sleep(duration)  # 模拟耗时操作
    result = duration * 10  # 模拟计算结果
    print(f"  [进程 {pid}] {name}: 工作完成，结果 = {result}")
    return result


if __name__ == "__main__":
    print(f"主进程 PID: {os.getpid()}")
    print("=" * 50)

    # 创建进程
    p1 = multiprocessing.Process(target=worker, args=("任务A", 2))
    p2 = multiprocessing.Process(target=worker, args=("任务B", 3))

    start = time.perf_counter()

    # 启动进程
    p1.start()
    p2.start()

    # 等待进程完成
    p1.join()
    p2.join()

    elapsed = time.perf_counter() - start
    print(f"=" * 50)
    print(f"总耗时: {elapsed:.2f} 秒")  # 约 3 秒，因为两个进程并行
```

输出示例：

```
主进程 PID: 1234
==================================================
  [进程 1235] 任务A: 开始工作，预计耗时 2 秒
  [进程 1236] 任务B: 开始工作，预计耗时 3 秒
  [进程 1235] 任务A: 工作完成，结果 = 20
  [进程 1236] 任务B: 工作完成，结果 = 30
==================================================
总耗时: 3.01 秒
```

### 进程间通信（IPC）

由于进程的内存是隔离的，进程之间传递数据需要专门的通信机制：

```python
import multiprocessing


def sender(queue):
    """发送端：向队列中放入数据"""
    messages = ["你好", "来自", "另一个进程"]
    for msg in messages:
        queue.put(msg)
        print(f"  [发送] {msg}")
    queue.put(None)  # 发送结束信号


def receiver(queue):
    """接收端：从队列中读取数据"""
    while True:
        msg = queue.get()  # 阻塞等待
        if msg is None:
            break
        print(f"  [接收] {msg}")


if __name__ == "__main__":
    # 创建一个进程安全的队列
    queue = multiprocessing.Queue()

    p_send = multiprocessing.Process(target=sender, args=(queue,))
    p_recv = multiprocessing.Process(target=receiver, args=(queue,))

    p_send.start()
    p_recv.start()

    p_send.join()
    p_recv.join()

    print("通信完成")
```

常见的进程间通信方式：

| 方式 | 原理 | 适用场景 |
|------|------|---------|
| Queue（队列） | 基于管道 + 锁 | 生产者-消费者模式 |
| Pipe（管道） | 内核缓冲区 | 双向通信 |
| Shared Memory | 映射同一物理内存 | 大量数据共享 |
| Manager | 基于代理对象 | 复杂数据结构 |
| 文件 | 磁盘文件 | 简单数据交换 |

### 进程的优缺点总结

```
┌──────────────────────────────────────────────────────────────────┐
│                        进程的优缺点                               │
├──────────────────────────┬───────────────────────────────────────┤
│          优点             │              缺点                     │
├──────────────────────────┼───────────────────────────────────────┤
│ ✓ 隔离性强，一个进程崩溃  │ ✗ 创建成本高（分配内存、加载程序）     │
│   不会影响其他进程        │                                       │
│                          │ ✗ 占用内存多（每进程独立地址空间）      │
│ ✓ 可以利用多个 CPU 核心   │                                       │
│   真正并行                │ ✗ 进程切换成本高（切换内存映射）        │
│                          │                                       │
│ ✓ 适合 CPU 密集型任务     │ ✗ 进程间通信麻烦（需要 IPC 机制）      │
│                          │                                       │
│ ✓ 天然的安全边界          │ ✗ 创建/销毁速度慢                     │
└──────────────────────────┴───────────────────────────────────────┘
```

## 1.3 线程是什么

线程（Thread）是操作系统**调度执行的基本单位**。一个进程中可以包含多个线程，它们共享进程的大部分资源。

### 线程的定义

如果说进程是一家餐厅，那么线程就是这家餐厅里的厨师。多个厨师在同一个厨房里工作，共用同一套厨具和食材。

```
┌─────────────────────────────────────────────────────────────────┐
│                        进程 A                                    │
│                                                                 │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐                  │
│  │  线程 1   │    │  线程 2   │    │  线程 3   │                  │
│  │          │    │          │    │          │                   │
│  │ 独有的:   │    │ 独有的:   │    │ 独有的:   │                  │
│  │ ·调用栈   │    │ ·调用栈   │    │ ·调用栈   │                  │
│  │ ·局部变量 │    │ ·局部变量 │    │ ·局部变量 │                  │
│  │ ·程序计数器│   │ ·程序计数器│   │ ·程序计数器│                 │
│  └─────┬────┘    └─────┬────┘    └─────┬────┘                  │
│        │               │               │                        │
│        └───────────────┼───────────────┘                        │
│                        │                                        │
│                 ┌──────▼──────┐                                 │
│                 │   共享资源    │                                 │
│                 │  ·全局变量    │                                 │
│                 │  ·堆内存      │                                 │
│                 │  ·文件描述符  │                                 │
│                 │  ·代码段      │                                 │
│                 └─────────────┘                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 线程共享什么、不共享什么

| 资源 | 是否共享 | 说明 |
|------|---------|------|
| 代码段 | 共享 | 执行相同的程序代码 |
| 全局变量 | 共享 | 可以直接读写，但需要注意同步 |
| 堆内存 | 共享 | 动态分配的对象互相可见 |
| 文件描述符 | 共享 | 一个线程打开的文件，其他线程也能读写 |
| 调用栈 | **不共享** | 每个线程有自己的调用栈 |
| 局部变量 | **不共享** | 函数内的变量是线程私有的 |
| 程序计数器 | **不共享** | 每个线程执行到不同的位置 |

### 使用 threading.Thread 创建线程

```python
import threading
import time


def download(url, duration):
    """模拟下载任务"""
    thread_name = threading.current_thread().name
    print(f"  [{thread_name}] 开始下载: {url}")
    time.sleep(duration)  # 模拟网络 I/O 等待
    size = duration * 1024  # 模拟下载的数据量
    print(f"  [{thread_name}] 下载完成: {url} ({size}KB)")
    return size


# 创建两个线程
thread1 = threading.Thread(target=download, args=("https://example.com/a.jpg", 2), name="下载线程-1")
thread2 = threading.Thread(target=download, args=("https://example.com/b.jpg", 2), name="下载线程-2")

start = time.perf_counter()

# 启动线程
thread1.start()
thread2.start()

# 等待线程完成
thread1.join()
thread2.join()

elapsed = time.perf_counter() - start
print(f"总耗时: {elapsed:.2f} 秒")  # 约 2 秒，因为两个线程并行等待
```

输出：

```
  [下载线程-1] 开始下载: https://example.com/a.jpg
  [下载线程-2] 开始下载: https://example.com/b.jpg
  [下载线程-1] 下载完成: https://example.com/a.jpg (2048KB)
  [下载线程-2] 下载完成: https://example.com/b.jpg (2048KB)
总耗时: 2.00 秒
```

### `start()` 和 `join()` 的作用

```python
import threading
import time


def task(name, duration):
    print(f"  {name}: 开始执行")
    time.sleep(duration)
    print(f"  {name}: 执行完毕")


# ─── 演示 start() 的作用 ───
print("=== start() 的作用 ===")
t = threading.Thread(target=task, args=("线程A", 1))
print("调用 start() 之前: 线程还没有开始运行")
t.start()  # start() 启动线程，让操作系统调度它
print("调用 start() 之后: 线程已经开始运行（但可能还没执行完）")
t.join()

# ─── 演示 join() 的作用 ───
print("\n=== join() 的作用 ===")
t = threading.Thread(target=task, args=("线程B", 1))
t.start()
print("调用 join() 之前: 主线程继续执行，不等待")
t.join()  # join() 阻塞当前线程，直到目标线程完成
print("调用 join() 之后: 线程B 已经完成")
```

```
=== start() 的作用 ===
调用 start() 之前: 线程还没有开始运行
  线程A: 开始执行
调用 start() 之后: 线程已经开始运行（但可能还没执行完）
  线程A: 执行完毕

=== join() 的作用 ===
调用 join() 之前: 主线程继续执行，不等待
  线程B: 开始执行
  线程B: 执行完毕
调用 join() 之后: 线程B 已经完成
```

**关键概念：**
- `start()` 启动一个新线程，让它进入可运行状态。调用 `start()` 后，新线程和当前线程**并发执行**。
- `join()` 让当前线程**阻塞等待**，直到目标线程执行完毕。类似于"等你做完我再继续"。

### 线程之间的竞态条件

线程共享内存既是优势也是风险。当多个线程同时修改共享数据时，可能出现竞态条件：

```python
import threading
import time


# 共享变量
counter = 0


def increment(n):
    """对 counter 执行 n 次 +1 操作"""
    global counter
    for _ in range(n):
        # 这里包含三个步骤：读取、加 1、写回
        # 如果两个线程同时执行，可能出现交错
        current = counter       # 步骤 1: 读取当前值
        # ── 此时另一个线程可能插入 ──
        current += 1            # 步骤 2: 加 1
        counter = current       # 步骤 3: 写回


# 创建两个线程，各执行 100000 次
t1 = threading.Thread(target=increment, args=(100000,))
t2 = threading.Thread(target=increment, args=(100000,))

t1.start()
t2.start()
t1.join()
t2.join()

print(f"期望值: 200000")
print(f"实际值: {counter}")  # 通常小于 200000！
```

输出（每次运行结果可能不同）：

```
期望值: 200000
实际值: 138492  ← 竞态条件导致数据丢失！
```

为什么结果不对？因为两个线程可能同时读取到相同的 `counter` 值，各自加 1 后写回，导致一次加法"丢失"了。

```
时间线：
  线程 1: 读取 counter=100 → 计算 100+1=101 → 写入 counter=101
  线程 2:          读取 counter=100 → 计算 100+1=101 → 写入 counter=101
  结果：counter=101，但实际应该 +2，变成了 101
```

### 使用锁解决竞态条件

```python
import threading


counter = 0
lock = threading.Lock()  # 创建一把锁


def safe_increment(n):
    """安全的计数器递增"""
    global counter
    for _ in range(n):
        with lock:  # 获取锁（如果锁被占用，等待）
            counter += 1  # 在锁的保护下操作共享变量
        # 退出 with 块时自动释放锁


t1 = threading.Thread(target=safe_increment, args=(100000,))
t2 = threading.Thread(target=safe_increment, args=(100000,))

t1.start()
t2.start()
t1.join()
t2.join()

print(f"期望值: 200000")
print(f"实际值: {counter}")  # 现在总是 200000
```

### 线程的优缺点总结

```
┌──────────────────────────────────────────────────────────────────┐
│                        线程的优缺点                               │
├──────────────────────────┬───────────────────────────────────────┤
│          优点             │              缺点                     │
├──────────────────────────┼───────────────────────────────────────┤
│ ✓ 创建成本比进程低        │ ✗ 存在竞态条件风险                    │
│                          │                                       │
│ ✓ 共享内存，通信方便      │ ✗ 需要使用锁等同步机制                │
│                          │                                       │
│ ✓ I/O 等待时可以让出 CPU  │ ✗ 调试困难（线程交错执行）            │
│                          │                                       │
│ ✓ 切换成本比进程低        │ ✗ 受 GIL 影响，CPU 密集型任务         │
│                          │   无法真正并行                        │
└──────────────────────────┴───────────────────────────────────────┘
```

## 1.4 Python 的 GIL 是什么

<div data-component="GILLimitDemo"></div>

GIL 是 Python 并发编程中最重要、也最容易被误解的概念之一。理解 GIL 是理解 Python 异步编程必要性的关键。

### GIL 的全称和定义

**GIL** = **G**lobal **I**nterpreter **L**ock（全局解释器锁）

它是 CPython 解释器中的一把**全局锁**，保护 Python 对象的访问安全。

核心规则：

```
┌─────────────────────────────────────────────────────────────────┐
│                        GIL 的核心规则                            │
│                                                                 │
│  在同一个 CPython 进程中，同一时刻只有一个线程持有 GIL，           │
│  也就是只有一个线程在执行 Python 字节码。                          │
│                                                                 │
│  ┌───────┐   ┌───────┐   ┌───────┐                             │
│  │线程 1  │   │线程 2  │   │线程 3  │                             │
│  │(执行中)│   │(等待中)│   │(等待中)│                             │
│  │ 🔒GIL  │   │       │   │       │                             │
│  └───────┘   └───────┘   └───────┘                             │
│                                                                 │
│  线程 1 持有 GIL → 线程 1 可以执行                                │
│  线程 2 没有 GIL → 线程 2 必须等待                                │
│  线程 3 没有 GIL → 线程 3 必须等待                                │
└─────────────────────────────────────────────────────────────────┘
```

### GIL 存在的原因

为什么 CPython 要有 GIL？主要原因有两个：

**1. 简化 CPython 的实现**

CPython 的内存管理使用引用计数（reference counting）。每个 Python 对象都有一个引用计数器，记录有多少变量引用了这个对象：

```python
import sys

a = [1, 2, 3]        # 列表对象的引用计数 = 1
b = a                # 引用计数 = 2
print(sys.getrefcount(a))  # 输出 3（getrefcount 自身也算一次）

del b                # 引用计数 = 2
# 当引用计数降为 0 时，对象被自动回收
```

如果没有 GIL，两个线程可能同时修改同一个对象的引用计数，导致内存泄漏或崩溃。GIL 通过"同一时刻只有一个线程执行"来避免这个问题。

**2. 保护 C 扩展库的安全**

很多 Python 的 C 扩展库（如 NumPy）在设计时假设同一时刻只有一个线程执行 Python 代码。GIL 保证了这些库的线程安全。

### GIL 对 CPU 密集型任务的影响

对于纯 Python 的 CPU 密集型代码，多线程**几乎不能**提升性能，甚至可能更慢：

```python
import threading
import time


def cpu_task(n):
    """CPU 密集型任务：计算累加和"""
    total = 0
    for i in range(n):
        total += i
    return total


# ─── 单线程执行 ───
N = 100_000_000
start = time.perf_counter()
cpu_task(N)
single_time = time.perf_counter() - start
print(f"单线程: {single_time:.3f} 秒")

# ─── 双线程执行 ───
start = time.perf_counter()
t1 = threading.Thread(target=cpu_task, args=(N // 2,))
t2 = threading.Thread(target=cpu_task, args=(N // 2,))
t1.start()
t2.start()
t1.join()
t2.join()
multi_time = time.perf_counter() - start
print(f"双线程: {multi_time:.3f} 秒")

# ─── 对比 ───
speedup = single_time / multi_time
print(f"加速比: {speedup:.2f}x")  # 通常 < 1.0，甚至可能更慢！
```

典型输出：

```
单线程: 5.234 秒
双线程: 5.891 秒
加速比: 0.89x  ← 双线程反而更慢了！
```

为什么更慢？因为线程切换本身有开销，而 GIL 又阻止了真正的并行：

```
单线程时间线：
  [====================== 计算 ======================]
  总时间: T

双线程时间线（受 GIL 限制）：
  线程1: [=== 计算 ===][切换][=== 计算 ===][切换][=== 计算 ===]
  线程2:              [切换][=== 计算 ===][切换][=== 计算 ===]
  总时间: T + 切换开销 > T
```

### GIL 对 I/O 密集型任务的影响

好消息是：**I/O 等待时，线程会释放 GIL！**

当一个线程执行 `time.sleep()`、网络请求、文件读写等 I/O 操作时，它会主动释放 GIL，让其他线程有机会执行：

```python
import threading
import time


def io_task(name, duration):
    """I/O 密集型任务：模拟网络请求"""
    print(f"  {name}: 开始请求")
    time.sleep(duration)  # I/O 等待，释放 GIL
    print(f"  {name}: 请求完成")


# ─── 单线程顺序执行 ───
start = time.perf_counter()
io_task("任务1", 2)
io_task("任务2", 2)
io_task("任务3", 2)
single_time = time.perf_counter() - start
print(f"单线程总耗时: {single_time:.2f} 秒")  # 约 6 秒

# ─── 多线程并发执行 ───
start = time.perf_counter()
t1 = threading.Thread(target=io_task, args=("任务1", 2))
t2 = threading.Thread(target=io_task, args=("任务2", 2))
t3 = threading.Thread(target=io_task, args=("任务3", 2))
t1.start()
t2.start()
t3.start()
t1.join()
t2.join()
t3.join()
multi_time = time.perf_counter() - start
print(f"多线程总耗时: {multi_time:.2f} 秒")  # 约 2 秒
```

输出：

```
  任务1: 开始请求
  任务1: 请求完成
  任务2: 开始请求
  任务2: 请求完成
  任务3: 开始请求
  任务3: 请求完成
单线程总耗时: 6.00 秒

  任务1: 开始请求
  任务2: 开始请求
  任务3: 开始请求
  任务1: 请求完成
  任务2: 请求完成
  任务3: 请求完成
多线程总耗时: 2.00 秒
```

I/O 等待时的 GIL 行为：

```
时间线（I/O 密集型，3 个线程）：

  线程1: [请求][~~~等待 GIL~~~][处理]
  线程2:          [请求][~~~等待 GIL~~~][处理]
  线程3:                   [请求][~~~等待 GIL~~~][处理]
                 ↑          ↑          ↑
           线程1 sleep    线程2 sleep   线程3 sleep
           释放 GIL       释放 GIL      释放 GIL
           线程2 获得      线程3 获得     线程1 恢复
```

### GIL 的工作原理（简化版）

```
┌─────────────────────────────────────────────────────────────────┐
│                    GIL 的工作周期                                 │
│                                                                 │
│  1. 线程获取 GIL                                                 │
│     └── 执行 Python 字节码                                       │
│                                                                 │
│  2. 每执行一定数量的字节码（默认 5ms 或 100 条指令）                │
│     └── 线程释放 GIL                                             │
│                                                                 │
│  3. 所有等待 GIL 的线程竞争                                       │
│     └── 获胜者得到 GIL，回到步骤 1                                │
│                                                                 │
│  特殊情况：                                                       │
│  - I/O 操作前：线程主动释放 GIL                                   │
│  - I/O 完成后：线程重新竞争 GIL                                   │
│  - C 扩展代码：可以在执行 C 代码时释放 GIL（如 NumPy）              │
└─────────────────────────────────────────────────────────────────┘
```

### GIL 的本质总结

```python
# 这段代码说明了 GIL 的核心限制
import threading

# CPU 密集型：GIL 限制 → 多线程不能并行
def cpu_bound():
    # 每一步都在执行 Python 字节码
    # GIL 保证同一时刻只有一个线程在执行
    # 所以多线程 == 交替执行，不是并行
    for i in range(10_000_000):
        _ = i * i

# I/O 密集型：GIL 不是瓶颈
def io_bound():
    # 大部分时间在等待 I/O
    # 等待时释放 GIL，其他线程可以执行
    import time
    time.sleep(1)  # 释放 GIL
    # I/O 完成后重新竞争 GIL
```

> **记住这个口诀：**
> - **CPU 密集型** + 多线程 = 受 GIL 限制，几乎无法加速
> - **I/O 密集型** + 多线程 = GIL 不是瓶颈，可以有效并发

### Python 3.13 的 Free-Threaded CPython

值得注意的是，Python 3.13 引入了实验性的 Free-Threaded 模式（PEP 703），可以禁用 GIL：

```bash
# Python 3.13+ 编译时启用 free-threading
# 这是一个实验性特性，需要在编译 Python 时指定
./configure --disable-gil
make
```

在 Free-Threaded 模式下，多线程可以真正并行执行 CPU 密集型任务。但这是一个渐进的过程，整个 Python 生态系统还在适配中。目前（2026年）大多数生产环境仍然使用标准 CPython（带 GIL）。

## 1.5 协程是什么

协程（Coroutine）是一种**可以暂停和恢复的函数**。它是 Python 异步编程的核心概念。

### 协程的定义

协程是一种特殊的函数，它可以在执行过程中**主动暂停**，让出控制权给其他协程，然后在适当的时候**从暂停的地方继续执行**。

```python
# 这就是一个协程
async def fetch_data(url):
    print(f"开始请求: {url}")
    result = await http_get(url)  # 执行到这里时，协程暂停
    print(f"请求完成: {url}")     # I/O 完成后，协程从这里继续
    return result
```

关键点：
- 使用 `async def` 定义协程函数
- 使用 `await` 标记"可以让出控制权"的点
- `await` 只能在 `async def` 函数内部使用

### 协程不是线程

这是一个非常重要的概念区分：

```
线程模型：
  ┌───────┐ ┌───────┐ ┌───────┐
  │线程 1  │ │线程 2  │ │线程 3  │    ← 3 个线程，由操作系统调度
  └───────┘ └───────┘ └───────┘
  多个线程可以同时运行（受 GIL 限制）

协程模型：
  ┌─────────────────────────────────────┐
  │            线程（只有一个）           │
  │  ┌───────┐ ┌───────┐ ┌───────┐    │
  │  │协程 A  │ │协程 B  │ │协程 C  │    │ ← 多个协程，由事件循环调度
  │  └───────┘ └───────┘ └───────┘    │
  └─────────────────────────────────────┘
  多个协程交替执行，同一时刻只有一个在运行
```

### 从生成器到协程

协程的概念其实源于生成器。理解生成器的 `yield` 机制有助于理解协程的 `await`：

```python
# 生成器：可以暂停和恢复的函数
def countdown(n):
    """一个简单的生成器"""
    print(f"开始倒计时: {n}")
    while n > 0:
        yield n  # 暂停，返回 n，等待 next() 调用
        n -= 1
    print("倒计时结束")


# 使用生成器
gen = countdown(3)
print(next(gen))  # 输出: 开始倒计时: 3 \n 3
print(next(gen))  # 输出: 2
print(next(gen))  # 输出: 1
print(next(gen))  # 输出: 倒计时结束，然后抛出 StopIteration
```

生成器的执行流程：

```
调用 countdown(3)  → 创建生成器对象（不执行代码）
next(gen)          → 执行到 yield n → 暂停，返回 3
next(gen)          → 从 yield 之后继续 → 执行到 yield n → 暂停，返回 2
next(gen)          → 从 yield 之后继续 → 执行到 yield n → 暂停，返回 1
next(gen)          → 循环结束 → 抛出 StopIteration
```

协程的 `await` 和生成器的 `yield` 在概念上类似：

```python
# 生成器版本
def gen_example():
    print("开始")
    result = yield some_value  # 暂停，等待 send() 传入值
    print(f"收到: {result}")

# 协程版本
async def coro_example():
    print("开始")
    result = await some_coroutine()  # 暂停，等待异步操作完成
    print(f"收到: {result}")
```

区别在于：
- 生成器主要用于**产生值**（迭代器模式）
- 协程主要用于**等待异步操作**（异步编程模式）
- 协程有专门的事件循环来调度，而生成器需要手动调用 `next()`

### async/await 的语法

```python
import asyncio


# 定义一个协程函数
async def hello():
    print("Hello")
    await asyncio.sleep(1)  # 暂停 1 秒，让出控制权
    print("World")


# 运行协程
asyncio.run(hello())
```

输出：

```
Hello
（等待 1 秒）
World
```

### 协程的暂停与恢复

让我们用一个详细的例子来理解协程的暂停和恢复：

```python
import asyncio


async def task_a():
    print("A: 开始执行")
    print("A: 执行第一步")
    await asyncio.sleep(1)  # 暂停：让出控制权给事件循环
    # ── 此时 task_b 可能开始执行 ──
    print("A: 从暂停处继续执行")
    print("A: 执行第二步")
    await asyncio.sleep(1)  # 再次暂停
    print("A: 执行完毕")


async def task_b():
    print("B: 开始执行")
    await asyncio.sleep(0.5)  # 暂停 0.5 秒
    print("B: 从暂停处继续执行")
    await asyncio.sleep(0.5)
    print("B: 执行完毕")


async def main():
    # 并发运行两个协程
    await asyncio.gather(task_a(), task_b())


asyncio.run(main())
```

输出：

```
A: 开始执行
A: 执行第一步
B: 开始执行
B: 从暂停处继续执行
A: 从暂停处继续执行
A: 执行第二步
B: 执行完毕
A: 执行完毕
```

执行时间线：

```
时间 ─────────────────────────────────────────────────────►

task_a: [执行][──sleep 1s──][执行][──sleep 1s──][执行]
task_b:      [执行][─sleep 0.5s─][执行][─sleep 0.5s─][执行]

事件循环在 task_a sleep 时切换到 task_b，实现了并发
```

### 协程的本质：可等待对象

在 Python 中，协程是一个**可等待对象（Awaitable）**。当你写 `await xxx` 时，`xxx` 必须是一个可等待对象。

有三种可等待对象：

```python
import asyncio


# 1. 协程（Coroutine）—— 最常用
async def my_coroutine():
    await asyncio.sleep(1)
    return 42

# 2. 任务（Task）—— 对协程的包装，可以并发执行
async def demo():
    task = asyncio.create_task(my_coroutine())  # 创建任务
    result = await task  # 等待任务完成
    return result

# 3. Future —— 底层的可等待对象，通常不需要直接使用
# Task 继承自 Future
```

### 为什么协程不是线程

这是一个常见的误解，需要澄清：

```
┌──────────────────────────────────────────────────────────────────┐
│                    协程 vs 线程的核心区别                          │
├──────────────────┬───────────────────────────────────────────────┤
│      线程         │                 协程                          │
├──────────────────┼───────────────────────────────────────────────┤
│ 由操作系统调度    │ 由程序中的事件循环调度                         │
│                  │                                               │
│ 可能随时被切换    │ 只在 await 处切换                              │
│ （抢占式）       │ （协作式）                                     │
│                  │                                               │
│ 需要锁保护共享数据│ 通常不需要锁（单线程内执行）                    │
│                  │                                               │
│ 有系统开销       │ 几乎无开销                                     │
│                  │                                               │
│ 可以利用多核     │ 单个事件循环在一个线程中                         │
└──────────────────┴───────────────────────────────────────────────┘
```

## 1.6 谁负责调度

进程和线程由操作系统调度，协程由程序中的事件循环调度。这个区别决定了它们的行为特征。

### 操作系统调度（进程 & 线程）

```
┌─────────────────────────────────────────────────────────────────┐
│                    操作系统调度器                                  │
│                                                                 │
│   就绪队列：  [线程A] [线程B] [线程C] [线程D]                    │
│                              │                                  │
│                              ▼                                  │
│                    ┌─────────────────┐                          │
│                    │   调度算法       │                          │
│                    │ (时间片轮转等)   │                          │
│                    └────────┬────────┘                          │
│                             │                                   │
│                             ▼                                   │
│                    ┌─────────────────┐                          │
│                    │   CPU 执行       │                          │
│                    │   选中的线程     │                          │
│                    └─────────────────┘                          │
│                                                                 │
│   操作系统决定哪个线程在什么时候运行，程序无法控制                   │
└─────────────────────────────────────────────────────────────────┘
```

操作系统调度的特点：
- **抢占式调度**：操作系统可以在任何时候暂停一个线程，切换到另一个线程
- **时间片轮转**：每个线程分配一个时间片（通常几毫秒），用完就被切换
- **不可预测**：程序无法知道什么时候会被切换
- **公平性**：操作系统保证每个线程都能获得 CPU 时间

### 事件循环调度（协程）

```
┌─────────────────────────────────────────────────────────────────┐
│                      事件循环 (Event Loop)                       │
│                                                                 │
│   待执行队列：  [协程A] [协程B] [协程C]                          │
│                              │                                  │
│                              ▼                                  │
│                    ┌─────────────────┐                          │
│                    │   事件循环       │                          │
│                    │   检查哪个协程   │                          │
│                    │   可以执行       │                          │
│                    └────────┬────────┘                          │
│                             │                                   │
│                             ▼                                   │
│                    ┌─────────────────┐                          │
│                    │   执行协程 A     │                          │
│                    │   直到遇到 await │                          │
│                    └────────┬────────┘                          │
│                             │                                   │
│                             ▼                                   │
│                    ┌─────────────────┐                          │
│                    │   协程 A 暂停    │                          │
│                    │   切换到协程 B   │                          │
│                    └─────────────────┘                          │
│                                                                 │
│   程序自己控制什么时候切换，切换点是明确的（await）                  │
└─────────────────────────────────────────────────────────────────┘
```

事件循环调度的特点：
- **协作式调度**：协程主动让出控制权（通过 `await`）
- **确定性**：切换点是程序员指定的，行为可预测
- **无抢占**：一个协程不执行 `await`，就不会让出控制权
- **高效**：切换在用户态完成，不需要系统调用

### 调度方式对比

| 特性 | 操作系统（进程/线程） | 事件循环（协程） |
|------|---------------------|----------------|
| 调度类型 | 抢占式 | 协作式 |
| 切换时机 | 任意时刻 | 仅在 `await` 处 |
| 切换控制 | 操作系统决定 | 程序员决定 |
| 切换开销 | 需要系统调用 | 用户态切换 |
| 可预测性 | 低 | 高 |
| 公平性 | 操作系统保证 | 需要程序员保证 |
| 调试难度 | 较高（并发不确定） | 较低（切换点明确） |

### 一个典型的事件循环周期

```python
import asyncio


async def show_scheduling():
    """展示事件循环的调度过程"""
    print("1. 协程开始执行")
    await asyncio.sleep(0)  # 让出控制权，但不等待
    print("2. 从 await 恢复，继续执行")
    await asyncio.sleep(0)
    print("3. 再次恢复")


# asyncio.run() 的内部流程：
# 1. 创建事件循环
# 2. 把协程加入事件循环
# 3. 事件循环开始运行：
#    - 取出一个协程
#    - 执行到 await
#    - 检查是否有其他协程可以执行
#    - 重复以上过程
# 4. 所有协程执行完毕，关闭事件循环
asyncio.run(show_scheduling())
```

## 1.7 上下文切换成本

<div data-component="ContextSwitchCost"></div>

上下文切换是指 CPU 从一个执行流切换到另一个执行流的过程。不同类型的切换，成本差异巨大。

### 什么是上下文切换

当 CPU 从执行线程 A 切换到执行线程 B 时，需要：

1. 保存线程 A 的执行状态（寄存器、程序计数器、栈指针等）
2. 加载线程 B 的执行状态
3. 可能需要刷新 CPU 缓存
4. 恢复线程 B 的执行

这个"保存-加载"的过程就是上下文切换。

### 三种切换的成本对比

```
┌───────────────────────────────────────────────────────────────────┐
│                    上下文切换成本对比                               │
│                                                                   │
│   切换成本（对数刻度）                                              │
│   │                                                               │
│   │  ██████████████████████████████████████████  进程 (~1-10 μs)  │
│   │  ████████████████████                       线程 (~1-2 μs)   │
│   │  ██                                          协程 (~0.1 μs)  │
│   │                                                               │
│   └─────────────────────────────────────────────────────────────  │
│                                                                   │
│   进程切换：需要切换内存空间、刷新 TLB、切换页表                     │
│   线程切换：需要保存/恢复寄存器、可能刷新缓存                        │
│   协程切换：只需要保存/恢复少量寄存器，在用户态完成                   │
└───────────────────────────────────────────────────────────────────┘
```

### 进程切换的成本

进程切换是最昂贵的，因为它涉及：

```
进程切换步骤：
  1. 保存进程 A 的所有寄存器状态
  2. 保存进程 A 的页表基地址
  3. 切换到进程 B 的页表  ← 这一步很昂贵
  4. 刷新 TLB (Translation Lookaside Buffer)
  5. 加载进程 B 的寄存器状态
  6. 可能需要从磁盘换入内存页
  7. 恢复进程 B 的执行

额外开销：
  - CPU 缓存失效（L1/L2/L3 缓存中的数据可能属于进程 A）
  - TLB 缓存失效（地址翻译缓存需要重建）
  - 内存带宽消耗（页表切换）
```

### 线程切换的成本

线程切换比进程切换轻量，因为它们共享同一地址空间：

```
线程切换步骤：
  1. 保存线程 A 的寄存器状态
  2. 保存线程 A 的栈指针
  3. 加载线程 B 的寄存器状态
  4. 加载线程 B 的栈指针
  5. 恢复线程 B 的执行

不需要：
  - 切换页表
  - 刷新 TLB
  - 切换内存空间

但仍需要：
  - 操作系统介入（系统调用）
  - 可能的 CPU 缓存失效
  - 内核态/用户态切换
```

### 协程切换的成本

协程切换是最轻量的，完全在用户态完成：

```python
import asyncio
import time


async def benchmark_switch(n):
    """测量协程切换的性能"""
    async def empty_coroutine():
        pass

    start = time.perf_counter()
    for _ in range(n):
        await empty_coroutine()
    elapsed = time.perf_counter() - start

    per_switch = elapsed / n * 1_000_000  # 微秒
    print(f"协程切换 {n} 次")
    print(f"总耗时: {elapsed:.4f} 秒")
    print(f"每次切换: {per_switch:.4f} 微秒")


asyncio.run(benchmark_switch(100_000))
```

典型输出：

```
协程切换 100000 次
总耗时: 0.0234 秒
每次切换: 0.2340 微秒
```

协程切换为什么这么快？

```
协程切换步骤：
  1. 保存当前协程的执行位置（程序计数器）
  2. 保存局部变量（栈帧）
  3. 切换到另一个协程的执行位置
  4. 恢复另一个协程的局部变量
  5. 继续执行

不需要：
  - 操作系统介入
  - 系统调用
  - 切换内存空间
  - 刷新任何缓存

全部在用户态完成！
```

### 实测对比

```python
import threading
import asyncio
import time
import multiprocessing


def measure_thread_switch(n):
    """测量线程切换的性能"""
    event1 = threading.Event()
    event2 = threading.Event()

    def worker():
        for _ in range(n):
            event1.wait()  # 等待信号
            event1.clear()
            event2.set()  # 发送信号

    t = threading.Thread(target=worker)
    t.start()

    start = time.perf_counter()
    for _ in range(n):
        event1.set()  # 发送信号
        event2.wait()  # 等待信号
        event2.clear()
    elapsed = time.perf_counter() - start

    t.join()
    return elapsed / n * 1_000_000  # 微秒


async def measure_coroutine_switch(n):
    """测量协程切换的性能"""
    async def a():
        for _ in range(n):
            await asyncio.sleep(0)  # 让出控制权

    start = time.perf_counter()
    await a()
    elapsed = time.perf_counter() - start
    return elapsed / n * 1_000_000  # 微秒


if __name__ == "__main__":
    N = 10_000

    # 线程切换
    thread_cost = measure_thread_switch(N)
    print(f"线程切换:   {thread_cost:.2f} 微秒/次")

    # 协程切换
    coro_cost = asyncio.run(measure_coroutine_switch(N))
    print(f"协程切换:   {coro_cost:.2f} 微秒/次")

    print(f"差距: {thread_cost / coro_cost:.1f} 倍")
```

典型输出：

```
线程切换:   5.23 微秒/次
协程切换:   0.31 微秒/次
差距: 16.9 倍
```

### 切换成本总结

| 切换类型 | 典型耗时 | 是否需要操作系统 | 主要开销来源 |
|---------|---------|----------------|-------------|
| 进程切换 | 1-10 μs | 是 | 内存空间切换、TLB 刷新 |
| 线程切换 | 1-5 μs | 是 | 内核态切换、寄存器保存 |
| 协程切换 | 0.1-0.5 μs | 否 | 用户态栈帧切换 |

> **为什么协程适合高并发？** 当你需要同时处理 10000 个连接时，每个连接的切换频率可能很高。线程切换的开销会成为瓶颈，而协程切换的开销几乎可以忽略不计。

## 1.8 I/O 密集型任务

<div data-component="IOBoundVsCPUBound"></div>

理解任务的类型是选择并发方案的关键。我们先来看 I/O 密集型任务。

### 什么是 I/O 密集型任务

I/O 密集型任务的特征是：**大部分时间花在等待外部资源上，CPU 计算时间占比很小。**

```
一个典型的 I/O 密集型任务的时间分布：

  ┌─────────────────────────────────────────────────────────────┐
  │  任务：请求一个 API 并处理响应                                 │
  │                                                             │
  │  时间 ─────────────────────────────────────────────────────►│
  │  │                                                          │
  │  │  [计算 2ms][────── 等待网络响应 500ms ──────][处理 3ms]   │
  │  │  准备请求                                解析 JSON       │
  │  │                                                          │
  │  │  CPU 使用率: (2 + 3) / 505 ≈ 1%                         │
  │  │  等待时间占比: 500 / 505 ≈ 99%                           │
  └─────────────────────────────────────────────────────────────┘

  结论：这个任务 99% 的时间都在等待，只有 1% 在做计算
```

### 常见的 I/O 密集型任务

```python
# 1. 网络请求（最常见）
import requests

response = requests.get("https://api.example.com/data")  # 等待网络
data = response.json()  # 很快

# 2. 数据库查询

conn = sqlite3.connect("database.db")
cursor = conn.execute("SELECT * FROM users WHERE id = ?", (1,))
row = cursor.fetchone()  # 等待磁盘 I/O

# 3. 文件读写
with open("large_file.txt", "r") as f:
    content = f.read()  # 等待磁盘 I/O

# 4. 用户输入
user_input = input("请输入: ")  # 等待用户

# 5. 调用外部 API
import json
import urllib.request

response = urllib.request.urlopen("https://httpbin.org/delay/1")  # 等待 1 秒
```

```
  创建 1000 个协程

  - 协程切换开销极小
  - 可以轻松处理数万并发

内存对比（1000 并发）：

  │  线程：  ████████████████████████████████████  8000 MB       │

  └──────────────────────────────────────────────────────────────┘
```

## 1.9 CPU 密集型任务

CPU 密集型任务的特征是：**大部分时间花在 CPU 计算上，几乎没有 I/O 等待。**

### 什么是 CPU 密集型任务

```
一个典型的 CPU 密集型任务的时间分布：

  ┌─────────────────────────────────────────────────────────────┐
  │  任务：计算一亿个数字的平方和                                  │

  │  │  等待时间占比: 0%                                         │
```

### CPU 密集型任务的时间分析

```python
import time


def analyze_cpu_task():
    """分析 CPU 密集型任务的时间分布"""

    start_total = time.perf_counter()

    # 步骤 1: 加载数据（少量 I/O）
    start = time.perf_counter()
    data = list(range(10_000_000))  # 生成数据
```

```
  进程 1: [════ 计算 ════]         ← CPU 核心 1
  进程 2: [════ 计算 ════]         ← CPU 核心 2
  进程 3: [════ 计算 ════]         ← CPU 核心 3
```

## 1.10 为什么 asyncio 不适合直接跑大量计算

这是初学者常犯的错误：认为只要写成 `async def`，代码就会自动变快。实际上，`async` 不是魔法粉。

### 问题演示

```python
import asyncio
import time


async def bad_cpu_task(n):
    """错误示范：在协程中做大量计算"""
    total = 0
    for i in range(n):
        total += i
    return total


async def another_task():
    """这个任务想并发执行，但会被阻塞"""
    print("另一个任务: 开始")
    await asyncio.sleep(0.1)
    print("另一个任务: 完成")


async def main():
    start = time.perf_counter()

    # 期望：两个任务并发执行
    # 实际：bad_cpu_task 会独占事件循环
    await asyncio.gather(
        bad_cpu_task(100_000_000),
        another_task()
    )

    elapsed = time.perf_counter() - start
    print(f"总耗时: {elapsed:.2f} 秒")


asyncio.run(main())
```

输出：

```
另一个任务: 开始
（等待 5 秒...）
另一个任务: 完成  ← 被延迟了 5 秒！
总耗时: 5.23 秒
```

为什么 `another_task` 被延迟了？因为 `bad_cpu_task` 中没有 `await`，它会持续占用事件循环，不让其他协程执行。

### 事件循环的工作原理

```
正常情况（有 await）：
  协程 A: [执行][await]        [执行][await]        [执行]
  协程 B:       [执行][await]        [执行][await]
  事件循环在 await 处切换，实现并发

问题情况（无 await）：
  协程 A: [══════════════ 执行 5 秒 ══════════════]
  协程 B:                                    [等待 5 秒...][执行]
  协程 A 独占事件循环，协程 B 无法执行
```

### 正确的做法

```python
import asyncio
import time


async def good_cpu_task(n, chunk_size=1_000_000):
    """正确做法：分块计算，定期让出控制权"""
    total = 0
    for i in range(0, n, chunk_size):
        # 计算一小块
        end = min(i + chunk_size, n)
        for j in range(i, end):
            total += j
        # 定期让出控制权
        await asyncio.sleep(0)
    return total


async def another_task():
    """这个任务现在可以并发执行了"""
    print("另一个任务: 开始")
    await asyncio.sleep(0.1)
    print("另一个任务: 完成")


async def main():
    start = time.perf_counter()

    # 现在两个任务可以并发执行了
    await asyncio.gather(
        good_cpu_task(100_000_000),
        another_task()
    )

    elapsed = time.perf_counter() - start
    print(f"总耗时: {elapsed:.2f} 秒")


asyncio.run(main())
```

输出：

```
另一个任务: 开始
另一个任务: 完成  ← 现在可以并发执行了！
总耗时: 5.31 秒（稍有增加，但不阻塞其他任务）
```

### 更好的方案：使用进程池

```python
import asyncio
import time
from concurrent.futures import ProcessPoolExecutor


def cpu_heavy(n):
    """CPU 密集型计算（在进程中运行）"""
    total = 0
    for i in range(n):
        total += i
    return total


async def another_task():
    """这个任务可以并发执行"""
    print("另一个任务: 开始")
    await asyncio.sleep(0.1)
    print("另一个任务: 完成")


async def main():
    start = time.perf_counter()

    # 使用进程池执行 CPU 密集型任务
    loop = asyncio.get_event_loop()
    with ProcessPoolExecutor() as executor:
        # 在进程中执行 CPU 密集型计算
        future = loop.run_in_executor(executor, cpu_heavy, 100_000_000)

        # 同时执行其他协程任务
        await another_task()

        # 等待 CPU 密集型任务完成
        result = await future

    elapsed = time.perf_counter() - start
    print(f"总耗时: {elapsed:.2f} 秒")
    print(f"计算结果: {result}")


asyncio.run(main())
```

### 总结：async 不是魔法粉

```
┌──────────────────────────────────────────────────────────────────┐
│                    async 不是魔法粉                                │
│                                                                  │
│  ✗ 错误理解：async def 会自动并行                                  │
│  ✓ 正确理解：async def 只是声明"这个函数可以暂停"                    │
│                                                                  │
│  ✗ 错误理解：把同步代码改成 async 就会变快                          │
│  ✓ 正确理解：只有 I/O 等待时才能让出控制权                          │
│                                                                  │
│  ✗ 错误理解：async 可以替代多进程                                  │
│  ✓ 正确理解：CPU 密集型任务仍然需要多进程                           │
│                                                                  │
│  记住：                                                           │
│  - async 的价值在于 I/O 并发，不在于 CPU 并行                      │
│  - 没有 await 的 async 函数等于同步阻塞                            │
│  - CPU 密集型任务应该使用多进程                                    │
└──────────────────────────────────────────────────────────────────┘
```

## 1.11 三种模型对比

### 综合对比表

| 对比维度 | 进程 (Process) | 线程 (Thread) | 协程 (Coroutine) |
|---------|---------------|--------------|-----------------|
| **调度者** | 操作系统 | 操作系统 | 事件循环（程序） |
| **内存空间** | 独立 | 共享进程内存 | 共享线程内存 |
| **创建成本** | 高（~10ms） | 中（~1ms） | 低（~0.01ms） |
| **切换成本** | 高（~1-10μs） | 中（~1-5μs） | 低（~0.1-0.5μs） |
| **内存占用** | 高（独立地址空间） | 中（共享地址空间） | 低（几 KB） |
| **CPU 并行** | ✓ 可以（多核） | ✗ 受 GIL 限制 | ✗ 单线程 |
| **I/O 并发** | 可以但成本高 | ✓ 适合 | ✓✓ 非常适合 |
| **数据共享** | 麻烦（需要 IPC） | 方便（共享内存） | 方便（共享内存） |
| **隔离性** | 强（互不影响） | 弱（共享资源） | 弱（共享资源） |
| **同步需求** | 低（隔离） | 高（需要锁） | 低（单线程） |
| **调试难度** | 中 | 高（并发不确定） | 低（切换点明确） |
| **最大并发数** | 受限于系统资源 | 受限于内存和 OS | 可达数万 |
| **适用场景** | CPU 密集型 | I/O 密集（阻塞库） | I/O 密集（异步库） |

```python
def choose_concurrency_model(task_description):
    """
    根据任务特征选择并发模型

    Args:
        task_description: 任务描述字典，包含以下字段：
            - io_bound: bool, 是否是 I/O 密集型
            - has_async_lib: bool, 是否有异步库支持
            - cpu_bound: bool, 是否是 CPU 密集型
            - need_isolation: bool, 是否需要进程隔离

    Returns:
        str: 推荐的并发模型
    """

    if task_description.get("cpu_bound"):
        return "multiprocessing"

    if task_description.get("io_bound"):
        if task_description.get("has_async_lib"):
            return "asyncio"
        else:
            return "threading"

    if task_description.get("need_isolation"):
        return "multiprocessing"

    # 默认：简单任务用线程
    return "threading"


# 测试
examples = [
    {"io_bound": True, "has_async_lib": True, "description": "请求 100 个 API"},
    {"io_bound": True, "has_async_lib": False, "description": "读取 50 个文件"},
    {"cpu_bound": True, "description": "压缩 10 个视频"},
    {"io_bound": True, "has_async_lib": True, "description": "下载 1000 张图片"},
    {"cpu_bound": True, "description": "计算矩阵乘法"},
```

## 1.13 混合场景

实际工程中，一个任务往往同时包含 I/O 和 CPU 计算。这时候需要混合使用多种并发模型。

### 场景：下载图片、压缩、保存

假设我们需要处理 100 张网络图片：
1. 从网络下载图片（I/O 密集型）
2. 对图片进行压缩（CPU 密集型）
3. 保存到磁盘（I/O 密集型）

```python
import asyncio
import aiohttp
from PIL import Image
from concurrent.futures import ProcessPoolExecutor
import io
import os


# CPU 密集型：图片压缩（在进程中运行）
def compress_image(image_data, quality=60):
    """
    压缩图片（CPU 密集型）

    Args:
        image_data: 原始图片数据（bytes）
        quality: 压缩质量（1-100）

    Returns:
        压缩后的图片数据（bytes）
    """
    img = Image.open(io.BytesIO(image_data))

    # 调整大小（如果太大）
    max_size = (1920, 1080)
    img.thumbnail(max_size, Image.Resampling.LANCZOS)

    # 压缩
    output = io.BytesIO()
    img.save(output, format='JPEG', quality=quality, optimize=True)
    return output.getvalue()


# I/O 密集型：下载图片
async def download_image(url, session):
    """
    异步下载图片

    Args:
        url: 图片 URL
        session: aiohttp 客户端会话

    Returns:
        图片数据（bytes）
    """
    async with session.get(url) as response:
        if response.status == 200:
            return await response.read()
        return None


# I/O 密集型：保存图片
async def save_image(data, filename):
    """
    异步保存图片到磁盘

    Args:
        data: 图片数据（bytes）
        filename: 保存路径
    """
    loop = asyncio.get_event_loop()
    # 使用 run_in_executor 将文件 I/O 放到线程池
    await loop.run_in_executor(None, lambda: open(filename, 'wb').write(data))


# 主流程：混合使用 asyncio 和多进程
async def process_images(urls, output_dir, max_concurrent=10):
    """
    处理多张图片：下载 → 压缩 → 保存

    Args:
        urls: 图片 URL 列表
        output_dir: 输出目录
        max_concurrent: 最大并发数
    """
    os.makedirs(output_dir, exist_ok=True)

    # 创建进程池（用于 CPU 密集型的图片压缩）
    process_pool = ProcessPoolExecutor(max_workers=4)

    # 创建信号量（控制并发数）
    semaphore = asyncio.Semaphore(max_concurrent)

    async def process_one(url, index):
        """处理单张图片"""
        async with semaphore:  # 限制并发数
            async with aiohttp.ClientSession() as session:
                # 步骤 1: 下载图片（I/O，使用 asyncio）
                print(f"  [{index}] 开始下载: {url}")
                image_data = await download_image(url, session)

                if image_data is None:
                    print(f"  [{index}] 下载失败")
                    return

                print(f"  [{index}] 下载完成，大小: {len(image_data)} bytes")

                # 步骤 2: 压缩图片（CPU，使用进程池）
                loop = asyncio.get_event_loop()
                compressed = await loop.run_in_executor(
                    process_pool,
                    compress_image,
                    image_data
                )
                print(f"  [{index}] 压缩完成，大小: {len(compressed)} bytes")

                # 步骤 3: 保存图片（I/O，使用 asyncio）
                filename = os.path.join(output_dir, f"image_{index:03d}.jpg")
                await save_image(compressed, filename)
                print(f"  [{index}] 保存完成: {filename}")

    # 并发处理所有图片
    tasks = [process_one(url, i) for i, url in enumerate(urls)]
    await asyncio.gather(*tasks)

    # 关闭进程池
    process_pool.shutdown()


# 使用示例
async def main():
    urls = [
        "https://example.com/image1.jpg",
        "https://example.com/image2.jpg",
        # ... 更多 URL
    ]

    print("开始处理图片...")
    await process_images(urls, "output/", max_concurrent=5)
    print("所有图片处理完成！")
```

## 1.14 常见误区

在学习并发编程的过程中，有一些常见的误解需要澄清。

### 误区 1：线程一定是并行的

```python
# 错误理解：创建 4 个线程就能同时使用 4 个 CPU 核心
# 现实：受 GIL 限制，同一时刻只有一个线程执行 Python 字节码

import threading
import time


def cpu_task():
    """CPU 密集型任务"""
    total = 0
    for i in range(10_000_000):
        total += i
    return total


# 测试：单线程 vs 多线程
start = time.perf_counter()
cpu_task()
single = time.perf_counter() - start

start = time.perf_counter()
threads = [threading.Thread(target=cpu_task) for _ in range(4)]
for t in threads:
    t.start()
for t in threads:
    t.join()
multi = time.perf_counter() - start

print(f"单线程: {single:.2f} 秒")
print(f"4 线程: {multi:.2f} 秒")
print(f"加速比: {single/multi:.2f}x")  # 通常 < 1.0！
```

**正确理解：**
- 线程是**并发**的（看起来同时进行），但不一定是**并行**的（真正同时执行）
- 对于 CPU 密集型任务，多线程受 GIL 限制，几乎无法加速
- 对于 I/O 密集型任务，多线程可以有效并发（因为 I/O 等待时释放 GIL）

### 误区 2：协程就是轻量级线程

```python
# 错误理解：协程和线程一样，只是更轻量
# 现实：协程的调度方式完全不同

# 线程：抢占式调度（操作系统决定什么时候切换）
# 协程：协作式调度（程序决定什么时候切换）

import asyncio


async def bad_example():
    """协程不会被抢占，可能独占事件循环"""
    # 这个循环没有 await，会一直执行，不会让出控制权
    while True:
        pass  # 其他协程永远得不到执行机会！


async def good_example():
    """正确的协程写法：定期让出控制权"""
    while True:
        # 做一些工作
        await asyncio.sleep(0)  # 让出控制权
```

**正确理解：**
- 协程是**协作式**的，必须主动让出控制权（通过 `await`）
- 线程是**抢占式**的，操作系统可以在任何时候切换
- 协程如果不让出控制权，会独占事件循环

### 误区 3：进程一定比线程快

```python
# 错误理解：进程可以并行，所以一定比线程快
# 现实：进程创建和通信成本更高

import multiprocessing
import threading
import time


def light_task():
    """轻量级任务"""
    return sum(range(1000))


if __name__ == "__main__":
    N = 100

    # 测试线程
    start = time.perf_counter()
    threads = [threading.Thread(target=light_task) for _ in range(N)]
    for t in threads:
        t.start()
    for t in threads:
        t.join()
    thread_time = time.perf_counter() - start

    # 测试进程
    start = time.perf_counter()
    with multiprocessing.Pool(4) as pool:
        pool.map(light_task, range(N))
    process_time = time.perf_counter() - start

    print(f"线程 ({N} 个): {thread_time:.3f} 秒")
    print(f"进程 ({N} 个): {process_time:.3f} 秒")
```

**正确理解：**
- 进程创建成本比线程高（需要分配内存空间）
- 进程间通信成本比线程高（需要 IPC 机制）
- 对于轻量级任务，进程的开销可能超过并行带来的收益
- 进程适合 CPU 密集型任务，线程适合 I/O 密集型任务

### 误区 4：CPU 密集型写成 async def 就会变快

```python
# 错误理解：async def 会自动并行
# 现实：async 只是声明"可以暂停"，不会改变执行速度

import asyncio
import time


async def cpu_heavy_async(n):
    """错误：在协程中做 CPU 密集型计算"""
    total = 0
    for i in range(n):
        total += i
    return total


def cpu_heavy_sync(n):
    """正确的 CPU 密集型函数（同步）"""
    total = 0
    for i in range(n):
        total += i
    return total


async def main():
    N = 10_000_000

    # 测试 async 版本
    start = time.perf_counter()
    await cpu_heavy_async(N)
    async_time = time.perf_counter() - start

    # 测试 sync 版本
    start = time.perf_counter()
    cpu_heavy_sync(N)
    sync_time = time.perf_counter() - start

    print(f"async 版本: {async_time:.3f} 秒")
    print(f"sync 版本: {sync_time:.3f} 秒")
    print(f"差异: {abs(async_time-sync_time)*1000:.1f}ms（几乎相同）")


asyncio.run(main())
```

**正确理解：**
- `async def` 只是定义一个协程函数，不会改变代码的执行速度
- `await` 只在 I/O 等待时有意义，对 CPU 计算没有帮助
- CPU 密集型任务应该使用多进程，不是 asyncio

### 误区 5：单线程就不会有数据竞争

```python
# 错误理解：单线程 + 协程不会有数据竞争
# 现实：协程在 await 点可能被其他协程插入

import asyncio


class BankAccount:
    """银行账户（演示协程中的竞态条件）"""

    def __init__(self, balance=0):
        self.balance = balance

    async def withdraw(self, amount):
        """取款（有问题的实现）"""
        if self.balance >= amount:
            # ── 这里有一个 await，可能被其他协程插入 ──
            await asyncio.sleep(0)  # 模拟异步操作
            self.balance -= amount
            return True
        return False


async def main():
    account = BankAccount(balance=100)

    # 两个协程同时取款
    results = await asyncio.gather(
        account.withdraw(80),
        account.withdraw(80)
    )

    print(f"取款结果: {results}")
    print(f"最终余额: {account.balance}")  # 可能是 -60！


asyncio.run(main())
```

输出：

```
取款结果: [True, True]
最终余额: -60  ← 数据竞争！余额变成负数了！
```

**正确理解：**
- 单线程中的协程也可能有数据竞争
- 关键在于 `await` 点：其他协程可能在这个点插入执行
- 解决方案：使用 `asyncio.Lock` 或避免在修改共享状态时 `await`

```python
# 正确实现：使用 asyncio.Lock
class SafeBankAccount:

    def __init__(self, balance=0):
        self.balance = balance
        self.lock = asyncio.Lock()

    async def withdraw(self, amount):
        async with self.lock:  # 获取锁
            if self.balance >= amount:
                await asyncio.sleep(0)
                self.balance -= amount
                return True
            return False
```

## 本节检查题

### 第 1 题

判断下面任务属于 I/O 密集型还是 CPU 密集型：

1. 请求 500 个网页
2. 计算一亿个数字的平方和

4. 对 1000 张图片执行复杂滤镜
5. 等待用户输入
6. 解压一个大文件
7. 对文本进行正则匹配
8. 调用外部 API 并处理响应

### 第 2 题

为什么多个协程可以并发执行，却通常不能称为并行执行？

### 第 3 题

在 CPython 中，为什么多线程通常不能明显加速纯 Python 的 CPU 密集型代码？

### 第 4 题

为下面任务选择方案：
- A. 调用 100 个网络 API
- B. 用同步库处理 20 个文件
- C. 编码 10 个大型视频
- D. 同时下载 1000 张图片并压缩
- E. 实时处理 WebSocket 消息

### 第 5 题

解释协程的 `await` 和生成器的 `yield` 有什么相似之处和不同之处。

---

**答案：**

**第 1 题：**

| 任务 | 类型 | 原因 |
|------|------|------|
| 请求 500 个网页 | I/O 密集型 | 大部分时间在等待网络响应 |
| 计算一亿个数字的平方和 | CPU 密集型 | 持续的数学计算 |
| 查询数据库 | I/O 密集型 | 等待数据库返回结果 |
| 对 1000 张图片执行复杂滤镜 | CPU 密集型 | 大量的像素计算 |
| 等待用户输入 | I/O 密集型 | 等待用户操作 |
| 解压一个大文件 | CPU 密集型 | 大量的解压计算 |
| 对文本进行正则匹配 | CPU 密集型 | 大量的字符串匹配计算 |
| 调用外部 API 并处理响应 | I/O 密集型 | 大部分时间在等待网络 |

**第 2 题：**

因为多个协程在同一时刻通常只有一个在执行 Python 代码。它们是在**交替执行**，而不是**同时执行**。并发（Concurrency）是指多个任务在重叠的时间段内推进，并行（Parallelism）是指多个任务在同一时刻真正同时执行。协程通过在 I/O 等待时切换来实现并发，但在任意时刻只有一个协程在运行。

**第 3 题：**

因为 CPython 有 **GIL（全局解释器锁）**，同一时刻通常只有一个线程执行 Python 字节码。即使创建多个线程，它们也是在轮流获得 GIL，而不是真正并行。对于 CPU 密集型任务，这种轮流执行不仅没有加速效果，反而因为线程切换的开销可能导致性能下降。

**第 4 题：**

| 任务 | 推荐方案 | 原因 |
|------|---------|------|
| A. 调用 100 个网络 API | asyncio | I/O 密集型，有异步库支持 |
| B. 用同步库处理 20 个文件 | 多线程 | I/O 密集型，但库只支持同步 |
| C. 编码 10 个大型视频 | 多进程 | CPU 密集型，需要多核并行 |
| D. 下载 1000 张图片并压缩 | asyncio + 多进程 | 下载用 asyncio，压缩用多进程 |
| E. 实时处理 WebSocket 消息 | asyncio | I/O 密集型，需要低延迟 |

**第 5 题：**

**相似之处：**
- 都可以在函数执行过程中暂停
- 都可以在暂停后从暂停点继续执行
- 都使用特殊语法标记暂停点（`yield` 和 `await`）

**不同之处：**
- `yield` 主要用于**产生值**（迭代器模式），`await` 主要用于**等待异步操作**
- `yield` 需要手动调用 `next()` 或 `send()` 驱动，`await` 由事件循环自动驱动
- `yield` 暂停后需要外部恢复，`await` 暂停后事件循环会自动切换到其他协程
- 生成器主要用于数据生产，协程主要用于异步流程控制

```python
import psutil
import os


def analyze_task(task_func, *args):
    """
    分析任务的执行特征

    Args:
        task_func: 要分析的函数
        *args: 函数的参数

    Returns:
        dict: 包含执行时间和任务类型的字典

    start = time.perf_counter()
    result = task_func(*args)
    elapsed = time.perf_counter() - start

    # 判断任务类型
    # 简单启发式：如果执行时间相对于计算量很短，可能是 I/O 密集型
    # 这里用函数名作为简单判断
    func_name = task_func.__name__.lower()

    elif elapsed > 0.01:  # 超过 10ms 可能是 CPU 密集型
        task_type = "cpu_bound"
    else:
        task_type = "unknown"

    return {
        "function": task_func.__name__,
        "elapsed_ms": elapsed * 1000,
        "task_type": task_type,
        "result": result,
    }
```

```python
import asyncio
import aiohttp
import time


async def download(url, session):
    """
    异步下载单个 URL

    Args:
        url: 要下载的 URL
        session: aiohttp 客户端会话

    Returns:
        tuple: (url, status_code, content_length)
    """
    try:
        async with session.get(url) as response:
            content = await response.read()
            return (url, response.status, len(content))
    except Exception as e:
        return (url, -1, 0)


async def download_all(urls, max_concurrent=5):
    """
    并发下载多个 URL

    Args:
        urls: URL 列表
        max_concurrent: 最大并发数

    Returns:
        list: 下载结果列表
    """
    semaphore = asyncio.Semaphore(max_concurrent)

    async def download_with_limit(url, session):
        async with semaphore:
            return await download(url, session)

    async with aiohttp.ClientSession() as session:
        tasks = [download_with_limit(url, session) for url in urls]
        results = await asyncio.gather(*tasks)

    return results
```

## 本章小结

通过这一章的学习，我们建立了 Python 并发编程的基础知识体系：

### 核心概念回顾

```
┌──────────────────────────────────────────────────────────────────┐
│                    三种并发模型的本质                              │
│                                                                  │
│  进程：操作系统的独立执行单元                                      │
│  ├── 独立内存空间，隔离性强                                       │
│  ├── 可以利用多核 CPU 真正并行                                    │
│  └── 适合 CPU 密集型任务                                          │
│                                                                  │
│  线程：进程内的执行流                                             │
│  ├── 共享进程内存，通信方便                                       │
│  ├── 受 GIL 限制，CPU 密集型无法并行                              │
│  └── 适合 I/O 密集型任务（阻塞库）                                │
│                                                                  │
│  协程：程序主动调度的轻量级任务                                    │
│  ├── 切换成本极低，适合高并发                                     │
│  ├── 需要异步库支持                                               │
│  └── 适合 I/O 密集型任务（异步库）                                │
└──────────────────────────────────────────────────────────────────┘
```

### GIL 的影响

```
┌──────────────────────────────────────────────────────────────────┐
│                    GIL 的关键影响                                  │
│                                                                  │
│  CPU 密集型任务：                                                 │
│  - 多线程：受 GIL 限制，无法加速                                  │
│  - 多进程：可以利用多核，推荐方案                                  │
│                                                                  │
│  I/O 密集型任务：                                                 │
│  - 多线程：I/O 等待时释放 GIL，可以有效并发                        │
│  - asyncio：更轻量，适合大规模并发                                 │
│                                                                  │
│  记住：                                                          │
│  - GIL 限制的是 CPU 并行，不是 I/O 并发                           │
│  - I/O 等待时释放 GIL，所以多线程对 I/O 有效                       │
│  - asyncio 是 I/O 并发的最佳选择                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 选择并发模型的口诀

```
┌──────────────────────────────────────────────────────────────────┐
│                    并发模型选择口诀                                │
│                                                                  │
│         大量 I/O + 有异步库  →  asyncio                          │
│         大量 I/O + 只有同步库 →  多线程                           │
│         大量计算（CPU）       →  多进程                           │
│                                                                  │
│  更简洁地记：                                                     │
│         大量 I/O → 协程，阻塞 I/O → 线程，重 CPU → 进程           │
└──────────────────────────────────────────────────────────────────┘
```

### 上下文切换成本

```
┌──────────────────────────────────────────────────────────────────┐
│                    切换成本从高到低                                │
│                                                                  │
│         进程切换 > 线程切换 > 协程切换                             │
│                                                                  │
│  进程切换：~1-10 μs（需要切换内存空间）                            │
│  线程切换：~1-5 μs（需要系统调用）                                 │
│  协程切换：~0.1-0.5 μs（用户态完成）                               │
│                                                                  │
│  协程的低切换成本是它适合高并发的根本原因                           │
└──────────────────────────────────────────────────────────────────┘
```

### 关键要点

1. **进程是资源分配的单位，线程是调度的单位，协程是协作的单位**
2. **GIL 限制的是 CPU 并行，不是 I/O 并发**
3. **async 不是魔法粉，CPU 密集型任务仍然需要多进程**
4. **协程在 await 点切换，线程在任意时刻可能被切换**
5. **实际工程中经常混合使用三种模型**

## 思考题

### 1. 进程和线程最大的区别是什么？

进程拥有独立内存空间，线程共享所属进程的内存。进程间通信需要显式机制（IPC），线程间可直接访问共享变量。

从操作系统的角度看：
- 进程是资源分配的基本单位
- 线程是调度执行的基本单位
- 一个进程可以包含多个线程
- 进程崩溃不会影响其他进程，线程崩溃可能导致整个进程崩溃

### 2. 为什么协程的切换成本最低？

协程切换在用户态完成，不需要操作系统介入，主要只需保存执行位置和局部变量。

具体来说：
- 进程切换需要切换内存空间、刷新 TLB、切换页表（涉及内核态切换）
- 线程切换需要保存/恢复寄存器、可能刷新 CPU 缓存（涉及系统调用）
- 协程切换只需要保存/恢复少量寄存器和栈帧（完全在用户态）

协程切换的开销比线程切换低一个数量级，比进程切换低两个数量级。

### 3. 一个程序能否同时使用 asyncio 和多进程？

可以。例如用 asyncio 处理网络 I/O，同时用进程池处理 CPU 密集型计算。

```python
import asyncio
from concurrent.futures import ProcessPoolExecutor


def cpu_heavy(n):
    """CPU 密集型计算"""
    return sum(i * i for i in range(n))


async def main():
    loop = asyncio.get_event_loop()

    # 使用进程池执行 CPU 密集型任务
    with ProcessPoolExecutor() as pool:
        # 在进程中执行计算
        future = loop.run_in_executor(pool, cpu_heavy, 10_000_000)

        # 同时执行其他 asyncio 任务
        await asyncio.sleep(1)

        # 等待计算完成
        result = await future
        print(f"计算结果: {result}")


asyncio.run(main())
```

这种混合模式在实际工程中很常见，asyncio 作为调度中心，进程池作为执行引擎。

### 4. 如果一个库只提供同步接口，如何在异步程序中使用它？

可以使用 `asyncio.to_thread()` 将同步调用放到工作线程中执行。

```python
import asyncio
import time


def blocking_io():
    """阻塞的同步函数"""
    time.sleep(1)
    return "result"


async def main():
    # 将阻塞函数放到线程池中执行
    result = await asyncio.to_thread(blocking_io)
    print(f"结果: {result}")


asyncio.run(main())
```

`asyncio.to_thread()` 会将同步函数放到默认的线程池中执行，不会阻塞事件循环。

### 5. 为什么说"单线程也可能有数据竞争"？

协程在修改共享状态时如果遇到 `await`，其他协程可能插入执行并修改同一状态，导致竞态条件。

```python
import asyncio


class Counter:
    def __init__(self):
        self.count = 0

    async def increment(self):
        current = self.count
        await asyncio.sleep(0)  # 其他协程可能在这里插入！
        self.count = current + 1


async def main():
    counter = Counter()
    await asyncio.gather(*[counter.increment() for _ in range(10)])
    print(f"期望: 10, 实际: {counter.count}")  # 可能小于 10


asyncio.run(main())
```

解决方案：使用 `asyncio.Lock` 或避免在修改共享状态时 `await`。

### 6. GIL 会被移除吗？

Python 3.13 引入了实验性的 Free-Threaded 模式（PEP 703），可以禁用 GIL。但这需要整个 Python 生态系统的适配，是一个渐进的过程。目前大多数生产环境仍然使用标准 CPython（带 GIL）。

### 7. 协程能替代多线程吗？

不能完全替代。协程适合 I/O 密集型任务，但对于以下场景，线程仍然是更好的选择：

1. 使用只提供同步接口的库
2. 需要在等待 I/O 时执行 CPU 计算
3. 需要与操作系统线程交互（如某些 GUI 框架）
4. 代码中没有异步版本，改造成异步成本太高

实际工程中，asyncio 和线程经常一起使用，各取所长。
