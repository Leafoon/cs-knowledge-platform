# 第二十二章 中断系统

中断是CPU响应外部或内部事件的机制，是计算机系统中最重要的控制方式之一。本章详细讲解中断的分类、判优、响应、处理和多重中断。**本章是考研重点章节**，中断处理流程和多重中断是高频考点。

## 22.1 中断系统概述

### 22.1.1 中断的定义

**中断**（Interrupt）：CPU在执行程序过程中，由于内部或外部事件的请求，暂停当前程序，转去处理该事件，处理完毕后返回原程序继续执行的过程。

### 22.1.2 中断的作用

1. **提高CPU利用率**：CPU不必忙等待I/O设备
2. **实现实时处理**：及时响应紧急事件
3. **实现人机交互**：键盘、鼠标等输入
4. **实现故障处理**：电源故障、硬件错误
5. **实现系统调用**：用户程序请求操作系统服务

### 22.1.3 中断与子程序调用的区别

| 特性 | 中断 | 子程序调用 |
|------|------|-----------|
| 触发方式 | 随机事件 | 程序中的CALL指令 |
| 返回地址 | PC（断点） | CALL的下一条指令地址 |
| 保存内容 | PC + PSW + 寄存器 | PC + 部分寄存器 |
| 嵌套处理 | 需要中断屏蔽 | 由程序控制 |
| 响应时机 | 指令执行完毕 | 立即 |

## 22.2 中断的分类

### 22.2.1 外部中断（硬件中断）

由CPU外部事件引起的中断。

| 类型 | 来源 | 可屏蔽性 | 典型示例 |
|------|------|---------|---------|
| 可屏蔽中断（INTR） | 外设 | **可屏蔽** | 键盘、定时器、网卡 |
| 不可屏蔽中断（NMI） | 紧急事件 | **不可屏蔽** | 电源故障、内存校验错 |

**INTR（可屏蔽中断）**：
- 可以通过中断允许标志IF屏蔽
- IF=1时允许响应，IF=0时禁止响应
- 通常用于普通外设中断

**NMI（不可屏蔽中断）**：
- 不能被屏蔽，必须响应
- 用于紧急事件（如电源故障、硬件错误）
- 通常有最高的优先级

### 22.2.2 内部中断（异常）

由CPU内部事件引起的中断。

| 类型 | 原因 | 可恢复性 | 典型示例 |
|------|------|---------|---------|
| 故障（Fault） | 可恢复的错误 | **可恢复** | 缺页异常、除零错误 |
| 陷阱（Trap） | 主动陷入 | **可恢复** | 系统调用（INT指令） |
| 终止（Abort） | 不可恢复的错误 | **不可恢复** | 硬件故障、严重错误 |

**Fault（故障）**：
- 由执行指令时的错误引起
- 修复后可以重新执行引起故障的指令
- 典型：缺页异常（Page Fault）

**Trap（陷阱）**：
- 由执行特定指令主动触发
- 处理完后执行下一条指令
- 典型：系统调用（System Call）

**Abort（终止）**：
- 由严重的、不可恢复的错误引起
- 终止当前程序的执行
- 典型：硬件故障

<div data-component="InterruptClassificationDemo"></div>

### 22.2.3 软中断

由软件指令触发的中断（如x86的`INT n`指令）。

- 属于同步中断（在指令执行时同步发生）
- 通常用于系统调用

## 22.3 中断请求的标记

### 22.3.1 中断请求寄存器（IRR）

记录哪些中断源有请求。

| 位 | 中断源 | 说明 |
|----|--------|------|
| IR0 | 定时器 | 周期性中断 |
| IR1 | 键盘 | 按键中断 |
| IR2 | 串口 | 数据到达 |
| ... | ... | ... |
| IR7 | 并口 | 打印完成 |

### 22.3.2 中断屏蔽寄存器（IMR）

记录哪些中断被屏蔽。

- 对应位为1：该级中断被屏蔽（不响应）
- 对应位为0：该级中断允许响应

### 22.3.3 中断服务寄存器（ISR）

记录正在服务的中断。

- 对应位为1：该级中断正在服务
- 用于实现中断嵌套

### 22.3.4 三个寄存器的关系

```
中断源 → IRR（记录请求）
              ↓
         判优逻辑（比较IRR和IMR）
              ↓
         ISR（记录正在服务的中断）
              ↓
         CPU响应中断
```

## 22.4 中断判优

### 22.4.1 为什么需要判优

多个中断源可能同时发出请求，CPU需要决定先响应哪个中断。

### 22.4.2 硬件判优

**一、菊花链判优（链式查询）**

```
BR（总线请求）→ 所有设备
BG（总线允许）→ 设备1 → 设备2 → 设备3 → ...
```

- 优先级：离CPU越近优先级越高
- 优点：简单
- 缺点：不公平，对故障敏感

**二、独立请求判优**

```
设备1: BR1, BG1
设备2: BR2, BG2
设备3: BR3, BG3
```

- 每个设备独立的请求/允许线
- 优先级：由硬件逻辑决定
- 优点：速度快，灵活
- 缺点：线数多

**三、编码器判优**

用编码器直接产生最高优先级的中断号。

### 22.4.3 软件判优

程序按优先级逐一检查每个中断源。

```c
if (IRR & 0x01) handle_interrupt_0();      // 最高优先级
else if (IRR & 0x02) handle_interrupt_1();
else if (IRR & 0x04) handle_interrupt_2();
// ...
```

- 优点：灵活，可动态调整优先级
- 缺点：速度慢

### 22.4.4 中断优先级

**固定优先级**：

每个中断源有固定的优先级，不可改变。

**动态优先级**：

中断源的优先级可以动态改变。

**常见的优先级设置**：

| 优先级 | 中断类型 | 原因 |
|--------|---------|------|
| 最高 | 电源故障/硬件错误 | 必须立即处理 |
| 高 | 时钟中断 | 系统调度需要 |
| 中 | I/O中断 | 设备数据传输 |
| 低 | 用户程序中断 | 可以延迟 |

## 22.5 中断响应

### 22.5.1 中断响应的条件

CPU响应中断需要满足以下条件：

1. **有中断请求**：IRR ≠ 0
2. **中断允许**：IF = 1（或该中断为不可屏蔽中断）
3. **无更高优先级中断在服务**：当前中断优先级高于正在服务的中断
4. **当前指令执行完毕**：不能在指令执行中途响应中断

> [!NOTE]
> **中断响应的时机**：在每条指令执行完毕后，CPU检查是否有中断请求。如果有满足条件的中断请求，则进入中断响应周期。

### 22.5.2 中断响应周期

中断响应周期是CPU从响应中断到开始执行中断服务程序之间的特殊周期。

**中断响应的微操作序列**：

```
1. 关中断（0 → IF）          ← 防止在保存现场时被新的中断打断
2. 保存断点（PC → 栈）       ← 保存返回地址
3. 保存PSW（PSW → 栈）       ← 保存程序状态字
4. 形成中断服务程序入口地址    ← 通过中断向量或查询中断向量表
5. 转中断服务程序             ← PC = 入口地址
```

### 22.5.3 中断入口地址的形成

**方法一：中断向量法**

每个中断源对应一个**中断向量**（入口地址），存储在**中断向量表**中。

```
中断向量表：
┌──────────────┬──────────────────┐
│ 中断向量号    │ 入口地址          │
├──────────────┼──────────────────┤
│ 0            │ 0x1000           │
│ 1            │ 0x1050           │
│ 2            │ 0x1100           │
│ ...          │ ...              │
└──────────────┴──────────────────┘
```

**方法二：查询法**

用软件逐一检查中断源，跳转到对应的处理程序。

## 22.6 中断服务程序

### 22.6.1 中断服务程序的基本结构

```
中断服务程序：
1. 保护现场（保存通用寄存器）
2. 开中断（允许中断嵌套）      ← 如果支持多重中断
3. 执行中断服务（实际处理）
4. 关中断                      ← 防止恢复现场时被打断
5. 恢复现场（恢复通用寄存器）
6. 开中断                      ← 恢复中断响应能力
7. 中断返回（恢复PC和PSW）
```

### 22.6.2 保护现场与恢复现场

**保护现场**：

将中断服务程序中可能用到的寄存器保存到栈中。

```assembly
PUSH R1
PUSH R2
PUSH R3
; ... 执行中断服务 ...
POP R3
POP R2
POP R1
```

**恢复现场**：

从栈中恢复寄存器（按相反顺序）。

### 22.6.3 中断返回

执行中断返回指令（如x86的`IRET`），恢复PC和PSW，返回到被中断的程序继续执行。

## 22.7 多重中断

### 22.7.1 单重中断与多重中断

<div data-component="NestedInterruptDemo"></div>

**单重中断**：

在中断服务过程中，不响应任何新的中断请求。

**多重中断**（中断嵌套）：

在中断服务过程中，允许更高优先级的中断打断当前服务。

### 22.7.2 多重中断的实现条件

1. 在中断服务程序中**开中断**（STI指令）
2. 有**中断优先级**机制
3. 有**中断屏蔽**机制

### 22.7.3 中断屏蔽字

用**屏蔽字**控制哪些中断可以打断当前服务。

**屏蔽字规则**：
- 屏蔽字中，**当前中断及更低优先级的中断**对应位为0（屏蔽）
- **更高优先级的中断**对应位为1（允许）

**【例 22-1】** 设有4级中断，优先级 $A > B > C > D$。

| 当前服务 | 屏蔽字（A,B,C,D） | 效果 |
|----------|------------------|------|
| 无 | 1111 | 所有中断都响应 |
| A | 1000 | 只有A级可打断 |
| B | 1100 | A、B级可打断 |
| C | 1110 | A、B、C级可打断 |
| D | 1111 | 所有级可打断 |

**【例 22-2】** 分析以下中断处理过程：

| 中断 | 优先级 | 处理时间 | 到达时间 |
|------|--------|---------|---------|
| A | 最高 | 10μs | 0μs |
| B | 中 | 20μs | 5μs |
| C | 低 | 15μs | 8μs |

**时间线**：

```
0μs:  A到达，开始服务A
5μs:  B到达，但A优先级更高，B等待
8μs:  C到达，等待
10μs: A服务完成，开始服务B
30μs: B服务完成，开始服务C
45μs: C服务完成
```

**如果支持中断嵌套**（A可以打断B和C）：

```
0μs:  A到达，开始服务A
5μs:  B到达，A可以打断B → 继续服务A
8μs:  C到达，等待
10μs: A服务完成，开始服务B
30μs: B服务完成，开始服务C
45μs: C服务完成
```

结果相同（因为A优先级最高且先到达）。

## 22.8 保护断点 vs 保护现场

| 操作 | 内容 | 执行方式 | 时机 |
|------|------|---------|------|
| **保护断点** | PC（和PSW） | **硬件自动** | 中断响应时 |
| **保护现场** | 通用寄存器 | **软件**（中断服务程序） | 中断服务开始时 |

> [!WARNING]
> **保护断点与保护现场的区别**是考研中的经典易错点。保护断点是硬件自动完成的（在中断响应周期），保护现场是软件完成的（在中断服务程序开头）。

## 22.9 中断系统的性能分析

### 22.9.1 中断响应时间

从中断请求发出到CPU开始执行中断服务程序第一条指令的时间。

$$
\text{中断响应时间} = \text{当前指令剩余执行时间} + \text{中断响应周期}
$$

### 22.9.2 中断延迟

从中断请求发出到中断服务程序开始执行的时间。

$$
\text{中断延迟} = \text{中断响应时间} + \text{排队等待时间}
$$

### 22.9.3 中断开销

中断处理带来的额外时间开销。

$$
\text{中断开销} = \text{保护断点} + \text{保护现场} + \text{恢复现场} + \text{中断返回}
$$

## 22.10 本章典型例题

### 例题1：中断处理流程

**题目**：描述CPU响应外部中断的完整流程。

**解答**：

```
1. 外设发出中断请求（INTR信号）
2. IRR记录中断请求
3. CPU在当前指令执行完毕后检查中断
4. 判优逻辑确定最高优先级的中断
5. 检查中断条件：
   - IF = 1（中断允许）
   - 该中断优先级高于当前服务的中断
6. 进入中断响应周期：
   a. 关中断（0 → IF）
   b. 保护断点（PC → 栈）
   c. 保存PSW
   d. 形成中断入口地址（查中断向量表）
   e. 转中断服务程序
7. 执行中断服务程序：
   a. 保护现场（寄存器入栈）
   b. 开中断（允许嵌套）
   c. 执行中断服务
   d. 关中断
   e. 恢复现场（寄存器出栈）
   f. 开中断
8. 中断返回（IRET）：恢复PC和PSW
9. 返回原程序继续执行
```

### 例题2：多重中断与屏蔽字

**题目**：设有3级中断，优先级 $1 > 2 > 3$。当前正在服务中断2，此时中断1和中断3同时请求。分析处理过程和屏蔽字设置。

**解答**：

**屏蔽字设置**（当前服务中断2）：

| 中断级 | 屏蔽位 | 说明 |
|--------|--------|------|
| 1 | 1（允许） | 优先级更高 |
| 2 | 0（屏蔽） | 正在服务 |
| 3 | 0（屏蔽） | 优先级更低 |

屏蔽字 = `100`

**处理过程**：

1. 中断2正在服务
2. 中断1和中断3同时请求
3. 判优：中断1优先级高于中断3
4. 检查屏蔽字：中断1的屏蔽位=1（允许），中断3的屏蔽位=0（屏蔽）
5. 响应中断1：
   a. 保存中断2的断点和现场
   b. 转中断1服务程序
6. 中断1服务完成后：
   a. 恢复中断2的现场
   b. 继续中断2的服务
7. 中断2服务完成后：
   c. 响应中断3

### 例题3：中断响应条件判断

**题目**：以下哪种情况CPU会响应中断？

A. 当前正在执行中断服务程序，IF=0
B. 当前正在执行中断服务程序，IF=1
C. 当前指令执行到一半
D. 当前执行的是乘法指令（多周期）

**解答**：

A. 不会响应（IF=0，中断被屏蔽）
B. 会响应（IF=1，且如果新中断优先级更高则可以嵌套）
C. 不会响应（必须等当前指令执行完毕）
D. 会响应（多周期指令在最后一个周期执行完毕后可以响应）

答案：**B和D**

## 22.11 本章常见误区与易错点

1. **保护断点是硬件自动完成的**：不需要程序员编写代码。

2. **保护现场是软件完成的**：需要在中断服务程序中用PUSH/POP指令。

3. **关中断的时机**：在保护断点之前关中断（防止保存过程被打断），在保护现场之后开中断（允许嵌套）。

4. **中断响应的条件**：IF=1只是条件之一，还需要有中断请求、当前指令执行完毕、优先级足够高。

5. **Fault可以重新执行**：缺页异常是Fault，修复后重新执行引起故障的指令。

6. **Trap执行下一条指令**：系统调用是Trap，处理完后执行系统调用的下一条指令。

7. **中断向量表的地址**：中断向量号 × 每个表项的大小 = 表项的地址。

8. **多重中断需要开中断**：如果中断服务程序中不开中断，就不会有中断嵌套。

## 22.12 本章总结

### 中断处理完整流程

```
中断请求 → 判优 → 响应（关中断、保护断点）→ 服务（保护现场、开中断、服务、关中断、恢复现场）→ 返回（开中断、恢复断点）
```

### 关键概念对比

| 概念 | 说明 |
|------|------|
| 保护断点 vs 保护现场 | 硬件自动 vs 软件 |
| 中断 vs 异步 | 外部事件 vs 内部事件 |
| Fault vs Trap | 可重新执行 vs 执行下一条 |
| 单重中断 vs 多重中断 | 不允许嵌套 vs 允许嵌套 |
| IF=0 vs IF=1 | 关中断 vs 开中断 |

## 22.14 深入理解中断系统

### 22.14.1 中断的分类详解

**外部中断**（硬件中断）：

由CPU外部事件引起的中断。

| 类型 | 来源 | 可屏蔽性 | 典型示例 |
|------|------|---------|---------|
| 可屏蔽中断（INTR） | 外设 | **可屏蔽** | 键盘、定时器、网卡 |
| 不可屏蔽中断（NMI） | 紧急事件 | **不可屏蔽** | 电源故障、内存校验错 |

**内部中断**（异常）：

由CPU内部事件引起的中断。

| 类型 | 原因 | 可恢复性 | 典型示例 |
|------|------|---------|---------|
| 故障（Fault） | 可恢复的错误 | **可恢复** | 缺页异常、除零错误 |
| 陷阱（Trap） | 主动陷入 | **可恢复** | 系统调用（INT指令） |
| 终止（Abort） | 不可恢复的错误 | **不可恢复** | 硬件故障、严重错误 |

### 22.14.2 中断判优详解

**硬件判优**：

**一、菊花链判优**

```
BR（总线请求）→ 所有设备
BG（总线允许）→ 设备1 → 设备2 → 设备3 → ...
BS（总线忙）
```

- 优先级：离仲裁器越近优先级越高
- 优点：简单
- 缺点：不公平，对故障敏感

**二、独立请求判优**

```
设备1: BR1, BG1
设备2: BR2, BG2
设备3: BR3, BG3
```

- 优先级：由硬件逻辑决定
- 优点：速度快，灵活
- 缺点：线数多

**三、编码器判优**

用编码器直接产生最高优先级的中断号。

### 22.14.3 中断响应详解

**中断响应的条件**：

1. **有中断请求**：IRR ≠ 0
2. **中断允许**：IF=1（或该中断为不可屏蔽中断）
3. **无更高优先级中断在服务**：当前中断优先级高于正在服务的中断
4. **当前指令执行完毕**：不能在指令执行中途响应中断

**中断响应的微操作序列**：

```
1. 关中断（0 → IF）          ← 防止在保存现场时被新的中断打断
2. 保护断点（PC → 栈）       ← 保存返回地址
3. 保存PSW（PSW → 栈）       ← 保存程序状态字
4. 形成中断服务程序入口地址    ← 通过中断向量或查询中断向量表
5. 转中断服务程序             ← PC = 入口地址
```

### 22.14.4 中断服务程序详解

**中断服务程序的基本结构**：

```
1. 保护现场（保存通用寄存器）
2. 开中断（允许中断嵌套）
3. 执行中断服务（实际处理）
4. 关中断
5. 恢复现场（恢复通用寄存器）
6. 开中断
7. 中断返回（恢复PC和PSW）
```

**保护现场与恢复现场**：

```assembly
; 保护现场
PUSH R1
PUSH R2
PUSH R3
; ... 执行中断服务 ...
POP R3
POP R2
POP R1
```

### 22.14.5 多重中断详解

**多重中断的条件**：

1. 在中断服务程序中**开中断**（STI指令）
2. 有**中断优先级**机制
3. 有**中断屏蔽**机制

**中断屏蔽字**：

用屏蔽字控制哪些中断可以打断当前服务。

| 当前服务 | 屏蔽字（A,B,C,D） | 效果 |
|----------|------------------|------|
| 无 | 1111 | 所有中断都响应 |
| A | 1000 | 只有A级可打断 |
| B | 1100 | A、B级可打断 |
| C | 1110 | A、B、C级可打断 |
| D | 1111 | 所有级可打断 |

### 22.15.1 中断响应时间

从中断请求发出到CPU开始执行中断服务程序第一条指令的时间。

$$
\text{中断响应时间} = \text{当前指令剩余执行时间} + \text{中断响应周期}
$$

### 22.15.2 中断延迟

从中断请求发出到中断服务程序开始执行的时间。

$$
\text{中断延迟} = \text{中断响应时间} + \text{排队等待时间}
$$

### 22.15.3 中断开销

中断处理带来的额外时间开销。

$$
\text{中断开销} = \text{保护断点} + \text{保护现场} + \text{恢复现场} + \text{中断返回}
$$

## 22.18 中断控制器的硬件结构

### 22.18.1 8259A（PIC）架构

<div data-component="InterruptControllerViz"></div>

8259A 是 Intel 推出的可编程中断控制器（Programmable Interrupt Controller），负责管理外部设备的中断请求，将其汇总后发送给 CPU。

**内部寄存器与模块：**

```
┌─────────────────────────────────────────────────────┐
│                    8259A 内部结构                     │
│                                                     │
│  IRQ0 ──┐                                           │
│  IRQ1 ──┤    ┌───────────┐    ┌──────────────┐      │
│  IRQ2 ──┤    │   IRR     │    │    IMR       │      │
│  IRQ3 ──┼───→│ 中断请求  │───→│ 中断屏蔽     │      │
│  IRQ4 ──┤    │ 寄存器    │    │ 寄存器       │      │
│  IRQ5 ──┤    └─────┬─────┘    └──────┬───────┘      │
│  IRQ6 ──┤          │                 │              │
│  IRQ7 ──┘          ▼                 ▼              │
│              ┌──────────────────────────┐            │
│              │      优先级解析器        │            │
│              │   (Priority Resolver)    │            │
│              └────────────┬─────────────┘            │
│                           ▼                          │
│              ┌──────────────────────────┐            │
│              │    ISR                   │            │
│              │  正在服务寄存器           │            │
│              └────────────┬─────────────┘            │
│                           ▼                          │
│              ┌──────────────────────────┐            │
│              │      控制逻辑            │            │
│              │   (Control Logic)        │──→ INT     │
│              └──────────────────────────┘            │
│                           ▲                          │
│              ┌────────────┴─────────────┐            │
│              │  INTA  │  D0-D7 │  CAS  │            │
│              └──────────────────────────┘            │
└─────────────────────────────────────────────────────┘
```

**核心寄存器说明：**

| 寄存器 | 全称 | 位宽 | 功能 |
|--------|------|------|------|
| **IRR** | Interrupt Request Register | 8位 | 锁存各 IRQ 线的中断请求，1=有请求 |
| **IMR** | Interrupt Mask Register | 8位 | 屏蔽对应 IRQ，1=屏蔽该中断 |
| **ISR** | In-Service Register | 8位 | 记录当前正在处理的中断，1=正在服务 |
| **PR** | Priority Resolver | — | 比较 IRR 与 ISR，选出最高优先级中断 |

> [!TIP]
> IRR 记录"谁在请求"，ISR 记录"谁正在被服务"，IMR 记录"谁被屏蔽"。三者配合实现中断的优先级管理和嵌套。

### 22.18.2 PIC 级联（Master/Slave）

当外部中断超过 8 个时，采用主从级联方式扩展：

```
                    ┌──────────┐
        IRQ0-7 ───→ │  Master  │ ───→ INT → CPU INTR
                    │  8259A   │
        INTA ──────→│          │
                    └──┬───┬───┘
                  CAS0-2│   │INTA
              ┌────────┘   └────────┐
              ▼                     ▼
        ┌──────────┐          ┌──────────┐
        │  Slave 0 │          │  Slave 1 │
        │  8259A   │          │  8259A   │
        └──────────┘          └──────────┘
        IRQ8-15                IRQ16-23
```

- **CAS0-CAS2**：级联信号线，3 位编码可寻址 8 个从片
- **SP/EN**：主片接高电平，从片接低电平
- 级联后最多支持 **8 × 8 = 64** 个中断源（实际 x86 系统常用 2 片，支持 15 个外部中断）

### 22.18.3 APIC 架构

现代 x86 处理器采用高级可编程中断控制器（Advanced PIC），由两部分组成：

**Local APIC（本地 APIC）：**
- 集成在每个 CPU 核心内部
- 接收本地中断（定时器、性能计数器等）
- 接收来自 I/O APIC 的中断
- 支持处理器间中断（IPI）

**I/O APIC：**
- 位于南桥/PCH 芯片中
- 管理外部设备中断
- 将中断路由到指定的 CPU 核心

```
┌─────────────────────────────────────────────────────────┐
│                  x86 中断路由架构                        │
│                                                         │
│   ┌──────────────────────────────────────────────┐      │
│   │               CPU 芯片                       │      │
│   │  ┌────────────┐  ┌────────────┐              │      │
│   │  │ Core 0     │  │ Core 1     │   ...        │      │
│   │  │┌──────────┐│  │┌──────────┐│              │      │
│   │  ││Local APIC││  ││Local APIC││              │      │
│   │  ││   #0     ││  ││   #1     ││              │      │
│   │  │└────┬─────┘│  │└────┬─────┘│              │      │
│   │  └─────┼──────┘  └─────┼──────┘              │      │
│   │        │    APIC Bus    │                     │      │
│   │        └────────┬───────┘                     │      │
│   └─────────────────┼─────────────────────────────┘      │
│                     │                                    │
│              ┌──────┴──────┐                             │
│              │  I/O APIC   │                             │
│              │ (南桥/PCH)  │                             │
│              └──────┬──────┘                             │
│         ┌───────────┼───────────┐                        │
│         ▼           ▼           ▼                        │
│      设备0       设备1       设备2    ...                │
│    (PCIe/USB/SATA 等外部设备)                             │
└─────────────────────────────────────────────────────────┘
```

### 22.18.4 8259A 与 APIC 对比

| 特性 | 8259A (PIC) | APIC |
|------|-------------|------|
| 最大中断源 | 8（级联 64） | 256+ |
| 多核支持 | 不支持 | 原生支持 |
| 中断路由 | 固定到 CPU0 | 可路由到任意核心 |
| IPI 支持 | 不支持 | 支持 |
| MSI 支持 | 不支持 | 支持 |
| 中断类型 | 仅边沿/电平触发 | 边沿/电平/消息触发 |
| 现代使用 | 兼容模式/Legacy | 主流模式 |

> [!NOTE]
> 现代操作系统启动时通常先通过 8259A 兼容模式初始化，随后切换到 APIC 模式。Linux 内核默认使用 APIC，但在虚拟机中可能退回 8259A 模式。

## 22.19 中断判优电路的实现

### 22.19.1 优先编码器电路

<div data-component="PriorityArbiter"></div>

优先编码器（Priority Encoder）是最基本的硬件判优电路，将多个中断请求编码为二进制优先级编号。

**74148 优先编码器（8-3 线）：**

```
         I7 ──→┐
         I6 ──→┤
         I5 ──→┤    ┌──────────────┐
         I4 ──→┤    │   74148     │
         I3 ──→┼───→│  优先编码器  │───→ A2 A1 A0 (3位编码)
         I2 ──→┤    │             │───→ GS (输出有效)
         I1 ──→┤    │             │───→ EO (使能输出)
         I0 ──→┘    └──────────────┘
                    EI ← (使能输入)
```

| 输入有效位 | A2 A1 A0 | 含义 |
|-----------|----------|------|
| I7 | 1 1 1 | 最高优先级 |
| I6 | 1 1 0 | |
| I5 | 1 0 1 | |
| I4 | 1 0 0 | |
| I3 | 0 1 1 | |
| I2 | 0 1 0 | |
| I1 | 0 0 1 | |
| I0 | 0 0 0 | 最低优先级 |

> [!TIP]
> 优先编码器同时只输出最高优先级的请求编号。低位输入（如 I0）只有在所有高位（I1-I7）都无请求时才会被编码输出。

### 22.19.2 菊花链硬件电路

菊花链（Daisy Chain）通过串行传递中断响应信号来实现判优：

```
                    ┌─────────────────────────────────────────────┐
                    │              CPU / 中断控制器               │
                    │                                             │
                    │  INT ←─── 所有设备的 INT 线"线或"           │
                    │  INTA ──→ 向链首发送响应                    │
                    └──────┬──────────────────────────────────────┘
                           │ INTA
                           ▼
                    ┌──────────────┐
                    │   设备 0     │  ← 最高优先级
                    │  ┌────────┐  │
                    │  │ 判优   │  │
                    │  │ 电路   │  │
                    │  └───┬────┘  │
        INT0 ──→ INT │    │       │ ──→ INTA_out
                    └────┼───────┘       │
                         │               │ INTA
                         ▼               ▼
                    ┌──────────────┐
                    │   设备 1     │
                    │  ┌────────┐  │
                    │  │ 判优   │  │
                    │  │ 电路   │  │
                    │  └───┬────┘  │
        INT1 ──→ INT │    │       │ ──→ INTA_out
                    └────┼───────┘       │
                         │               │ INTA
                         ▼               ▼
                    ┌──────────────┐
                    │   设备 2     │  ← 最低优先级
                    │  ┌────────┐  │
                    │  │ 判优   │  │
                    │  └────────┘  │
        INT2 ──→ INT              │
                    └──────────────┘
```

**工作原理：**
1. INTA 信号从链首设备开始传递
2. 每个设备检查自己是否有中断请求
3. 若有请求，该设备截获 INTA，不再向后传递，同时将自己的中断类型码放到数据总线上
4. 若无请求，INTA 继续传递给下一个设备

**优缺点：**

| 优点 | 缺点 |
|------|------|
| 电路简单，连线少 | 优先级固定，不可动态调整 |
| 易于扩展 | 链越长，响应延迟越大 |
| 硬件开销小 | 链首设备故障影响全链 |

### 22.19.3 并行优先级中断硬件

独立请求方式（Independent Request）为每个中断源提供独立的请求线和响应线：

```
    IRQ0 ──────┐
    IRQ1 ──────┤     ┌──────────────┐
    IRQ2 ──────┤     │  优先级      │     INT → CPU
    IRQ3 ──────┼────→│  判优电路    │
    IRQ4 ──────┤     │              │     ← INTA (CPU)
    IRQ5 ──────┤     │  输出编码    │──→ 设备编号
    IRQ6 ──────┤     └──────────────┘
    IRQ7 ──────┘
```

**与菊花链对比：**

| 特性 | 菊花链 | 并行优先级 |
|------|--------|-----------|
| 连线数量 | 少（3根） | 多（2N根） |
| 响应速度 | 慢（逐级传递） | 快（并行判优） |
| 优先级灵活性 | 固定 | 可编程调整 |
| 故障隔离 | 差（链式影响） | 好（独立线路） |

### 22.19.4 8259A 中的可编程优先级

8259A 支持多种优先级设置方式：

**固定优先级（默认）：** IRQ0 最高，IRQ7 最低

**旋转优先级（自动旋转）：** 当某个中断被服务后，其优先级降为最低

```
服务前:  IRQ0 > IRQ1 > IRQ2 > IRQ3 > IRQ4 > IRQ5 > IRQ6 > IRQ7
                ↑
              服务 IRQ1

服务后:  IRQ2 > IRQ3 > IRQ4 > IRQ5 > IRQ6 > IRQ7 > IRQ0 > IRQ1
```

**特殊旋转优先级（通过 OCW2 设置）：** 可指定最低优先级

> [!NOTE]
> 旋转优先级适用于多个设备重要性相近的场景，避免某个低优先级设备长期得不到服务（饥饿问题）。Linux 系统中常采用类似的中断负载均衡策略。

## 22.20 中断响应的硬件时序

### 22.20.1 中断确认总线周期

当 CPU 响应中断时，需要执行特殊的中断确认（Interrupt Acknowledge, INTA）总线周期来获取中断向量号。

**8086/x86 的两次 INTA 周期：**

```
时钟:   T1    T2    T1    T2    T3    T4
        ├─────┼─────┼─────┼─────┼─────┼─────┤
        │           │           │           │
INTA#:  ┃___________┃           ┃___________┃
        ┃  第1个     ┃           ┃  第2个     ┃
        ┃  INTA周期  ┃           ┃  INTA周期  ┃
        │           │           │           │
AD0-7:  │  高阻态   │           │← 向量号 →│
        │           │           │  (Type)   │
        │           │           │           │
LOCK#:  ┃______________________________________┃
        ┃          总线锁定                     ┃
```

**两次 INTA 周期的作用：**

| 周期 | 动作 | 说明 |
|------|------|------|
| 第1个 INTA | CPU 发出第一个 INTA 脉冲 | 通知中断控制器：CPU 已响应 |
| 空闲周期 | 插入等待状态 | 给中断控制器准备时间 |
| 第2个 INTA | CPU 发出第二个 INTA 脉冲 | 中断控制器将向量号放到数据总线上 |

> [!TIP]
> 为什么需要两个 INTA 周期？第一个周期通知 PIC "我要响应了"，PIC 用这个时间进行内部判优并确定最高优先级中断；第二个周期 PIC 才将 8 位向量号送上数据总线。

### 22.20.2 完整硬件信号序列

```
时序流程:

设备请求          判优           响应           取向量         执行服务
  │               │              │              │              │
  ▼               ▼              ▼              ▼              ▼
┌─────┐      ┌─────────┐    ┌────────┐    ┌─────────┐    ┌──────────┐
│设备  │      │         │    │        │    │         │    │          │
│发出  │─INT─→│  8259A  │    │  CPU   │    │  8259A  │    │  CPU    │
│请求  │      │  判优   │    │  检测  │    │ 放置    │    │ 取得    │
│     │      │         │    │  INTR  │    │ 向量号  │    │ 向量号  │
└─────┘      └────┬────┘    └───┬────┘    └────┬────┘    └────┬────┘
                  │             │              │              │
                  │        ┌────┴────┐    ┌────┴────┐         │
                  │        │INTR=1  │    │ INTA x2 │         │
                  │        │IF=1    │    │ D0-D7=  │         │
                  │        │无NMI   │    │ 向量号  │         │
                  │        └─────────┘    └─────────┘         │
                  │                                            │
                  │              ┌──────────────────────────────┘
                  │              │ 通过向量号查 IDT/IVT
                  │              ▼
                  │         ┌──────────┐
                  │         │ 取中断   │
                  │         │ 服务程序 │
                  │         │ 入口地址 │
                  │         └──────────┘
                  │
```

### 22.20.3 关键信号说明

| 信号 | 方向 | 功能 |
|------|------|------|
| **INTR** | PIC → CPU | 中断请求输入，高电平有效 |
| **INTA#** | CPU → PIC | 中断确认，低电平有效，输出两个脉冲 |
| **LOCK#** | CPU → 总线 | 总线锁定，防止 DMA 争用 |
| **D0-D7** | PIC → CPU | 数据总线，传递 8 位中断向量号 |

> [!NOTE]
> NMI（不可屏蔽中断）不需要 INTA 周期，其向量号固定为 2。可屏蔽中断 INTR 才需要通过 INTA 周期从 8259A 获取向量号。

## 22.21 中断屏蔽寄存器的硬件实现

### 22.21.1 8259A 中的 IMR

IMR（Interrupt Mask Register）是 8259A 内部的一个 8 位寄存器，每一位对应一个 IRQ 线：

```
IMR 寄存器各位:

  Bit7  Bit6  Bit5  Bit4  Bit3  Bit2  Bit1  Bit0
  ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐
  │M7   │M6   │M5   │M4   │M3   │M2   │M1   │M0   │
  │IRQ7 │IRQ6 │IRQ5 │IRQ4 │IRQ3 │IRQ2 │IRQ1 │IRQ0 │
  └─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┘

  1 = 屏蔽该中断    0 = 允许该中断
```

**写入 IMR：** 通过 OCW1（操作命令字 1）写入，端口地址为 A0=1

```
; 屏蔽 IRQ2 和 IRQ5，允许其他中断
MOV AL, 00100100B   ; M5=1, M2=1
OUT 21H, AL          ; 写入主片 OCW1
```

### 22.21.2 优先级与屏蔽字的硬件比较

8259A 内部的优先级判优逻辑同时考虑 IRR、IMR 和 ISR：

```
┌─────────────────────────────────────────────────────┐
│              判优逻辑的输入组合                       │
│                                                     │
│  IRR (请求)    IMR (屏蔽)    ISR (在服务)             │
│  ┌───┐        ┌───┐        ┌───┐                    │
│  │110│        │001│        │010│                    │
│  └─┬─┘        └─┬─┘        └─┬─┘                    │
│    │            │            │                      │
│    ▼            ▼            ▼                      │
│  ┌─────────────────────────────────┐                │
│  │  有效请求 = IRR AND (NOT IMR)   │                │
│  │  110 AND 110 = 110             │                │
│  └──────────────┬──────────────────┘                │
│                 ▼                                   │
│  ┌─────────────────────────────────┐                │
│  │  可响应 = 有效请求 AND           │                │
│  │          (优先级 > ISR最高级)    │                │
│  │  优先级比较器输出                │                │
│  └──────────────┬──────────────────┘                │
│                 ▼                                   │
│            最终响应的中断                            │
└─────────────────────────────────────────────────────┘
```

**判优公式：**

> 有效中断请求 = IRR · IMR̄（取反）
> 可响应中断 = 有效请求中优先级高于 ISR 中最高级的中断

### 22.21.3 x86 IF 标志与 IMR 的关系

```
┌────────────────────────────────────────────────────┐
│              中断屏蔽层次结构                        │
│                                                    │
│   外部设备 IRQ                                     │
│       │                                            │
│       ▼                                            │
│   ┌──────────┐                                     │
│   │   IMR    │ ← 8259A 级屏蔽（OCW1 设置）         │
│   │ 逐位屏蔽 │   可屏蔽特定 IRQ 线                  │
│   └────┬─────┘                                     │
│        │ 剩余未屏蔽的请求                           │
│        ▼                                            │
│   ┌──────────┐                                     │
│   │ IF 标志  │ ← CPU 级屏蔽（CLI/STI 指令）        │
│   │ 全局开关 │   IF=0 屏蔽所有 INTR                 │
│   └────┬─────┘                                     │
│        │ IF=1 时放行                                │
│        ▼                                            │
│   CPU 响应中断                                      │
│                                                    │
│   ┌──────────────────────────────┐                 │
│   │ NMI 不受 IMR 和 IF 影响！    │                 │
│   │ NMI 直接到达 CPU              │                 │
│   └──────────────────────────────┘                 │
└────────────────────────────────────────────────────┘
```

| 屏蔽级别 | 控制方式 | 粒度 | 影响范围 |
|---------|---------|------|---------|
| IMR | OCW1 写入 | 逐位（8个IRQ） | 特定 IRQ 线 |
| IF 标志 | CLI / STI 指令 | 1 位（全局） | 所有 INTR |
| NMI | 无屏蔽机制 | — | 不可屏蔽 |

> [!TIP]
> 操作系统关中断（CLI）后，外设仍可以向 8259A 发送请求（IRR 仍被置位），只是 CPU 不响应。开中断（STI）后，之前积压的请求会被立即处理。

## 22.22 向量中断的硬件支持

### 22.22.1 中断向量生成硬件

向量中断方式下，每个中断源有唯一的中断向量号（0-255），CPU 通过向量号直接定位中断服务程序入口。

**8259A 向量号生成：**

```
ICW2 设置的高 5 位 (T7-T3)  +  IRQ 编号 (低 3 位)

例: ICW2 = 08H = 00001000B  (T7-T3 = 00001)

  IRQ0 → 00001 000 = 08H
  IRQ1 → 00001 001 = 09H
  IRQ2 → 00001 010 = 0AH
  IRQ3 → 00001 011 = 0BH
  IRQ4 → 00001 100 = 0CH
  IRQ5 → 00001 101 = 0DH
  IRQ6 → 00001 110 = 0EH
  IRQ7 → 00001 111 = 0FH
```

### 22.22.2 IDT（中断描述符表）结构

x86 保护模式下使用 IDT 存储中断处理程序的信息：

```
IDT 结构 (每个表项 8 字节):

  63        48 47  45 44 40 39    32 31         16 15          0
  ┌───────────┬──────┬────────────┬──────────────┬──────────────┐
  │ 偏移量    │ P DPL│ 保留/类型  │   段选择子   │  偏移量      │
  │ (高16位)  │  0   │            │              │  (低16位)    │
  └───────────┴──────┴────────────┴──────────────┴──────────────┘

  P:    存在位 (1=有效)
  DPL:  描述符特权级 (0=内核, 3=用户)
  段选择子: 指向 GDT/LDT 中的代码段描述符
  偏移量:   中断处理程序在代码段内的偏移地址
```

**IDT 寄存器（IDTR）：**

```
  ┌──────────────────┬───────────────┐
  │   IDT 基地址     │   IDT 限长    │
  │   (32/64 位)     │   (16 位)     │
  └──────────────────┴───────────────┘
  通过 LIDT 指令加载
```

### 22.22.3 硬件向量获取机制

```
中断响应时 CPU 的内部动作:

  ┌──────────────┐     ┌──────────────────────────────┐
  │ 获得向量号 N │────→│ 计算 IDT 表项地址             │
  │ (来自 8259A) │     │ = IDT_base + N × 8           │
  └──────────────┘     └──────────────┬───────────────┘
                                      │
                                      ▼
                      ┌──────────────────────────────┐
                      │ 读取 IDT 表项                 │
                      │ 获取段选择子和偏移量           │
                      └──────────────┬───────────────┘
                                      │
                                      ▼
                      ┌──────────────────────────────┐
                      │ 查 GDT 获取段基地址           │
                      │ 入口地址 = 段基址 + 偏移量    │
                      └──────────────┬───────────────┘
                                      │
                                      ▼
                      ┌──────────────────────────────┐
                      │ 保护现场 → 压栈 CS:EIP/EFLAGS│
                      │ 跳转到中断处理程序             │
                      └──────────────────────────────┘
```

### 22.22.4 IVT 布局（实模式）

实模式下使用 IVT（中断向量表），位于内存最低 1KB（地址 00000H-003FFH）：

```
IVT 内存布局 (实模式):

地址          向量号    说明
00000H ┌───────────┐
       │  IP_0     │  向量 0: 除法错误 (#DE)
00004H │  CS_0     │
       ├───────────┤
       │  IP_1     │  向量 1: 单步调试 (#DB)
00008H │  CS_1     │
       ├───────────┤
       │  IP_2     │  向量 2: NMI
0000CH │  CS_2     │
       ├───────────┤
       │  IP_3     │  向量 3: 断点 (#BP)
00010H │  CS_3     │
       ├───────────┤
       │    ...    │
       ├───────────┤
       │  IP_8     │  向量 8: IRQ0 (定时器)
00020H │  CS_8     │
       ├───────────┤
       │    ...    │
       ├───────────┤
       │  IP_13    │  向量 13: 一般保护错误 (#GP)
00034H │  CS_13    │
       ├───────────┤
       │    ...    │
003FCH │  IP_255   │  向量 255
       │  CS_255   │
003FFH └───────────┘

每个表项 = 4 字节 (CS:IP)
256 个表项 × 4 字节 = 1024 字节 = 1KB
```

> [!NOTE]
> 实模式 IVT 每个表项仅 4 字节（CS:IP），保护模式 IDT 每个表项 8 字节，支持段保护和特权级检查。64 位模式下 IDT 表项扩展到 16 字节。

## 22.23 多核环境下的中断处理

### 22.23.1 多核中断路由

在多核系统中，中断需要被合理分配到各个 CPU 核心：

```
┌───────────────────────────────────────────────────────────────┐
│                   多核中断路由架构                              │
│                                                               │
│  ┌──────────────────────────────────────────────────────┐    │
│  │                    CPU 芯片                          │    │
│  │                                                      │    │
│  │  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐ │    │
│  │  │ Core 0  │  │ Core 1  │  │ Core 2  │  │ Core 3  │ │    │
│  │  │┌───────┐│  │┌───────┐│  │┌───────┐│  │┌───────┐│ │    │
│  │  ││Local  ││  ││Local  ││  ││Local  ││  ││Local  ││ │    │
│  │  ││APIC 0 ││  ││APIC 1 ││  ││APIC 2 ││  ││APIC 3 ││ │    │
│  │  │└───┬───┘│  │└───┬───┘│  │└───┬───┘│  │└───┬───┘│ │    │
│  │  └────┼────┘  └────┼────┘  └────┼────┘  └────┼────┘ │    │
│  │       └──────┬─────┴──────┬─────┴──────┬─────┘      │    │
│  │              │  APIC 总线/环  │           │            │    │
│  └──────────────┼───────────────┼───────────┼────────────┘    │
│                 │               │           │                  │
│          ┌──────┴───────┐      │           │                  │
│          │   I/O APIC   │      │           │                  │
│          │   (PCH 中)   │      │           │                  │
│          └──────┬───────┘      │           │                  │
│     ┌───────────┼───────────┐  │           │                  │
│     ▼           ▼           ▼  ▼           ▼                  │
│   设备0      设备1       设备2            MSI                  │
│  (传统IRQ)  (传统IRQ)   (传统IRQ)     (消息信号中断)            │
└───────────────────────────────────────────────────────────────┘
```

### 22.23.2 IPI（处理器间中断）机制

IPI（Inter-Processor Interrupt）是多核系统中核心之间通信的重要机制：

| IPI 类型 | 向量 | 用途 |
|---------|------|------|
| Fixed | 自定义 | 普通 IPI，投递到指定核心 |
| Lowest Priority | 自定义 | 投递到当前优先级最低的核心 |
| SMI | 2 | 系统管理中断 |
| NMI | 4 | 不可屏蔽 IPI |
| INIT | 5 | 初始化核心 |
| Start-Up | 6 | 启动 AP 核心（SIPI） |

**典型应用场景：**

- **TLB Shootdown**：一个核心修改页表后，通知其他核心刷新 TLB
- **函数调用中断（Call Function Interrupt）**：内核要求所有核心执行某函数
- **定时器同步**：同步各核心的时间戳计数器

### 22.23.3 中断亲和性与负载均衡

**中断亲和性（IRQ Affinity）：**

```
/proc/irq/<IRQ号>/smp_affinity  (Linux)

示例: 将 IRQ 32 绑定到 Core 0 和 Core 2
echo 5 > /proc/irq/32/smp_affinity   ; 5 = 0101 (Core 0 和 Core 2)
```

**中断负载均衡策略：**

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| 固定绑定 | 指定 IRQ → 固定核心 | 高性能网络/存储 |
| 自动均衡 | 内核自动分配 | 通用桌面/服务器 |
| 轮询分配 | 每次中断轮流分配 | 均匀负载 |
| NUMA 感知 | 优先分配到本地 NUMA 节点 | 多路服务器 |

### 22.23.4 MSI（消息信号中断）

MSI（Message Signaled Interrupts）通过内存写操作传递中断，而非传统的 IRQ 线：

```
传统中断:
  设备 ──IRQ线──→ I/O APIC ──→ CPU

MSI 中断:
  设备 ──内存写──→ 特定地址 ──→ Local APIC ──→ CPU
         (带向量号)
```

**MSI vs 传统中断：**

| 特性 | 传统中断 | MSI |
|------|---------|-----|
| 中断数量 | 受 IRQ 线限制（15-24） | 最多 32 个/设备 |
| 共享 | 多设备可共享一条 IRQ | 每个 MSI 向量独立 |
| 延迟 | 需经 I/O APIC 路由 | 直接写入 Local APIC |
| 向量分配 | 由 I/O APIC 决定 | 设备驱动可指定 |
| 兼容性 | 所有系统支持 | 需要芯片组和驱动支持 |

> [!TIP]
> 现代高性能设备（NVMe SSD、万兆网卡）优先使用 MSI-X，单设备最多支持 2048 个中断向量，可将不同队列的中断绑定到不同 CPU 核心，实现真正的并行处理。

### 22.23.5 APIC 中断投递模式

| 投递模式 | 说明 | 应用 |
|---------|------|------|
| **Fixed** | 投递到指定目标核心 | 常规中断 |
| **Lowest Priority** | 投递到当前负载最低的核心 | 负载均衡 |
| **SMI** | 系统管理中断 | 电源管理 |
| **NMI** | 不可屏蔽中断 | 硬件故障 |
| **INIT** | 初始化目标核心 | 多核启动 |
| **ExtINT** | 8259A 兼容模式 | Legacy 设备 |

### 22.23.6 多核中断处理流程

```
┌──────────┐     ┌──────────┐     ┌────────────────────────────────┐
│ 外部设备 │     │ I/O APIC │     │         多核 CPU               │
│          │     │          │     │                                │
│ 发送中断 │────→│ 查路由表 │     │  Core 0        Core 1          │
│          │     │ 确定目标 │     │  ┌──────┐     ┌──────┐        │
└──────────┘     │ CPU 核心 │     │  │Local │     │Local │        │
                 │          │────→│  │APIC 0│     │APIC 1│        │
                 └──────────┘     │  └──┬───┘     └──┬───┘        │
                                  │     │             │            │
                                  │     ▼             ▼            │
                                  │  ┌──────┐     ┌──────┐        │
                                  │  │判断  │     │判断  │        │
                                  │  │IF=1? │     │IF=1? │        │
                                  │  └──┬───┘     └──┬───┘        │
                                  │     │             │            │
                                  │   IF=1          IF=1           │
                                  │     │             │            │
                                  │     ▼             ▼            │
                                  │  ┌──────────────────────┐     │
                                  │  │  查 IDT 获取入口地址  │     │
                                  │  │  执行中断服务程序      │     │
                                  │  │  EOI → Local APIC    │     │
                                  │  └──────────────────────┘     │
                                  └────────────────────────────────┘
```

**多核中断处理的关键步骤：**

1. 设备发送中断请求到 I/O APIC
2. I/O APIC 根据路由表和投递模式选择目标 CPU 核心
3. 目标核心的 Local APIC 接收中断
4. 核心检查 IF 标志和中断优先级
5. 若条件满足，核心通过 IDT 获取中断处理程序入口
6. 执行中断服务程序
7. 服务完成后发送 EOI（End of Interrupt）到 Local APIC
8. Local APIC 通知 I/O APIC 可以接受下一个中断

> [!NOTE]
> 在 NUMA 架构中，中断处理的 CPU 选择还需考虑内存访问的局部性。将中断分配给离设备最近的 CPU 核心可以减少跨节点内存访问延迟，显著提升 I/O 性能。
