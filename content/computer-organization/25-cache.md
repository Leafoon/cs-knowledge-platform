# 第二十五章 高速缓冲存储器（Cache）

Cache是CPU与主存之间的高速缓冲层，利用局部性原理将主存中的热点数据复制到高速存储器中，从而大幅提高访存速度。**本章是考研中最重要的章节之一**，几乎每年都有大题涉及Cache的地址划分、命中率计算和性能分析。

## 25.1 Cache的基本原理

### 25.1.1 为什么需要Cache

**存储器墙问题**：CPU速度远快于主存速度。

| 部件 | 典型访问时间 | 说明 |
|------|------------|------|
| CPU寄存器 | <1ns | 最快 |
| L1 Cache | 1~2ns | 片内 |
| L2 Cache | 5~10ns | 片内/片外 |
| 主存（DRAM） | 50~100ns | 慢10~100倍 |

如果不使用Cache，CPU每次访存都要等待50~100ns，严重浪费CPU性能。

### 25.1.2 局部性原理

Cache能够有效工作的理论基础是**局部性原理**：

- **时间局部性**（Temporal Locality）：最近访问的数据**很可能再次被访问**
  - 例：循环中的变量、频繁调用的函数
- **空间局部性**（Spatial Locality）：相邻地址的数据**很可能被连续访问**
  - 例：数组遍历、顺序执行的指令

### 25.1.3 Cache的工作原理

1. CPU访问某个地址时，先查Cache
2. **命中（Hit）**：数据在Cache中 → 直接读取（快）
3. **缺失（Miss）**：数据不在Cache中 → 从主存读取，同时将包含该数据的整个**数据块**复制到Cache

### 25.1.4 Cache的性能公式

$$
\boxed{\text{平均访存时间} = h \times t_c + (1-h) \times t_m}
$$

其中：
- $h$ = 命中率（Hit Rate）
- $t_c$ = Cache访问时间
- $t_m$ = 主存访问时间

**【例 25-1】** Cache命中率95%，Cache访问时间1ns，主存访问时间100ns。

$$
\text{平均访存时间} = 0.95 \times 1 + 0.05 \times 100 = 0.95 + 5 = 5.95\text{ns}
$$

相比直接访问主存（100ns），速度提高了约**17倍**！

**【例 25-2】** 若命中率从95%提高到99%，性能提升多少？

- 95%时：$0.95 \times 1 + 0.05 \times 100 = 5.95$ ns
- 99%时：$0.99 \times 1 + 0.01 \times 100 = 1.99$ ns
- 提升：$5.95 / 1.99 \approx 3$ 倍

> [!TIP]
> 命中率从95%提高到99%，看似只提高了4个百分点，但性能提升了3倍！这说明**提高命中率的收益非常大**。

## 25.2 Cache的基本结构

### 25.2.1 Cache行（Cache Line/Block）

Cache以**行**（也叫块/block）为单位进行数据传输。

每个Cache行包含：
- **有效位（Valid Bit）**：该行是否包含有效数据
- **标记（Tag）**：标识该行对应主存的哪个块
- **脏位（Dirty Bit）**：该行是否被修改过（写回策略使用）
- **数据（Data）**：实际存储的数据

```
┌────────┬────────┬────────┬────────────┐
│ Valid  │ Dirty  │  Tag   │    Data    │
│  1位   │  1位   │ t位    │   块大小    │
└────────┴────────┴────────┴────────────┘
```

### 25.2.2 Cache的参数

| 参数 | 符号 | 说明 |
|------|------|------|
| Cache容量 | $C$ | Cache的总数据容量 |
| 块大小 | $B$ | 每个Cache行的数据字节数 |
| Cache行数 | $L$ | $L = C / B$ |
| 组数 | $S$ | 组相联时 $S = L / n$ |
| 相联度 | $n$ | 每组包含的行数 |

## 25.3 Cache与主存的映射方式

映射方式决定了主存块如何放置到Cache中，是Cache设计的核心问题。

### 25.3.1 直接映射（Direct Mapped Cache）

**映射规则**：每个主存块只能映射到Cache的**唯一一个位置**。

$$
\text{Cache行号} = \text{主存块号} \mod \text{Cache行数}
$$

**地址划分**：

```
┌──────────┬──────────────┬──────────────┐
│   Tag    │    Index     │   Offset     │
│  标记位   │   行索引      │  块内偏移     │
└──────────┴──────────────┴──────────────┘
```

- **Tag（标记）**：高位部分，标识是哪个主存块
- **Index（索引）**：中间部分，确定Cache行号
- **Offset（偏移）**：低位部分，确定块内哪个字节

**位数计算**：

$$
\text{Offset位数} = \log_2 B
$$

$$
\text{Index位数} = \log_2 L
$$

$$
\text{Tag位数} = \text{地址总位数} - \text{Index位数} - \text{Offset位数}
$$

**【例 25-3】** 主存地址32位，Cache容量16KB，块大小64B，直接映射。

- Cache行数 $L = 16\text{KB} / 64\text{B} = 256$ 行
- Offset = $\log_2 64 = 6$ 位
- Index = $\log_2 256 = 8$ 位
- Tag = $32 - 8 - 6 = 18$ 位

**访问过程**：

1. 用Index字段找到Cache行
2. 比较Tag字段与Cache行中存储的Tag
3. 如果Tag匹配且Valid=1 → **命中**，用Offset读取数据
4. 否则 → **缺失**，从主存调入数据块

**优点**：实现简单，查找速度快（只需比较1个Tag）

**缺点**：冲突率高。如果两个主存块映射到同一Cache行，即使Cache有空闲行也会发生冲突。

**【例 25-4】** 直接映射的冲突问题。

假设Cache有4行（Index=2位），主存块0和块4的Index都是0（$0 \mod 4 = 0$，$4 \mod 4 = 0$）。

如果程序交替访问块0和块4：
- 访问块0 → 命中
- 访问块4 → 缺失，调入块4，替换块0
- 访问块0 → 缺失，调入块0，替换块4
- ...

每次访问都缺失！命中率为0！这就是**抖动**（Thrashing）现象。

### 25.3.2 全相联映射（Fully Associative Cache）

**映射规则**：主存块可以映射到Cache的**任意一行**。

**地址划分**：

```
┌──────────────────┬──────────────┐
│       Tag        │   Offset     │
│     标记位        │  块内偏移     │
└──────────────────┴──────────────┘
```

没有Index字段！所有位都用于Tag。

**访问过程**：

1. 将地址的Tag与Cache中**所有行**的Tag同时比较
2. 如果某行Tag匹配且Valid=1 → **命中**
3. 否则 → **缺失**

**优点**：冲突率最低，空间利用率最高

**缺点**：需要比较所有行的Tag，硬件复杂（需要 $L$ 个比较器），速度慢

**适用场景**：小容量Cache（如TLB）

### 25.3.3 组相联映射（Set Associative Cache）

**映射规则**：Cache分为若干**组**（Set），每组包含 $n$ 行。主存块映射到某个组，但可以放在组内的任意一行。

$$
\text{组号} = \text{主存块号} \mod \text{组数}
$$

**地址划分**：

```
┌──────────┬──────────────┬──────────────┐
│   Tag    │  Set Index   │   Offset     │
│  标记位   │   组索引      │  块内偏移     │
└──────────┴──────────────┴──────────────┘
```

**参数关系**：

$$
n\text{路组相联}：\text{组数} S = L / n
$$

**【例 25-5】** Cache容量16KB，块大小64B，4路组相联。

- Cache行数 $L = 16\text{KB} / 64\text{B} = 256$ 行
- 组数 $S = 256 / 4 = 64$ 组
- Offset = $\log_2 64 = 6$ 位
- Set Index = $\log_2 64 = 6$ 位
- Tag = $32 - 6 - 6 = 20$ 位

**访问过程**：

1. 用Set Index找到Cache组
2. 将Tag与组内**所有 $n$ 行**的Tag同时比较
3. 如果匹配且Valid=1 → **命中**
4. 否则 → **缺失**

**特殊关系**：

| 相联度 $n$ | 组数 $S$ | 说明 |
|-----------|---------|------|
| 1 | $L$ | **直接映射** |
| $L$ | 1 | **全相联映射** |
| $1 < n < L$ | $1 < S < L$ | **组相联映射** |

### 25.3.4 三种映射方式的对比

<div data-component="CacheMappingDemo"></div>

| 特性 | 直接映射 | 全相联 | 组相联 |
|------|---------|--------|--------|
| 映射位置 | 唯一 | 任意 | 组内任意 |
| 比较次数 | 1 | $L$ | $n$ |
| 冲突率 | 高 | 最低 | 中 |
| 硬件复杂度 | 低 | 高 | 中 |
| 速度 | 快 | 慢 | 中 |
| 典型用途 | L1 Cache | TLB | L2/L3 Cache |

### 25.3.5 地址划分的计算方法

**通用公式**：

$$
\text{地址总位数} = \text{Tag位数} + \text{Index位数} + \text{Offset位数}
$$

**计算步骤**：

1. 确定Offset位数：$\log_2(\text{块大小})$
2. 确定Index位数：
   - 直接映射：$\log_2(\text{Cache行数})$
   - 组相联：$\log_2(\text{组数}) = \log_2(\text{Cache行数} / \text{相联度})$
   - 全相联：0
3. Tag位数 = 地址总位数 - Index位数 - Offset位数

**【例 25-6】** 主存地址32位，Cache容量64KB，块大小128B，8路组相联。

1. Cache行数 = $64\text{KB} / 128\text{B} = 512$ 行
2. 组数 = $512 / 8 = 64$ 组
3. Offset = $\log_2 128 = 7$ 位
4. Set Index = $\log_2 64 = 6$ 位
5. Tag = $32 - 6 - 7 = 19$ 位

**【例 25-7】** 给定地址 `0x1A3C5`，求其在4路组相联Cache（64组，块大小64B）中的映射位置。

地址 `0x1A3C5` = `0001 1010 0011 1100 0101`（20位）

- Offset（6位）= `000101` = 5（块内第5字节）
- Set Index（6位）= `110001` = 49（第49组）
- Tag（8位）= `00011010` = 26

该地址映射到第49组，Tag为26。

## 25.4 Cache的替换算法

当Cache行已满且需要调入新块时，需要选择一行进行替换。

### 25.4.1 随机替换（Random）

随机选择一行替换。

- 优点：实现简单
- 缺点：可能替换掉频繁使用的行，效果不稳定

### 25.4.2 先进先出（FIFO）

替换**最早进入**Cache的行。

- 实现：为每行维护一个计数器，记录进入时间
- 缺点：可能替换掉频繁使用的行（不考虑访问频率）

**Belady异常**：FIFO算法在某些情况下，增加Cache行数反而会增加缺失率！

### 25.4.3 最近最少使用（LRU）

替换**最长时间未被访问**的行。

**实现方式**：

**计数器法**：
- 每行维护一个计数器
- 每次访问某行时，该行计数器清零，其他行计数器加1
- 替换时选择计数器值最大的行

**堆栈法**：
- 维护一个栈，最近访问的行号放在栈顶
- 替换时选择栈底的行号

**【例 25-8】** 4路组相联，LRU替换。访问序列：A, B, C, D, A, B, E, A

| 访问 | 组内容 | 操作 |
|------|--------|------|
| A | A | 缺失，调入A |
| B | A, B | 缺失，调入B |
| C | A, B, C | 缺失，调入C |
| D | A, B, C, D | 缺失，调入D |
| A | A, B, C, D | 命中 |
| B | A, B, C, D | 命中 |
| E | A, B, C, E | 缺失，替换D（LRU） |
| A | A, B, C, E | 命中 |

<div data-component="LRUCacheSim"></div>

### 25.4.4 近似LRU：时钟算法（Clock Algorithm）

LRU的精确实现需要大量计数器，硬件开销大。**时钟算法**是LRU的近似实现。

**原理**：
- 每行有一个**引用位**（Reference Bit）
- 所有行排列成一个"时钟"，有一个指针
- 访问某行时，置该行引用位为1
- 需要替换时，指针扫描：
  - 引用位为1 → 清零，指针前移
  - 引用位为0 → 替换该行

### 25.4.5 替换算法对比

| 算法 | 实现复杂度 | 命中率 | Belady异常 |
|------|-----------|--------|-----------|
| 随机替换 | 最低 | 低 | 无 |
| FIFO | 低 | 中 | **有** |
| LRU | 高 | 高 | 无 |
| 时钟算法 | 中 | 较高 | 无 |

## 25.5 Cache的写策略

<div data-component="WritePolicyComparison"></div>

### 25.5.1 写命中（Write Hit）

当CPU写入的数据在Cache中时：

**写直达（Write-Through）**：

同时写Cache和主存。

- 优点：Cache和主存数据一致，实现简单
- 缺点：每次写都要访问主存，速度慢
- 优化：使用**写缓冲**（Write Buffer），CPU写入Cache和写缓冲后立即返回，由写缓冲异步写入主存

**写回（Write-Back）**：

只写Cache，不写主存。当该行被替换时才写回主存。

- 优点：写速度快（只写Cache）
- 缺点：需要**脏位**（Dirty Bit）标记是否被修改，数据一致性维护复杂

| 特性 | 写直达 | 写回 |
|------|--------|------|
| 写操作 | 同时写Cache和主存 | 只写Cache |
| 写速度 | 慢 | 快 |
| 脏位 | 不需要 | 需要 |
| 数据一致性 | 好 | 差（替换时才写回） |
| 总线流量 | 高 | 低 |
| 硬件复杂度 | 低 | 高 |

### 25.5.2 写不命中（Write Miss）

当CPU写入的数据不在Cache中时：

**写分配（Write-Allocate）**：

先将主存块调入Cache，再写Cache。

**非写分配（No-Write-Allocate）**：

直接写主存，不调入Cache。

### 25.5.3 常见组合

| 写命中策略 | 写不命中策略 | 常见搭配 |
|-----------|------------|---------|
| 写直达 | 非写分配 | 常见搭配 |
| 写回 | 写分配 | 常见搭配 |

> [!TIP]
> **记忆方法**：写回通常搭配写分配（因为写回策略下，调入Cache后可以减少后续的写主存操作）；写直达通常搭配非写分配（因为写直达已经保证了数据一致性）。

## 25.6 Cache缺失的分类（3C模型）

Cache缺失可以分为三类：

### 25.6.1 强制缺失（Compulsory Miss）

**首次访问**某个数据块时的缺失（冷启动缺失）。

- 无法避免（第一次访问时Cache为空）
- 可以通过**预取**（Prefetching）减少

### 25.6.2 容量缺失（Capacity Miss）

Cache**容量不足**，无法容纳工作集。

- 可以通过增大Cache容量减少

### 25.6.3 冲突缺失（Conflict Miss）

由于**映射方式的限制**，多个主存块竞争同一Cache行。

- 直接映射和组相联映射才有冲突缺失
- 全相联映射没有冲突缺失
- 可以通过增大相联度减少

> [!NOTE]
> **全相联Cache只有强制缺失和容量缺失，没有冲突缺失**。因为任何主存块都可以放在Cache的任意位置。

| 写回 + 写分配 | LRU/时钟 | 现代处理器 |

## 25.8 多级Cache

### 25.8.1 为什么需要多级Cache

单级Cache面临两难：
- 大容量Cache → 命中率高但速度慢
- 小容量Cache → 速度快但命中率低

**解决方案**：使用多级Cache，L1追求速度，L2/L3追求命中率。

### 25.8.2 多级Cache的层次

```
CPU ←→ L1 Cache ←→ L2 Cache ←→ L3 Cache ←→ 主存
```

| 层级 | 容量 | 速度 | 位置 |
|------|------|------|------|
| L1 | 32~64KB | 1~2ns | CPU片内 |
| L2 | 256KB~1MB | 5~10ns | CPU片内 |
| L3 | 4~64MB | 15~30ns | CPU片内/共享 |
| 主存 | GB级 | 50~100ns | 主板 |

### 25.8.3 包含性与排他性

- **包含性**（Inclusive）：L2包含L1的所有数据（L1 ⊂ L2）
- **排他性**（Exclusive）：L2不包含L1的数据（L1 ∩ L2 = ∅）

## 25.9 本章典型例题

### 例题1：地址划分

**题目**：某计算机主存容量1MB（$2^{20}$B），Cache容量16KB（$2^{14}$B），块大小64B（$2^6$B），采用2路组相联映射。求：
（1）Cache有多少组？
（2）地址各字段如何划分？
（3）若主存地址为 `0x1A3C5`，它映射到Cache的哪一组？

**解答**：

（1）Cache行数 = $16\text{KB} / 64\text{B} = 256$ 行

组数 = $256 / 2 = 128$ 组

（2）地址划分（主存地址20位）：

- Offset = $\log_2 64 = 6$ 位
- Set Index = $\log_2 128 = 7$ 位
- Tag = $20 - 7 - 6 = 7$ 位

```
┌───────┬───────┬──────┐
│  Tag  │ Index │Offset│
│  7位  │  7位  │  6位 │
└───────┴───────┴──────┘
```

（3）地址 `0x1A3C5` = `0001 1010 0011 1100 0101`（20位）

- Offset（6位）= `000101` = 5
- Set Index（7位）= `1000111` = 71
- Tag（7位）= `0001101` = 13

该地址映射到**第71组**，Tag为13。

### 例题2：命中率计算

**题目**：某Cache采用直接映射，块大小16B，Cache容量1KB（$2^{10}$B）。程序依次访问以下地址（十进制）：0, 16, 32, 0, 16, 32, 48, 0, 16, 32。求命中率。

**解答**：

Cache行数 = $1024 / 16 = 64$ 行

各地址的Cache行号（$地址 / 16 \mod 64$）：

| 地址 | 块号 | Cache行号 | 命中/缺失 |
|------|------|----------|----------|
| 0 | 0 | 0 | 缺失 |
| 16 | 1 | 1 | 缺失 |
| 32 | 2 | 2 | 缺失 |
| 0 | 0 | 0 | **命中** |
| 16 | 1 | 1 | **命中** |
| 32 | 2 | 2 | **命中** |
| 48 | 3 | 3 | 缺失 |
| 0 | 0 | 0 | **命中** |
| 16 | 1 | 1 | **命中** |
| 32 | 2 | 2 | **命中** |

命中次数 = 6，总访问次数 = 10

命中率 = $6 / 10 = 60\%$

### 例题3：写策略分析

**题目**：某Cache采用写回+写分配策略。初始时Cache为空，依次执行以下操作（地址用块号表示）：读块0，写块0，读块1，写块0，读块0。

分析每步的Cache状态和主存访问情况。

**解答**：

| 操作 | Cache状态 | 主存访问 | 说明 |
|------|----------|---------|------|
| 读块0 | 块0调入（V=1,D=0） | 读主存 | 缺失 |
| 写块0 | 块0修改（V=1,D=1） | 无 | 命中，写Cache |
| 读块1 | 块1调入（V=1,D=0） | 读主存 | 缺失 |
| 写块0 | 块0修改（V=1,D=1） | 无 | 命中 |
| 读块0 | 块0在Cache | 无 | 命中 |

如果之后块0被替换（脏位D=1），需要**写回主存**。

### 例题4：综合计算

**题目**：某系统Cache命中率90%，Cache访问时间10ns，主存访问时间200ns。采用写直达策略，写操作占30%，写缓冲效率95%（95%的写操作不需要等待）。求平均访存时间。

**解答**：

读操作（70%）：
- 命中：$10\text{ns}$（访问Cache）
- 缺失：$10 + 200 = 210\text{ns}$（访问Cache+主存）
- 平均读时间：$0.9 \times 10 + 0.1 \times 210 = 9 + 21 = 30\text{ns}$

写操作（30%）：
- 写缓冲命中（95%）：$10\text{ns}$（只写Cache和写缓冲）
- 写缓冲未命中（5%）：$10 + 200 = 210\text{ns}$
- 平均写时间：$0.95 \times 10 + 0.05 \times 210 = 9.5 + 10.5 = 20\text{ns}$

总平均访存时间：$0.7 \times 30 + 0.3 \times 20 = 21 + 6 = 27\text{ns}$

## 25.10 本章常见误区与易错点

1. **地址位数的计算**：Offset位数由块大小决定，Index位数由组数（或行数）决定，Tag是剩余位数。

2. **直接映射的Index是行号，组相联的Index是组号**：两者含义不同。

3. **全相联没有Index字段**：整个地址除了Offset都是Tag。

4. **LRU替换的是"最久未访问"的行**：不是"最早进入"的行（那是FIFO）。

5. **写回策略需要脏位**：脏位为1表示该行被修改过，替换时需要写回主存。

6. **写分配和非写分配是针对写不命中的策略**：不要和写命中策略混淆。

7. **Cache行的有效位**：初始时所有行的有效位为0（无效），第一次访问时才设为1。

8. **组相联映射中，同一主存块可以映射到不同组**：不对！每个主存块只能映射到唯一的一个组（由Index决定），但可以放在组内的任意一行。

## 25.13 深入理解Cache的工作原理

### 25.13.1 Cache的访问流程详解

当CPU访问某个地址时，Cache的完整工作流程：

```
1. CPU发出地址
2. 用Index字段找到Cache组
3. 将Tag与组内所有行的Tag同时比较
4. 如果匹配且Valid=1 → 命中
   a. 用Offset读取数据
   b. 返回数据给CPU
5. 否则 → 缺失
   a. 从主存读取整个数据块
   b. 选择一行进行替换（如果需要）
   c. 将数据块写入Cache
   d. 更新Tag、Valid、Dirty位
   e. 返回数据给CPU
```

### 25.13.2 Cache行的结构详解

每个Cache行包含以下字段：

| 字段 | 位数 | 说明 |
|------|------|------|
| Valid | 1位 | 该行是否包含有效数据 |
| Dirty | 1位 | 该行是否被修改过 |
| Tag | t位 | 标识对应主存块 |
| Data | 块大小 | 实际存储的数据 |

**Valid位**：
- 初始时为0（无效）
- 第一次访问时设为1
- 替换时可能设为0

**Dirty位**：
- 写回策略使用
- 写操作时设为1
- 替换时如果Dirty=1，需要写回主存

### 25.13.3 Cache的替换过程

当Cache行已满且需要调入新块时：

1. 根据替换算法选择一行
2. 如果被替换行的Dirty=1：
   a. 将该行数据写回主存
   b. 然后调入新块
3. 如果Dirty=0：
   a. 直接调入新块
4. 更新Tag、Valid、Dirty位

## 25.14 Cache的写策略深入分析

### 25.14.1 写直达（Write-Through）

**原理**：每次写操作同时写Cache和主存。

**优点**：
- Cache和主存数据一致
- 实现简单
- 缺失时不需要写回

**缺点**：
- 每次写都要访问主存，速度慢
- 总线流量大

**优化**：写缓冲（Write Buffer）

```
CPU → Cache + 写缓冲
                ↓
              主存（异步）
```

CPU写入Cache和写缓冲后立即返回，由写缓冲异步写入主存。

### 25.14.2 写回（Write-Back）

**原理**：只写Cache，不写主存。当该行被替换时才写回主存。

**优点**：
- 写速度快（只写Cache）
- 总线流量小
- 利用局部性（多次写可能只需要一次写回）

**缺点**：
- 需要脏位（Dirty Bit）
- 数据一致性维护复杂
- 缺失时可能需要写回

**脏位的作用**：
- 脏位=1：该行被修改过，替换时需要写回主存
- 脏位=0：该行未被修改，替换时可以直接丢弃

### 25.14.3 写分配（Write-Allocate）

**原理**：写不命中时，先将主存块调入Cache，再写Cache。

**特点**：
- 利用空间局部性（写入的数据可能被再次访问）
- 通常与写回策略配合使用

### 25.14.4 非写分配（No-Write-Allocate）

**原理**：写不命中时，直接写主存，不调入Cache。

**特点**：
- 简单
- 通常与写直达策略配合使用

### 25.14.5 写策略组合

| 写命中策略 | 写不命中策略 | 常见搭配 | 特点 |
|-----------|------------|---------|------|
| 写直达 | 非写分配 | 常见 | 简单，一致性好 |
| 写回 | 写分配 | 常见 | 高效，利用局部性 |

## 25.15 Cache的缺失分析（3C模型）

### 25.15.1 强制缺失（Compulsory Miss）

**定义**：首次访问某个数据块时的缺失。

**原因**：Cache初始为空，第一次访问任何数据都会缺失。

**减少方法**：
- **预取**（Prefetching）：提前将数据调入Cache
- **增大块大小**：利用空间局部性

### 25.15.2 容量缺失（Capacity Miss）

**定义**：由于Cache容量不足导致的缺失。

**原因**：程序的工作集超过了Cache容量。

**减少方法**：
- 增大Cache容量
- 优化程序（减少工作集）

### 25.15.3 冲突缺失（Conflict Miss）

**定义**：由于映射方式限制导致的缺失。

**原因**：多个主存块映射到同一Cache行。

**减少方法**：
- 增大相联度
- 使用全相联映射

### 25.15.4 3C模型总结

| 缺失类型 | 原因 | 减少方法 |
|----------|------|---------|
| 强制缺失 | 首次访问 | 预取、增大块大小 |
| 容量缺失 | Cache太小 | 增大容量 |
| 冲突缺失 | 映射冲突 | 增大相联度 |

**全相联Cache**只有强制缺失和容量缺失，没有冲突缺失。

### 练习5：3C模型分析

**题目**：某Cache的总缺失率为10%，其中强制缺失2%，容量缺失5%。如果将Cache容量翻倍，容量缺失减少一半。求优化后的总缺失率。

**解答**：

当前冲突缺失 = 10% - 2% - 5% = 3%

优化后容量缺失 = 5% / 2 = 2.5%

优化后总缺失率 = 2% + 2.5% + 3% = 7.5%

### 练习6：Cache综合计算

**题目**：某系统L1 Cache命中率90%，L1访问时间1ns；L2 Cache局部缺失率20%，L2访问时间10ns；主存访问时间100ns。求平均访存时间。

**解答**：

L1缺失率 = 10%

L2全局缺失率 = 10% × 20% = 2%

平均访存时间 = $0.9 \times 1 + 0.08 \times (1+10) + 0.02 \times (1+10+100)$

$= 0.9 + 0.88 + 2.22 = 4$ ns

## 25.21 Cache的常见错误

### 25.21.1 地址划分错误

**错误1**：混淆行号和组号。

**正确**：
- 直接映射：Index是行号
- 组相联：Index是组号

**错误2**：Offset位数计算错误。

**正确**：Offset位数 = $\log_2$(块大小)，不是 $\log_2$(块大小/字节数)

### 25.21.2 替换算法错误

**错误1**：LRU和FIFO混淆。

**正确**：
- LRU：替换最久未访问的行
- FIFO：替换最早进入的行

**错误2**：LRU的计数器更新错误。

**正确**：每次访问某行时，该行计数器清零，其他行计数器加1。

### 25.21.3 写策略错误

**错误1**：写直达和写回混淆。

**正确**：
- 写直达：同时写Cache和主存
- 写回：只写Cache，替换时写回

**错误2**：脏位的含义错误。

**正确**：脏位=1表示该行被修改过，替换时需要写回主存。

### 25.21.4 性能计算错误

**错误1**：平均访存时间公式错误。

**正确**：$t_{avg} = h \times t_c + (1-h) \times t_m$

**错误2**：多级Cache的缺失率计算错误。

**正确**：全局缺失率 = 各级局部缺失率之积
