# 第十一章 指令格式与操作码设计

指令是计算机执行操作的基本命令。本章讲解指令的基本格式、地址码的设计、操作码的编码方式（定长与变长），以及扩展操作码的设计方法。**扩展操作码设计是考研高频题型**。

---

## 11.1 指令的基本格式

### 11.1.1 指令的组成

一条指令由**操作码**和**地址码**两部分组成：

```
┌──────────┬───────────────────┐
│  操作码   │     地址码         │
│  (OP)    │   (Address)       │
└──────────┴───────────────────┘
```

- **操作码**（Opcode）：指出指令要执行什么操作（加、减、传送、跳转等）
- **地址码**（Address）：指出操作数的来源和结果的去向

### 11.1.2 指令字长

- **等长指令字**：所有指令长度相同（RISC常用）
- **变长指令字**：不同指令长度不同（CISC常用）

---

## 11.2 地址码的位数与指令格式分类

### 11.2.1 三地址指令

$$
(A_1) \text{ op } (A_2) \to A_3
$$

```
┌─────┬─────┬─────┬─────┐
│ OP  │ A1  │ A2  │ A3  │
└─────┴─────┴─────┴─────┘
```

- 两个源操作数地址 + 一个目的地址
- 优点：表达能力强
- 缺点：指令字长较长

### 11.2.2 二地址指令

$$
(A_1) \text{ op } (A_2) \to A_1
$$

```
┌─────┬─────┬─────┐
│ OP  │ A1  │ A2  │
└─────┴─────┴─────┘
```

- 结果覆盖其中一个源操作数
- 最常见的指令格式

### 11.2.3 一地址指令

$$
(\text{AC}) \text{ op } (A_1) \to \text{AC}
$$

```
┌─────┬─────┐
│ OP  │ A1  │
└─────┴─────┘
```

- 另一个操作数隐含在累加器（AC）中
- 指令字长短

### 11.2.4 零地址指令

```
┌─────┐
│ OP  │
└─────┘
```

- 无显式地址，如停机指令（HALT）、空操作（NOP）
- 栈结构计算机中的运算指令（操作数隐含在栈顶）

### 11.2.5 各格式对比

| 格式 | 地址数 | 访存次数 | 指令长度 | 适用场景 |
|------|--------|---------|---------|---------|
| 三地址 | 3 | 4 | 长 | RISC |
| 二地址 | 2 | 4 | 中 | 通用 |
| 一地址 | 1 | 3 | 短 | 早期/简单 |
| 零地址 | 0 | 1~2 | 最短 | 栈/特殊指令 |

## 11.3 操作码的设计

### 11.3.1 定长操作码

所有指令的操作码**等长**。

- 设操作码为 $k$ 位，则最多可表示 $2^k$ 条指令
- **优点**：译码简单、速度快
- **缺点**：浪费编码空间（指令数量少时大部分编码未使用）

### 11.3.2 变长操作码（扩展操作码）

短操作码表示**常用指令**，长操作码表示**不常用指令**。

**设计原则**：
- 短操作码的编码中，某些编码作为"前缀"，表示后面还有更多位属于操作码
- 地址字段减少时，操作码可以更长

**【例 11-1】** 设指令字长16位，设计4位基本操作码的扩展操作码方案：

- 4位地址指令（操作码4位）：最多16条指令
- 若某条4位操作码指令只需3个地址，则操作码可扩展为 $4+4=8$ 位
- 若某条指令只需2个地址，操作码可扩展为 $4+4+4=12$ 位
- 若只需1个地址，操作码可扩展为16位

```
0000 ─┐
0001  │ 4位OP，4地址指令
...   │（16条）
1111 ─┘

0000 0000 ─┐
0000 0001  │ 8位OP，3地址指令
...        │
1111 1111 ─┘

0000 0000 0000 ─┐
0000 0000 0001  │ 12位OP，2地址指令
...             │
1111 1111 1111 ─┘
```

> [!TIP]
> **扩展操作码的关键**：用短操作码中的一些编码作为"转义前缀"，指示后续更多位属于操作码。地址数越少，操作码越长。

## 11.4 操作码的优化 — 哈夫曼编码

### 11.4.1 思想

根据指令的**使用频率**，给高频指令分配短操作码，低频指令分配长操作码。

### 11.4.2 哈夫曼编码步骤

1. 统计每条指令的使用频率
2. 将频率最低的两条指令合并，形成一棵二叉树
3. 重复步骤2直到只剩一个节点
4. 从根到每个叶子的路径即为该指令的编码

**【例 11-2】** 6条指令的使用频率：

| 指令 | 频率 | 哈夫曼编码 | 操作码长度 |
|------|------|-----------|-----------|
| I1 | 45% | 0 | 1位 |
| I2 | 30% | 10 | 2位 |
| I3 | 15% | 110 | 3位 |
| I4 | 5% | 1110 | 4位 |
| I5 | 3% | 11110 | 5位 |
| I6 | 2% | 11111 | 5位 |

平均操作码长度 = $0.45 \times 1 + 0.30 \times 2 + 0.15 \times 3 + 0.05 \times 4 + 0.03 \times 5 + 0.02 \times 5 = 1.95$ 位

相比定长3位编码（$\log_2 6 \approx 3$），节省了约35%的空间。

## 11.5 本章典型例题

### 例题1：扩展操作码设计

**题目**：某指令系统有15条指令，指令字长16位，地址码4位。设计一个包含1条零地址指令、2条一地址指令、3条二地址指令的扩展操作码方案，最多可有多少条三地址指令？

**解答**：

- 三地址指令：操作码 $16 - 3 \times 4 = 4$ 位，最多 $2^4 = 16$ 条
- 但需要保留一些编码给短操作码指令：
  - 二地址指令3条：占用3个操作码 → 三地址指令 $16 - 1 = 15$ 条（保留1个前缀）
  - 一地址指令2条：占用2个编码
  - 零地址指令1条：占用1个编码

设三地址指令有 $x$ 条，则：

$x$ 条三地址指令占用 $x$ 个4位操作码。

保留 $16 - x$ 个前缀用于扩展。每个前缀可扩展出 $2^4 = 16$ 个8位操作码（二地址）。

需要3条二地址指令：$16 - x \geq 1$（至少1个前缀）

需要2条一地址指令：从二地址的编码中再保留前缀。

需要1条零地址指令：从一地址的编码中再保留前缀。

答案：$x = 16 - 1 = 15$ 条三地址指令（保留1个前缀）。

## 11.8 深入理解指令格式

### 11.8.1 指令格式的设计考虑

设计指令格式时需要考虑以下因素：

| 因素 | 说明 | 影响 |
|------|------|------|
| 指令数量 | 操作码能表示的指令数 | 操作码位数 |
| 寻址方式 | 支持的寻址方式数量 | 地址码位数 |
| 地址空间 | 能访问的存储器大小 | 地址码位数 |
| 指令字长 | 指令的总位数 | 取指效率 |
| 代码密度 | 程序占用的存储空间 | 指令格式的紧凑性 |

### 11.8.2 定长指令 vs 变长指令

**定长指令**：

所有指令的长度相同。

| 优点 | 缺点 |
|------|------|
| 取指简单 | 空间浪费 |
| 译码快 | 功能受限 |
| 流水线友好 | |

**变长指令**：

不同指令的长度可以不同。

| 优点 | 缺点 |
|------|------|
| 空间效率高 | 取指复杂 |
| 功能灵活 | 译码慢 |
| 代码密度高 | 流水线不友好 |

### 11.8.3 指令格式的发展趋势

**RISC风格**：
- 定长指令（通常32位）
- 简单的指令格式
- Load/Store架构

**CISC风格**：
- 变长指令（1~15字节）
- 复杂的指令格式
- 内存操作灵活

**现代融合**：
- 外部接口：CISC（兼容性）
- 内部实现：RISC（性能）

## 11.9 扩展操作码的深入分析

### 11.9.1 扩展操作码的原理

扩展操作码的核心思想：**用短操作码中的一些编码作为"转义前缀"**，指示后续更多位属于操作码。

**设计原则**：
- 短操作码表示常用指令（数量少但使用频率高）
- 长操作码表示不常用指令（数量多但使用频率低）

### 11.9.2 扩展操作码的编码规则

**规则1**：短操作码不能是长操作码的前缀

**规则2**：地址数越少，操作码越长

**规则3**：保留码字用于扩展

### 11.9.3 扩展操作码的设计示例

**【例 11-3】** 设指令字长16位，设计以下指令系统：
- 15条三地址指令
- 14条二地址指令
- 31条一地址指令
- 16条零地址指令

**设计过程**：

**三地址指令**：操作码4位，$2^4 = 16$，用15个编码，保留1个（1111）用于扩展。

**二地址指令**：操作码8位，前缀1111，后4位可表示16个编码，用14个，保留2个（1111 1110、1111 1111）用于扩展。

**一地址指令**：操作码12位，前缀1111 111x，后4位可表示16个编码，但需要两个前缀，所以 $2 \times 16 = 32$ 个编码，用31个，保留1个用于扩展。

**零地址指令**：操作码16位，前缀1111 1111 1111，后4位可表示16个编码，全部使用。

**验证**：
- 三地址指令：15条
- 二地址指令：14条
- 一地址指令：31条
- 零地址指令：16条

### 11.9.4 扩展操作码的计算方法

**给定指令系统，求最多可有多少条某类指令**：

**方法**：
1. 确定操作码位数
2. 减去已占用的编码
3. 保留必要的前缀

**【例 11-4】** 指令字长16位，地址码4位。已有1条零地址指令、2条一地址指令。最多可有多少条三地址指令？

三地址指令操作码：$16 - 3 \times 4 = 4$ 位，最多 $2^4 = 16$ 条。

需要保留编码给短操作码指令：
- 一地址指令2条：需要从二地址指令的编码中保留前缀
- 零地址指令1条：需要从一地址指令的编码中保留前缀

设三地址指令有 $x$ 条，二地址指令有 $y$ 条：

- 三地址：$x$ 个编码
- 二地址：需要从 $16 - x$ 个前缀中选择
- 一地址：需要从二地址的编码中保留前缀
- 零地址：需要从一地址的编码中保留前缀

约束条件：
- $x + y + \text{前缀} \leq 16$
- 需要足够的前缀给短操作码指令

答案取决于具体的设计方案。

## 11.10 哈夫曼编码的深入分析

### 11.10.1 哈夫曼编码的原理

哈夫曼编码是一种**最优前缀编码**，根据符号的使用频率分配编码长度。

**核心思想**：高频符号用短编码，低频符号用长编码。

### 11.10.2 哈夫曼编码的步骤

1. 统计每条指令的使用频率
2. 将频率最低的两条指令合并，形成一棵二叉树
3. 重复步骤2直到只剩一个节点
4. 从根到每个叶子的路径即为该指令的编码

### 11.10.3 哈夫曼编码的示例

**【例 11-5】** 8条指令的使用频率：

| 指令 | 频率 |
|------|------|
| I1 | 30% |
| I2 | 25% |
| I3 | 15% |
| I4 | 10% |
| I5 | 8% |
| I6 | 6% |
| I7 | 4% |
| I8 | 2% |

**构建哈夫曼树**：

```
第1步：合并I7(4%)和I8(2%) → 节点6%
第2步：合并I5(8%)和I6(6%) → 节点14%  （或合并6%和I6(6%)）
...
```

最终得到的哈夫曼编码：

| 指令 | 频率 | 编码 | 长度 |
|------|------|------|------|
| I1 | 30% | 00 | 2位 |
| I2 | 25% | 01 | 2位 |
| I3 | 15% | 100 | 3位 |
| I4 | 10% | 101 | 3位 |
| I5 | 8% | 1100 | 4位 |
| I6 | 6% | 1101 | 4位 |
| I7 | 4% | 1110 | 4位 |
| I8 | 2% | 1111 | 4位 |

**平均编码长度**：

$0.30 \times 2 + 0.25 \times 2 + 0.15 \times 3 + 0.10 \times 3 + 0.08 \times 4 + 0.06 \times 4 + 0.04 \times 4 + 0.02 \times 4 = 2.74$ 位

### 11.10.4 哈夫曼编码的特点

| 特性 | 说明 |
|------|------|
| 前缀性质 | 任何编码都不是其他编码的前缀 |
| 最优性 | 平均编码长度最短 |
| 唯一可译 | 编码序列可以唯一解码 |

## 11.11 本章综合练习

### 练习1：扩展操作码设计

**题目**：指令字长16位，地址码4位。设计一个包含2条零地址指令、3条一地址指令、4条二地址指令的扩展操作码方案，最多可有多少条三地址指令？

**解答**：

三地址指令操作码：$16 - 3 \times 4 = 4$ 位，最多 $2^4 = 16$ 条。

保留编码：
- 二地址指令4条：需要1个前缀（保留1个三地址编码）
- 一地址指令3条：需要从二地址编码中保留1个前缀
- 零地址指令2条：需要从一地址编码中保留1个前缀

设三地址指令有 $x$ 条：

- 三地址：$x$ 个编码
- 保留前缀：$16 - x$ 个
- 二地址：从 $16 - x$ 个前缀中扩展，每个前缀可扩展 $2^4 = 16$ 个编码
- 需要4条二地址指令：$16 - x \geq 1$
- 需要3条一地址指令：从二地址编码中保留前缀
- 需要2条零地址指令：从一地址编码中保留前缀

答案：$x = 15$ 条三地址指令（保留1个前缀）

### 练习2：哈夫曼编码

**题目**：4条指令的使用频率：I1=50%，I2=30%，I3=15%，I4=5%。求哈夫曼编码和平均编码长度。

**解答**：

**构建哈夫曼树**：

1. 合并I3(15%)和I4(5%) → 节点20%
2. 合并I2(30%)和20% → 节点50%
3. 合并I1(50%)和50% → 根节点100%

**编码**：

| 指令 | 频率 | 编码 | 长度 |
|------|------|------|------|
| I1 | 50% | 0 | 1位 |
| I2 | 30% | 10 | 2位 |
| I3 | 15% | 110 | 3位 |
| I4 | 5% | 111 | 3位 |

**平均编码长度**：

$0.50 \times 1 + 0.30 \times 2 + 0.15 \times 3 + 0.05 \times 3 = 1.70$ 位

### 练习3：指令格式分析

**题目**：某指令系统有64条指令，支持8种寻址方式，地址空间为64KB，数据线8位。设计指令格式。

**解答**：

- 操作码：$\log_2 64 = 6$ 位
- 寻址方式：$\log_2 8 = 3$ 位
- 地址码：$\log_2 64\text{KB} = 16$ 位
- 指令字长：$6 + 3 + 16 = 25$ 位

可以采用32位指令字长（对齐）。

## 11.12 常见错误与易错点

1. **扩展操作码的前缀**：短操作码不能是长操作码的前缀。

2. **地址数越少，操作码越长**：不是地址数越少，操作码越短。

3. **哈夫曼编码的前缀性质**：任何编码都不是其他编码的前缀。

4. **指令字长与地址码的关系**：指令字长 = 操作码位数 + 地址码位数 × 地址数。

5. **定长指令的空间浪费**：当指令功能差异大时，定长指令会浪费空间。

6. **变长指令的译码复杂**：需要先确定指令长度，再译码。

7. **哈夫曼编码的最优性**：在给定频率下，哈夫曼编码的平均长度最短。

## 11.13 指令格式的工程实践

### 11.13.1 现代处理器的指令格式

**x86指令格式**：

```
┌────────┬────────┬────────┬─────┬────────┬────────┐
│Prefix  │Opcode  │ModR/M │ SIB │Displacement│Immediate│
│ 0-4字节│ 1-3字节│ 0-1字节│0-1字节│ 0-4字节  │ 0-4字节 │
└────────┴────────┴────────┴─────┴────────┴────────┘
```

**MIPS指令格式**：

```
R型：[opcode(6)] [rs(5)] [rt(5)] [rd(5)] [shamt(5)] [funct(6)]
I型：[opcode(6)] [rs(5)] [rt(5)] [immediate(16)]
J型：[opcode(6)] [address(26)]
```

**ARM指令格式**：

```
[cond(4)] [opcode(20)] [Rd(4)] [Rn(4)] [operand2(12)]
```

### 11.13.2 指令格式的设计原则

| 原则 | 说明 | 示例 |
|------|------|------|
| 正交性 | 操作码和寻址方式独立 | ADD可用任意寻址方式 |
| 规整性 | 格式一致 | 定长指令 |
| 兼容性 | 向后兼容 | x86的16/32/64位兼容 |
| 可扩展性 | 留有扩展空间 | 操作码保留编码 |

### 11.13.3 指令格式的优化

**代码密度优化**：
- 使用变长指令
- 常用指令用短编码
- 使用压缩指令集（如Thumb、RVC）

**译码速度优化**：
- 使用定长指令
- 简单的指令格式
- 避免复杂前缀

## 11.14 指令格式与编译器

### 11.14.1 编译器对指令格式的影响

编译器需要根据指令格式生成高效的机器代码：

| 指令格式特点 | 编译器策略 |
|-------------|-----------|

## 11.15 本章扩展练习

### 练习4：扩展操作码设计（复杂）

**题目**：指令字长16位，地址码4位。设计一个包含以下指令的扩展操作码方案：
- 1条零地址指令
- 2条一地址指令
- 3条二地址指令
- 尽可能多的三地址指令

**解答**：

**三地址指令**：操作码4位，最多 $2^4 = 16$ 条。

需要保留编码给短操作码指令：
- 二地址指令3条：需要1个前缀（保留1个三地址编码）
- 一地址指令2条：需要从二地址编码中保留1个前缀
- 零地址指令1条：需要从一地址编码中保留1个前缀

设三地址指令有 $x$ 条：

- 三地址：$x$ 个编码
- 保留前缀：$16 - x$ 个
- 二地址：从 $16 - x$ 个前缀中扩展，每个前缀可扩展 $2^4 = 16$ 个编码
- 需要3条二地址指令：$16 - x \geq 1$
- 需要2条一地址指令：从二地址编码中保留前缀
- 需要1条零地址指令：从一地址编码中保留前缀

答案：$x = 15$ 条三地址指令（保留1个前缀）

### 练习5：哈夫曼编码（复杂）

**题目**：6条指令的使用频率：I1=40%，I2=25%，I3=15%，I4=10%，I5=6%，I6=4%。求哈夫曼编码和平均编码长度。

**解答**：

**构建哈夫曼树**：

1. 合并I5(6%)和I6(4%) → 节点10%
2. 合并I4(10%)和10% → 节点20%
3. 合并I3(15%)和I2(25%) → 节点40%
4. 合并20%和40% → 节点60%
5. 合并I1(40%)和60% → 根节点100%

**编码**：

| 指令 | 频率 | 编码 | 长度 |
|------|------|------|------|
| I1 | 40% | 0 | 1位 |
| I2 | 25% | 10 | 2位 |
| I3 | 15% | 110 | 3位 |
| I4 | 10% | 1110 | 4位 |
| I5 | 6% | 11110 | 5位 |
| I6 | 4% | 11111 | 5位 |

**平均编码长度**：

$0.40 \times 1 + 0.25 \times 2 + 0.15 \times 3 + 0.10 \times 4 + 0.06 \times 5 + 0.04 \times 5 = 2.24$ 位

### 练习6：指令格式分析

**题目**：某指令系统有128条指令，支持16种寻址方式，地址空间为1MB，数据线32位。设计指令格式。

**解答**：

- 操作码：$\log_2 128 = 7$ 位
- 寻址方式：$\log_2 16 = 4$ 位
- 地址码：$\log_2 1\text{MB} = 20$ 位
- 指令字长：$7 + 4 + 20 = 31$ 位

可以采用32位指令字长（对齐），剩余1位用于其他用途。

## 11.18 指令格式的常见错误

### 11.18.1 扩展操作码错误

**错误1**：前缀冲突。

**正确**：短操作码不能是长操作码的前缀。

**错误2**：保留码字错误。

**正确**：需要保留足够的码字用于扩展。

**错误3**：地址数与操作码长度关系错误。

**正确**：地址数越少，操作码越长。

### 11.18.2 哈夫曼编码错误

**错误1**：合并顺序错误。

**正确**：每次合并频率最低的两个节点。

**错误2**：编码分配错误。

**正确**：从根到叶子的路径即为编码。

### 11.18.3 指令字长错误

**错误1**：指令字长计算遗漏。

**正确**：指令字长 = 操作码位数 + 地址码位数 × 地址数。

**错误2**：地址空间计算错误。

**正确**：地址空间 = $2^{\text{地址码位数}}$。

## 11.21 指令译码器的内部结构与工作原理

### 11.21.1 指令译码的基本流程

指令译码器（Instruction Decoder）是控制单元的核心部件，负责将二进制指令码翻译为一组控制信号。其基本工作流程如下：

```
指令存储器 → 指令寄存器IR → 操作码字段提取 → 译码器 → 控制信号
```

**译码过程的三个阶段**：

| 阶段 | 操作 | 说明 |
|------|------|------|
| 取指 | IR ← M(PC) | 将指令从存储器装入指令寄存器 |
| 分离 | OP ← IR[opcode字段] | 从IR中提取操作码字段 |
| 译码 | 控制信号 ← Decode(OP) | 将操作码映射为控制信号 |

### 11.21.2 指令寄存器（IR）

指令寄存器是CPU中保存当前正在执行指令的专用寄存器。

**IR的位域划分**（以MIPS为例）：

```
┌──────────────────────────────────────────────────────────┐
│                    指令寄存器 IR（32位）                    │
├──────┬─────┬─────┬─────┬──────┬──────┤
│opcode│ rs  │ rt  │ rd  │shamt │ func │
│ 6位  │ 5位 │ 5位 │ 5位 │ 5位  │ 6位  │
│[31:26]│[25:21]│[20:16]│[15:11]│[10:6]│[5:0] │
└──────┴─────┴─────┴─────┴──────┴──────┘
```

IR的输出直接连接到控制单元和数据通路的各个多路选择器，无需额外的寄存器传输。

> [!NOTE]
> IR的内容在整个指令执行周期内保持不变，直到下一条指令被取指。这保证了控制信号在整个执行过程中稳定。

### 11.21.3 操作码字段提取

操作码字段的提取通过硬布线连接实现——IR的特定位直接连到译码器的输入端。

**定长操作码**（RISC）：

```
IR[31:26] → 主译码器 → 主控制信号
IR[5:0]   → ALU译码器 → ALU控制信号（R型指令）
```

**变长操作码**（CISC）：

```
IR[7:0] → 第一级译码 → 判断是否需要额外字节
IR[15:8] → 第二级译码（若需要）
IR[23:16] → 第三级译码（若需要）
```

变长操作码需要**多级译码**，第一级译码器根据前缀字节判断指令长度，再逐级译码后续字段。

### 11.21.4 译码器的硬件实现

**一、硬布线译码器**

硬布线译码器使用组合逻辑电路（AND门阵列）直接将操作码映射为控制信号。

```
        ┌─────────────────────┐
opcode ─┤                     ├─ RegWrite
  位    │   PLA / ROM         ├─ MemRead
  输入   │   译码阵列           ├─ MemWrite
        │                     ├─ ALUOp
        │                     ├─ Branch
        └─────────────────────┘
```

**PLA实现**：可编程逻辑阵列，每个输出对应一个控制信号，输入为操作码各位及其反码。

**ROM实现**：以操作码为地址，ROM中存储对应的控制信号字。

| 实现方式 | 优点 | 缺点 |
|----------|------|------|
| PLA | 灵活可修改 | 速度稍慢 |
| ROM | 速度快 | 不灵活 |
| 纯门电路 | 最快 | 设计复杂 |

**二、微程序译码器**

CISC处理器使用微程序控制，译码器输出的是微程序入口地址：

```
操作码 → 微程序入口地址 → 控制存储器 → 微指令 → 控制信号
```

### 11.21.5 两级译码结构

现代高性能处理器通常采用两级译码：

**第一级译码**：将操作码映射为内部微操作类型

**第二级译码**：根据微操作类型生成具体的控制信号

```
操作码 → [一级译码] → 微操作码 → [二级译码] → 控制信号
```

这种结构将译码延迟分摊到两个阶段，每一级的逻辑深度较浅，有利于提高时钟频率。

> [!TIP]
> RISC处理器的译码器通常只需一级，因为操作码格式规整，译码逻辑简单。CISC处理器的译码器需要多级，因为变长操作码需要先确定指令边界。

## 11.22 指令预取单元的工作机制

### 11.22.1 预取的基本原理

指令预取（Instruction Prefetch）是指在当前指令执行的同时，提前从存储器取出后续指令，隐藏访存延迟。

**基本思想**：CPU执行速度远高于存储器访问速度，如果不预取，CPU会在取指阶段频繁等待。

```
时间 →
CPU：  [执行I1] [执行I2] [执行I3] ...
取指：  [取I1]   [取I2]   [取I3] ...
预取：  [取I2]   [取I3]   [取I4] ...  ← 与执行并行
```

### 11.22.2 预取单元的组成

```
┌────────────────────────────────────────┐
│            指令预取单元                  │
│  ┌──────────┐  ┌──────────────────┐    │
│  │ 预取地址  │  │  指令缓冲队列     │    │
│  │ 生成器   │→│  (Instruction     │    │
│  │          │  │   Queue/Buffer)  │    │
│  └──────────┘  └──────────────────┘    │
│       ↑                ↓               │
│    分支预测器      取指/译码单元         │
└────────────────────────────────────────┘
```

**预取地址生成器**：

| 情况 | 地址来源 | 说明 |
|------|---------|------|
| 顺序执行 | PC + 指令长度 | 最简单 |
| 分支跳转 | 分支预测器 | 预测目标地址 |
| 函数调用 | 返回地址栈 | 预测返回地址 |

### 11.22.3 预取策略

**一、顺序预取**

每次预取下一条顺序指令，适用于线性代码。

**二、分支感知预取**

结合分支预测，预取预测路径上的指令。

**三、提前预取**

在指令缓存缺失时，提前发出预取请求，减少停顿。

| 预取策略 | 复杂度 | 命中率 | 适用场景 |
|----------|--------|--------|---------|
| 顺序预取 | 低 | 中 | 线性代码 |
| 分支感知 | 高 | 高 | 含分支代码 |
| 提前预取 | 中 | 中 | 缓存密集访问 |

### 11.22.4 预取与指令格式的关系

指令格式对预取效率有重要影响：

| 指令格式 | 预取难度 | 原因 |
|----------|---------|------|
| 定长指令 | 低 | 每次预取固定字节数 |
| 变长指令 | 高 | 需先确定指令边界 |

> [!NOTE]
> RISC的定长指令（如MIPS的32位）使得预取单元设计简单——每次从I-Cache取一个32位字即为一条完整指令。x86的变长指令则需要复杂的预取对齐逻辑。

## 11.23 指令队列与指令缓冲的设计

### 11.23.1 指令队列的作用

指令队列（Instruction Queue）位于取指单元和译码单元之间，起到**解耦**和**缓冲**的作用。

```
┌──────┐    ┌──────────┐    ┌──────┐
│ 取指  │ → │ 指令队列  │ → │ 译码  │
│ 单元  │    │ (Buffer) │    │ 单元  │
└──────┘    └──────────┘    └──────┘
  快速         缓冲           可能停顿
```

**核心功能**：
- 平衡取指和译码的速度差异
- 在译码单元停顿（如数据冒险）时继续预取
- 支持多指令同时译码（超标量）

### 11.23.2 队列的组织方式

**一、FIFO队列**

最基本的组织方式，先取的指令先译码。

```
┌────┬────┬────┬────┬────┬────┬────┬────┐
│ I1 │ I2 │ I3 │ I4 │    │    │    │    │
└────┴────┴────┴────┴────┴────┴────┴────┘
  ↑                                 ↑
 入队(取指)                        出队(译码)
```

**二、指令行缓冲（Line Buffer）**

缓存一整行I-Cache数据，通常包含多条指令。

| 设计参数 | 典型值 | 说明 |
|----------|--------|------|
| 队列深度 | 4~16条指令 | 决定可缓冲的指令数 |
| 队列宽度 | 1~4条/周期 | 决定取指带宽 |
| 对齐方式 | 按缓存行对齐 | 简化硬件 |

### 11.23.3 超标量处理器的指令队列

在超标量处理器中，指令队列需要支持**多指令并行取指和译码**：

```
┌──────────────────────────────────┐
│          指令队列（宽度=4）        │
│ ┌────┬────┬────┬────┐            │
│ │ I1 │ I2 │ I3 │ I4 │  → 译码器  │
│ └────┴────┴────┴────┘            │
│ ┌────┬────┬────┬────┐            │
│ │ I5 │ I6 │ I7 │ I8 │  → 译码器  │
│ └────┴────┴────┴────┘            │
└──────────────────────────────────┘
```

### 11.23.4 指令缓冲与分支处理

当发生分支时，预测路径上的指令需要从队列中清除（分支误预测冲刷）：

```
分支预测正确：继续使用队列中的指令
分支预测错误：清空队列，从正确地址重新取指
```

| 处理方式 | 恢复速度 | 硬件复杂度 |
|----------|---------|-----------|
| 全部冲刷 | 慢 | 低 |
| 部分保留 | 快 | 高 |
| 路径记录 | 最快 | 最高 |

> [!TIP]
> 指令队列深度是性能和面积的权衡——更深的队列能更好地隐藏延迟，但占用更多芯片面积和功耗。现代处理器通常使用8~16项的指令队列。

## 11.24 指令格式对CPU数据通路设计的影响

### 11.24.1 指令字长与数据通路宽度

指令字长直接影响数据通路的宽度：

| 指令字长 | 数据通路宽度 | 取指周期 | 说明 |
|----------|------------|---------|------|
| 16位 | 16位 | 1周期 | 简单嵌入式 |
| 32位 | 32位 | 1周期 | RISC标准 |
| 变长 | 64/128位 | 1~2周期 | CISC需预取 |

定长指令使得取指单元可以每次取一条完整指令，数据通路设计简单。

变长指令需要预取多个字节才能确定指令边界，数据通路中需要额外的**指令对齐逻辑**。

### 11.24.2 操作码格式与译码通路

**定长操作码**（RISC）：

```
IR[31:26] ──→ 6-to-64译码器 ──→ 64条控制信号
                  (简单，快速)
```

**变长操作码**（CISC）：

```
IR[7:0] ──→ 第一级译码 ──→ 判断指令长度
    ↓
IR[15:8] ──→ 第二级译码（若需要）
    ↓
IR[23:16] ──→ 第三级译码（若需要）
    ↓
最终控制信号
```

| 操作码格式 | 译码级数 | 译码延迟 | 控制信号数 |
|-----------|---------|---------|-----------|
| 定长6位 | 1级 | 1个门延迟 | 64条 |
| 定长8位 | 1级 | 2个门延迟 | 256条 |
| 变长1~3字节 | 2~3级 | 3~5个门延迟 | 可变 |

### 11.24.3 地址码字段与ALU输入通路

<div data-component="AddressFieldDesigner"></div>

地址码字段的数量决定了ALU输入端口和寄存器文件读端口的数量：

| 地址码数 | 寄存器读端口 | ALU输入数 | 数据通路复杂度 |
|----------|------------|----------|-------------|
| 三地址 | 2读+1写 | 2 | 中 |
| 二地址 | 1读+1写 | 2 | 低 |
| 一地址 | 1读+1写 | 1+AC | 最低 |

三地址指令需要同时读取两个源寄存器，要求寄存器文件有**两个读端口**：

```
寄存器文件
  ├─ 读端口1 ──→ rs ──→ ALU输入A
  ├─ 读端口2 ──→ rt ──→ ALU输入B
  └─ 写端口1 ──← rd ←── ALU输出
```

### 11.24.4 指令格式对流水线的影响

指令格式的规整性直接影响流水线效率：

| 特性 | 对流水线的影响 |
|------|-------------|
| 定长指令 | 每周期取指数量固定，流水线规整 |
| 变长指令 | 取指数量不固定，需要对齐逻辑 |
| 固定寄存器字段位置 | 译码阶段可并行读寄存器 |
| 可变寄存器字段位置 | 需先译码再读寄存器 |
| Load/Store架构 | 访存只在特定流水级，简化控制 |

> [!NOTE]
> MIPS的三种指令格式（R/I/J）中，rs和rt字段的位置始终在[25:21]和[20:16]，这使得寄存器文件可以在译码阶段直接读取，无需等待译码完成。这是RISC指令格式对流水线友好的典型体现。

### 11.24.5 指令格式对数据通路延迟的影响

数据通路中各部件的延迟与指令格式密切相关：

```
取指 → 译码 → 寄存器读 → ALU → 访存 → 写回
 │       │       │        │      │       │
 I-Cache  译码器   寄存器    ALU   D-Cache  寄存器
 延迟     延迟     延迟     延迟   延迟     延迟
```

| 流水级 | 典型延迟 | 与指令格式的关系 |
|--------|---------|---------------|
| 取指 | 1~2ns | 定长指令更快 |
| 译码 | 0.5~2ns | 变长指令更慢 |
| 寄存器读 | 0.5~1ns | 字段位置固定更快 |
| ALU | 0.5~1ns | 与格式关系较小 |
| 访存 | 1~5ns | Load/Store格式影响地址计算 |
| 写回 | 0.5~1ns | 字段位置固定更快 |

> [!TIP]
> 指令格式的设计是ISA设计中最关键的决策之一。它不仅影响软件的代码密度和编译器复杂度，更直接决定了CPU数据通路的硬件复杂度、译码延迟和流水线效率。RISC的定长格式是"以空间换时间"的典型体现。
