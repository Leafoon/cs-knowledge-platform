# 第三十二章 程序中断方式

中断驱动I/O是程序查询方式的重要改进，CPU不再忙等待，而是去做其他工作，设备准备好后通过中断通知CPU。

---

## 32.1 中断驱动I/O的基本原理

### 32.1.1 工作流程

<div data-component="InterruptDrivenIO"></div>

```
1. CPU发送I/O命令 → 继续执行其他指令
2. 设备独立准备数据
3. 设备完成 → 发中断请求
4. CPU响应中断 → 执行中断服务程序（数据传输）
5. 中断返回 → 继续原程序
```

### 32.1.2 与程序查询的对比

| 特性 | 程序查询 | 中断驱动 |
|------|---------|---------|
| CPU等待 | **忙等待** | **不等待** |
| CPU利用率 | **低** | **高** |
| 实时性 | 差 | **好** |
| 硬件复杂度 | 低 | 中 |
| 适用 | 低速设备 | 中低速设备 |
| 数据传输单位 | 字/字节 | 字/字节 |
| CPU与设备并行 | **无** | **有** |
| 软件开销 | 轮询循环 | 中断服务程序 |
| 硬件开销 | 状态寄存器 | 中断控制器+请求/屏蔽逻辑 |
| 响应延迟 | 等待轮询周期结束 | **指令执行完毕即响应** |
| 实现难度 | 简单 | 中等 |

### 32.1.3 中断驱动I/O的系统组成

## 32.2 中断驱动I/O的完整过程

### 32.2.1 CPU发送命令

CPU向I/O接口写入控制字：
- 设备地址
- 读/写命令
- 传输数据量
- 启动设备

然后CPU继续执行其他指令。

### 32.2.2 设备准备数据

设备独立工作，CPU无需等待。

### 32.2.3 中断请求与响应

设备完成后：
1. 设备置完成标志
2. 设备发出中断请求
3. CPU在当前指令执行完毕后检查中断
4. 如果条件满足，响应中断

### 32.2.4 中断处理

```
1. 关中断
2. 保护断点（PC → 栈）
3. 保护现场（寄存器 → 栈）
4. 开中断
5. 执行中断服务程序（数据传输）
6. 关中断
7. 恢复现场
8. 开中断
9. 中断返回
```

## 32.3 中断硬件结构

### 32.3.1 中断请求寄存器（IRR）

中断请求寄存器记录各设备的中断请求状态。

```
           ┌───────────────────────────────────────────────┐
           │          中断请求寄存器 (IRR)                   │
           │  ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┐ │
           │  │ IR7 │ IR6 │ IR5 │ IR4 │ IR3 │ IR2 │ IR1 │ │
           │  └──┬──┴──┬──┴──┬──┴──┬──┴──┬──┴──┬──┴──┬──┘ │
           │     │     │     │     │     │     │     │     │
           └─────┼─────┼─────┼─────┼─────┼─────┼─────┼─────┘
                 │     │     │     │     │     │     │
                 ▼     ▼     ▼     ▼     ▼     ▼     ▼
              设备7  设备6  设备5  设备4  设备3  设备2  设备1
```

**IRR的工作机制**：
- 当设备 $i$ 发出中断请求时，$IRR_i \leftarrow 1$
- CPU响应中断后，对应位被清除
- 多个位可以同时为1（多个设备同时请求）

### 32.3.2 中断屏蔽寄存器（IMR）

中断屏蔽寄存器控制哪些中断允许被响应。

```
           ┌───────────────────────────────────────────────┐
           │          中断屏蔽寄存器 (IMR)                   │
           │  ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┐ │
           │  │ M7  │ M6  │ M5  │ M4  │ M3  │ M2  │ M1  │ │
           │  └─────┴─────┴─────┴─────┴─────┴─────┴─────┘ │
           │                                               │
           │  Mi = 1 → 屏蔽设备i的中断                      │
           │  Mi = 0 → 允许设备i的中断                      │
           └───────────────────────────────────────────────┘
```

**屏蔽逻辑**：

$$
\text{有效请求} = IRR \wedge \overline{IMR}
$$

即：只有未被屏蔽的中断请求才能被CPU响应。

### 32.3.3 中断服务寄存器（ISR）

中断服务寄存器记录当前正在被服务的中断。

```
           ┌───────────────────────────────────────────────┐
           │          中断服务寄存器 (ISR)                   │
           │  ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┐ │
           │  │ IS7 │ IS6 │ IS5 │ IS4 │ IS3 │ IS2 │ IS1 │ │
           │  └─────┴─────┴─────┴─────┴─────┴─────┴─────┘ │
           │                                               │
           │  ISi = 1 → 设备i的中断正在被服务                │
           │  ISi = 0 → 设备i的中断未在服务                  │
           └───────────────────────────────────────────────┘
```

**ISR的工作机制**：
- CPU响应设备 $i$ 的中断时，$ISR_i \leftarrow 1$
- 中断服务程序执行完毕后，$ISR_i \leftarrow 0$
- 用于实现中断嵌套时的优先级判断

### 32.3.4 三个寄存器的协同工作

```
                设备中断请求
                     │
                     ▼
            ┌────────────────┐
            │  IRR: 记录请求  │
            └───────┬────────┘
                    │
                    ▼
            ┌────────────────┐     ┌────────────────┐
            │  与非门逻辑     │◄────│  IMR: 屏蔽字    │
            │  IRR ∧ ~IMR    │     └────────────────┘
            └───────┬────────┘
                    │
                    ▼
            ┌────────────────┐     ┌────────────────┐
            │  优先级判优器   │◄────│  ISR: 当前服务  │
            └───────┬────────┘     └────────────────┘
                    │
                    ▼
            ┌────────────────┐
            │  向CPU发INTR    │
            └────────────────┘
```

> [!NOTE]
> IRR、IMR、ISR三个寄存器是中断控制器（如8259A）的核心组成部分。IRR记录谁在请求，IMR决定谁能请求，ISR记录谁正在被服务。

## 32.4 中断优先级判优电路

### 32.4.1 硬件判优的基本原理

<div data-component="PriorityArbitrationCircuit"></div>

当多个设备同时发出中断请求时，需要硬件判优电路确定最高优先级。

**判优方式分类**：

| 判优方式 | 原理 | 优点 | 缺点 |
|---------|------|------|------|
| 菊花链判优 | 串行优先级 | 简单 | 速度慢，离CPU越近优先级越高 |
| 并行判优 | 编码器+比较器 | 速度快 | 硬件复杂 |
| 专用中断控制器 | 可编程优先级 | 灵活 | 成本高 |

### 32.4.2 菊花链判优电路

```
        CPU
         │
    ┌────┴────┐
    │ INTA信号 │ ─────────────────────────────────────────→
    └────┬────┘
         │                    菊花链（串行传递）
         │    ┌──────────┐    ┌──────────┐    ┌──────────┐
         │    │ 设备0    │    │ 设备1    │    │ 设备2    │
         │    │ ┌──────┐ │    │ ┌──────┐ │    │ ┌──────┐ │
    INTA─┼───→│ │有请求?│─┼───→│ │有请求?│─┼───→│ │有请求?│ │
         │    │ └──┬───┘ │    │ └──┬───┘ │    │ └──┬───┘ │
         │    │    │     │    │    │     │    │    │     │
         │    │  有│无   │    │  有│无   │    │  有│无   │
         │    │    ↓  ↓  │    │    ↓  ↓  │    │    ↓  ↓  │
         │    │ 截止 传递│    │ 截止 传递│    │ 截止 传递│
         │    └──────────┘    └──────────┘    └──────────┘
```

**工作过程**：
1. INTA信号沿菊花链串行传递
2. 每个设备检查自己是否有中断请求
3. 第一个有请求的设备截获INTA，不再向下传递
4. 该设备将中断类型号放到数据总线上

> [!TIP]
> 菊花链判优中，离CPU最近的设备优先级最高。如果设备0和设备2同时请求，设备0先获得INTA。

### 32.4.3 并行判优电路

```
    IR7 ──→┐
    IR6 ──→┤
    IR5 ──→┤    ┌─────────────┐     ┌─────────────┐     ┌──────────┐
    IR4 ──→┼───→│  优先级编码器 │────→│   比较器     │────→│ 中断类型号│
    IR3 ──→┤    │  (7→3编码)   │     │ (与ISR比较)  │     └──────────┘
    IR2 ──→┤    └─────────────┘     └──────┬──────┘
    IR1 ──→┘                               │
                                    ISR7~ISR1 (当前服务)
```

**优先级编码器真值表**（部分）：

| IR7 | IR6 | IR5 | IR4 | IR3 | IR2 | IR1 | 编码输出 | 响应设备 |
|-----|-----|-----|-----|-----|-----|-----|---------|---------|
| 1 | × | × | × | × | × | × | 111 | 设备7 |
| 0 | 1 | × | × | × | × | × | 110 | 设备6 |
| 0 | 0 | 1 | × | × | × | × | 101 | 设备5 |
| 0 | 0 | 0 | 1 | × | × | × | 100 | 设备4 |
| 0 | 0 | 0 | 0 | 1 | × | × | 011 | 设备3 |
| 0 | 0 | 0 | 0 | 0 | 1 | × | 010 | 设备2 |
| 0 | 0 | 0 | 0 | 0 | 0 | 1 | 001 | 设备1 |

### 32.4.4 可编程中断控制器（8259A）的判优

8259A使用ISR和IMR配合实现可编程优先级：

```
            ┌─────────────────────────────────────────┐
            │              8259A                       │
            │                                         │
            │  ┌─────┐   ┌─────┐   ┌─────────────┐   │
            │  │ IRR │   │ IMR │   │  优先级      │   │
            │  │     │   │     │   │  判优电路     │   │
            │  └──┬──┘   └──┬──┘   └──────┬──────┘   │
            │     │         │             │           │
            │     └────┬────┘             │           │
            │          ▼                  ▼           │
            │     ┌─────────┐      ┌──────────┐      │
            │     │IRR∧~IMR │─────→│与ISR比较  │      │
            │     └─────────┘      └────┬─────┘      │
            │                          │             │
            │                     ┌────┴────┐        │
            │                     │ INTR输出 │        │
            │                     └────┬────┘        │
            └──────────────────────────┼─────────────┘
                                       │
                                       ▼
                                      CPU
```

**优先级判断规则**：
1. 有效请求 = $IRR \wedge \overline{IMR}$
2. 在有效请求中，找出优先级最高的
3. 与ISR中当前服务的中断比较
4. 若新请求优先级更高，发出INTR信号

## 32.5 中断响应的硬件时序

### 32.5.1 中断周期的微操作

CPU在每个指令执行完毕后检查中断请求信号。如果满足响应条件，进入中断周期。

```
┌─────────────────────────────────────────────────────────────────┐
│                    指令周期与中断周期                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────────────────┐ │
│  │ 取指 │→│ 译码 │→│ 执行 │→│ 访存 │→│ 中断周期（若有中断）│ │
│  └──────┘  └──────┘  └──────┘  └──────┘  └──────────────────┘ │
│                                                  │              │
│                                      ┌───────────┴───────────┐  │
│                                      │  无中断 → 返回取指     │  │
│                                      └───────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

### 32.5.2 中断周期的详细微操作序列

```
中断周期（Interrupt Cycle, INT）：

T0: MAR ← SP          // 栈指针送地址寄存器
T1: MDR ← PC          // 保存PC到MDR
    PC  ← 0           // PC清零（准备接收中断向量）
T2: M[MAR] ← MDR      // PC写入栈（保护断点）
    SP  ← SP - 1      // 栈指针减1
T3: MAR ← 中断向量地址 // 取中断向量
T4: MDR ← M[MAR]      // 读取中断服务程序入口地址
T5: PC  ← MDR         // PC指向中断服务程序
    INTE ← 0          // 关中断
```

> [!NOTE]
> 以上微操作序列是典型的冯·诺依曼体系结构下的中断响应过程。不同CPU的具体实现可能有所不同，但基本步骤一致。

### 32.5.3 中断响应的条件

CPU响应中断必须同时满足以下条件：

| 条件 | 说明 |
|------|------|
| 有中断请求 | $INTR = 1$（至少一个设备请求） |
| CPU允许中断 | $INTE = 1$（中断允许触发器置1） |
| 指令执行完毕 | 当前指令的最后一个机器周期结束 |
| 无更高优先级 | 无DMA请求等更高优先级事件 |
| 无总线请求 | 无HOLD等总线请求信号 |

> [!TIP]
> CPU在每个指令周期的最后检查INTR信号。如果条件满足，下一个周期就是中断周期，而不是取指周期。

### 32.5.4 中断响应的时序图

```
时钟    ──┐ ┌──┐ ┌──┐ ┌──┐ ┌──┐ ┌──┐ ┌──┐ ┌──┐ ┌──┐ ┌──┐ ┌──
          │ │  │ │  │ │  │ │  │ │  │ │  │ │  │ │  │ │  │ │
          └─┘  └─┘  └─┘  └─┘  └─┘  └─┘  └─┘  └─┘  └─┘  └─┘

INTR    ────────────────────────────────┐
                                        │ 高电平表示有中断请求
                                        └─────────────────────

INTA    ───────────────────────────────────────┐
                                               │ CPU发出应答信号
                                               └──────────

PC      ════════════════════════════════╗
                                        ║ PC值保存到栈
                                        ╚══════════════════

SP      ═══════════════════════════════════════╗
                                               ║ SP减1
                                               ╚═══════════

状态     │ 取指 │ 执行 │ 取指 │ 执行 │ INT │ INT │ INT │ 取指 │
         │      │      │      │      │ T0  │ T1  │ T2  │(服务)│
```

## 32.6 中断服务程序的完整流程

### 32.6.1 中断服务程序的标准框架

```
中断服务程序入口：
    ┌───────────────────────────────────────┐
    │  ① 保护现场                            │
    │     PUSH  ACC                         │
    │     PUSH  PSW                         │
    │     PUSH  通用寄存器                    │
    └──────────────────┬────────────────────┘
                       │
                       ▼
    ┌───────────────────────────────────────┐
    │  ② 开中断（允许中断嵌套）               │
    │     STI  /  设置INTE=1                 │
    └──────────────────┬────────────────────┘
                       │
                       ▼
    ┌───────────────────────────────────────┐
    │  ③ 执行中断服务（数据传输等）            │
    │     从设备读取数据 / 向设备写入数据       │
    └──────────────────┬────────────────────┘
                       │
                       ▼
    ┌───────────────────────────────────────┐
    │  ④ 关中断                              │
    │     CLI  /  设置INTE=0                 │
    └──────────────────┬────────────────────┘
                       │
                       ▼
    ┌───────────────────────────────────────┐
    │  ⑤ 恢复现场                            │
    │     POP  通用寄存器                     │
    │     POP  PSW                           │
    │     POP  ACC                           │
    └──────────────────┬────────────────────┘
                       │
                       ▼
    ┌───────────────────────────────────────┐
    │  ⑥ 开中断                              │
    │     STI  /  设置INTE=1                 │
    └──────────────────┬────────────────────┘
                       │
                       ▼
    ┌───────────────────────────────────────┐
    │  ⑦ 中断返回                            │
    │     IRET  /  PC ← 栈顶                 │
    └───────────────────────────────────────┘
```

### 32.6.2 保护现场与恢复现场

**保护现场**：将CPU中所有可能被中断服务程序修改的寄存器内容保存到栈中。

```
          保护现场前               保护现场后
          ┌─────────┐             ┌─────────┐
    SP →  │  旧栈   │       SP →  │  R3     │  ← 最后压入
          │  内容   │             │  R2     │
          │  ...    │             │  R1     │
          └─────────┘             │  ACC    │
                                  │  PSW    │
                                  │  PC     │  ← 断点
                                  │  旧栈   │
                                  │  内容   │
                                  │  ...    │
                                  └─────────┘
```

**需要保护的寄存器**：

| 寄存器类型 | 具体寄存器 | 是否必须保护 |
|-----------|-----------|-------------|
| 程序计数器 | PC | **必须**（硬件自动） |
| 程序状态字 | PSW/FLAGS | **必须** |
| 累加器 | ACC | **必须** |
| 通用寄存器 | R0~Rn | 按需（只保护被修改的） |
| 段寄存器 | CS/DS/SS | 按需 |
| 栈指针 | SP | 通常不需要 |

### 32.6.3 中断服务程序中的数据传输

以从设备读取一个字节为例：

```
中断服务程序（读取设备数据）：
    ; 保护现场
    PUSH ACC
    PUSH R1
    
    ; 读取设备数据
    IN   R1, 设备数据端口    ; 从设备端口读取数据到R1
    MOV  [缓冲区指针], R1    ; 将数据存入缓冲区
    INC  缓冲区指针          ; 指针加1
    DEC  剩余计数            ; 计数减1
    
    ; 清除中断请求
    OUT  设备控制端口, 0x00  ; 清除设备的中断请求标志
    
    ; 恢复现场
    POP  R1
    POP  ACC
    IRET                   ; 中断返回
```

## 32.7 中断方式的性能分析

<div data-component="IOInterruptLatency"></div>

### 32.7.1 CPU利用率分析

**中断方式下的CPU利用率**：

$$
\text{CPU利用率} = 1 - \frac{\text{中断处理时间}}{\text{总时间}}
$$

其中：

$$
\text{中断处理时间} = \text{中断次数} \times \text{每次中断开销}
$$

### 32.7.2 中断开销的详细组成

| 阶段 | 操作 | 典型周期数 |
|------|------|-----------|
| 中断响应 | 保存PC、获取向量 | 5~10 |
| 保护现场 | 压栈寄存器 | 10~30 |
| 中断服务 | 数据传输 | 10~50 |
| 恢复现场 | 弹栈寄存器 | 10~30 |
| 中断返回 | 恢复PC | 3~5 |
| **总计** | | **38~125** |

### 32.7.3 性能计算实例

**【例 32-4】** 某系统CPU时钟频率500MHz，设备数据传输率500KB/s，每次中断传输1字节，中断开销80个时钟周期。

**求解**：

每秒中断次数：

$$
N = 500 \times 1024 = 512000 \text{次/秒}
$$

每秒中断处理时间：

$$
T_{int} = 512000 \times 80 / 500 \times 10^6 = 81.92\text{ms}
$$

CPU用于中断的时间占比：

$$
\text{占比} = \frac{81.92\text{ms}}{1000\text{ms}} = 8.19\%
$$

> [!TIP]
> 当设备数据传输率很高时，中断方式的CPU开销会急剧增加。例如传输率提高到5MB/s，CPU开销将达到81.9%，这就是高速设备需要使用DMA方式的原因。

### 32.7.4 中断方式的数据传输率上限

设CPU时钟频率为 $f$，中断开销为 $C$ 个时钟周期，每次传输 $n$ 字节，则中断方式的最大数据传输率为：

$$
R_{max} = \frac{f \times n}{C}
$$

**【例 32-5】** CPU时钟1GHz，中断开销60周期，每次传输1字节。

$$
R_{max} = \frac{1 \times 10^9 \times 1}{60} = 16.67 \text{MB/s}
$$

> [!NOTE]
> 实际传输率还要考虑中断响应延迟、现场保护开销等因素，实际值会低于理论最大值。

## 32.8 多设备中断处理的详细方案

### 32.8.1 单级中断与多级中断

| 特性 | 单级中断 | 多级中断 |
|------|---------|---------|
| 中断嵌套 | **不允许** | **允许** |
| 优先级处理 | 按顺序处理 | 高优先级可打断低优先级 |
| 现场保护 | 只需一套 | 需要多套（嵌套深度） |
| 实现复杂度 | 低 | 高 |
| 实时性 | 一般 | **好** |

### 32.8.2 中断嵌套的实现

<div data-component="NestedInterruptHandler"></div>

│                │       │                                    │
│                ▼       ▼                                    │
│  服务B   ████████████████░░░░░░░░░░░░░░░░████████████████  │
│                       │                     ▲               │

```

**嵌套处理的关键步骤**：
1. 设备B中断主程序，开始服务B
2. 服务B过程中开中断（$INTE \leftarrow 1$）
3. 设备A请求中断（优先级高于B）
4. 保存B的现场，转去服务A
5. A服务完成，恢复B的现场
6. 继续服务B直至完成
7. 恢复主程序现场

### 32.8.3 中断屏蔽字的应用

**屏蔽字的作用**：
- 控制哪些中断可以打断当前服务
- 实现动态优先级调整

**【例 32-6】** 三个设备A、B、C，优先级A>B>C。

| 当前服务 | 屏蔽字 (A,B,C) | 说明 |
|---------|----------------|------|
| 无 | 0,0,0 | 所有中断都可以响应 |
| 服务C | 0,0,1 | C被屏蔽，A和B可以打断 |
| 服务B | 0,1,1 | B和C被屏蔽，只有A可以打断 |
| 服务A | 1,1,1 | 所有中断都被屏蔽 |

**屏蔽字的计算规则**：

$$
M_i = \begin{cases} 1 & \text{设备}i\text{的优先级} \leq \text{当前服务设备的优先级} \\ 0 & \text{设备}i\text{的优先级} > \text{当前服务设备的优先级} \end{cases}
$$

## 32.9 向量中断的硬件实现

### 32.9.1 向量中断的基本概念

<div data-component="VectorInterruptViz"></div>

**向量中断**：每个中断源对应一个唯一的中断向量号，CPU根据向量号直接找到中断服务程序的入口地址。

```
中断向量表（存储在主存中）：

地址      内容（中断服务程序入口地址）
┌────────┬──────────────────────────┐
│ 0x00   │ ISR_0 入口地址            │  ← 向量0
├────────┼──────────────────────────┤
│ 0x04   │ ISR_1 入口地址            │  ← 向量1
├────────┼──────────────────────────┤
│ 0x08   │ ISR_2 入口地址            │  ← 向量2
├────────┼──────────────────────────┤
│ 0x0C   │ ISR_3 入口地址            │  ← 向量3
├────────┼──────────────────────────┤
│ ...    │ ...                      │
├────────┼──────────────────────────┤
│ 0x7C   │ ISR_31 入口地址           │  ← 向量31
└────────┴──────────────────────────┘
```

### 32.9.2 向量中断的硬件实现

```
┌──────────────────────────────────────────────────────────────┐
│                    向量中断硬件结构                            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  设备中断请求                                                  │
│       │                                                      │
│       ▼                                                      │
│  ┌──────────┐     ┌──────────┐     ┌──────────────────┐     │
│  │ 中断请求  │────→│ 优先级   │────→│ 中断类型号寄存器  │     │
│  │ 寄存器   │     │ 判优器   │     │ (向量号)         │     │
│  └──────────┘     └──────────┘     └────────┬─────────┘     │
│                                             │               │
│                                             ▼               │
│  ┌──────────────────────────────────────────────────────┐   │
│  │                   数据总线                             │   │
│  └──────────────────────────┬───────────────────────────┘   │
│                             │                               │
│                             ▼                               │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  CPU                                                │   │
│  │  ┌─────────┐    ┌─────────┐    ┌─────────────────┐  │   │
│  │  │ 向量号  │───→│ 向量表  │───→│ 中断服务程序入口 │  │   │
│  │  │ × 4     │    │ 基地址  │    │ 地址            │  │   │
│  │  └─────────┘    └─────────┘    └─────────────────┘  │   │
│  └──────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

### 32.9.3 向量中断的处理过程

```
1. 设备发出中断请求
2. 中断控制器判优，确定最高优先级
3. CPU响应中断，发出INTA信号
4. 中断控制器将向量号放到数据总线
5. CPU接收向量号
6. 计算向量表地址：向量表基地址 + 向量号 × 4
7. 从向量表读取中断服务程序入口地址
8. PC ← 入口地址，开始执行中断服务程序
```

**向量地址计算**：

$$
\text{向量地址} = \text{向量表基地址} + \text{向量号} \times \text{每项大小}
$$

例如：向量表基地址为0x00000000，每个入口占4字节，向量号为5：

$$
\text{向量地址} = 0x00000000 + 5 \times 4 = 0x00000014
$$

### 32.9.4 向量中断与非向量中断

| 特性 | 向量中断 | 非向量中断 |
|------|---------|-----------|
| 入口地址获取 | 硬件自动（查表） | 软件查询 |
| 响应速度 | **快** | 慢 |
| 硬件复杂度 | 高 | 低 |
| 适用场景 | 高性能系统 | 简单系统 |

> [!NOTE]
> 现代计算机系统普遍采用向量中断方式。x86架构的中断描述符表（IDT）就是一种向量中断机制。

## 32.11 本章典型例题

### 32.3.1 有效数据传输率

$$
\text{有效数据传输率} = \frac{\text{数据量}}{\text{传输时间} + \text{中断开销}}
$$

### 32.3.2 中断开销

- 保护现场：约20个时钟周期
- 恢复现场：约20个时钟周期
- 中断返回：约5个时钟周期

### 32.3.3 中断方式的效率

## 32.4 多设备中断处理

### 32.4.1 中断优先级

多个设备同时请求中断时，按优先级处理。

### 32.4.2 中断嵌套

**【例 32-2】** 分析以下中断处理过程：

| 设备 | 优先级 | 处理时间 |
|------|--------|---------|
| 磁盘 | 高 | 50μs |

```

若磁盘在打印机服务过程中请求：

```
0μs:  打印机开始服务
30μs: 磁盘请求（优先级更高）
      保存打印机现场
      开始服务磁盘
80μs: 磁盘服务完成
      恢复打印机现场
      继续服务打印机
170μs: 打印机服务完成
```

## 32.9 深入理解中断驱动I/O

### 32.9.1 中断驱动I/O的工作流程详解

**CPU发送命令**：

CPU向I/O接口写入控制字：
- 设备地址
- 读/写命令
- 传输数据量
- 启动设备

然后CPU继续执行其他指令。

**设备准备数据**：

设备独立工作，CPU无需等待。

**中断请求与响应**：

设备完成后：
1. 设备置完成标志
2. 设备发出中断请求
3. CPU在当前指令执行完毕后检查中断
4. 如果条件满足，响应中断

**中断处理**：

```
1. 关中断
2. 保护断点（PC → 栈）
3. 保护现场（寄存器 → 栈）
4. 开中断
5. 执行中断服务程序（数据传输）
6. 关中断
7. 恢复现场
8. 开中断
9. 中断返回
```

### 32.9.2 中断的性能分析详解

**有效数据传输率**：

$$
\text{有效数据传输率} = \frac{\text{数据量}}{\text{传输时间} + \text{中断开销}}
$$

**中断开销**：

中断开销包括：
- 保护断点：约10个时钟周期
- 保护现场：约20个时钟周期
- 恢复现场：约20个时钟周期
- 中断返回：约5个时钟周期
- **总计**：约55个时钟周期

### 32.9.3 多设备中断处理详解

**中断优先级**：

多个设备同时请求中断时，按优先级处理。

**中断嵌套**：

在中断服务程序中开中断，允许更高优先级的中断。

**【例 32-3】** 分析以下中断处理过程：

| 设备 | 优先级 | 处理时间 |
|------|--------|---------|
| 磁盘 | 高 | 50μs |
| 打印机 | 中 | 100μs |
| 键盘 | 低 | 20μs |

若磁盘和打印机同时请求中断：

```
0μs:  磁盘和打印机同时请求
      判优：磁盘优先级高
      开始服务磁盘
50μs: 磁盘服务完成
      开始服务打印机
150μs: 打印机服务完成
```

### 32.9.4 中断方式的局限性

| 局限性 | 说明 |
|--------|------|
| 每次传输都需要CPU | 数据必须经过CPU |
| 高速设备频繁中断 | 中断开销累积 |
| 传输单位小 | 每次只能传输一个字 |

## 32.12 中断方式的高级话题

### 32.12.1 中断延迟分析

**中断延迟**：从中断请求发生到CPU开始执行中断服务程序的时间。

$$
\text{中断延迟} = \text{指令剩余执行时间} + \text{中断响应时间} + \text{现场保护时间}
$$

| 组成部分 | 典型值 | 说明 |
|---------|-------|------|
| 指令剩余执行时间 | 0~几个周期 | 取决于当前指令 |
| 中断响应时间 | 2~5个周期 | 检测中断、发INTA |
| 现场保护时间 | 10~30个周期 | 保存寄存器到栈 |
| **总延迟** | **12~38个周期** | |

> [!TIP]
> 对于实时系统，中断延迟的确定性比平均延迟更重要。最坏情况下的中断延迟必须满足实时性要求。

### 32.12.2 中断抖动

**中断抖动**：中断响应时间的变化范围。

$$
\text{中断抖动} = \text{最大中断延迟} - \text{最小中断延迟}
$$

影响中断抖动的因素：
- 当前指令的执行时间差异
- Cache命中/缺失
- 流水线状态
- 其他中断的嵌套

### 32.12.3 中断合并

当设备频繁产生中断时，可以采用中断合并技术减少中断次数：

```
传统方式（每次中断传输1字节）：
    设备 → 中断 → 1字节 → 中断 → 1字节 → 中断 → 1字节 → ...

中断合并方式（累积后一次性传输）：
    设备 → 等待 → 累积N字节 → 中断 → N字节 → 等待 → ...
```

**中断合并的优缺点**：

| 方面 | 优点 | 缺点 |
|------|------|------|
| CPU开销 | 减少中断次数 | 增加响应延迟 |
| 吞吐量 | 提高有效传输率 | 可能丢失实时性 |
| 适用场景 | 批量数据传输 | 实时性要求高的场景 |

### 32.12.4 中断亲和性

在多核处理器系统中，中断可以绑定到特定的CPU核心：

```
┌─────────────────────────────────────────┐
│              多核系统                     │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  │
│  │  CPU 0  │  │  CPU 1  │  │  CPU 2  │  │
│  └────┬────┘  └────┬────┘  └────┬────┘  │
│       │            │            │        │
│  ┌────┴────┐  ┌────┴────┐  ┌────┴────┐  │
│  │中断:0,1 │  │中断:2,3 │  │中断:4,5 │  │
│  └─────────┘  └─────────┘  └─────────┘  │
└─────────────────────────────────────────┘
```

**中断亲和性的优势**：
- 减少CPU间缓存同步开销
- 提高中断处理的局部性
- 负载均衡

### 32.12.5 中断与异常的对比

| 特性 | 中断（Interrupt） | 异常（Exception） |
|------|------------------|------------------|
| 来源 | **外部设备** | **CPU内部** |
| 同步性 | 异步 | 同步 |
| 可屏蔽 | 可以 | 不可以（大多数） |
| 示例 | 键盘输入、磁盘完成 | 除零、缺页、溢出 |
| 返回点 | 下一条指令 | 当前指令或下一条 |

## 32.14 中断方式的工程实践

### 32.14.1 中断服务程序的编写原则

| 原则 | 说明 |
|------|------|
| **尽量短** | 减少关中断时间，降低对其他中断的影响 |
| **不要阻塞** | 不要等待慢速设备，使用缓冲区 |
| **保护现场** | 保存所有可能被修改的寄存器 |
| **清除中断源** | 处理完毕后清除中断请求标志 |
| **避免嵌套过深** | 控制嵌套层数，防止栈溢出 |

### 32.14.2 中断调试技巧

```
调试中断问题的常用方法：

1. 逻辑分析仪
   - 捕获中断信号的时序
   - 分析中断请求和响应的时序关系

2. 中断计数器
   - 统计各设备的中断次数
   - 检测中断丢失或重复

3. 中断延迟测量
   - 在中断入口和出口打时间戳
   - 分析中断处理时间

4. 中断日志
   - 记录中断发生的顺序和时间
   - 分析中断嵌套情况
```

### 32.14.3 常见中断问题及解决

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 中断丢失 | 中断被屏蔽过久 | 缩短关中断时间 |
| 中断风暴 | 设备频繁中断 | 中断合并、降低频率 |
| 死锁 | 中断嵌套不当 | 仔细设计优先级和屏蔽字 |
| 栈溢出 | 中断嵌套太深 | 限制嵌套层数 |
| 数据竞争 | 中断与主程序共享数据 | 使用原子操作或关中断保护 |
