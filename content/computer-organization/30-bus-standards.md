## 30.2 总线通信控制

<div data-component="BusCommunicationTiming"></div>

### 30.2.1 同步通信

**统一时钟信号**，所有操作在一个时钟周期内完成。

```
时钟  ─┐ ┌─┐ ┌─┐ ┌─┐ ┌─
      ─┘ └─┘ └─┘ └─┘ └─
地址  ──── X ────────────
数据  ────────── X ──────
R/W   ──── X ────────────
```

**优点**：简单、快速

**缺点**：不能适应速度差异大的设备

### 30.2.2 异步通信

**不需要统一时钟**，通过握手信号协调。

**一、不互锁方式**

```
主设备：请求 ────────────
        ↑
从设备：应答 ──────
```

- 主设备发出请求后**不等待应答就撤除**
- 从设备发出应答后**不等待请求撤除就撤除**
- 速度最快，但不可靠

**二、半互锁方式**

```
主设备：请求 ────────────────────
        ↑                ↑
从设备：应答 ──────
```

- 主设备发出请求后**等待应答**，收到应答后撤除请求
- 从设备发出应答后**不等待请求撤除就撤除**
- 比不互锁可靠

**三、全互锁方式**

```
主设备：请求 ────────────────────────────
        ↑                ↑        ↑
从设备：应答 ──────────────────
```

- 主设备发出请求后**等待应答**，收到应答后撤除请求
- 从设备**等待请求撤除后**才撤除应答
- 最可靠，但速度最慢

### 30.2.3 三种异步通信方式对比

| 方式 | 主设备 | 从设备 | 速度 | 可靠性 |
|------|--------|--------|------|--------|
| 不互锁 | 发请求后不等应答 | 发应答后不等请求撤除 | **最快** | 差 |
| 半互锁 | 等应答后撤请求 | 发应答后不等请求撤除 | 中 | 中 |
| 全互锁 | 等应答后撤请求 | 等请求撤除后撤应答 | 慢 | **最好** |

### 30.2.4 半同步通信

**同步+异步混合**，在同步通信基础上增加READY/WAIT信号。

```
时钟  ─┐ ┌─┐ ┌─┐ ┌─┐ ┌─
      ─┘ └─┘ └─┘ └─┘ └─
READY ────────────── ────
WAIT  ────── ────────────
```

- 正常情况：同步通信（READY=1）
- 设备未准备好：插入WAIT周期

### 30.2.5 分离式通信

将一次总线事务分成**两个独立的子事务**：

```
阶段1：主设备发送地址和命令 → 释放总线
        ↓（设备准备数据）
阶段2：设备主动发送数据 → 释放总线
```

**优点**：总线利用率最高

**缺点**：实现复杂

## 30.4 本章典型例题

### 例题1：异步通信方式判断

**题目**：某总线通信中，主设备发出请求后立即撤除，从设备收到请求后发出应答，然后立即撤除。这是哪种异步通信方式？

**解答**：

- 主设备：发请求后不等应答就撤除 → 不互锁
- 从设备：发应答后不等请求撤除就撤除 → 不互锁

答案：**不互锁方式**

### 例题2：总线带宽计算

**题目**：PCIe Gen3 x16的带宽是多少？

**解答**：

- Gen3每通道速率：8 GT/s
- 每通道有效带宽：8 GT/s × 8/10（8b/10b编码）= 6.4 Gbps
- x16带宽：6.4 Gbps × 16 = 102.4 Gbps = **12.8 GB/s**

（实际单向带宽约15.75 GB/s，考虑编码效率）

## 30.10 PCI总线的详细协议与信号定义

### 30.10.1 PCI总线信号组

PCI总线定义了丰富的信号线，按功能分组如下：

| 信号组 | 信号名 | 方向 | 功能 |
|--------|--------|------|------|
| 系统信号 | CLK | IN | 33/66MHz时钟 |
| | RST# | IN | 复位信号 |
| 地址/数据 | AD[31:0] | T/S | 地址/数据复用 |
| | C/BE#[3:0] | T/S | 命令/字节使能 |
| 接口控制 | FRAME# | S/T/S | 帧信号（传输进行中） |
| | IRDY# | S/T/S | 主设备就绪 |
| | TRDY# | S/T/S | 从设备就绪 |
| | STOP# | S/T/S | 从设备请求停止 |
| | IDSEL | IN | 初始化设备选择 |
| | DEVSEL# | S/T/S | 设备选择 |
| 仲裁 | REQ# | T/S | 总线请求 |
| | GNT# | T/S | 总线授权 |
| 错误报告 | PERR# | S/T/S | 奇偶校验错误 |
| | SERR# | O/D | 系统错误 |
| 中断 | INTA#~INTD# | O/D | 中断请求 |

> [!NOTE]
> 信号名后缀 `#` 表示低电平有效，`T/S` 表示三态，`S/T/S` 表示持续三态，`O/D` 表示开漏极。

### 30.10.2 PCI命令类型

C/BE#[3:0]在地址阶段指示命令类型：

| C/BE#[3:0] | 命令 | 说明 |
|------------|------|------|
| 0000 | Interrupt Acknowledge | 中断应答 |
| 0001 | Special Cycle | 特殊周期 |
| 0010 | I/O Read | I/O读 |
| 0011 | I/O Write | I/O写 |
| 0110 | Memory Read | 存储器读 |
| 0111 | Memory Write | 存储器写 |
| 1010 | Configuration Read | 配置读 |
| 1011 | Configuration Write | 配置写 |
| 1100 | Memory Read Multiple | 存储器多行读 |
| 1110 | Memory Read Line | 存储器行读 |
| 1111 | Memory Write and Invalidate | 存储器写并无效 |

### 30.10.3 PCI读操作时序

```
时钟   ─┐┌─┐┌─┐┌─┐┌─┐┌─┐┌─┐┌─┐┌─┐┌─
       ─┘└─┘└─┘└─┘└─┘└─┘└─┘└─┘└─┘└─
       │C1│C2│C3│C4│C5│C6│C7│
FRAME# ──┐            ┌─────────────
       ──┘            └─────────────
AD     ──X══ADDR══X═══X═══X══DATA══X──
C/BE#  ──X══CMD═══X═══X═══X══BE#═══X──
IRDY#  ──────────────┐         ┌─────
       ──────────────┘         └─────
TRDY#  ──────────────────┐     ┌─────
       ──────────────────┘     └─────
DEVSEL#──┐                       ┌───
       ──┘                       └───
```

**PCI传输阶段说明**：

| 阶段 | 周期 | 操作 |
|------|------|------|
| 地址阶段 | C1 | 主设备驱动AD和C/BE#，FRAME#有效 |
| 等待阶段 | C2~C3 | 等待从设备译码和DEVSEL#响应 |
| 数据阶段 | C4~C7 | IRDY#和TRDY#同时有效时传输数据 |

### 30.10.4 PCI突发传输

PCI支持突发传输，一次地址阶段后可连续传输多个数据：

```
时钟   ─┐┌─┐┌─┐┌─┐┌─┐┌─┐┌─┐┌─
       ─┘└─┘└─┘└─┘└─┘└─┘└─┘└─
       │A│D1│D2│D3│D4│
FRAME# ─┐              ┌──────
       ─┘              └──────
AD     ─X═ADDR═X═D1═X═D2═X═D3═X
IRDY#  ─────────┐         ┌───
       ─────────┘         └───
TRDY#  ─┐          ┌─┐  ┌─┐
       ─┘          └─┘  └─┘
```

**突发传输的优势**：

| 项目 | 单次传输（4次） | 突发传输（4数据） |
|------|---------------|-----------------|
| 地址阶段 | 4次 | 1次 |
| 仲裁次数 | 4次 | 1次 |
| 总线周期 | 4×5=20 | 1+4+1=6 |
| 效率 | 20% | 67% |

## 30.11 PCIe的层次结构

### 30.11.1 PCIe协议栈

<div data-component="PCIeLayerViz"></div>

PCIe采用分层协议架构：

```
┌──────────────────────────────────────┐
│           软件/驱动层                  │
├──────────────────────────────────────┤
│          事务层（Transaction Layer）    │
│  ┌────────────────────────────────┐  │
│  │  TLP（事务层包）的封装与解析     │  │
│  │  读/写请求、完成包              │  │
│  └────────────────────────────────┘  │
├──────────────────────────────────────┤
│          数据链路层（Data Link Layer） │
│  ┌────────────────────────────────┐  │
│  │  DLLP（数据链路层包）           │  │
│  │  CRC校验、ACK/NAK、流控        │  │
│  └────────────────────────────────┘  │
├──────────────────────────────────────┤
│          物理层（Physical Layer）      │
│  ┌────────────────────────────────┐  │
│  │  链路训练、符号传输              │  │
│  │  8b/10b或128b/130b编码         │  │
│  │  差分信号传输                   │  │
│  └────────────────────────────────┘  │
└──────────────────────────────────────┘
```

### 30.11.2 事务层（Transaction Layer）

**TLP包格式**：

```
┌────────┬────────┬────────┬────────┐
│  Byte0 │  Byte1 │  Byte2 │  Byte3 │
├────────┼────────┼────────┼────────┤
│ Fmt[1:0]│Type[4:0]│TC[2:0] │Attr[1:0]│TH│TD│EP│Attr│Length[9:0]│
├────────┴────────┴────────┴────────┤
│        Requester ID (16位)         │
│        Tag (8位)                   │
│        Last DW BE / First DW BE   │
├────────────────────────────────────┤
│        Address [63:2]              │
│        (或 Completion ID)          │
├────────────────────────────────────┤
│        Data Payload                │
│        (可变长度)                   │
├────────────────────────────────────┤
│        ECRC (32位，可选)            │
└────────────────────────────────────┘
```

**TLP类型**：

| Fmt | Type | TLP类型 | 说明 |
|-----|------|---------|------|
| 00 | 00000 | MRd | 存储器读请求 |
| 01 | 00000 | MRdLk | 存储器读锁定 |
| 00/01 | 00001 | MWr | 存储器写 |
| 00 | 00010 | IORd | I/O读 |
| 00 | 00011 | IOWr | I/O写 |
| 10 | 01010 | Cpl | 完成包 |
| 10 | 01011 | CplD | 带数据的完成包 |

### 30.11.3 数据链路层（Data Link Layer）

**DLLP包格式**：

```
┌────────┬────────┬────────┬────────┐
│  DLLP Type  │  序列号  │  CRC   │
│  (8位)      │ (12位)  │ (16位) │
└────────┴────────┴────────┴────────┘
```

**DLLP功能**：

| 功能 | 说明 |
|------|------|
| ACK | 确认接收到正确的TLP |
| NAK | 报告接收到错误的TLP |
| 流控 | 缓冲区信用管理 |
| 电源管理 | 链路电源状态管理 |

**序列号与重传机制**：

```
发送端                          接收端
  │                              │
  │  TLP (Seq=N) ──────────────→│
  │                              │  CRC校验
  │  ←────────────── ACK (N)    │  正确：发送ACK
  │                              │
  │  TLP (Seq=N+1) ────────────→│
  │                              │  CRC校验
  │  ←────────────── NAK (N+1)  │  错误：发送NAK
  │                              │
  │  TLP (Seq=N+1) ────────────→│  重传
```

### 30.11.4 物理层（Physical Layer）

**PCIe物理层特性**：

| 特性 | Gen1 | Gen2 | Gen3 |
|------|------|------|------|
| 速率 | 2.5 GT/s | 5 GT/s | 8 GT/s |
| 编码 | 8b/10b | 8b/10b | 128b/130b |
| 效率 | 80% | 80% | 98.5% |
| 信号 | 差分 | 差分 | 差分 |

**链路训练过程**：

```
Detect → Polling → Configuration → L0（正常工作）
   ↑                                    │
   └────────── Recovery ←──────────────┘
```

| 状态 | 说明 |
|------|------|
| Detect | 检测对端设备是否存在 |
| Polling | 协商链路参数 |
| Configuration | 配置链路宽度 |
| L0 | 正常工作状态 |
| Recovery | 链路错误恢复 |

> [!TIP]
> PCIe使用差分信号传输，每条通道由一对发送线（Tx+/Tx-）和一对接收线（Rx+/Rx-）组成。差分信号抗干扰能力强，适合高速传输。

```
┌──────┬──────┬──────┬──────┐

```

**USB数据包（Data Packet）**：

```
┌──────┬──────┬──────────────┬───────┐
│ SYNC │ PID  │   DATA       │ CRC16 │
│ 8位  │ 8位  │ 0~1023字节   │ 16位  │
```

**PID类型**：

| PID[3:0] | 类型 | 说明 |
|----------|------|------|
| 0001 | OUT | 主机→设备 |
| 1001 | IN | 设备→主机 |
| 0101 | SOF | 帧开始 |
| 1101 | SETUP | 建立事务 |
| 0011 | DATA0 | 数据包0 |
| 1011 | DATA1 | 数据包1 |
| 0010 | ACK | 确认 |
| 1010 | NAK | 不确认 |
| 1110 | STALL | 停止 |

### 30.12.4 USB事务流程

**USB控制传输事务序列**：

```
阶段1：建立事务（Setup）
  主机 → SETUP令牌包 → 设备
  主机 → DATA0包（建立数据）→ 设备
  设备 → ACK → 主机

阶段2：数据事务（可选）
  主机 → IN/OUT令牌包 → 设备
  设备/主机 → DATA1包 → 主机/设备
  接收方 → ACK → 发送方

阶段3：状态事务
  主机 → IN令牌包 → 设备
  设备 → DATA1（零长度）→ 主机
  主机 → ACK → 设备
```

```
│  (SATA PHY) │                 │  (PCIe PHY) │
```

### 30.13.2 性能对比

| 参数 | SATA III | NVMe |
|------|----------|------|
| 接口带宽 | 6 Gbps | 32+ Gbps (PCIe Gen4 x4) |
| 协议开销 | 较高（AHCI） | 较低 |
| 队列深度 | 1个命令队列，32条命令 | 65535个队列，每队列65536条命令 |
| 延迟 | 较高 | 较低 |
| CPU占用 | 较高 | 较低 |
| 最大IOPS | ~100K | ~1000K+ |

### 30.13.3 命令处理对比

**SATA (AHCI)命令处理**：

```
CPU → 内存（命令表）→ AHCI控制器 → SATA设备
        │                    │
        └─ 单命令队列 ────────┘
           深度32
```

**NVMe命令处理**：

```
CPU → 内存（提交队列）→ NVMe控制器 → SSD
   ←─── 内存（完成队列）←──────────┘
        │                    │
        └─ 多队列（每核一个）──┘
           每队列深度65536
```

> [!NOTE]
> NVMe支持多队列设计，每个CPU核心可以有独立的命令队列，避免了队列锁竞争，充分发挥了多核CPU和SSD内部并行性。

### 30.13.4 适用场景

| 场景 | 推荐接口 | 原因 |
|------|---------|------|
| 传统HDD | SATA | HDD速度远低于SATA带宽上限 |
| 普通SSD | SATA | 成本低，兼容性好 |
| 高性能SSD | NVMe | 充分发挥SSD性能 |
| 数据中心 | NVMe | 低延迟、高IOPS |
| 嵌入式/旧系统 | SATA | 兼容性、成熟度 |

## 30.14 四种通信方式的详细时序

### 30.14.1 同步通信详细时序

**同步读操作时序（带等待状态）**：

```
时钟    ─┐ ┌─┐ ┌─┐ ┌─┐ ┌─┐ ┌─┐ ┌─┐ ┌─
        ─┘ └─┘ └─┘ └─┘ └─┘ └─┘ └─┘ └─
        │T1│T2│T3│Tw│Tw│T4│
地址    ──X══════════════════════════════
CS      ──┐                         ┌───
        ──┘                         └───
R/W     ──┐ (读)
        ──┘
READY   ──────────────────┐         ┌───
        ──────────────────┘         └───
数据    ────────────────────────X════════
                             ↑
                          数据有效
```

**同步写操作时序**：

```
时钟    ─┐ ┌─┐ ┌─┐ ┌─┐ ┌─┐ ┌─
        ─┘ └─┘ └─┘ └─┘ └─┘ └─
        │T1│T2│T3│T4│
地址    ──X════════════════════
CS      ──┐                 ┌──
        ──┘                 └──
R/W     ──┐ (写)
        ──┘
数据    ──X════════════════════
```

### 30.14.2 异步通信详细时序

**不互锁方式详细时序**：

```
主设备 REQ ──┐                           ┌──
            └───────────────────────────┘
                 t_req（请求脉冲宽度）
从设备 ACK ──────┐                 ┌──────
                 └─────────────────┘
                    t_ack（应答脉冲宽度）

时序分析：
  主设备发出REQ后，经过 t1 后撤除（不等ACK）
  从设备检测到REQ，经过 t2 后发出ACK
  从设备发出ACK后，经过 t3 后撤除（不等REQ撤除）
  
  总时间 = max(t1, t2 + t3)
  风险：如果ACK宽度 < 主设备检测窗口，可能丢失应答
```

**半互锁方式详细时序**：

```
主设备 REQ ──┐                               ┌──
            └───────────────────────────────┘
                 ↑                       ↑
                 │ REQ发出               │ 收到ACK后撤除
从设备 ACK ────────┐               ┌──────────
                   └───────────────┘
                      ↑           ↑
                      │ 检测到REQ  │ 发ACK后即撤除
                      │ 发出ACK   │（不等REQ撤除）

时序分析：
  主设备发出REQ，等待ACK
  从设备检测到REQ，发出ACK
  主设备检测到ACK，撤除REQ
  从设备发出ACK后立即撤除
  
  总时间 = t_req_to_ack + t_ack_width + t_ack_remove
```

**全互锁方式详细时序**：

```
主设备 REQ ──┐                                   ┌──
            └───────────────────────────────────┘
                 ↑                           ↑
                 │ REQ发出                   │ 收到ACK后撤除
从设备 ACK ────────┐                   ┌──────────
                   └───────────────────┘
                      ↑               ↑
                      │ 检测到REQ      │ 检测到REQ撤除后
                      │ 发出ACK       │ 才撤除ACK

时序分析：
  主设备发出REQ，等待ACK
  从设备检测到REQ，发出ACK
  主设备检测到ACK，撤除REQ
  从设备检测到REQ撤除，撤除ACK
  
  总时间 = t1 + t2 + t3 + t4
  保证：每个信号都被对方确认
```

### 30.14.3 半同步通信详细时序

**正常情况（设备就绪）**：

```
时钟    ─┐ ┌─┐ ┌─┐ ┌─┐ ┌─┐ ┌─
        ─┘ └─┘ └─┘ └─┘ └─┘ └─
READY   ────────────────────────（高电平）
        │T1│T2│T3│T4│
地址    ──X════════════════════
数据    ────────────X══════════
```

**慢速设备情况（插入等待）**：

```
时钟    ─┐ ┌─┐ ┌─┐ ┌─┐ ┌─┐ ┌─┐ ┌─┐ ┌─
        ─┘ └─┘ └─┘ └─┘ └─┘ └─┘ └─┘ └─
READY   ──────────────┐             ┌────
        ──────────────┘             └────
WAIT    ──────┐                 ┌────────
        ──────┘                 └────────
        │T1│T2│  Tw  │  Tw  │T3│T4│
地址    ──X══════════════════════════════
数据    ─────────────────────────X════════
```

> [!TIP]
> 半同步通信结合了同步和异步的优点：正常情况下按同步方式高速传输，设备未准备好时通过WAIT信号插入等待周期，兼顾速度和可靠性。

### 30.14.4 分离式通信详细时序

```
时间轴：
├──── 阶段1 ────┤── 空闲 ──┤──── 阶段2 ────┤
│               │          │               │
│ 主设备发送    │ 总线     │ 从设备发送    │
│ 地址和命令    │ 释放     │ 数据          │
│               │          │               │

时钟    ─┐┌─┐┌─┐┌─┐┌─┘ ─┐┌─┐┌─┐┌─┐┌─
        ─┘└─┘└─┘└─┘└─┘   ─┘└─┘└─┘└─┘└─┘
        │阶段1│空│阶段2│
主设备   │发地址│  │     │
从设备   │     │  │发数据│
总线     │主占用│  │从占用│
```

**分离式通信的工作流程**：

| 步骤 | 执行者 | 操作 | 总线状态 |
|------|--------|------|---------|
| 1 | 主设备 | 发送地址、命令 | 主设备占用 |
| 2 | 主设备 | 释放总线 | 空闲 |
| 3 | 从设备 | 准备数据 | 空闲 |
| 4 | 从设备 | 请求总线，发送数据 | 从设备占用 |
| 5 | 从设备 | 释放总线 | 空闲 |

**分离式通信的优点**：

| 优点 | 说明 |
|------|------|
| 总线利用率高 | 空闲期间其他设备可使用总线 |
| 无等待周期 | 不需要WAIT/READY信号 |
| 适合高速系统 | CPU不被慢速设备阻塞 |

## 30.17 扩展例题

### 例题3：PCI传输分析

**题目**：PCI总线宽度32位，频率33MHz，进行一次突发传输读取4个32位数据。地址阶段1个周期，等待3个周期，每个数据阶段1个周期，间隔1个周期。求传输时间和带宽。

**解答**：

$$
T_{总} = 1 + 3 + 4 \times 1 + 1 = 9 \text{个时钟周期}
$$

$$
T_{时间} = 9 \times \frac{1}{33\text{MHz}} = 272.7\text{ns}
$$

$$
\text{传输数据量} = 4 \times 4 = 16\text{字节}
$$

$$
BW = \frac{16\text{B}}{272.7\text{ns}} = 58.7\text{MB/s}
$$

### 例题4：PCIe编码效率

**题目**：比较PCIe Gen2和Gen3的有效带宽利用率。

**解答**：

| 参数 | Gen2 | Gen3 |
|------|------|------|
| 原始速率 | 5 GT/s | 8 GT/s |
| 编码方式 | 8b/10b | 128b/130b |
| 编码效率 | 80% | 98.5% |
| 有效速率 | 4 GT/s | 7.88 GT/s |

虽然Gen3原始速率只提升60%，但由于编码效率提升，有效速率提升接近一倍。

### 例题5：USB传输时间计算

**题目**：USB 2.0批量传输，数据包最大512字节。传输10KB数据需要几个事务？每个事务包括令牌包（3字节）、数据包（512字节+CRC）和握手包（1字节），USB 2.0速率480Mbps。估算传输时间。

**解答**：

$$
\text{事务数} = \lceil 10240 / 512 \rceil = 20 \text{个事务}
$$

每个事务的数据量：

$$
3 + (512 + 2) + 1 = 518\text{字节}
$$

总数据量：

$$
20 \times 518 = 10360\text{字节}
$$

传输时间（不含协议延迟）：

$$
T = \frac{10360 \times 8}{480 \times 10^6} = 172.7\mu\text{s}
$$

实际时间需加上协议处理延迟（约数百微秒到毫秒级）。

### 例题6：NVMe vs SATA性能分析

**题目**：某SSD随机4KB读取，队列深度1。SATA接口延迟100μs，NVMe接口延迟20μs。分别计算IOPS。

**解答**：

$$
\text{IOPS}_{SATA} = \frac{1}{100\mu\text{s}} = 10000
$$

$$
\text{IOPS}_{NVMe} = \frac{1}{20\mu\text{s}} = 50000
$$

NVMe的IOPS是SATA的5倍。如果考虑多队列并发，差距更大。
