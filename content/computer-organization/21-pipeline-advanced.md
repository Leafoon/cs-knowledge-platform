# 第二十一章 流水线的高级技术

本章讲解超越基本流水线的高级技术：超标量、动态调度、推测执行和多线程。这些技术是现代处理器的核心。

---

## 21.1 超标量流水线

### 21.1.1 概念

<div data-component="SuperscalarViz"></div>

**超标量**（Superscalar）：每个时钟周期**发射多条指令**（通常2~8条）。

```
周期  | 发射的指令
1     | I1, I2
2     | I3, I4
3     | I5, I6
```

### 21.1.2 超标量的实现条件

1. **多个功能部件**：ALU、Load/Store、分支等
2. **多端口寄存器文件**：同时读写多个寄存器
3. **多端口Cache**：同时取多条指令
4. **冒险检测逻辑**：检测多条指令之间的依赖

### 21.1.3 超标量 vs 超流水线

| 特性 | 超标量 | 超流水线 |
|------|--------|---------|
| 并行方式 | 多条指令同时执行 | 每周期多个阶段 |
| 硬件需求 | 多个功能部件 | 更细的流水线 |
| 典型 | 每周期2~8条 | 每周期1条但阶段更多 |
| 时钟频率 | 中等 | 更高 |

### 21.1.4 发射策略

| 策略 | 说明 | 复杂度 |
|------|------|--------|
| 顺序发射顺序完成 | 最简单 | 低 |
| 顺序发射乱序完成 | 允许乱序完成 | 中 |
| 乱序发射乱序完成 | 最灵活 | 高 |

## 21.2 动态调度

### 21.2.1 静态调度 vs 动态调度

| 类型 | 方法 | 优点 | 缺点 |
|------|------|------|------|
| 静态调度 | 编译器重排指令 | 硬件简单 | 无法预测运行时行为 |
| 动态调度 | 硬件在运行时调度 | 适应性强 | 硬件复杂 |

### 21.2.2 记分牌（Scoreboard）算法

**核心思想**：硬件跟踪指令状态和功能部件状态，动态决定指令何时可以执行。

**四个阶段**：

| 阶段 | 功能 | 条件 |
|------|------|------|
| 发射 | 指令进入记分牌 | 功能部件空闲 |
| 读操作数 | 等待数据就绪 | 源操作数可用 |
| 执行 | 在功能部件上执行 | 操作数已读取 |
| 写结果 | 写回结果 | 无WAR冲突 |

**数据结构**：
- **指令状态表**：每条指令当前处于哪个阶段
- **功能部件状态表**：每个功能部件的状态
- **寄存器结果状态表**：哪个功能部件将写入哪个寄存器

### 21.2.3 记分牌的优缺点

| 优点 | 缺点 |
|------|------|
| 允许乱序执行 | 不能消除WAR和WAW |
| 提高功能部件利用率 | 硬件复杂 |
| | 不支持寄存器重命名 |

## 21.3 Tomasulo算法

### 21.3.1 核心思想

<div data-component="TomasuloAlgorithm"></div>

**保留站**（Reservation Station）+ **寄存器重命名**

- 保留站：为每条指令分配一个保留站，存放操作数或操作数来源
- 寄存器重命名：消除WAR和WAW冒险
- CDB（公共数据总线）：广播运算结果

### 21.3.2 三个阶段

**一、发射（Issue）**

1. 从指令队列取指令
2. 检查是否有空闲的保留站
3. 如果有：分配保留站，读取操作数或记录来源
4. 如果没有：暂停发射

**二、执行（Execute）**

1. 等待所有操作数就绪
2. 在功能部件上执行
3. 监听CDB获取等待的操作数

**三、写结果（Write Result）**

1. 通过CDB广播结果
2. 所有等待该结果的保留站和寄存器同时更新

### 21.3.3 Tomasulo的硬件结构

```
指令队列 → 保留站 → 功能部件 → CDB
              ↑                    ↓
         寄存器文件 ←←←←←←←←←←←←←←
```

### 21.3.4 Tomasulo vs 记分牌

| 特性 | 记分牌 | Tomasulo |
|------|--------|----------|
| WAR冒险 | 不能消除 | **能消除** |
| WAW冒险 | 不能消除 | **能消除** |
| 寄存器重命名 | 无 | **有** |
| 结果转发 | 无 | **CDB广播** |
| 硬件复杂度 | 中 | 高 |

## 21.4 推测执行

### 21.4.1 概念

在分支结果确定之前，**推测性地执行**分支目标指令。

### 21.4.2 ROB（重排序缓冲区）

<div data-component="ReorderBufferDemo"></div>

**ROB**（Reorder Buffer）保证指令按**程序顺序完成**。

```
发射 → 执行 → 写入ROB → 按序提交
```

- 推测执行的结果暂存在ROB中
- 分支结果确定后：
  - 正确 → 从ROB提交到寄存器文件
  - 错误 → 撤销ROB中的结果

### 21.4.3 推测执行的流程

```
1. 指令发射到保留站
2. 推测执行（即使分支结果未定）
3. 结果写入ROB
4. 分支结果确定：
   - 预测正确 → ROB提交结果
   - 预测错误 → 清空ROB，恢复状态
```

## 21.5 多线程技术

### 21.5.1 为什么需要多线程

单线程处理器中，Cache缺失、分支误预测等导致流水线停顿，功能部件空闲。

### 21.5.2 多线程的类型

| 类型 | 切换粒度 | 开销 | 并行度 |
|------|---------|------|--------|
| 细粒度多线程 | 每周期切换 | 低 | 中 |
| 粗粒度多线程 | 长延迟事件切换 | 中 | 低 |
| 同时多线程(SMT) | 每周期多线程同时执行 | 高 | **高** |

### 21.5.3 同时多线程（SMT）

<div data-component="SMTVisualizer"></div>

**SMT**（Simultaneous Multithreading）：一个物理核**同时执行多个线程**的指令。

- Intel超线程（Hyper-Threading）是SMT的一种实现
- 一个物理核呈现为两个逻辑核
- 共享功能部件，利用空闲资源

## 21.11 超标量处理器的发射策略（详细）

<div data-component="OutOfOrderExecution"></div>

### 21.11.1 顺序发射顺序完成（In-order Issue, In-order Completion）

最简单的策略，严格按照程序顺序发射和完成指令。

**时序图示例**（假设两条指令，I1为长延迟操作如乘法，I2为短延迟操作如加法）：

```
        | C1  | C2  | C3  | C4  | C5  | C6  | C7  |
--------|-----|-----|-----|-----|-----|-----|-----|
I1 MUL  | IF  | ID  | EX1 | EX2 | EX3 | WB  |     |
I2 ADD  |     |     |     |     |     | IF  | ID  |
                                        ↑
                              I2必须等待I1完成后才能发射
```

**关键特点**：
- I2必须等I1完成所有阶段后才能进入流水线
- 即使I2与I1无数据依赖，也被阻塞
- 硬件最简单，但性能最低

> [!NOTE]
> 顺序发射顺序完成的问题在于：后面的短指令被前面的长指令阻塞，即使两者之间没有数据依赖关系。这导致功能部件利用率很低。

### 21.11.2 顺序发射乱序完成（In-order Issue, Out-of-order Completion）

允许指令在完成执行后立即写回结果，不必等待前面的指令完成。

**时序图示例**：

```
        | C1  | C2  | C3  | C4  | C5  | C6  | C7  |
--------|-----|-----|-----|-----|-----|-----|-----|
I1 MUL  | IF  | ID  | EX1 | EX2 | EX3 | WB  |     |
I2 ADD  |     | IF  | ID  | EX  | WB  |     |     |
                            ↑         ↑
                        I2开始执行   I2先于I1完成
```

**关键特点**：
- 按程序顺序发射（IF → ID按序）
- 但I2可以在I1之前完成并写回结果
- 需要解决WAR和WAW冒险

**WAR冒险示例**：

```
I1: MUL R1, R2, R3    ; 读R2, R3，写R1（长延迟）
I2: ADD R4, R1, R5    ; 读R1, R5，写R4（短延迟）
I3: SUB R2, R6, R7    ; 写R2（短延迟）

如果I3先于I1完成写回R2，则I1可能读到错误的R2值（WAR冒险）
```

> [!TIP]
> 顺序发射乱序完成需要额外的硬件来检测和解决WAR冒险。一种方法是推迟写回（直到没有WAR风险），另一种方法是使用寄存器重命名。

### 21.11.3 乱序发射乱序完成（Out-of-order Issue, Out-of-order Completion）

最灵活的策略，指令可以在满足依赖关系后立即发射和执行。

**时序图示例**：

```
        | C1  | C2  | C3  | C4  | C5  | C6  | C7  | C8  |
--------|-----|-----|-----|-----|-----|-----|-----|-----|
I1 MUL  | IF  | ID  | EX1 | EX2 | EX3 | WB  |     |     |
I2 ADD  |     | IF  | ID  | EX  | WB  |     |     |     |
I3 LD   |     |     | IF  | ID  | EX  | MEM | WB  |     |
I4 ADD  |     |     |     | IF  | ID  | EX  | WB  |     |
                                ↑
                    I4可以在I1完成前发射（若无依赖）
```

**关键特点**：
- 指令不必按程序顺序发射
- 保留站 + 寄存器重命名消除WAR/WAW
- 需要ROB保证精确异常
- 硬件最复杂，性能最高

### 21.11.4 发射策略性能对比

| 策略 | CPI理想值 | 硬件复杂度 | WAR处理 | WAW处理 | 精确异常 |
|------|----------|-----------|---------|---------|---------|
| 顺序发射顺序完成 | 高 | 低 | 不需要 | 不需要 | 天然支持 |
| 顺序发射乱序完成 | 中 | 中 | 推迟写回 | 推迟写回 | 困难 |
| 乱序发射乱序完成 | **低** | **高** | 寄存器重命名 | 寄存器重命名 | 需要ROB |

### 21.11.5 发射宽度对性能的影响

**发射宽度**（Issue Width）：每周期最多发射的指令数。

设发射宽度为 $w$，理想CPI为：

$$CPI_{ideal} = \frac{1}{w}$$

但实际性能受限于：
1. **指令级并行度（ILP）**：程序中可并行的指令数有限
2. **数据依赖**：RAW依赖限制了并行发射
3. **资源冲突**：功能部件数量有限
4. **分支预测准确率**：误预测导致流水线排空

**Amdahl定律的应用**：

$$Speedup = \frac{1}{(1 - f) + \frac{f}{w}}$$

其中 $f$ 为可并行化的指令比例。

| 发射宽度 | 理论加速比（f=0.8） | 实际加速比 |
|---------|-------------------|-----------|
| 1 | 1.0 | 1.0 |
| 2 | 1.67 | 1.3~1.5 |
| 4 | 2.50 | 1.5~2.0 |
| 8 | 3.08 | 1.8~2.5 |

> [!NOTE]
> 发射宽度增加到一定程度后，性能收益递减。这就是为什么现代处理器通常采用4~8发射宽度，而不是更大的发射宽度。功耗和复杂度的增长远超性能收益。

## 21.12 Tomasulo算法的硬件实现（详细）

### 21.12.1 保留站字段详解

每个保留站条目包含以下字段：

| 字段 | 位宽 | 说明 |
|------|------|------|
| Op | 4~6位 | 操作类型（ADD, SUB, MUL, DIV等） |
| Vj | 32/64位 | 源操作数1的值（已就绪时） |
| Vk | 32/64位 | 源操作数2的值（已就绪时） |
| Qj | 3~5位 | 产生源操作数1的保留站编号（未就绪时） |
| Qk | 3~5位 | 产生源操作数2的保留站编号（未就绪时） |
| Busy | 1位 | 该保留站是否被占用 |
| A | 32/64位 | 地址（Load/Store指令使用） |

**操作数就绪判断**：
- 若 Qj = 0 且 Vj 有值 → 源操作数1已就绪
- 若 Qj ≠ 0 → 等待Qj指向的保留站产生结果

### 21.12.2 CDB协议与时序

**CDB**（Common Data Bus，公共数据总线）是Tomasulo算法的核心数据通路。

```
                    ┌──────────────────────────────────┐
                    │           CDB（公共数据总线）       │
                    │                                    │
    ┌──────────┐   │  ┌─────┐  ┌─────┐  ┌─────┐      │
    │ 保留站1  │←──│──│ Tag │  │Data │  │Valid│      │
    ├──────────┤   │  └─────┘  └─────┘  └─────┘      │
    │ 保留站2  │←──│                                    │
    ├──────────┤   └──────────────────────────────────┘
    │ 保留站3  │←──           ↑
    ├──────────┤              │
    │ 寄存器文件│←──    ┌──────┴──────┐
    └──────────┘       │  功能部件    │
                       └─────────────┘
```

**CDB传输格式**：
- **Tag**（标签）：产生结果的保留站编号
- **Data**（数据）：运算结果
- **Valid**（有效位）：数据是否有效

**CDB时序**（单周期广播）：

```
时钟周期  | 功能部件状态        | CDB内容         | 保留站更新
----------|--------------------|-----------------|----------------
C5        | EX完成，准备写回    | Tag=RS1,Data=42 | 所有监听RS1的
          |                    |                 | 保留站更新Vj/Vk
C6        | 下一条指令开始EX    | （空闲）        | 寄存器文件更新
```

> [!TIP]
> CDB是共享总线，同一时钟周期只能传输一个结果。如果有多个功能部件同时完成，需要仲裁机制决定谁先使用CDB。这是Tomasulo算法的一个潜在瓶颈。

### 21.12.3 寄存器重命名详解

**寄存器重命名**的本质：将体系结构寄存器（如R1）映射到物理寄存器或保留站，消除假依赖。

**示例**：

```
指令序列：
I1: MUL R1, R2, R3    ; 写R1
I2: ADD R4, R1, R5    ; 读R1（RAW，真依赖）
I3: SUB R1, R6, R7    ; 写R1（WAW，假依赖）
I4: ADD R8, R1, R9    ; 读R1（WAR，假依赖）
```

**无重命名时**：
- I3必须等I2读完R1后才能写（WAR）
- I3必须等I1写完R1后才能写（WAW）
- 严重的流水线停顿

**有重命名时**（保留站充当重命名）：

| 指令 | 分配的保留站 | 重命名映射 |
|------|------------|-----------|
| I1: MUL R1, R2, R3 | RS1 | R1 → RS1 |
| I2: ADD R4, R1, R5 | RS2 | R1 → RS1（读RS1的值） |
| I3: SUB R1, R6, R7 | RS3 | R1 → RS3（新的映射） |
| I4: ADD R8, R1, R9 | RS4 | R1 → RS3（读RS3的值） |

I2读的是RS1中的旧值，I3写的是新的RS3，两者互不影响 → WAR消除。
I1和I3写不同的保留站 → WAW消除。

### 21.12.4 寄存器状态表

寄存器状态表跟踪每个寄存器的值来源：

| 寄存器 | Qi（产生该寄存器值的保留站） | 说明 |
|--------|---------------------------|------|
| R0 | 0 | 值已就绪（在寄存器文件中） |
| R1 | RS1 | 等待RS1产生结果 |
| R2 | 0 | 值已就绪 |
| R3 | 0 | 值已就绪 |
| R4 | RS2 | 等待RS2产生结果 |
| R5 | 0 | 值已就绪 |

- Qi = 0：寄存器中的值是最新的，可以直接读取
- Qi ≠ 0：等待指定保留站产生结果，结果将通过CDB写入

## 21.13 推测执行的硬件支持（详细）

### 21.13.1 ROB字段与结构

**ROB**（Reorder Buffer，重排序缓冲区）是一个环形FIFO缓冲区。

每个ROB条目包含以下字段：

| 字段 | 位宽 | 说明 |
|------|------|------|
| Instruction Type | 2位 | 指令类型（分支/Store/Load/寄存器操作） |
| Destination | 5~6位 | 目标寄存器编号或存储地址 |
| Value | 32/64位 | 运算结果（就绪后填入） |
| Ready | 1位 | 结果是否已计算完成 |
| Busy | 1位 | 该条目是否被占用 |
| Exception | 1位 | 是否产生异常 |

### 21.13.2 ROB条目格式

```
┌──────┬──────────────┬────────────┬───────┬───────┬───────────┐
│ 编号 │ 指令类型      │ 目标地址    │ 值    │ Ready │ Exception │
├──────┼──────────────┼────────────┼───────┼───────┼───────────┤
│  0   │ 寄存器操作    │ R5         │ 42    │  1    │    0      │
│  1   │ Load         │ F0         │ -     │  0    │    0      │
│  2   │ 分支         │ -          │ T/NT  │  0    │    0      │
│  3   │ Store        │ 0x1000     │ 100   │  1    │    0      │
│ ...  │ ...          │ ...        │ ...   │ ...   │ ...       │
└──────┴──────────────┴────────────┴───────┴───────┴───────────┘
```

### 21.13.3 提交阶段行为

指令必须按程序顺序从ROB头部提交（Commit/Retire）：

```
ROB头部
  ↓
┌──────┬───────┬────────┬───────┬───────┐
│ 编号 │ 类型  │ 目标   │ 值    │ Ready │
├──────┼───────┼────────┼───────┼───────┤
│ HEAD │ REG   │ R5     │ 42    │  1    │ ← Ready=1，可以提交
│      │ LOAD  │ F0     │ 3.14  │  1    │ ← HEAD=1后，下周期变为HEAD
│      │ BRANCH│ -      │ Taken │  1    │ ← 分支已解析
│ TAIL │ STORE │ 0x1000 │ -     │  0    │ ← 尚未完成
└──────┴───────┴────────┴───────┴───────┘
```

**提交规则**：
1. 只有ROB头部的条目才能提交
2. 头部条目必须Ready=1才能提交
3. 提交后，ROB头部指针前移
4. 如果头部条目有异常，触发异常处理

**提交动作**：

| 指令类型 | 提交时的动作 |
|---------|-------------|
| 寄存器操作 | ROB中的值写入体系结构寄存器文件 |
| Load | ROB中的值写入体系结构寄存器文件 |
| Store | ROB中的值写入存储器 |
| 分支 | 如果误预测，清空ROB并恢复状态 |

### 21.13.4 误预测恢复

**分支误预测时的恢复步骤**：

```
1. 检测到分支预测错误
         ↓
2. 清空ROB中该分支之后的所有条目
         ↓
3. 清空保留站中对应的条目
         ↓
4. 从正确的分支目标重新取指
         ↓
5. 恢复体系结构寄存器状态
```

**恢复机制对比**：

| 方法 | 恢复速度 | 硬件复杂度 | 说明 |
|------|---------|-----------|------|
| ROB回滚 | 快 | 中 | 利用ROB中的旧值恢复 |
| 检查点 | **最快** | **高** | 在分支处保存寄存器快照 |
| 重命名表回滚 | 中 | 中 | 回滚寄存器重命名映射表 |

> [!NOTE]
> ROB保证了精确异常：即使指令乱序执行，提交时按程序顺序进行。如果发生异常，只需清空ROB中异常指令之后的所有条目，处理器状态是精确的。

## 21.14 超流水线的设计

### 21.14.1 超流水线概念

**超流水线**（Superpipelining）：将流水线阶段进一步细分，使每个阶段更短，从而提高时钟频率。

**基本思想**：将原来的一个阶段拆分为多个子阶段。

```
基本流水线（5级）：
IF → ID → EX → MEM → WB

超流水线（10级）：
IF1 → IF2 → ID1 → ID2 → EX1 → EX2 → MEM1 → MEM2 → WB1 → WB2
```

### 21.14.2 阶段拆分技术

**阶段拆分原则**：
1. 找出关键路径（最长的阶段）
2. 在关键路径的中间插入流水线寄存器
3. 确保每个子阶段的延迟大致相等

**EX阶段拆分示例**：

```
原始EX阶段（包含多级ALU操作）：
┌─────────────────────────────────┐
│  取操作数 → ALU第1级 → ALU第2级  │
└─────────────────────────────────┘

拆分后：
┌──────────────┐  ┌──────────────┐
│EX1:取操作数   │→│EX2:ALU计算   │
│  + ALU第1级  │  │  + 结果锁存  │
└──────────────┘  └──────────────┘
```

### 21.14.3 频率与深度的权衡

设基本流水线的时钟周期为 $T$，阶段数为 $N$，每个阶段的延迟为 $t_{stage}$，流水线寄存器延迟为 $t_{reg}$：

$$T = t_{stage} + t_{reg}$$

拆分为 $k$ 倍后：

$$T' = \frac{t_{stage}}{k} + t_{reg}$$

频率提升比：

$$\frac{f'}{f} = \frac{T}{T'} = \frac{t_{stage} + t_{reg}}{t_{stage}/k + t_{reg}}$$

当 $t_{reg}$ 相对 $t_{stage}$ 较小时，频率提升接近 $k$ 倍；当 $t_{reg}$ 较大时，频率提升受限。

**示例计算**：

| 参数 | 值 |
|------|------|
| $t_{stage}$ | 200ps |
| $t_{reg}$ | 50ps |
| 基本周期 $T$ | 250ps |
| 基本频率 $f$ | 4GHz |

拆分为2倍后：

$$T' = \frac{200}{2} + 50 = 150ps, \quad f' = 6.67GHz$$

频率提升：$6.67 / 4 = 1.67$ 倍（而非理想的2倍）

拆分为4倍后：

$$T' = \frac{200}{4} + 50 = 100ps, \quad f' = 10GHz$$

频率提升：$10 / 4 = 2.5$ 倍（远低于理想的4倍）

> [!TIP]
> 流水线寄存器的延迟 $t_{reg}$ 是超流水线的"天花板"。当阶段拆分到一定程度，$t_{reg}$ 占据周期的大部分，继续拆分带来的频率提升微乎其微，而冒险惩罚和功耗却大幅增加。

### 21.14.4 Pentium 4案例分析

Intel Pentium 4（NetBurst微架构）是超流水线的典型案例：

| 特性 | Willamette/Northwood | Prescott |
|------|---------------------|----------|
| 流水线深度 | 20级 | **31级** |
| 时钟频率 | 1.4~3.4GHz | 2.8~3.8GHz |
| 分支误预测惩罚 | ~20周期 | ~30周期 |
| 功耗 | 55~89W | 89~115W |

**Pentium 4的教训**：
1. **深流水线的分支惩罚高**：31级流水线的分支误预测需要排空大量指令
2. **功耗急剧增加**：高频率导致功耗与频率成超线性关系 $P \propto C \cdot V^2 \cdot f$
3. **实际IPC低**：虽然频率高，但每个周期完成的有效工作少
4. **被Core架构取代**：Intel后来转向更短流水线、更高IPC的Core架构

### 21.14.5 超标量 vs 超流水线 vs 超标量超流水线对比

| 特性 | 标量流水线 | 超标量 | 超流水线 | 超标量超流水线 |
|------|----------|--------|---------|-------------|
| 每周期发射数 | 1 | $n$（2~8） | 1 | $n$ |
| 流水线深度 | $k$ | $k$ | $m \cdot k$（更深） | $m \cdot k$ |
| 时钟频率 | 基准 | 中等 | 更高 | **最高** |
| CPI理想值 | 1 | $1/n$ | 1 | $1/n$ |
| IPC理想值 | 1 | $n$ | 1 | $n$ |
| 冒险惩罚 | 基准 | 较大 | **最大** | 最大 |
| 硬件复杂度 | 低 | 中 | 中 | **高** |
| 典型代表 | 早期CPU | Pentium Pro | Pentium 4 | 现代CPU |

> [!NOTE]
> 现代处理器通常同时采用超标量和超流水线技术，但更侧重超标量（多发射）而非超流水线（深流水线）。这是因为IPC提升比频率提升更有效，且功耗更可控。

## 21.15 多线程技术（详细）

### 21.15.1 细粒度多线程机制

**细粒度多线程**（Fine-grained Multithreading）：每个时钟周期切换线程。

**工作机制**：

```
周期  | 线程    | 指令
------|---------|------
C1    | Thread1 | I1_1
C2    | Thread2 | I2_1
C3    | Thread3 | I3_1
C4    | Thread1 | I1_2
C5    | Thread2 | I2_2
C6    | Thread3 | I3_2
```

**硬件需求**：
- 每个线程独立的PC（程序计数器）
- 每个线程独立的寄存器文件（或寄存器重命名）
- 共享功能部件和Cache

**优点**：
- 某线程停顿时，其他线程可继续执行
- 简单的调度逻辑
- 隐藏各种延迟

**缺点**：
- 即使某线程无停顿，也必须让出周期给其他线程
- 单线程性能可能下降

### 21.15.2 粗粒度多线程机制

**粗粒度多线程**（Coarse-grained Multithreading）：仅在长延迟事件（如Cache缺失）时切换线程。

**工作机制**：

```
周期  | 线程    | 指令      | 说明
------|---------|----------|------
C1    | T1      | I1_1     |
C2    | T1      | I1_2     |
C3    | T1      | I1_LD   | Cache缺失！
C4    | T1      | stall    | 等待访存
------|---------|----------|------← 切换点
C5    | T2      | I2_1     | 切换到T2
C6    | T2      | I2_2     |
------|---------|----------|------← 切换点
C15   | T1      | I1_LD完成| T1数据到达
C16   | T1      | I1_3     | 恢复T1
```

**优点**：
- 仅在长延迟事件时切换，减少不必要的切换
- 单线程性能影响较小

**缺点**：
- 切换开销较大（需要排空流水线）
- 短延迟事件（如整数运算停顿）无法利用

### 21.15.3 SMT硬件需求

**SMT**（Simultaneous Multithreading，同时多线程）需要以下硬件支持：

| 硬件资源 | 说明 | 每线程独立/共享 |
|---------|------|----------------|
| PC寄存器 | 每线程独立的程序计数器 | **独立** |
| 寄存器文件 | 每线程独立的寄存器集 | **独立** |
| 重命名表 | 每线程独立的映射表 | **独立** |
| ROB | 每线程独立的重排序缓冲区 | **独立** |
| 保留站 | 线程间共享或分区 | **共享/分区** |
| 功能部件 | 线程间共享 | **共享** |
| Cache | 线程间共享 | **共享** |
| 分支预测器 | 线程间共享 | **共享** |

### 21.15.4 Intel超线程技术细节

Intel Hyper-Threading（HT）是SMT的一种实现，通常支持2个逻辑核：

**第一代HT（Pentium 4）**：
- 2个逻辑核共享执行资源
- 独立的体系结构状态（寄存器、APIC等）
- 共享执行引擎、Cache、分支预测器

**现代HT（Core系列）**：
- 更精细的资源分区和优先级控制
- 支持动态资源分配
- 当只有一个逻辑核活跃时，可获得接近完整的资源

**资源竞争与优先级**：

```
┌──────────────────────────────────────────┐
│            物理核资源                      │
│                                          │
│  ┌──────────┐      ┌──────────┐         │
│  │逻辑核 0  │      │逻辑核 1  │         │
│  │ (高优先级)│      │ (低优先级)│         │
│  └────┬─────┘      └────┬─────┘         │
│       │                  │               │
│       └──────┬───────────┘               │
│              ↓                           │
│  ┌──────────────────────────────────┐   │
│  │        共享执行资源               │   │
│  │  ALU0  ALU1  MUL  LD   ST  FPU  │   │
│  └──────────────────────────────────┘   │
└──────────────────────────────────────────┘
```

### 21.15.5 TLP vs ILP

| 特性 | ILP（指令级并行） | TLP（线程级并行） |
|------|------------------|------------------|
| 并行粒度 | 指令级 | 线程级 |
| 检测方式 | 硬件（动态调度） | 操作系统/硬件 |
| 依赖分析 | 编译器/硬件 | 线程间无依赖 |
| 典型技术 | 超标量、VLIW | 多核、SMT |
| 上限 | 受限于指令依赖 | 受限于线程数 |
| 复杂度 | 高（硬件复杂） | 中（需要同步） |

**ILP的局限性**：
- 程序中的指令级并行度有限（通常IPC < 3~4）
- 依赖链限制了可并行的指令数
- 分支预测和推测执行的收益递减

**TLP的优势**：
- 线程间通常无数据依赖
- 可以利用多个独立的执行上下文
- 更容易扩展到大量并行

### 21.15.6 SMT中的资源共享

SMT的关键挑战是资源共享的公平性和效率：

**资源分配策略**：

| 策略 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| 完全共享 | 所有资源自由竞争 | 利用率高 | 可能不公平 |
| 静态分区 | 资源按比例分配 | 公平 | 利用率可能低 |
| 动态分区 | 根据需求动态调整 | 灵活 | 硬件复杂 |

**Cache共享问题**：
- 两个线程竞争Cache空间
- 一个线程可能"污染"另一个线程的Cache
- 解决方案：Cache分区或优先级控制

### 21.15.7 SMT性能指标

**吞吐率提升**：

$$Throughput_{SMT} = \frac{IPC_{Thread1} + IPC_{Thread2}}{IPC_{Single}}$$

典型SMT的吞吐率提升：

| 场景 | 吞吐率提升 |
|------|-----------|
| 两个计算密集型线程 | 1.1~1.3倍 |
| 一个计算密集 + 一个访存密集 | 1.3~1.5倍 |
| 两个访存密集型线程 | 1.0~1.2倍 |

**单线程性能影响**：

$$IPC_{SMT} = \frac{IPC_{shared}}{2} + \frac{IPC_{single}}{2}$$

理想情况下，SMT不应显著降低单线程性能，同时提升总体吞吐率。

> [!TIP]
> SMT的收益取决于工作负载的互补性。如果两个线程都是计算密集型，竞争功能部件，收益有限；如果一个线程是访存密集型（Cache缺失多），另一个是计算密集型，两者可以互补，收益最大。

## 21.16 现代处理器的流水线实例

### 21.16.1 Intel Core系列流水线

**Intel Core（酷睿）系列**采用较深但高效的乱序流水线：

| 微架构 | 流水线深度 | 发射宽度 | 乱序窗口 | 年份 |
|--------|----------|---------|---------|------|
| Core (Merom) | 14级 | 4-wide | 96条指令 | 2006 |
| Nehalem | 14级 | 4-wide | 128条指令 | 2008 |
| Sandy Bridge | 14~19级 | 4-wide | 168条指令 | 2011 |
| Haswell | 14~19级 | 4-wide | 192条指令 | 2013 |
| Skylake | 14~19级 | 4-wide | 224条指令 | 2015 |
| Alder Lake | 14~20级 | 6-wide | 512条指令 | 2021 |

**Core系列流水线阶段**：

```
┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐
│  F  │  D  │Rename│Alloc│Issue│ EX  │  WR  │ ROB  │Retire│
│取指 │译码 │重命名│分配 │发射 │执行  │写结果│重排序│ 退役 │
└─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┘
  1-2   1-4    1     1    1-多  1-多   1     多     1
  级    级     级    级    级    级    级    级     级
```

**关键技术**：
- **Macro-op Fusion**：将某些指令对融合为单个操作
- **Micro-op Fusion**：将复杂指令分解为更少的微操作
- **Loop Stream Detector**：检测循环并直接从缓冲区发射
- **AVX/SSE向量指令**：单条指令处理多个数据

### 21.16.2 ARM Cortex-A系列流水线

ARM Cortex-A系列广泛用于移动设备：

| 处理器 | 流水线深度 | 发射宽度 | 乱序/顺序 | 年份 |
|--------|----------|---------|----------|------|
| Cortex-A8 | 13级 | 2-wide | 顺序 | 2005 |
| Cortex-A9 | 8~11级 | 2-wide | **乱序** | 2007 |
| Cortex-A15 | 15级 | 3-wide | **乱序** | 2011 |
| Cortex-A53 | 8级 | 2-wide | 顺序 | 2013 |
| Cortex-A57 | 15级 | 3-wide | **乱序** | 2013 |
| Cortex-A76 | 13级 | 4-wide | **乱序** | 2018 |
| Cortex-A78 | 11级 | 4-wide | **乱序** | 2020 |

**Cortex-A76流水线**：

```
┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐
│Fetch│Fetch│Decode│Rename│Issue│Execute│Write│Commit│
│ 1   │ 2   │ 1-3  │  1   │ 1-2 │ 1-4   │  1  │  1   │
└─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┘
```

**ARM特点**：
- 固定长度指令（32位ARM模式），简化译码
- 条件执行减少分支数量
- Load/Store架构，只有Load/Store访问内存
- NEON/SVE向量扩展

### 21.16.3 RISC-V流水线

RISC-V是开源指令集架构，流水线实现灵活：

**典型RISC-V流水线（5级）**：

```
┌─────┬─────┬─────┬─────┬─────┐
│ IF  │ ID  │ EX  │ MEM │ WB  │
│取指 │译码 │执行 │访存 │写回 │
└─────┴─────┴─────┴─────┴─────┘
```

**高级RISC-V实现**（如BOOM、Rocket）：

| 实现 | 流水线深度 | 发射宽度 | 乱序/顺序 | 说明 |
|------|----------|---------|----------|------|
| Rocket | 5级 | 1-wide | 顺序 | 嵌入式/教学 |
| BOOM v2 | 9~16级 | 2~4-wide | **乱序** | 高性能 |
| SiFive U74 | 8级 | 2-wide | 顺序 | 嵌入式 |
| SiFive P670 | 12级 | 4-wide | **乱序** | 高性能 |

**RISC-V流水线优势**：
- 简洁的指令格式，简化译码
- 无条件码，减少依赖
- 灵活的扩展（M/A/F/D/C/V等）

### 21.16.4 现代处理器对比表

| 特性 | Intel Core i9 | ARM Cortex-A78 | RISC-V BOOM |
|------|--------------|----------------|-------------|
| 流水线深度 | 14~20级 | 11级 | 9~16级 |
| 发射宽度 | 6-wide | 4-wide | 2~4-wide |
| 乱序窗口 | 512条 | 160条 | 64~128条 |
| ROB大小 | 512 | 160 | 64~128 |
| 功能部件 | 10+ | 8 | 6~8 |
| 分支预测 | 极复杂 | 复杂 | 中等 |
| 功耗 | 125W+ | 1~3W | 0.5~5W |
| 目标市场 | 桌面/服务器 | 移动/嵌入式 | 嵌入式/教学/高性能 |
| 指令集 | x86-64 | ARMv8/v9 | RISC-V |

> [!NOTE]
> 不同指令集架构的流水线设计哲学不同：x86需要复杂的译码器将CISC指令转换为微操作；ARM利用固定长度指令简化译码；RISC-V的简洁设计使得译码更简单，但高性能实现仍需复杂的乱序执行逻辑。

## 21.17 动态调度的深入分析

### 21.17.1 记分牌详细执行跟踪

**示例程序**：

```
I1: LD    F0, 0(R1)     ; Load, 延迟=1
I2: MUL   F4, F0, F2    ; 乘法, 延迟=5, 依赖I1
I3: ADD   F6, F4, F2    ; 加法, 延迟=2, 依赖I2
I4: DIV   F8, F6, F2    ; 除法, 延迟=10, 依赖I3
I5: ADD   F10, F0, F2   ; 加法, 延迟=2, 依赖I1
```

**记分牌状态跟踪**：

假设功能部件：1个Load、1个乘法器、1个加法器、1个除法器。

**周期1：I1发射**
```
指令状态：I1=发射, I2~I5=等待
功能部件状态：
  Load1: Busy=1, Op=LD, Fi=F0
寄存器结果状态：
  F0: Load1
```

**周期2：I1执行，I2发射**
```
指令状态：I1=执行, I2=发射, I3~I5=等待
功能部件状态：
  Load1: Busy=1, Op=LD, Fi=F0
  Mult1: Busy=1, Op=MUL, Fi=F4, Fj=F0(Qj=Load1), Fk=F2(就绪)
寄存器结果状态：
  F0: Load1
  F4: Mult1
```

**周期3：I1完成，I5发射**
```
指令状态：I1=写结果, I2=读操作数(等待F0), I5=发射
功能部件状态：
  Load1: 写回F0=mem[0+R1], 然后空闲
  Mult1: 等待F0
  Add2: Busy=1, Op=ADD, Fi=F10, Fj=F0(就绪), Fk=F2(就绪)
寄存器结果状态：
  F0: 0 (Load1写回完成)
  F4: Mult1
  F10: Add2
```

**周期4：I5执行，I2读操作数**
```
指令状态：I2=读操作数(F0已就绪), I5=执行
Mult1: Fj←F0值, 开始执行
Add2: 执行中
```

**周期5：I2执行，I5完成，I3发射**
```
指令状态：I2=执行, I5=写结果, I3=发射
Mult1: 执行中
Add2: 写回F10, 空闲
Add1: Busy=1, Op=ADD, Fi=F6, Fj=F4(Qj=Mult1), Fk=F2(就绪)
寄存器结果状态：
  F4: Mult1
  F6: Add1
  F10: 0
```

**注意**：I5在I2之前完成，体现了乱序完成。

> [!TIP]
> 记分牌的关键限制：当I3的写结果阶段需要写回F6时，必须检查是否有后续指令正在读F6（WAR检查）。如果有，必须推迟写回直到读操作完成。这是记分牌不能消除WAR冒险的根本原因。

### 21.17.2 寄存器重命名硬件

**重命名硬件的两种实现方式**：

**方式一：使用保留站编号重命名（Tomasulo方式）**

```
体系结构寄存器  →  保留站编号
     R1         →     RS3
     R2         →     RS5
     R3         →     0（值在寄存器文件中）
```

每条指令发射时，将目标寄存器映射到分配的保留站编号。后续指令读该寄存器时，通过保留站编号获取值。

**方式二：使用物理寄存器重命名（现代处理器方式）**

```
体系结构寄存器  →  物理寄存器
     R1         →     P17
     R2         →     P23
     R3         →     P8
```

**重命名表（RAT，Register Alias Table）**：

| 体系结构寄存器 | 物理寄存器映射 | 状态 |
|--------------|--------------|------|
| R0 | P0 | 有效 |
| R1 | P17 | 有效 |
| R2 | P23 | 有效 |
| R3 | P8 | 有效 |
| R4 | P31 | 有效 |
| ... | ... | ... |

**重命名流程**：

```
1. 指令译码，读取源操作数的物理寄存器映射
         ↓
2. 从空闲列表分配新的物理寄存器给目标操作数
         ↓
3. 更新RAT：目标体系结构寄存器→新物理寄存器
         ↓
4. 将指令中的体系结构寄存器替换为物理寄存器编号
         ↓
5. 将指令发射到保留站
```

**示例**：

```
指令：ADD R1, R2, R3

执行前RAT：
  R1→P10, R2→P5, R3→P7

重命名步骤：
  1. 源操作数：R2→P5, R3→P7
  2. 分配新物理寄存器：P15（空闲列表中取出）
  3. 更新RAT：R1→P15
  4. 发射指令：ADD P15, P5, P7

执行后RAT：
  R1→P15, R2→P5, R3→P7
  P10变为"旧值"，待ROB提交后释放回空闲列表
```

### 21.17.3 存储器消歧

**存储器消歧**（Memory Disambiguation）：解决Load和Store指令之间的地址依赖问题。

**问题**：

```
I1: ST  R1, 0(R2)     ; Store: 内存[0+R2] ← R1
I2: LD  R3, 0(R4)     ; Load:  R3 ← 内存[0+R4]
```

如果 R2 == R4，则I2必须等待I1完成（地址冲突）。
如果 R2 ≠ R4，则I2可以与I1并行执行。

**消歧策略**：

| 策略 | 说明 | 准确性 | 复杂度 |
|------|------|--------|--------|
| 保守策略 | Load必须等待所有前面的Store完成 | 最安全 | 低 |
| 地址预测 | 预测Load与前面的Store不冲突 | 较高 | 中 |
| 运行时检查 | 比较Load和Store的地址 | **最高** | 高 |

**运行时消歧硬件**：

```
┌────────────────────────────────────┐
│         Store缓冲区                 │
│  ┌──────┬────────┬──────────┐     │
│  │ 条目 │ 地址   │ 数据     │     │
│  ├──────┼────────┼──────────┤     │
│  │ S1   │ 0x1000 │ 42       │     │
│  │ S2   │ 0x1008 │ 100      │     │
│  └──────┴────────┴──────────┘     │
└────────────────────────────────────┘
          ↑
          │ 地址比较
          ↓
┌────────────────────────────────────┐
│         Load队列                    │
│  ┌──────┬────────┬──────────┐     │
│  │ 条目 │ 地址   │ 状态     │     │
│  ├──────┼────────┼──────────┤     │
│  │ L1   │ 0x1000 │ 冲突！   │     │
│  │ L2   │ 0x2000 │ 无冲突   │     │
│  └──────┴────────┴──────────┘     │
└────────────────────────────────────┘
```

**消歧执行流程**：

```
1. Load计算出地址
         ↓
2. 与Store缓冲区中的所有地址比较
         ↓
3. 如果找到匹配地址：
   a. Store已完成 → 从Store缓冲区读取数据（Store-to-Load Forwarding）
   b. Store未完成 → 等待Store完成
         ↓
4. 如果无匹配地址 → Load可以继续执行
         ↓
5. 后续Store到达时，如果发现与已执行的Load地址冲突
   → 触发流水线冲刷，重新执行Load（误推测恢复）
```

> [!NOTE]
> 存储器消歧是现代乱序处理器中最复杂的部分之一。Load可能在Store地址确定之前就需要执行（推测性Load）。如果事后发现地址冲突，必须冲刷流水线并重新执行。这与分支推测类似，但处理的是数据依赖而非控制依赖。
