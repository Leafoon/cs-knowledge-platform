## 12.1 多级反馈队列（MLFQ）

### 12.1.1 设计目标

MLFQ（Multi-Level Feedback Queue）是 UNIX 系统最初使用的调度算法，试图同时优化**周转时间**和**响应时间**，而不需要预先知道进程的执行时间。

MLFQ 的核心思想是：通过观察进程的**行为**来动态调整优先级。它不像 SJF 那样需要预测执行时间，而是通过实际运行来"学习"进程的特征。

**设计目标**：
1. 优化周转时间——让短进程快速完成
2. 优化响应时间——让交互式进程快速响应
3. 不需要预知执行时间——通过观察行为调整
4. 防止饥饿——长进程最终也能获得 CPU

### 12.1.2 基本规则

MLFQ 维护多个就绪队列，每个队列有不同的优先级和时间片：

```
优先级 3（最高）：时间片 = 8ms    [队列3: 新进程进入这里]
优先级 2：        时间片 = 16ms   [队列2]
优先级 1（最低）：时间片 = 32ms   [队列1: CPU 密集型进程最终到这里]
```

**规则 1**：优先级高的队列中的进程先运行。只有高优先级队列为空时，才运行低优先级队列。

**规则 2**：同一优先级队列内的进程使用轮转调度（RR）。

**规则 3**：新进程进入最高优先级队列。

**规则 4**：进程用完当前时间片后，优先级降低一级（移到下一级队列）。

**规则 5**：进程在时间片内主动让出 CPU（如 I/O 阻塞），优先级保持不变。

```
规则 4 示例（CPU 密集型进程）：
进程 A 进入优先级 3（时间片=8ms）
  → 运行 8ms，用完时间片 → 降到优先级 2（时间片=16ms）
  → 运行 16ms，用完时间片 → 降到优先级 1（时间片=32ms）
  → 在优先级 1 中与其他 CPU 密集型进程轮转

规则 5 示例（I/O 密集型进程）：
进程 B 进入优先级 3（时间片=8ms）
  → 运行 5ms，发起 I/O 阻塞 → 优先级保持在 3
  → I/O 完成，回到优先级 3 → 立即获得高优先级 CPU
```

### 12.1.3 MLFQ 的行为分析

规则 5 是 MLFQ 的精髓：它自然地实现了"短作业优先"的效果。

**I/O 密集型进程**（通常是交互式进程）：
- 只使用少量 CPU 时间就发起 I/O
- 在时间片内主动让出 CPU
- 优先级保持在高水平
- 每次 I/O 完成后都能快速获得 CPU

**CPU 密集型进程**（通常是批处理进程）：
- 使用完整个时间片
- 优先级逐渐降低
- 最终降到最低优先级队列
- 在低优先级队列中与其他 CPU 密集型进程轮转

这正是我们想要的行为：交互式进程（短进程）获得高优先级，快速响应；批处理进程（长进程）被降到低优先级，不阻塞短进程。

```
时间线示例：
进程 A（CPU 密集型，执行=100ms）
进程 B（I/O 密集型，每 10ms 计算 2ms 后 I/O）

优先级 3（时间片=8ms）：
  [B:2ms][IO]          [B:2ms][IO]          [B:2ms][IO]
优先级 2（时间片=16ms）：
  [A:8ms]
优先级 1（时间片=32ms）：
                      [A:16ms]             [A:32ms]

→ B 始终保持高优先级，快速响应
→ A 逐渐降低优先级，不阻塞 B
```

### 12.1.4 防止博弈（Gaming）

规则 4 存在一个漏洞：进程可以在时间片即将用完时主动发起一个"假的"I/O 操作（如读取 0 字节），以保持高优先级。这称为**博弈（Gaming）**。

```
博弈示例：
进程 C 在时间片的第 7ms（时间片=8ms）发起一个假的 I/O
→ 规则 5：在时间片内让出 CPU，优先级保持不变
→ C 始终保持在优先级 3，获得更多 CPU 时间

时间线：
C: [=7ms=][假IO] [=7ms=][假IO] [=7ms=][假IO] ...
优先级：3        3        3        3
```

**规则 6**（防博弈）：进程在某一优先级上累计使用的时间达到**提升周期**（Scheduling Period）后，将其优先级提升到最高。

```
提升周期 = 100ms

进程 C（博弈者）：
  运行 7ms → 假 I/O → 保持优先级 3（累计 7ms）
  运行 7ms → 假 I/O → 保持优先级 3（累计 14ms）
  ...
  运行 7ms → 假 I/O → 保持优先级 3（累计 98ms）
  运行 7ms → 假 I/O → 累计 105ms > 100ms → 提升到优先级 3（重置）
```

即使进程博弈，它也无法长期独占高优先级——提升周期会将其重置，其他进程有机会运行。

### 12.1.5 参数调优

MLFQ 的性能取决于几个关键参数：

**队列数量**：通常 4-8 个。太少区分度不够（如只有 2 个队列），太多管理开销大。

**每级时间片**：通常是倍增关系（如 8ms, 16ms, 32ms, 64ms）。这样 CPU 密集型进程在每个优先级上获得的时间逐渐增加，减少上下文切换。

**提升周期**：通常 1-10 秒。太短会导致频繁重置（所有进程都被提升），太长会导致长进程饥饿。

```
MLFQ 参数示例（BSD 调度器）：
- 32 个优先级队列
- 时间片：4ms × (priority + 1)
- 最高优先级：时间片 = 4ms
- 最低优先级：时间片 = 128ms
- 提升周期：10 秒
```

### 12.1.6 MLFQ 在真实系统中的应用

**BSD 调度器**：FreeBSD 的传统调度器基于 MLFQ，使用 32 个优先级队列。

**Windows 调度器**：Windows NT 到 Windows 7 使用 MLFQ，32 个优先级级别。

**Linux O(1) 调度器**（Linux 2.6.0-2.6.22）：使用 140 个优先级（0-99 实时，100-139 普通），基于 MLFQ 思想。

**Linux CFS**（Linux 2.6.23+）：取代了 O(1) 调度器，使用完全不同的设计哲学（见下一节）。

## 12.2 完全公平调度器（CFS）

### 12.2.1 设计哲学

CFS（Completely Fair Scheduler）是 Linux 2.6.23 引入的调度器，设计目标是实现**理想的多任务处理器**——在 N 个进程的情况下，每个进程获得 1/N 的 CPU 时间。

CFS 的设计哲学与 MLFQ 截然不同：
- MLFQ 通过**观察行为**来调整优先级
- CFS 通过**精确追踪 CPU 使用量**来实现公平

CFS 的核心思想是追踪每个进程的**虚拟运行时间（vruntime）**，并始终运行 vruntime 最小的进程。vruntime 越小，说明进程获得的 CPU 时间越少，应该被优先调度。

### 12.2.2 虚拟运行时间（vruntime）

vruntime 是 CFS 的核心概念：

```
vruntime += 实际运行时间 × (NICE_0_WEIGHT / 进程权重)
```

- **NICE_0_WEIGHT**：nice 值为 0 的进程的权重（通常 1024）
- **进程权重**：由 nice 值决定，nice 越低（优先级越高）权重越大

```
nice 值与权重的映射（简化）：
nice -20 → 权重 88761（最高优先级，vruntime 增长最慢）
nice  -10 → 权重 11059
nice   0 → 权重 1024（默认优先级）
nice +10 → 权重 110
nice +19 → 权重 15（最低优先级，vruntime 增长最快）
```

**关键洞察**：高优先级进程的 vruntime 增长慢，低优先级进程的 vruntime 增长快。CFS 总是运行 vruntime 最小的进程，因此高优先级进程获得更多 CPU 时间。

**vruntime 计算示例**：

```
两个进程 A(nice=0) 和 B(nice=0)，权重相同=1024

每运行 1ms：
  vruntime += 1ms × (1024/1024) = 1ms

A 运行 5ms：A.vruntime = 5ms
B 运行 5ms：B.vruntime = 5ms
→ vruntime 相同，CFS 交替运行，实现 50/50 CPU 分配

A(nice=-10, 权重=11059) 和 B(nice=0, 权重=1024)

A 运行 1ms：A.vruntime += 1ms × (1024/11059) = 0.093ms
B 运行 1ms：B.vruntime += 1ms × (1024/1024) = 1ms

→ A 的 vruntime 增长慢 10.8 倍
→ CFS 会分配给 A 更多 CPU 时间，实现优先级效果
```

### 12.2.3 红黑树实现就绪队列

CFS 使用**红黑树（Red-Black Tree）** 存储就绪进程，以 vruntime 为键。红黑树是一种自平衡二叉搜索树，支持 O(log n) 的插入、删除和查找最小值操作。

```
CFS 红黑树（以 vruntime 为键）：

         [vruntime=50]
        /              \
  [vruntime=20]    [vruntime=80]
  /         \           \
[10]      [30]        [100]

最左节点（vruntime=10）= 下一个运行的进程
```

CFS 维护一个缓存指针 `rb_leftmost`，指向红黑树中最左节点（vruntime 最小的进程）。选择下一个进程只需 O(1) 时间（读取缓存指针）。

**红黑树操作**：
- **插入**：O(log n) — 新进程就绪时
- **删除**：O(log n) — 进程阻塞或终止时
- **更新**：O(log n) — 进程运行后更新 vruntime（先删后插）
- **选择最小**：O(1) — 读取 `rb_leftmost` 缓存

### 12.2.4 调度延迟与最小粒度

CFS 的目标是让每个进程在**调度延迟**（Scheduling Latency）内至少运行一次。调度延迟通常为 6-24ms，取决于就绪进程数。

```
调度延迟 = max(目标延迟, 最小粒度 × 进程数)

目标延迟 = 6ms（可配置）
最小粒度 = 0.75ms（可配置）

1 个进程：每个进程获得 6ms 时间片
2 个进程：每个进程获得 3ms 时间片
4 个进程：每个进程获得 1.5ms 时间片
10 个进程：每个进程获得 max(0.6ms, 0.75ms) = 0.75ms 时间片
```

当进程数很多时，CFS 使用最小粒度作为下限，避免时间片太小导致切换过于频繁。

### 12.2.5 Nice 值与权重映射

Linux 的 nice 值范围为 -20 到 +19，对应不同的权重。权重决定了 vruntime 的增长速度：

```
权重映射（Linux 内核 sched_prio_to_weight[] 数组）：
nice -20:  weight = 88761
nice -10:  weight = 11059
nice   0:  weight = 1024
nice +10:  weight = 110
nice +19:  weight = 15

CPU 时间分配比例（两个进程 nice=-20 和 nice=0）：
权重比 = 88761 : 1024 ≈ 86.7 : 1
→ nice=-20 的进程获得 86.7 倍的 CPU 时间
```

### 12.2.6 组调度（Group Scheduling）

CFS 支持**组调度**：将进程分组，组间按权重分配 CPU 时间，组内再按权重分配。

```
组调度示例：
用户 A 的进程组（权重 60%）：进程 A1, A2
用户 B 的进程组（权重 40%）：进程 B1, B2, B3

CPU 时间分配：
  用户 A 组：60% → A1 和 A2 各 30%
  用户 B 组：40% → B1、B2、B3 各 13.3%
```

组调度防止了一个用户创建大量进程来"抢占"CPU 时间的问题。

## 12.3 实时调度

### 12.3.1 硬实时 vs 软实时

**硬实时系统**：必须在截止时间前完成任务，否则后果严重。典型例子：飞行控制系统、心脏起搏器、核电站控制。

**软实时系统**：尽量在截止时间前完成，偶尔错过可以接受。典型例子：视频播放、音频处理、Web 服务器。

### 12.3.2 速率单调调度（RMS）

RMS（Rate Monotonic Scheduling）是一种静态优先级调度算法：**周期越短的进程，优先级越高**。

```
进程 A：周期=10ms，执行时间=2ms
进程 B：周期=20ms，执行时间=4ms
进程 C：周期=40ms，执行时间=6ms

RMS 优先级：A > B > C（周期越短优先级越高）

时间线：
A: [2ms]      [2ms]      [2ms]      [2ms]
B:      [4ms]           [4ms]           [4ms]
C:                  [6ms]                     [6ms]
0    5   10   15   20   25   30   35   40ms
```

### 12.3.3 最早截止时间优先（EDF）

EDF（Earliest Deadline First）是一种动态优先级调度算法：**截止时间最近的进程，优先级越高**。

```
时刻 0：A(截止=10ms), B(截止=20ms) → 运行 A（截止更近）
时刻 2：A 完成，B 截止=20ms → 运行 B
时刻 6：B 完成
时刻 10：新 A 到达（截止=20ms）→ 运行 A
```

### 12.3.4 可调度性分析

**RMS 可调度性条件**（Liu & Layland 定理）：

```
n 个进程可调度的充分条件：
Σ(C_i / T_i) ≤ n × (2^(1/n) - 1)

其中 C_i = 执行时间，T_i = 周期

当 n=1 时：利用率 ≤ 100%
当 n=2 时：利用率 ≤ 82.8%
当 n=3 时：利用率 ≤ 78.0%
当 n→∞ 时：利用率 ≤ 69.3%
```

**EDF 可调度性条件**：

```
n 个进程可调度的充分必要条件：
Σ(C_i / T_i) ≤ 1.0

→ CPU 利用率不超过 100% 时，EDF 保证所有进程可调度
```

EDF 的 CPU 利用率上限（100%）高于 RMS（69.3%），但 EDF 实现更复杂，优先级动态变化。

### 12.3.5 Linux 实时调度

Linux 支持三种实时调度策略：

| 策略 | 说明 | 优先级范围 |
|------|------|-----------|
| SCHED_FIFO | 先进先出，不使用时间片 | 1-99 |
| SCHED_RR | 轮转，使用时间片 | 1-99 |
| SCHED_DEADLINE | EDF，指定截止时间 | 动态 |

## 12.4 多处理器调度

### 12.4.1 单队列 vs 多队列

**单队列调度（Single Queue）**：所有 CPU 共享一个就绪队列。

优点：天然负载均衡。
缺点：锁竞争严重，缓存局部性差。

**多队列调度（Multiple Queue）**：每个 CPU 有自己的就绪队列。

优点：无锁竞争，缓存局部性好。
缺点：需要负载均衡机制。

现代操作系统（Linux）使用**每 CPU 运行队列**（per-CPU run queue）。

### 12.4.2 负载均衡（Load Balancing）

多队列调度需要**负载均衡**机制来确保各 CPU 的负载大致相等：

**推（Push）**：当一个 CPU 的队列过长时，将进程推送到空闲 CPU。

**拉（Pull）**：当一个 CPU 空闲时，从繁忙 CPU 拉取进程。

Linux CFS 使用周期性的负载均衡器，检查各 CPU 的负载差异，并在需要时迁移进程。

### 12.4.3 处理器亲和性（CPU Affinity）

**处理器亲和性**是指进程倾向于在同一 CPU 上运行。这有利于缓存局部性——进程的数据和指令已经在该 CPU 的缓存中，切换到其他 CPU 会导致缓存失效。

```
软亲和性（Soft Affinity）：调度器尽量让进程在同一 CPU 运行，但不保证
硬亲和性（Hard Affinity）：进程明确绑定到特定 CPU
```

### 12.4.4 NUMA 感知调度

在 NUMA（Non-Uniform Memory Access）架构中，不同 CPU 访问不同内存区域的速度不同。NUMA 感知调度器会尽量将进程调度到离其内存最近的 CPU 上。

```
NUMA 节点 0：CPU 0-3，本地内存 16GB
NUMA 节点 1：CPU 4-7，本地内存 16GB

进程 A 的内存在节点 0 → 调度到 CPU 0-3
进程 B 的内存在节点 1 → 调度到 CPU 4-7
```

## 12.5 xv6 调度器分析

### 12.5.1 简单的 RR 调度器

xv6 使用最简单的调度策略：遍历进程表，找到第一个 RUNNABLE 进程运行。这本质上是**非抢占式 FCFS/RR**。

```c
// kernel/proc.c
void scheduler(void) {
    for (;;) {
        intr_on();
        for (p = proc; p < &proc[NPROC]; p++) {
            acquire(&p->lock);
            if (p->state == RUNNABLE) {
                p->state = RUNNING;
                c->proc = p;
                swtch(&c->context, &p->context);
                c->proc = 0;
            }
            release(&p->lock);
        }
    }
}
```

xv6 调度器的简单性体现了教学操作系统的设计哲学：**先理解核心思想，再学习优化**。

## 12.6 面试高频考点

**Q1：MLFQ 如何同时优化周转时间和响应时间？**

MLFQ 通过动态调整优先级：新进程进入最高优先级（快速响应），CPU 密集型进程逐渐降到低优先级（不阻塞短进程），I/O 密集型进程保持高优先级（快速响应 I/O 完成）。这自然实现了"短作业优先"的效果。

**Q2：CFS 如何实现公平性？**

CFS 追踪每个进程的 vruntime，始终运行 vruntime 最小的进程。vruntime 的增长速度与 nice 值相关——高优先级进程增长慢，低优先级增长快。相同 nice 值的进程 vruntime 增长速度相同，实现精确的 CPU 时间分配。

**Q3：RMS 和 EDF 的区别？**

RMS 是静态优先级（周期越短优先级越高），CPU 利用率上限 69.3%。EDF 是动态优先级（截止时间越近优先级越高），CPU 利用率上限 100%。EDF 更灵活但实现更复杂。

**Q4：多处理器调度的挑战是什么？**

负载均衡（各 CPU 负载大致相等）、处理器亲和性（缓存局部性）、NUMA 感知（内存访问延迟）。现代系统使用每 CPU 运行队列 + 负载均衡器。

**Q5：CFS 的红黑树有什么用？**

CFS 用红黑树存储就绪进程，以 vruntime 为键。红黑树支持 O(log n) 的插入/删除，O(1) 的查找最小值（通过 `rb_leftmost` 缓存）。这使得调度器的性能为 O(log n)。
