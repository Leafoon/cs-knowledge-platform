## 33.1 分布式系统基础

### 33.1.1 定义与挑战

**分布式系统（Distributed System）** 是由多台通过网络互联的独立计算机组成的系统，这些计算机协同工作，对外表现为一个统一的整体。

Leslie Lamport 的经典定义：

> "分布式系统就是这样一种系统：你甚至不知道有一台你从未听说过的计算机出了故障，就能让你自己的计算机变得不可用。"

```
分布式系统的基本架构：

┌─────────┐    ┌─────────┐    ┌─────────┐
│  Node A │◄──►│  Node B │◄──►│  Node C │
│ (Server)│    │ (Server)│    │ (Server)│
└────┬────┘    └────┬────┘    └────┬────┘
     │              │              │
     ▼              ▼              ▼
┌─────────┐    ┌─────────┐    ┌─────────┐
│ Storage │    │ Storage │    │ Storage │
└─────────┘    └─────────┘    └─────────┘
     ▲              ▲              ▲
     │              │              │
┌────┴────┐    ┌────┴────┐    ┌────┴────┐
│ Client  │    │ Client  │    │ Client  │
└─────────┘    └─────────┘    └─────────┘
```

**分布式系统的核心挑战**：

| 挑战 | 描述 | 影响 |
|------|------|------|
| **部分失败** | 系统中某些节点可能失败，而其他节点继续运行 | 难以区分节点故障与网络延迟 |
| **网络不可靠** | 消息可能丢失、重复、乱序或延迟 | 通信协议必须处理各种异常 |
| **时钟不同步** | 各节点的物理时钟存在漂移 | 难以确定事件的全局顺序 |
| **并发性** | 多个节点同时访问和修改共享数据 | 需要协调机制保证一致性 |
| **延迟** | 网络通信存在不可忽略的延迟 | 性能优化需要权衡一致性 |
| **安全** | 开放网络中的通信面临窃听和篡改风险 | 需要加密和认证机制 |

### 33.1.2 网络消息传递

分布式系统中的通信完全依赖网络消息传递。与单机系统中共享内存的通信方式不同，消息传递引入了不确定性和延迟。

**基本通信模型**：

```
发送方                    网络                     接收方
┌──────┐   send(msg)   ┌──────────┐   deliver(msg)  ┌──────┐
│Client│ ─────────────►│  Network │ ───────────────► │Server│
│      │               │ (不可靠)  │                  │      │
│      │◄─────────────│          │◄───────────────│      │
└──────┘   recv(ack)   └──────────┘   send(ack)     └──────┘
```

**RPC（远程过程调用）** 是分布式系统中最基本的通信范式：

```
RPC 调用流程：

Client                          Server
  │                               │
  │  1. 打包参数 (Marshalling)     │
  │ ──────────────────────────────►│
  │         RPC Request            │
  │                               │  2. 解包参数 (Unmarshalling)
  │                               │  3. 执行本地函数
  │                               │  4. 打包返回值
  │◄──────────────────────────────│
  │         RPC Response           │
  │  5. 解包返回值                  │
  │                               │

超时处理：
  │                               │
  │ ──────────────────────────────►│  (请求丢失)
  │                               │
  │  TIMEOUT! 重试?               │
  │ ──────────────────────────────►│
  │                               │
  │◄──────────────────────────────│
  │         Response               │
```

**RPC 的语义挑战**：

| 语义 | 含义 | 实现复杂度 |
|------|------|-----------|
| **At-most-once** | 最多执行一次，可能不执行 | 低 |
| **At-least-once** | 至少执行一次，可能重复执行 | 中 |
| **Exactly-once** | 恰好执行一次（理想目标） | 高（需要幂等性） |

### 33.1.3 部分失败（Partial Failure）

部分失败是分布式系统区别于单机系统最本质的特征。在单机系统中，要么整个系统正常工作，要么完全崩溃。但在分布式系统中，部分节点可能失败，而其他节点继续运行。

**部分失败的挑战**：

```
场景：Node B 突然无响应

┌─────────┐         ┌─────────┐         ┌─────────┐
│  Node A │───X────►│  Node B │◄──X────│  Node C │
│  (正常)  │  网络?  │  (故障?) │  网络?  │  (正常)  │
└─────────┘         └─────────┘         └─────────┘

Node A 的视角：
  - Node B 不响应 → 可能的原因：
    1. Node B 崩溃了
    2. 网络分区，A-B 之间断开
    3. Node B 处理缓慢，消息还在途中
    4. Node B 的响应消息丢失了
  → 无法确定真正的原因！
```

**不可能区分的三种情况**：

1. **节点崩溃**：目标节点真的停止工作了
2. **网络分区**：节点正常但网络连接断开
3. **处理延迟**：节点正在处理但响应缓慢

这种不确定性是许多分布式系统问题的根源。系统设计者必须做出假设（超时时间、故障模型等），并在此基础上构建可靠的协议。

### 33.1.4 CAP 定理

**CAP 定理**（Brewer's Theorem）是分布式系统设计中最基本的理论约束之一，由 Eric Brewer 于 2000 年提出，2002 年被 Gilbert 和 Lynch 证明。

```
CAP 定理：

                    Consistency (一致性)
                         /\
                        /  \
                       /    \
                      / CA   \ CP
                     / 理想但  \
                    / 不可能   \
                   /─────────────\
                  /  AP            \
                 /   最终一致性      \
                /____________________\
  Availability              Partition
  (可用性)              Tolerance (分区容忍性)

在分布式系统中，网络分区一定会发生
→ 只能在 C 和 A 之间选择！
```

**三个属性的定义**：

| 属性 | 含义 | 详细说明 |
|------|------|---------|
| **C (Consistency)** | 一致性 | 所有节点在同一时间看到相同的数据 |
| **A (Availability)** | 可用性 | 每个请求都能在有限时间内获得响应（非错误） |
| **P (Partition Tolerance)** | 分区容忍性 | 系统在网络分区时仍能继续运行 |

**关键理解**：

由于分布式系统中网络分区不可避免，实际上 CAP 的选择是：

- **CP 系统**（选择一致性，牺牲可用性）：网络分区时，为保证一致性而拒绝部分请求
  - 例：HBase、MongoDB（默认配置）、ZooKeeper、etcd

- **AP 系统**（选择可用性，牺牲一致性）：网络分区时，继续响应但可能返回旧数据
  - 例：Cassandra、DynamoDB、CouchDB、DNS

- **CA 系统**（选择一致性和可用性，放弃分区容忍性）：只能在单机或局域网环境中实现
  - 例：传统关系型数据库（单机部署）

**CAP 定理的实际应用**：

```
实际系统设计中的 CAP 权衡：

系统类型        │  选择  │  适用场景
───────────────┼────────┼────────────────────────
银行转账系统     │  CP    │  一致性优先，宁可拒绝服务
社交网络 Feed   │  AP    │  可用性优先，允许短暂不一致
购物车系统      │  AP    │  可用性优先，合并冲突
配置管理       │  CP    │  一致性优先，避免错误配置
```

> **面试高频考点**：CAP 定理的三个属性不是简单的"三选二"，而是在网络分区发生时在 C 和 A 之间做权衡。实际系统中，一致性是一个连续的光谱，而非二元选择。

## 33.2 分布式文件系统

分布式文件系统是分布式操作系统的核心组件，它允许多台计算机通过网络共享文件，就像访问本地文件一样。

### 33.2.1 NFS（Network File System）

NFS 由 Sun Microsystems 于 1984 年发布，是最经典的分布式文件系统之一。

**NFS 的核心设计原则**：

1. **无状态协议（Stateless Protocol）**：服务器不保存客户端的会话状态
2. **每个操作自包含**：每次请求包含完整的操作信息
3. **幂等性（Idempotent）**：重复执行同一操作结果相同

```
NFS 架构：

┌─────────────────────┐          ┌─────────────────────┐
│      Client          │          │      Server          │
│  ┌─────────────────┐ │          │  ┌─────────────────┐ │
│  │  Application    │ │          │  │  NFS Daemon     │ │
│  └────────┬────────┘ │          │  └────────┬────────┘ │
│  ┌────────┴────────┐ │          │  ┌────────┴────────┐ │
│  │  VFS Layer      │ │          │  │  VFS Layer      │ │
│  └────────┬────────┘ │          │  └────────┬────────┘ │
│  ┌────────┴────────┐ │  NFS     │  ┌────────┴────────┐ │
│  │  NFS Client     │◄├─Protocol─┤►│  NFS Server     │ │
│  │  (内核模块)      │ │  (UDP)   │  │  (内核模块)      │ │
│  └────────┬────────┘ │          │  └────────┬────────┘ │
│  ┌────────┴────────┐ │          │  ┌────────┴────────┐ │
│  │  Block Cache    │ │          │  │  Local FS       │ │
│  └─────────────────┘ │          │  │  (ext4/xfs)     │ │
│                      │          │  └─────────────────┘ │
└─────────────────────┘          └─────────────────────┘
```

**NFS 的缓存一致性问题**：

```
缓存一致性的时间窗口：

Client A                    Server                     Client B
   │                          │                          │
   │  write("hello")          │                          │
   │─────────────────────────►│  更新服务器数据            │
   │                          │                          │
   │  (缓存仍然有效？)         │     read()               │
   │                          │◄─────────────────────────│
   │                          │  返回旧数据！              │
   │                          │─────────────────────────►│
   │                          │                          │
   │  ┌────────────────────────────────────────────────┐│
   │  │  问题：Client B 可能读到旧数据                    ││
   │  │  NFS 解决方案：设置缓存超时时间 (如 3-30 秒)      ││
   │  └────────────────────────────────────────────────┘│
```

**NFS 的优缺点**：

| 优点 | 缺点 |
|------|------|
| 简单、易于实现 | 无状态限制了功能 |
| 服务器崩溃恢复容易 | 缓存一致性弱 |
| 广泛的平台支持 | 性能受网络延迟影响 |
| 良好的扩展性 | 安全模型较弱 |

### 33.2.2 AFS（Andrew File System）

AFS 由 Carnegie Mellon 大学于 1980 年代开发，采用了与 NFS 完全不同的设计理念。

**AFS 的核心设计原则**：

1. **全文件缓存（Whole-file Caching）**：客户端缓存整个文件
2. **回调机制（Callback）**：服务器在文件变化时通知客户端
3. **本地磁盘缓存**：缓存存储在本地磁盘而非内存

```
AFS 架构：

┌─────────────────────┐          ┌─────────────────────┐
│      Client          │          │      Server          │
│  ┌─────────────────┐ │          │  ┌─────────────────┐ │
│  │  Application    │ │          │  │  Vice (文件服务)  │ │
│  └────────┬────────┘ │          │  └────────┬────────┘ │
│  ┌────────┴────────┐ │          │  ┌────────┴────────┐ │
│  │  Cache Manager  │ │          │  │  Callback DB    │ │
│  │  (Venus)        │◄├──────────┤►│  (回调状态)      │ │
│  └────────┬────────┘ │ 回调通知  │  └────────┬────────┘ │
│  ┌────────┴────────┐ │          │  ┌────────┴────────┐ │
│  │  Local Disk     │ │          │  │  Vice Partition │ │
│  │  Cache          │ │          │  │  (服务器存储)    │ │
│  └─────────────────┘ │          │  └─────────────────┘ │
└─────────────────────┘          └─────────────────────┘
```

**AFS 的回调机制**：

```
回调 (Callback) 工作流程：

Client A                    Server                     Client B
   │                          │                          │
   │  open("file.txt")        │                          │
   │─────────────────────────►│                          │
   │◄─────────────────────────│                          │
   │  返回文件内容 + Callback  │                          │
   │                          │                          │
   │  open("file.txt")        │                          │
   │                          │◄─────────────────────────│
   │                          │─────────────────────────►│
   │                          │  返回文件内容 + Callback  │
   │                          │                          │
   │  write("file.txt")       │                          │
   │─────────────────────────►│                          │
   │                          │  Break Callback to B     │
   │                          │─────────────────────────►│
   │                          │  (B 的缓存失效)           │
   │                          │                          │
   │  读取仍在缓存中的文件     │                          │
   │  (A 的缓存仍然有效)       │                          │
```

**AFS vs NFS 对比**：

| 特性 | NFS | AFS |
|------|-----|-----|
| 缓存粒度 | 块级（Block） | 文件级（Whole File） |
| 缓存位置 | 内存 | 本地磁盘 |
| 一致性机制 | 超时检查 | 回调（Callback） |
| 状态 | 无状态 | 有状态 |
| 网络负载 | 频繁小请求 | 较少的大请求 |
| 离线访问 | 不支持 | 部分支持（缓存文件） |
| 适用场景 | 局域网 | 广域网、大规模部署 |

### 33.2.3 GFS（Google File System）

GFS 是 Google 设计的分布式文件系统，专为大规模数据处理而优化。

**GFS 的核心设计**：

```
GFS Master-Chunk 架构：

┌─────────────────────────────────────────────────────────┐
│                     GFS Master                            │
│  ┌───────────────────────────────────────────────────┐  │
│  │  文件命名空间 (Namespace)                          │  │
│  │  文件 → Chunk 映射                                 │  │
│  │  Chunk → ChunkServer 映射                          │  │
│  │  Chunk 副本位置信息                                │  │
│  └───────────────────────────────────────────────────┘  │
└──────────────────────┬──────────────────────────────────┘
                       │ 控制流
        ┌──────────────┼──────────────┐
        ▼              ▼              ▼
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│ ChunkServer1 │ │ ChunkServer2 │ │ ChunkServer3 │
│ ┌──────────┐ │ │ ┌──────────┐ │ │ ┌──────────┐ │
│ │ Chunk A  │ │ │ │ Chunk A  │ │ │ │ Chunk B  │ │
│ │ Chunk B  │ │ │ │ Chunk C  │ │ │ │ Chunk C  │ │
│ │ Chunk D  │ │ │ │ Chunk D  │ │ │ │ Chunk D  │ │
│ └──────────┘ │ │ └──────────┘ │ │ └──────────┘ │
└──────────────┘ └──────────────┘ └──────────────┘
        ▲              ▲              ▲
        │              │              │
        └──────────────┼──────────────┘
                       │ 数据流
                  ┌────┴────┐
                  │ Client  │
                  └─────────┘
```

**GFS 的关键特性**：

| 特性 | 设计选择 | 原因 |
|------|---------|------|
| 文件大小 | 大文件（GB 级别） | Google 工作负载以大文件为主 |
| Chunk 大小 | 64 MB | 减少 Master 元数据，减少客户端与 Master 交互 |
| 副本数 | 默认 3 副本 | 平衡可靠性和存储开销 |
| 一致性 | 松弛一致性 | 为追加操作优化，允许偶尔的不一致 |
| Master | 单点 | 简化设计，通过影子 Master 提供容错 |

**GFS 读写流程**：

```
GFS 写入流程：

Client          Master          Primary Replica    Secondary Replicas
   │               │                  │                   │
   │ 1. 请求Chunk │                  │                   │
   │──────────────►│                  │                   │
   │               │                  │                   │
   │ 2. 返回副本   │                  │                   │
   │ 位置和Primary │                  │                   │
   │◄──────────────│                  │                   │
   │               │                  │                   │
   │ 3. 推送数据到所有副本             │                   │
   │───────────────┼─────────────────►│───────────────────►│
   │               │                  │                   │
   │               │  4. 所有副本确认收到数据              │
   │◄──────────────┼──────────────────│◄──────────────────│
   │               │                  │                   │
   │               │  5. Client 向 Primary 发送写请求       │
   │───────────────┼─────────────────►│                   │
   │               │                  │                   │
   │               │  6. Primary 确定顺序，转发给 Secondary │
   │               │                  │──────────────────►│
   │               │                  │◄──────────────────│
   │               │                  │                   │
   │  7. 返回结果  │                  │                   │
   │◄──────────────┼──────────────────│                   │
```

### 33.2.4 HDFS（Hadoop Distributed File System）

HDFS 是 Apache Hadoop 项目的核心组件，是 GFS 的开源实现。

**HDFS 架构**：

```
HDFS 架构：

┌──────────────────────────────────────────────────┐
│                  NameNode (Active)                 │
│  ┌────────────────────────────────────────────┐  │
│  │  FsImage (文件系统元数据镜像)                │  │
│  │  EditLog (操作日志)                         │  │
│  │  Block 映射 (Block → DataNode)              │  │
│  └────────────────────────────────────────────┘  │
└──────────────────────┬───────────────────────────┘
                       │
         ┌─────────────┼─────────────┐
         ▼             ▼             ▼
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│  DataNode 1  │ │  DataNode 2  │ │  DataNode 3  │
│ ┌──────────┐ │ │ ┌──────────┐ │ │ ┌──────────┐ │
│ │ Block A  │ │ │ │ Block A  │ │ │ │ Block B  │ │
│ │ Block C  │ │ │ │ Block B  │ │ │ │ Block C  │ │
│ │ Block D  │ │ │ │ Block D  │ │ │ │ Block E  │ │
│ └──────────┘ │ │ └──────────┘ │ │ └──────────┘ │
└──────────────┘ └──────────────┘ └──────────────┘

Secondary NameNode: 定期合并 FsImage 和 EditLog
(不是热备，不能替代 NameNode)
```

**HDFS 的关键特性**：

| 特性 | HDFS | GFS |
|------|------|-----|
| 块大小 | 128 MB（默认） | 64 MB |
| 元数据管理 | NameNode（单点） | Master（单点） |
| 高可用 | NameNode HA（热备） | 影子 Master |
| 数据一致性 | 一次写入多次读取 | 松弛一致性 |
| 适用场景 | 大数据批处理 | Google 内部工作负载 |

## 33.3 时钟与时间

在分布式系统中，时间是一个复杂的问题。由于每台计算机都有自己的物理时钟，且这些时钟存在漂移，确定事件的全局顺序成为一项挑战。

### 33.3.1 物理时钟与 NTP 同步

**物理时钟的问题**：

```
时钟漂移现象：

理想情况（所有时钟完全同步）：
Node A:  |--1--|--2--|--3--|--4--|--5--|
Node B:  |--1--|--2--|--3--|--4--|--5--|
Node C:  |--1--|--2--|--3--|--4--|--5--|

实际情况（时钟漂移）：
Node A:  |--1--|--2--|--3--|--4--|--5--|
Node B:  |-1-|-2-|-3-|-4-|-5-|-6-|-7-|  (快了)
Node C:  |---1---|---2---|---3---|---4---|  (慢了)

晶振频率的微小差异会导致时钟逐渐偏离真实时间
典型漂移率：每天 1-10 毫秒
```

**NTP（Network Time Protocol）**：

NTP 是用于同步网络中各计算机时钟的协议。

```
NTP 同步原理：

Client                         NTP Server
   │                               │
   │  T1: 发送请求                  │
   │ ─────────────────────────────►│
   │                               │
   │                 T2: 服务器收到  │
   │                               │
   │                 T3: 服务器响应  │
   │◄──────────────────────────────│
   │  T4: 收到响应                  │
   │                               │

延迟计算：delay = (T4 - T1) - (T3 - T2)
偏移量：offset = ((T2 - T1) + (T3 - T4)) / 2

校正：Client 时钟 += offset
```

**NTP 层级（Stratum）**：

```
NTP 层级结构：

Stratum 0: 原子钟 / GPS 接收器（参考时钟）
    │
    ▼
Stratum 1: 直接连接 Stratum 0 的服务器
    │
    ▼
Stratum 2: 从 Stratum 1 同步的服务器
    │
    ▼
Stratum 3: 从 Stratum 2 同步的服务器
    ...
```

### 33.3.2 Lamport 逻辑时钟

**Lamport 逻辑时钟**（Lamport Logical Clock）由 Leslie Lamport 于 1978 年提出，用于为分布式系统中的事件建立偏序关系。

**核心思想**：

> 如果事件 a 发生在事件 b 之前（a → b），那么 LC(a) < LC(b)。

**Lamport 时钟规则**：

```
Lamport 时钟更新规则：

规则 1：本地事件
  - 在执行本地事件之前，LC = LC + 1

规则 2：发送消息
  - 在发送消息之前，LC = LC + 1
  - 将 LC 附加到消息中

规则 3：接收消息
  - 收到消息后，LC = max(本地LC, 消息中的LC) + 1
```

**Lamport 时钟示例**：

```
三个进程的 Lamport 时钟：

P1: a(1) ──────────── c(4) ──────────────── f(7)
                      │
P2:     b(2) ──────── │ ──── d(5)
                      │         │
P3:                  e(3) ──── │ ──────── g(6)
                               │
消息传递：
  P1 --c→ P2: 消息包含 LC=4
  P2 --d→ P3: 消息包含 LC=5

时间线：
  P1: a(1)────────c(4)────────f(7)
  P2:     b(2)────────d(5)
  P3:         e(3)───────g(6)

因果关系：a → c → d → g (LC: 1 → 4 → 5 → 6)
```

**Lamport 时钟的局限性**：

```
无法区分并发和因果关系：

场景：LC(a) < LC(b)，但 a 和 b 可能是并发的

P1: a(1) ────────────
P2:     b(2) ────────

问题：Lamport 时钟只能判断"发生在前"（→）
     无法判断"并发"（∥）
     即 LC(a) < LC(b) 不一定意味着 a → b
```

### 33.3.3 向量时钟（Vector Clocks）

**向量时钟**解决了 Lamport 时钟无法判断并发关系的问题。

**核心思想**：

> 每个进程维护一个向量，记录自己和其他进程的逻辑时间。通过比较向量，可以确定两个事件是因果关系还是并发关系。

**向量时钟规则**：

```
向量时钟更新规则（N 个进程）：

每个进程 i 维护一个向量 VC[1..N]

规则 1：本地事件
  - VC[i] = VC[i] + 1

规则 2：发送消息
  - VC[i] = VC[i] + 1
  - 将整个向量 VC 附加到消息中

规则 3：接收消息
  - 对于所有 j: VC[j] = max(VC[j], 消息中的VC[j])
  - VC[i] = VC[i] + 1
```

**向量时钟示例**：

```
三个进程的向量时钟：

P1: [1,0,0] ──────────── [4,0,0] ──────────── [7,0,0]
                         │
P2:     [0,2,0] ──────── │ ──── [4,5,0]
                         │              │
P3:                   [0,0,3] ──────── │ ──────── [4,5,6]

消息传递：
  P1 → P2: 消息包含 VC=[4,0,0]
  P2 → P3: 消息包含 VC=[4,5,0]

因果关系判断：
  [4,0,0] → [4,5,0] → [4,5,6]  (每个分量都 ≤)
  [1,0,0] ∥ [0,2,0]  (无法比较，并发关系)
```

**向量时钟 vs Lamport 时钟**：

| 特性 | Lamport 时钟 | 向量时钟 |
|------|-------------|---------|
| 数据结构 | 单一整数 | 大小为 N 的向量 |
| 空间开销 | O(1) | O(N) |
| 因果关系 | 只能判断"发生在前" | 可以精确判断因果和并发 |
| 实现复杂度 | 简单 | 较复杂 |
| 适用场景 | 简单排序需求 | 需要判断并发关系 |

### 33.3.4 因果关系

**因果关系（Causality）** 是分布式系统中理解事件顺序的核心概念。

**Happens-Before 关系（→）**：

```
Happens-Before 关系的定义（Lamport, 1978）：

1. 如果 a 和 b 是同一进程中的事件，且 a 在 b 之前发生，则 a → b

2. 如果 a 是发送消息的事件，b 是接收该消息的事件，则 a → b

3. 传递性：如果 a → b 且 b → c，则 a → c

4. 如果 a ↛ b 且 b ↛ a，则 a ∥ b（a 和 b 是并发的）
```

**因果一致性**：

```
因果一致性要求：

如果事件 a → b，那么所有进程必须以相同的顺序观察到 a 和 b

违反因果一致性的例子：

Process 1: write(x, 1) → send("x 被更新了")
Process 2: recv("x 被更新了") → read(x)

如果 Process 2 看到消息但 read(x) 得到旧值，
则违反了因果一致性
```

## 33.4 一致性模型

一致性模型定义了分布式系统中数据访问的语义保证，是理解分布式存储系统的关键。

### 33.4.1 强一致性（Strong Consistency）

**强一致性**（也称为线性一致性 Linearizability）是最强的一致性保证。

**定义**：

> 所有操作看起来像是在某个单一时间点原子地执行，且所有客户端都看到相同的操作顺序。

```
强一致性示例：

初始状态：x = 0

Client A: write(x, 1) ──────────────────────────►
Client B: ──────── read(x) → 1  (保证看到最新值)
Client C: ────────────── read(x) → 1

时间线：
t0: x = 0
t1: write(x, 1) 完成
t2: 任何 read(x) 都返回 1

强一致性保证：一旦写入完成，所有后续读取都能看到新值
```

**实现强一致性的方法**：

```
方法 1：两阶段提交 (2PC)

Coordinator          Participants
     │                    │
     │  PREPARE           │
     │───────────────────►│
     │                    │
     │  VOTE YES/NO       │
     │◄───────────────────│
     │                    │
     │  COMMIT/ABORT      │
     │───────────────────►│
     │                    │
     │  ACK               │
     │◄───────────────────│

方法 2：多数派协议 (Quorum)

写操作：W 个副本确认
读操作：R 个副本响应
保证：W + R > N（N 为总副本数）

例：N=3, W=2, R=2 → 强一致性
```

### 33.4.2 最终一致性（Eventual Consistency）

**最终一致性**是最弱的有用一致性模型，但提供了最高的可用性和性能。

**定义**：

> 如果没有新的更新，最终所有副本将收敛到相同的值。

```
最终一致性示例：

初始状态：x = 0

Client A: write(x, 1) ──────────────────────────►
Client B: ──────── read(x) → 0  (可能看到旧值)
Client C: ────────────── read(x) → 0  (可能看到旧值)

        ... 经过一段时间（不一致窗口）...

Client B: ────────────────────────── read(x) → 1
Client C: ───────────────────────────── read(x) → 1

最终一致性保证：最终所有读取都会返回 1
不保证：何时收敛，中间状态可能看到旧值
```

**最终一致性的变体**：

| 变体 | 描述 |
|------|------|
| **因果一致性** | 保证因果相关操作的顺序 |
| **读你所写 (Read-your-writes)** | 用户总能看到自己的写入 |
| **会话一致性 (Session Consistency)** | 在会话内保证因果一致性 |
| **单调读 (Monotonic Reads)** | 读取不会返回更旧的值 |
| **单调写 (Monotonic Writes)** | 写入按顺序执行 |

### 33.4.3 因果一致性（Causal Consistency）

**因果一致性**是介于强一致性和最终一致性之间的折中方案。

**定义**：

> 如果操作 A 因果地影响了操作 B（A → B），则所有进程都必须以 A 在 B 之前的顺序观察到这两个操作。并发操作可以被不同进程以不同顺序观察。

```
因果一致性示例：

初始状态：x = 0, y = 0

Process 1: write(x, 1)
           ↓ (因果关系：消息传递)
Process 2: read(x) → 1
           write(y, 1)

因果一致性保证：
  - 所有进程必须看到 write(x, 1) 在 write(y, 1) 之前
  - 如果某个进程看到 write(y, 1)，它必须也看到 write(x, 1)

违反因果一致性的场景：
  Process 3: read(y) → 1, read(x) → 0
  (看到 y 的新值但没看到 x 的新值，违反了因果关系)
```

**一致性模型的对比**：

```
一致性强度排序：

强一致性 (Linearizability)
    ▲
    │
顺序一致性 (Sequential Consistency)
    ▲
    │
因果一致性 (Causal Consistency)
    ▲
    │
最终一致性 (Eventual Consistency)

强度越高 → 一致性越好，但性能和可用性越差
强度越低 → 性能和可用性越好，但一致性越弱
```

## 33.5 分布式协调

分布式协调是分布式系统中最困难的问题之一，涉及多个节点就某个值达成一致。

### 33.5.1 共识问题

**共识问题（Consensus Problem）** 的定义：

> 多个进程（其中某些可能故障）需要就某个值达成一致的决策。

**共识的四个性质**：

| 性质 | 描述 |
|------|------|
| **终止性 (Termination)** | 每个正确的进程最终都会做出决定 |
| **协定性 (Agreement)** | 所有正确的进程做出相同的决定 |
| **完整性 (Integrity)** | 如果所有进程提议相同的值，则决定该值 |
| **有效性 (Validity)** | 决定的值必须是某个进程提议的值 |

**FLP 不可能定理**：

> 在异步系统中，即使只有一个进程可能崩溃，也不存在确定性的共识算法能保证在有限时间内达成共识。

```
FLP 不可能定理的影响：

实际系统中的解决方案：
1. 引入超时机制（部分同步模型）
2. 使用随机化算法
3. 故障检测器（不完美但实用）

→ Paxos 和 Raft 都是基于部分同步假设的解决方案
```

### 33.5.2 Paxos 算法

**Paxos** 由 Leslie Lamport 于 1989 年提出，是最经典的共识算法。

**Paxos 的三个角色**：

```
Paxos 角色：

┌─────────────────────────────────────────────────┐
│  Proposer (提议者)                               │
│  - 提出提案 (Proposal)                           │
│  - 每个提案有唯一编号 (Proposal Number)           │
├─────────────────────────────────────────────────┤
│  Acceptor (接受者)                               │
│  - 投票接受或拒绝提案                             │
│  - 必须接受收到的第一个提案                        │
├─────────────────────────────────────────────────┤
│  Learner (学习者)                                │
│  - 学习最终达成共识的值                           │
│  - 不参与投票过程                                 │
└─────────────────────────────────────────────────┘
```

**Paxos 的两阶段**：

```
Phase 1: Prepare (准备阶段)

Proposer                    Acceptors
    │                          │
    │  Prepare(n)              │
    │─────────────────────────►│
    │                          │
    │  Promise(n, prev_accepted_value)
    │◄─────────────────────────│
    │                          │
    │  如果收到多数派的响应      │
    │  进入 Phase 2            │

Phase 2: Accept (接受阶段)

Proposer                    Acceptors
    │                          │
    │  Accept(n, value)        │
    │─────────────────────────►│
    │                          │
    │  Accepted(n, value)      │
    │◄─────────────────────────│
    │                          │
    │  如果收到多数派的接受      │
    │  共识达成！               │
```

**Paxos 的挑战**：

```
活锁问题 (Livelock)：

Proposer 1: Prepare(1) ──────────►
Proposer 2: ◄────────── Promise(1)
            Prepare(2) ──────────►
Proposer 1: ◄────────── Promise(2)
            Prepare(3) ──────────►
Proposer 2: ◄────────── Promise(3)
            Prepare(4) ──────────►
...

两个 Proposer 不断互相抢占，导致无法达成共识
解决方案：引入随机退避、Leader 选举等
```

### 33.5.3 Raft 算法

**Raft** 由 Diego Ongaro 和 John Ousterhout 于 2014 年提出，设计目标是比 Paxos 更容易理解。

**Raft 的核心概念**：

```
Raft 节点的三种状态：

┌─────────────────────────────────────────────────┐
│  Follower (追随者)                               │
│  - 初始状态                                      │
│  - 接收 Leader 的心跳和日志                       │
│  - 超时未收到心跳 → 转为 Candidate                │
├─────────────────────────────────────────────────┤
│  Candidate (候选人)                              │
│  - 发起选举                                       │
│  - 收到多数派投票 → 转为 Leader                   │
│  - 发现更高任期的 Leader → 转为 Follower          │
├─────────────────────────────────────────────────┤
│  Leader (领导者)                                 │
│  - 处理所有客户端请求                              │
│  - 定期发送心跳维持权威                           │
│  - 发现更高任期 → 转为 Follower                   │
└─────────────────────────────────────────────────┘
```

#### Leader 选举

```
Raft Leader 选举流程：

初始状态：所有节点都是 Follower

Follower A    Follower B    Follower C
    │             │             │
    │  超时!       │             │
    │             │             │
    ▼             │             │
Candidate A       │             │
    │             │             │
    │  RequestVote(term=1)      │
    │────────────►│────────────►│
    │             │             │
    │  VoteGranted │  VoteGranted │
    │◄────────────│◄────────────│
    │             │             │
    ▼             │             │
Leader A          │             │
    │             │             │
    │  AppendEntries (心跳)     │
    │────────────►│────────────►│
    │             │             │
    │  收到心跳，重置选举超时    │
    │             │             │

选举超时：150-300ms（随机化避免同时选举）
```

**任期（Term）机制**：

```
任期的概念：

Term 1:    Leader A
           ──────────────
Term 2:         Leader B
                 ──────────────
Term 3:              Leader A
                      ──────────────

每个任期最多有一个 Leader
任期号单调递增
发现更高任期的节点会立即转为 Follower
```

#### 日志复制

```
Raft 日志复制流程：

Leader                    Followers
  │                          │
  │  客户端请求               │
  │  (SET x = 5)             │
  │                          │
  │  AppendEntries            │
  │  (term=1, index=1,       │
  │   entry=SET x=5)         │
  │─────────────────────────►│
  │                          │
  │  成功响应                  │
  │◄─────────────────────────│
  │                          │
  │  多数派确认后提交          │
  │  (commitIndex = 1)       │
  │                          │
  │  AppendEntries            │
  │  (leaderCommit=1)        │
  │─────────────────────────►│
  │                          │
  │  Followers 也提交          │

日志结构：
┌───────┬───────┬───────┬───────┐
│ index │ term  │ entry │ commit│
├───────┼───────┼───────┼───────┤
│   1   │   1   │ x=5   │  ✓   │
│   2   │   1   │ y=3   │  ✓   │
│   3   │   2   │ z=7   │      │
└───────┴───────┴───────┴───────┘
```

#### 安全性

**Raft 的安全性保证**：

```
安全性规则：

1. 选举安全：每个任期最多选出一个 Leader

2. 日志匹配：
   - 如果两个日志在某个 index 处有相同的 term
   - 则该 index 及之前的所有 entry 都相同

3. Leader 完整性：
   - 如果某个 entry 在某个 term 被提交
   - 则该 entry 会出现在所有更高 term 的 Leader 的日志中

4. 状态机安全：
   - 如果某个节点在某个 index 处应用了一个 entry
   - 则其他节点在该 index 处不会应用不同的 entry
```

### 33.5.4 ZooKeeper

**ZooKeeper** 是 Apache 开源的分布式协调服务，基于 ZAB 协议（类似 Raft）。

**ZooKeeper 的数据模型**：

```
ZooKeeper 的命名空间（类似文件系统）：

/
├── /app1
│   ├── /app1/config
│   │   └── data: "db_url=..."
│   └── /app1/leader
│       └── data: "node1"
├── /app2
│   ├── /app2/workers
│   │   ├── worker1
│   │   ├── worker2
│   │   └── worker3
│   └── /app2/tasks
└── /services
    └── /services/discovery
        └── data: "192.168.1.1:8080"

ZNode 类型：
- 持久节点 (Persistent)：显式删除才消失
- 临时节点 (Ephemeral)：客户端会话结束自动删除
- 顺序节点 (Sequential)：自动追加递增编号
```

**ZooKeeper 的典型用途**：

```
用途 1：Leader 选举

/workers
  ├── worker-0000000001 (临时顺序节点)
  ├── worker-0000000002
  └── worker-0000000003

最小编号的节点成为 Leader
如果 Leader 崩溃（临时节点消失），下一个节点成为 Leader

用途 2：配置管理

/config
  └── db_url = "jdbc:mysql://..."

所有节点监听 /config 的变化
配置更新时，ZooKeeper 通知所有监听者

用途 3：分布式锁

/locks
  └── resource-0000000001 (临时顺序节点)

所有竞争者创建顺序节点
最小编号的节点获得锁
释放锁时删除节点，下一个节点被通知
```

**ZooKeeper 的一致性保证**：

```
ZooKeeper 的一致性模型：

写操作：
  - 通过 Leader 顺序执行
  - 多数派确认后返回
  - 保证强一致性

读操作：
  - 可以从任何节点读取
  - 可能读到旧数据（非 Leader 节点）
  - sync() 操作可以保证读取最新数据

Session 保证：
  - 同一个客户端的操作按 FIFO 顺序执行
  - 客户端总是看到自己的写入
```

## 33.6 面试高频考点

### 考点 1：CAP 定理

**Q1: 为什么分布式系统不能同时满足 CAP 三个属性？**

A: 因为分布式系统中网络分区（P）不可避免。当网络分区发生时，系统必须在一致性（C）和可用性（A）之间做出选择：
- 选择 C：拒绝可能不一致的请求（牺牲 A）
- 选择 A：允许返回可能过时的数据（牺牲 C）

**Q2: CAP 定理中的"三选二"是否准确？**

A: 不准确。更准确的说法是：在网络分区发生时，在 C 和 A 之间做权衡。实际上，一致性是一个连续的光谱，而非二元选择。许多系统提供可调节的一致性级别。

### 考点 2：一致性模型

**Q3: 强一致性和最终一致性的区别是什么？**

A: 
- 强一致性：写入完成后，所有后续读取都能看到新值
- 最终一致性：写入完成后，可能在一段时间内读到旧值，但最终会收敛

**Q4: 什么时候应该选择最终一致性？**

A: 当系统需要高可用性和高性能，且业务允许短暂不一致时。例如：
- 社交网络的点赞数
- 购物车（可以合并冲突）
- CDN 缓存

### 考点 3：共识算法

**Q5: Paxos 和 Raft 的主要区别是什么？**

A: 
- Paxos 更理论化，难以理解和实现
- Raft 设计目标是易于理解，通过 Leader 选举、日志复制、安全性三个子问题分解共识问题
- Raft 强制所有请求通过 Leader，简化了日志复制

**Q6: Raft 如何保证安全性？**

A: 通过以下机制：
1. 选举安全：每个任期最多一个 Leader
2. 日志匹配：相同 index 和 term 的 entry 必须相同
3. Leader 完整性：已提交的 entry 会出现在所有后续 Leader 的日志中
4. 前 Leader 不能提交之前任期的 entry（通过提交当前任期的 entry 间接提交）

### 考点 4：分布式文件系统

**Q7: NFS 和 AFS 的主要区别是什么？**

A: 
- NFS：无状态协议，块级缓存，超时检查一致性
- AFS：有状态协议，全文件缓存，回调机制保证一致性

**Q8: GFS 为什么选择单 Master 架构？**

A: 简化设计，避免复杂的分布式元数据管理。通过以下方式缓解单点问题：
- Master 只存储元数据，不处理数据流
- 使用影子 Master 提供只读容错
- 元数据操作相对较少，不会成为瓶颈

### 考点 5：时钟同步

**Q9: Lamport 时钟和向量时钟的区别是什么？**

A: 
- Lamport 时钟：单一整数，只能判断偏序关系（"发生在前"），无法判断并发
- 向量时钟：大小为 N 的向量，可以精确判断因果关系和并发关系

**Q10: 为什么需要逻辑时钟？物理时钟不够吗？**

A: 物理时钟存在漂移，无法精确同步。逻辑时钟基于事件的因果关系，不依赖物理时间，更适合分布式系统中的事件排序。

## 33.7 扩展阅读

### 推荐论文

1. **"Time, Clocks, and the Ordering of Events in a Distributed System"** - Lamport, 1978
   - 分布式系统领域最重要的论文之一
   - 定义了 Happens-Before 关系和逻辑时钟

2. **"The Google File System"** - Ghemawat et al., 2003
   - GFS 的设计论文
   - 影响了 HDFS 等后续系统

3. **"In Search of an Understandable Consensus Algorithm"** - Ongaro & Ousterhout, 2014
   - Raft 算法的原始论文
   - 比 Paxos 更易于理解

4. **"A Brief History of Consensus"** - Rodrigues et al., 2011
   - 共识算法的历史综述

### 推荐书籍

1. **"Designing Data-Intensive Applications"** - Martin Kleppmann
   - 分布式系统设计的实用指南
   - 涵盖一致性、复制、分区等核心话题

2. **"Distributed Systems: Principles and Paradigms"** - Tanenbaum & Van Steen
   - 分布式系统的经典教材
   - 理论与实践结合

3. **"Distributed Algorithms"** - Nancy Lynch
   - 分布式算法的理论基础
   - 适合深入研究

### 在线资源

- **MIT 6.824: Distributed Systems** - 经典分布式系统课程
- **The Raft Consensus Algorithm** - raft.github.io
- **Jepsen** - 分布式系统一致性测试

> **本章总结**：
> - 分布式系统面临部分失败、网络不可靠、时钟不同步等核心挑战
> - CAP 定理约束了分布式系统设计的一致性和可用性权衡
> - 分布式文件系统（NFS/AFS/GFS/HDFS）有不同的设计理念和适用场景
> - 逻辑时钟和向量时钟是分布式系统中事件排序的核心工具
> - 一致性模型从强一致性到最终一致性是一个连续的光谱
> - Paxos 和 Raft 是解决分布式共识问题的核心算法
> - ZooKeeper 等协调服务为分布式系统提供了基础的协调原语
