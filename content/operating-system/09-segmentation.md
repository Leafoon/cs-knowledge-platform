## 9.1 分段机制

### 9.1.1 分段的动机：逻辑划分与保护

分页将地址空间划分为固定大小的页，这是一种**物理视角**的划分——页没有逻辑含义，只是物理内存的管理单位。但在程序的逻辑视角中，地址空间由几个**逻辑段**组成：代码段、数据段、堆段、栈段等。这些段的大小不同，增长方向不同，访问权限也不同。

**分段（Segmentation）** 的思想是：按照程序的**逻辑结构**将地址空间划分为若干个**段（Segment）**，每个段有独立的基址、界限和访问权限。

```
逻辑地址空间（分段视角）：

┌─────────────────┐
│  代码段          │  基址=0x1000, 界限=0x2000, 权限=r-x
│  (只读可执行)    │
├─────────────────┤
│  数据段          │  基址=0x5000, 界限=0x1000, 权限=rw-
│  (可读写)        │
├─────────────────┤
│  堆段            │  基址=0x8000, 界限=0x3000, 权限=rw-
│  (向上增长)      │
├─────────────────┤
│  栈段            │  基址=0xF000, 界限=0x1000, 权限=rw-
│  (向下增长)      │
└─────────────────┘
```

分段的优势在于它匹配了程序的逻辑结构：
- 每个段有独立的保护权限（代码段不可写，数据段不可执行）
- 段可以独立增长或缩小（堆和栈可以动态调整）
- 共享库可以作为独立的段在进程间共享

### 9.1.2 段表（Segment Table）

段表是分段机制的核心数据结构。每个条目描述一个段：

```
段表条目：
┌──────────┬──────────┬────────┬──────────┐
│ Base     │ Limit    │ Flags  │ Present  │
│ 段基址   │ 段界限   │ 权限位 │ 存在位   │
└──────────┴──────────┴────────┴──────────┘
```

- **Base**：段在物理内存中的起始地址
- **Limit**：段的长度（字节数或页数）
- **Flags**：访问权限（读/写/执行）、方向（向上/向下增长）
- **Present**：段是否在内存中（支持段的换出）

### 9.1.3 段地址翻译过程

```
逻辑地址 = (段选择器, 段内偏移)

翻译步骤：
1. 用段选择器索引段表
   SEG = 段表[段选择器]

2. 检查偏移是否在段界限内
   如果 offset ≥ SEG.Limit → 段错误（Segmentation Fault）

3. 检查访问权限
   如果权限不满足 → 段错误

4. 计算物理地址
   物理地址 = SEG.Base + offset
```

```
示例：
段表：
  段 0 (代码): Base=0x1000, Limit=0x2000, R-X
  段 1 (数据): Base=0x5000, Limit=0x1000, RW-
  段 2 (堆):   Base=0x8000, Limit=0x3000, RW-

逻辑地址 (段=1, 偏移=0x0500):
  SEG = 段表[1] = {Base=0x5000, Limit=0x1000}
  0x0500 < 0x1000 → 界限检查通过
  物理地址 = 0x5000 + 0x0500 = 0x5500

逻辑地址 (段=1, 偏移=0x1500):
  SEG = 段表[1] = {Base=0x5000, Limit=0x1000}
  0x1500 ≥ 0x1000 → 段错误！偏移超出段界限
```

## 9.2 分段的优缺点

### 9.2.1 优点

**逻辑分离**：分段匹配程序的逻辑结构，每个段有独立的含义（代码、数据、堆、栈）。这使得保护和共享更加自然——你可以共享整个代码段，但不共享数据段。

**自然的保护**：每个段有独立的访问权限。代码段可以设为只读可执行（防止代码被篡改），数据段可以设为可读写不可执行（防止数据被当作代码执行）。这比细粒度的分页保护更直观。

**支持共享**：多个进程可以共享同一个段。例如，共享库的代码段在物理内存中只有一份，多个进程的段表都指向同一物理地址。

**支持动态增长**：堆段和栈段可以独立增长。当堆需要更多空间时，操作系统只需分配新的物理内存并更新段的界限，不需要移动其他段。

### 9.2.2 缺点

**外部碎片严重**：段的大小是可变的，物理内存中会散布着各种大小的空闲块。当进程终止后留下的空闲区域可能太小，无法被新进程使用。这与连续内存分配的外部碎片问题相同。

**管理复杂**：需要维护每段的基址和界限，分配和释放算法比固定大小的分页复杂得多。

**不支持稀疏地址空间**：如果一个段很大但只使用了一小部分，整个段都需要分配连续的物理内存，浪费空间。

### 9.2.3 分段 vs 分页对比

| 维度 | 分段 | 分页 |
|------|------|------|
| **划分方式** | 按逻辑结构（可变大小） | 按物理大小（固定大小） |
| **外部碎片** | 严重 | 无 |
| **内部碎片** | 无 | 有（最后一页可能未满） |
| **共享** | 按逻辑段共享 | 按页共享 |
| **保护** | 段级保护（粗粒度） | 页级保护（细粒度） |
| **地址翻译** | 基址 + 偏移 | 页表查表 |
| **硬件支持** | 段表（段基址+界限寄存器） | 页表（PTBR） |
| **现代使用** | 退化（x86-64 平坦模式） | 主流（所有现代处理器） |

## 9.3 段页式管理

### 9.3.1 结合分段与分页的优势

段页式管理（Segmented Paging）结合了分段和分页的优点：

- **第一级：分段**——按逻辑结构划分地址空间，提供逻辑保护和共享
- **第二级：分页**——将每个段划分为固定大小的页，消除外部碎片

```
逻辑地址 = (段选择器, 页号, 页内偏移)

翻译过程：
1. 段选择器 → 段表 → 段基址
2. 段基址 + 页号 → 页表 → 物理页框号
3. 物理页框号 × 页大小 + 页内偏移 → 物理地址
```

### 9.3.2 x86 保护模式的段页式结构

x86 保护模式（32 位）使用段页式管理：

```
逻辑地址（x86 保护模式）：
┌──────────────────┬─────────────┬─────────────┬────────────┐
│ 段选择器(16位)    │ 段内偏移(32位)                                    │
└──────────────────┴─────────────┴─────────────┴────────────┘

翻译过程：
1. 段选择器 → GDT/LDT → 段描述符 → 段基址
2. 线性地址 = 段基址 + 段内偏移
3. 如果分页启用：线性地址 → 页表 → 物理地址
4. 如果分页未启用：线性地址 = 物理地址
```

### 9.3.3 GDT 与 LDT

**GDT（Global Descriptor Table）**：全局描述符表，存储系统级段描述符（内核代码段、内核数据段等）。每个系统只有一个 GDT。

**LDT（Local Descriptor Table）**：局部描述符表，存储进程级段描述符。每个进程可以有自己的 LDT。

段选择器的结构：
```
段选择器（16 位）：
┌─────────┬─────┬─────────┐
│ Index(13)│ TI(1)│ RPL(2) │
│ 描述符索引│表指示│请求特权级│
└─────────┴─────┴─────────┘
```

- **Index**：在 GDT 或 LDT 中的索引
- **TI**：0 = GDT，1 = LDT
- **RPL**：Requested Privilege Level（0=内核，3=用户）

## 9.4 现代系统的分段

### 9.4.1 x86-64 的平坦内存模型

在 x86-64（64 位模式）中，分段机制基本**退化**了。所有段寄存器（CS、DS、SS、ES）的基址被设为 0，界限被设为最大值。这意味着段地址翻译变成了一次恒等映射：`线性地址 = 0 + 偏移 = 偏移`。

```
x86-64 平坦内存模型：
CS.Base = 0, CS.Limit = MAX
DS.Base = 0, DS.Limit = MAX
SS.Base = 0, SS.Limit = MAX

→ 分段退化为恒等映射
→ 线性地址 = 逻辑地址
→ 所有保护由分页机制提供
```

为什么 x86-64 保留了分段但将其退化？
- **向后兼容**：x86-64 需要兼容 32 位和 16 位代码，这些代码使用分段
- **硬件复杂性**：完全移除分段需要重新设计指令集
- **成本低**：将段基址设为 0 几乎没有性能开销

### 9.4.2 FS/GS 段寄存器的特殊用途

虽然大部分段寄存器的基址被设为 0，但 **FS** 和 **GS** 段寄存器的基址可以被设置为非零值。这两个寄存器在现代系统中有特殊用途：

**线程局部存储（TLS）**：FS 段寄存器的基址指向当前线程的 TLS 区域。访问 `__thread` 变量时，编译器生成 `FS:` 前缀的内存访问指令。

```c
// C 代码
__thread int tls_var = 42;

// 编译后（x86-64 汇编）
mov eax, fs:[tls_var_offset]  // 通过 FS 段访问 TLS 变量
```

**系统调用中的内核栈指针**：在 Linux 中，GS 基址用于在系统调用入口快速找到当前 CPU 的内核栈。

```c
// Linux 系统调用入口（x86-64）
swapgs                    // 切换 GS 基址（用户 GS ↔ 内核 GS）
mov rsp, gs:[kernel_stack_offset]  // 加载内核栈指针
```

FS/GS 基址的设置通过 `arch_prctl()` 系统调用（Linux）或 `wrfsbase`/`wrgsbase` 指令完成。

## 9.5 面试高频考点

**Q1：分段和分页的区别？**

分段按逻辑结构划分（可变大小），提供逻辑保护和共享，但有外部碎片。分页按物理大小划分（固定大小），消除外部碎片，但粒度较粗。现代系统主要使用分页，分段在 x86-64 中已退化。

**Q2：为什么 x86-64 基本废弃了分段？**

平坦内存模型更简单高效。将所有段基址设为 0，界限设为最大，分段退化为恒等映射。所有内存保护由分页机制提供（页表中的 R/W/X 位）。分段只保留用于 TLS（FS/GS 寄存器）。

**Q3：段页式管理的地址翻译过程？**

两级翻译：首先用段选择器查段表得到段基址，加上段内偏移得到线性地址；然后用线性地址查页表得到物理地址。x86 保护模式使用这种机制。

**Q4：FS/GS 寄存器在现代系统中的作用？**

FS 用于线程局部存储（TLS），基址指向当前线程的 TLS 区域。GS 用于系统调用中快速定位内核栈。这两个寄存器是分段机制在现代系统中唯一有意义的用途。
