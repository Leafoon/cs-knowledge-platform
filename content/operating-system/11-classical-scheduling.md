## 11.1 先来先服务（FCFS / FIFO）

### 11.1.1 算法描述

FCFS（First-Come First-Served）是最简单的调度算法：按照进程到达就绪队列的顺序执行，先到的先运行。进程一旦获得 CPU，就一直运行直到完成或阻塞。

```c
// FCFS 调度伪代码
void fcfs_scheduler(void) {
    while (!empty(ready_queue)) {
        Process *p = dequeue(ready_queue);  // 取队头进程
        run(p);  // 运行直到完成或阻塞
    }
}
```

FCFS 使用一个简单的 FIFO 队列：
- 新进程到达时加入队尾
- 调度时从队头取出进程
- 进程运行直到完成或阻塞（非抢占式）

### 11.1.2 性能分析

**示例 1**：三个进程同时到达，执行时间分别为 A=24, B=3, C=3。

```
FCFS 调度（按 A→B→C 顺序）：
A: [===========执行 24ms ===========]
B:                                  [==执行 3ms==]
C:                                         [==执行 3ms==]
时间：0                          24    27    30

各进程指标：
  进程  执行时间  完成时间  周转时间  等待时间  响应时间
  A     24        24        24        0         0
  B     3         27        27        24        24
  C     3         30        30        27        27
  平均            -         27        17        17
```

如果改变顺序为 B→C→A：

```
B: [==执行 3ms==]
C:        [==执行 3ms==]
A:              [===========执行 24ms ===========]
时间：0  3    6              30

各进程指标：
  进程  执行时间  完成时间  周转时间  等待时间  响应时间
  B     3         3         3         0         0
  C     3         6         6         3         3
  A     24        30        30        6         6
  平均            -         13        3         3
```

**关键洞察**：仅改变进程顺序，平均周转时间从 27 降低到 13（减少 52%），平均等待时间从 17 降低到 3（减少 82%）。这说明 FCFS 的性能严重依赖于进程到达顺序和执行时间。

### 11.1.3 护航效应（Convoy Effect）

FCFS 的一个严重问题是**护航效应**：一个长进程在前，后面所有短进程都被迫等待，导致平均等待时间急剧增加。同时，I/O 设备空闲（短 I/O 密集型进程在等待），CPU 被一个长 CPU 密集型进程独占。

```
护航效应示例：
长进程 L(100ms) 在前，短进程 S1(1ms), S2(1ms), S3(1ms) 在后

L: [================================100ms=================================]
S1:                                                                      [=1=]
S2:                                                                         [=1=]
S3:                                                                            [=1=]
时间：0                                    100   101   102   103

平均等待时间 = (0 + 100 + 101 + 102) / 3 = 101ms

如果短进程在前：
S1: [=1=]
S2:     [=1=]
S3:         [=1=]
L:              [================================100ms=================================]
时间：0 1  2  3              103

平均等待时间 = (0 + 1 + 2 + 3) / 3 = 2ms
```

护航效应的影响：
- **CPU 利用率下降**：短进程等待时，它们的 I/O 设备空闲
- **平均等待时间急剧增加**：从 2ms 增加到 101ms
- **响应性极差**：短进程（通常是交互式进程）等待很长时间

### 11.1.4 FCFS 的优缺点

| 优点 | 缺点 |
|------|------|
| 实现极其简单（FIFO 队列） | 护航效应 |
| 公平（先到先服务） | 平均等待时间长 |
| 无饥饿（每个进程最终都会运行） | 不适合交互式系统 |
| 无额外开销（不需要预测执行时间） | 长进程阻塞短进程 |

**适用场景**：简单批处理系统，进程执行时间相近的场景，或者作为其他算法的组成部分（如 MLFQ 的最底层队列）。

## 11.2 最短作业优先（SJF）

### 11.2.1 算法描述

SJF（Shortest Job First）选择**执行时间最短**的进程运行。如果有多个进程执行时间相同，则使用 FCFS 打破平局。

```c
// SJF 调度伪代码
void sjf_scheduler(void) {
    while (!empty(ready_queue)) {
        // 选择执行时间最短的进程
        Process *p = find_shortest(ready_queue);
        remove(ready_queue, p);
        run(p);  // 运行直到完成（非抢占式）
    }
}
```

### 11.2.2 SJF 最优性证明

**定理**：在所有进程同时到达的情况下，SJF 最小化平均等待时间。

**证明**（反证法 + 交换论证）：

假设存在一个非 SJF 的调度顺序 S，其平均等待时间小于 SJF。

设 S 中有两个相邻的进程 P_i 和 P_j，其中 P_i 在 P_j 之前，但 P_i 的执行时间 > P_j 的执行时间（即 t_i > t_j）。

在 S 中：
- P_i 的等待时间 = w_i
- P_j 的等待时间 = w_i + t_i

如果我们交换 P_i 和 P_j（P_j 在前）：
- P_j 的等待时间 = w_i（减少 t_i）
- P_i 的等待时间 = w_i + t_j（增加 t_j）

总等待时间的变化：
```
Δ = (w_i + t_j) - (w_i + t_i) = t_j - t_i < 0（因为 t_i > t_j）
```

交换后总等待时间减少了 (t_i - t_j) > 0。

因此，任何违反 SJF 顺序的调度都可以通过交换相邻的长短进程来减少总等待时间。反复交换最终收敛到 SJF 顺序。SJF 是最优的。□

**推论**：SJF 也最小化平均周转时间，因为 周转时间 = 等待时间 + 执行时间，而执行时间的总和是固定的。

### 11.2.3 手算示例

```
进程：A(执行=6), B(执行=2), C(执行=8), D(执行=3)
所有进程同时到达

SJF 调度：B(2) → D(3) → A(6) → C(8)

B: [==2ms==]
D:        [==3ms==]
A:              [======6ms======]
C:                           [========8ms========]
时间：0  2    5          11              19

各进程指标：
  进程  执行时间  完成时间  周转时间  等待时间
  B     2         2         2         0
  D     3         5         5         2
  A     6         11        11        5
  C     8         19        19        11
  平均            -         9.25      4.5

对比 FCFS（A→B→C→D）：
  平均周转时间 = 12.25（SJF 为 9.25，减少 24%）
  平均等待时间 = 7.5（SJF 为 4.5，减少 40%）
```

### 11.2.4 饥饿问题

SJF 可能导致**饥饿（Starvation）**：如果不断有短进程到达，长进程可能永远无法获得 CPU。

```
时刻 0：长进程 L(100ms) 到达
时刻 1：短进程 S1(1ms) 到达
  → SJF 选择 S1（1ms < 100ms）
时刻 2：S1 完成，S2(1ms) 到达
  → SJF 选择 S2（1ms < 100ms）
...
时刻 100：S100(1ms) 到达
  → SJF 选择 S100（1ms < 100ms）
→ L 可能永远无法运行！
```

在实际系统中，如果不断有短交互式进程到达（如用户不断点击按钮），后台的长批处理进程可能永远得不到 CPU。这就是为什么 SJF 在实际系统中很少直接使用。

### 11.2.5 SJF 的实际应用

虽然纯粹的 SJF 很少使用，但它的思想被广泛应用：

**Linux CFS 的 nice 值**：高优先级（低 nice 值）的进程获得更多 CPU 时间，类似 SJF 的效果。

**MLFQ 的优先级降低**：CPU 密集型进程逐渐降低优先级，短进程保持高优先级，实现了类似 SJF 的效果。

**估算执行时间**：如前所述，使用 EWMA 预测进程的下一次执行时间，然后按预测时间排序。

## 11.3 最短剩余时间优先（SRTF / STCF）

### 11.3.1 SJF 的抢占式版本

SRTF（Shortest Remaining Time First）是 SJF 的抢占式版本：每当新进程到达时，调度器比较当前进程的**剩余时间**和新进程的执行时间，选择剩余时间最短的运行。

```c
// SRTF 调度伪代码
void srtf_scheduler(void) {
    while (!empty(ready_queue) || current != NULL) {
        // 检查是否有新进程到达
        check_new_arrivals();
        
        // 选择剩余时间最短的进程
        Process *p = find_shortest_remaining(ready_queue, current);
        if (p != current) {
            preempt(current);  // 抢占当前进程
            run(p);
        }
    }
}
```

### 11.3.2 详细示例

```
进程：A(到达=0, 执行=7), B(到达=2, 执行=4), C(到达=4, 执行=1), D(到达=5, 执行=4)

时刻 0：只有 A 就绪，A 剩余=7 → 运行 A
时刻 2：B 到达。A 剩余=5, B 剩余=4 → B 更短，抢占 A，运行 B
时刻 4：C 到达。B 剩余=2, C 剩余=1 → C 更短，抢占 B，运行 C
时刻 5：C 完成。D 到达。就绪：{A(5), B(2), D(4)}
  → B 剩余最短(2)，运行 B
时刻 7：B 完成。就绪：{A(5), D(4)}
  → D 剩余最短(4)，运行 D
时刻 11：D 完成。就绪：{A(5)}
  → 运行 A
时刻 16：A 完成

Gantt 图：
A: [=2=]                              [======5ms======]
B:     [==2=][==2==]
C:           [=1=]
D:                [====4ms====]
0   2   4   5   7                  11              16

各进程指标：
  进程  到达  执行  完成  周转时间  等待时间  响应时间
  A     0     7     16    16        9         0
  B     2     4     7     5         1         0
  C     4     1     5     1         0         0
  D     5     4     11    6         2         2
  平均                -     7         3         0.5
```

### 11.3.3 SRTF vs SJF 对比

```
同样的进程，非抢占式 SJF：

时刻 0：只有 A → 运行 A（直到完成）
时刻 7：A 完成。就绪：{B(4), C(1), D(4)}
  → C 最短(1)，运行 C
时刻 8：C 完成。就绪：{B(4), D(4)}
  → B 先到（FCFS），运行 B
时刻 12：B 完成 → 运行 D
时刻 16：D 完成

Gantt 图：
A: [=======7ms=======]
C:                     [=1=]
B:                        [====4ms====]
D:                                    [====4ms====]
0   2   4   5   7   8   9          12              16

平均周转时间 = (7 + 10 + 4 + 11) / 4 = 8
平均等待时间 = (0 + 6 + 3 + 7) / 4 = 4
```

SRTF 的平均等待时间（3）比 SJF（4）更小，因为它能更及时地切换到更短的进程。但 SRTF 的上下文切换次数更多。

### 11.3.4 SRTF 的问题

SRTF 优化了周转时间和等待时间，但有以下问题：

**饥饿**：与 SJF 相同，长进程可能被不断到达的短进程无限期推迟。

**上下文切换开销**：每当新进程到达都可能触发抢占，导致频繁的上下文切换。

**需要预测执行时间**：与 SJF 相同，需要估计进程的执行时间。

**响应时间不确定**：虽然平均响应时间可能不错，但某个特定进程的响应时间可能很长。

## 11.4 轮转调度（Round Robin / RR）

### 11.4.1 算法描述

轮转调度（Round Robin）是为交互式系统设计的调度算法。它给每个进程一个固定长度的**时间片（Time Quantum）**，时间片用完后，进程被抢占，放到就绪队列尾部，下一个进程获得 CPU。

```c
// RR 调度伪代码
void rr_scheduler(int time_quantum) {
    Queue *ready_queue = create_queue();
    while (!empty(ready_queue)) {
        Process *p = dequeue(ready_queue);
        run(p, time_quantum);  // 运行最多 time_quantum 时间
        
        if (p->remaining_time > 0) {
            // 时间片用完，进程未完成
            enqueue(ready_queue, p);  // 放到队尾
        }
        // 否则进程完成，不入队
    }
}
```

RR 的关键特性：
- **抢占式**：时间片用完后强制切换
- **公平**：每个进程获得相等的 CPU 时间份额
- **FIFO 队列**：新进程加入队尾，被抢占的进程也加入队尾

### 11.4.2 时间片大小的权衡

时间片的选择是 RR 的关键设计决策，需要在响应性和切换开销之间权衡：

**时间片太大**（如 100ms）：
- RR 退化为 FCFS
- 每个进程在时间片内完成或阻塞
- 响应性差——长进程占用 CPU 很长时间

**时间片太小**（如 0.1ms）：
- 上下文切换频繁
- 切换开销占总时间的比例增大
- 如果时间片为 0.1ms，上下文切换为 0.01ms，那么 10% 的 CPU 时间浪费在切换上

**时间片适中**（如 10-100ms）：
- 响应性好——用户感觉系统"即时"响应
- 切换开销可接受——通常小于 1%

```
时间片 = 20ms，上下文切换 = 1ms

A: [===20ms===][切换1ms]
B:                     [===20ms===][切换1ms]
C:                                         [===20ms===]
时间：0        20  21              41  42              62

切换开销 = 2ms / 62ms = 3.2%
```

**经验法则**：时间片应该远大于上下文切换时间（至少 100 倍），通常为 10-100ms。

### 11.4.3 详细示例

```
进程：A(执行=24), B(执行=3), C(执行=3)
时间片 = 4ms，所有进程同时到达

RR 调度：

时刻 0：就绪队列=[A,B,C]，运行 A（4ms）
时刻 4：A 用完时间片（剩余 20），就绪队列=[B,C,A]，运行 B（3ms）
时刻 7：B 完成，就绪队列=[C,A]，运行 C（3ms）
时刻 10：C 完成，就绪队列=[A]，运行 A（4ms）
时刻 14：A 剩余 16，就绪队列=[A]，运行 A（4ms）
时刻 18：A 剩余 12，就绪队列=[A]，运行 A（4ms）
时刻 22：A 剩余 8，就绪队列=[A]，运行 A（4ms）
时刻 26：A 剩余 4，就绪队列=[A]，运行 A（4ms）
时刻 30：A 完成

Gantt 图：
A: [====4ms====]                    [====4ms====]              [====4ms====]    [====4ms====]    [====4ms====]
B:               [=3ms=]
C:                    [=3ms=]
时间：0        4    7  10         14              18              22              26              30

各进程指标：
  进程  执行时间  完成时间  周转时间  等待时间  响应时间
  A     24        30        30        6         0
  B     3         7         7         4         4
  C     3         10        10        7         7
  平均            -         15.67     5.67      3.67
```

### 11.4.4 RR 的公平性分析

RR 的最大优势是**公平性**——每个进程获得相等的 CPU 时间份额。

**响应时间上界**：如果有 n 个进程，时间片为 q，那么任何进程的响应时间上界为 (n-1) × q。

```
3 个进程，时间片 = 4ms
响应时间上界 = (3-1) × 4 = 8ms
→ 任何进程最多等待 8ms 就能获得 CPU
```

这使得 RR 非常适合交互式系统：
- 每个用户进程都能及时获得 CPU 时间
- 响应时间有明确的上界
- 没有饥饿问题

### 11.4.5 RR vs SJF 的对比

| 维度 | SJF/SRTF | RR |
|------|----------|-----|
| 优化目标 | 周转时间 | 响应时间 |
| 公平性 | 差（长进程饥饿） | 好（相等份额） |
| 响应时间 | 差（长进程等待） | 好（有上界） |
| 适用场景 | 批处理 | 交互式 |
| 实现复杂度 | 中（需要预测执行时间） | 低 |
| 上下文切换 | 少（非抢占式 SJF） | 多（每个时间片切换） |
| 饥饿 | 是 | 否 |

## 11.5 优先级调度

### 11.5.1 算法描述

优先级调度为每个进程分配一个**优先级**，调度器总是运行优先级最高的进程。

```c
// 优先级调度伪代码
void priority_scheduler(void) {
    while (!empty(ready_queue)) {
        Process *p = extract_max(ready_queue);  // 取优先级最高的进程
        run(p);
    }
}
```

优先级可以是**静态的**（创建时确定，运行中不变）或**动态的**（运行中调整）。

### 11.5.2 静态优先级 vs 动态优先级

**静态优先级**：优先级在进程创建时确定，运行过程中不变。

优点：实现简单。
缺点：不能适应进程行为的变化。如果一个高优先级进程变成 I/O 密集型，它仍然保持高优先级。

**动态优先级**：优先级在运行过程中调整。例如：
- I/O 密集型进程获得更高的优先级（因为它们快速释放 CPU）
- CPU 密集型进程的优先级逐渐降低（因为它们长时间占用 CPU）
- 等待时间长的进程优先级逐渐提高（防止饥饿）

```c
// 动态优先级调整
void adjust_priority(Process *p) {
    if (p->io_count > threshold) {
        p->priority += 1;  // I/O 密集型，提高优先级
    }
    if (p->cpu_time > threshold) {
        p->priority -= 1;  // CPU 密集型，降低优先级
    }
    if (p->wait_time > aging_threshold) {
        p->priority += 1;  // 等待太久，提高优先级（老化）
    }
}
```

### 11.5.3 优先级反转（Priority Inversion）

优先级反转是优先级调度中的一个经典问题：**高优先级进程被低优先级进程间接阻塞**。

```
场景：
  进程 L（优先级=1，低）持有锁 S
  进程 H（优先级=3，高）尝试获取锁 S → 阻塞等待 L 释放
  进程 M（优先级=2，中）就绪 → 抢占 L（因为 M 优先级更高）
  → H 等待 L 释放 S，但 L 被 M 抢占，无法运行
  → H 间接被 M 阻塞（优先级反转！）

时间线：
L: [=持有锁S=]      [被M抢占]                    [恢复运行，释放S]
H:      [尝试获取S→阻塞]........................[获取S→继续]
M:                      [=运行=]
时间：  0              5                        15              20
```

优先级反转在实际系统中可能导致严重问题。1997 年，NASA 的火星探路者（Mars Pathfinder）就因为优先级反转导致系统频繁重启。

### 11.5.4 优先级继承（Priority Inheritance）

解决优先级反转的方案是**优先级继承**：当高优先级进程被低优先级进程阻塞时，临时提升低优先级进程的优先级到高优先级的水平。

```
优先级继承：
  L 持有锁 S，H 尝试获取 S → 阻塞
  L 的优先级临时提升到 H 的水平（优先级=3）
  M 到达但优先级=2 < L 的当前优先级=3 → 不抢占
  L 完成临界区，释放 S，优先级恢复为 1
  H 获得 S，继续运行

时间线：
L: [=持有锁S=优先级提升到3=]  [释放S，优先级恢复1]
H:      [尝试获取S→阻塞]....[获取S→继续]
M:                                    [=运行=]
时间：  0                     10      15
```

优先级继承的规则：
1. 当高优先级进程 H 被低优先级进程 L 阻塞时，L 的优先级临时提升到 H 的水平
2. L 释放锁后，优先级恢复到原来的水平
3. 如果 L 还被其他进程阻塞，优先级可以继续提升（传递性）

### 11.5.5 老化（Aging）技术

**老化**是防止饥饿的技术：随着时间推移，逐渐提高等待进程的优先级。

```c
// 老化伪代码
void aging(void) {
    for each process p in ready_queue {
        p->wait_time += clock_tick;
        if (p->wait_time > aging_threshold) {
            p->priority += 1;  // 优先级提升
            p->wait_time = 0;  // 重置等待时间
        }
    }
}
```

**示例**：

```
初始优先级：P_high=3, P_medium=2, P_low=1
老化阈值：每等待 5 秒，优先级 +1

时刻 0：P_low 优先级=1
时刻 5：P_low 优先级=2（等待 5 秒）
时刻 10：P_low 优先级=3（等待 10 秒）
→ P_low 获得与 P_high 相同的优先级，避免饥饿
```

老化保证了每个进程最终都能获得 CPU——即使它的初始优先级很低。

## 11.6 算法综合对比

| 算法 | 抢占 | 饥饿 | 平均等待时间 | 响应时间 | 公平性 | 适用场景 |
|------|------|------|-------------|---------|--------|---------|
| FCFS | 否 | 否 | 差 | 差 | 好 | 简单批处理 |
| SJF | 否 | 是 | 最优 | 差 | 差 | 批处理 |
| SRTF | 是 | 是 | 最优 | 中 | 差 | 批处理 |
| RR | 是 | 否 | 中 | 好 | 好 | 交互式 |
| 优先级 | 是 | 是 | 取决于 | 取决于 | 差 | 实时/混合 |

**关键洞察**：
- 没有"最好"的算法——选择取决于应用场景
- SJF 最优但有饥饿问题，RR 公平但周转时间长
- 现代系统（如 MLFQ、CFS）结合了多种算法的优点

## 11.7 面试高频考点

**Q1：SJF 为什么是最优的？**

在所有进程同时到达的情况下，SJF 最小化平均等待时间。证明思路：任何违反 SJF 顺序的调度都可以通过交换相邻的长短进程来减少总等待时间。

**Q2：时间片太大或太小会怎样？**

时间片太大：RR 退化为 FCFS，响应性差。时间片太小：上下文切换频繁，切换开销占比大。经验法则：时间片应远大于上下文切换时间（10-100ms）。

**Q3：什么是优先级反转？如何解决？**

高优先级进程被低优先级进程间接阻塞（低优先级持有锁，中优先级抢占低优先级）。解决方案：优先级继承——临时提升锁持有者的优先级。

**Q4：FCFS 的护航效应是什么？**

一个长 CPU 密集型进程在前，后面所有短 I/O 密集型进程都被迫等待，导致 CPU 利用率低。

**Q5：RR 和 FCFS 的区别？**

RR 是抢占式的（时间片用完强制切换），FCFS 是非抢占式的（运行直到完成或阻塞）。RR 响应性好、公平，但上下文切换多。FCFS 简单但响应性差。
