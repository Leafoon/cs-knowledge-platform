## 10.1 调度概述

### 10.1.1 为什么需要调度？

在第四章中，我们学习了上下文切换机制——操作系统如何保存旧进程状态、加载新进程状态。但有一个关键问题我们没有回答：**当有多个就绪进程时，应该选择哪个进程运行？** 这就是 CPU 调度（CPU Scheduling）要解决的问题。

CPU 调度的核心目标是**最大化 CPU 利用率**，同时满足多种性能需求。一个没有调度器的系统只能运行一个进程直到它结束——这会导致 CPU 在进程等待 I/O 时完全空闲，浪费了宝贵的计算资源。

考虑一个具体的场景：三个进程共享一个 CPU。

```
进程 A：到达时间=0，需要执行 10ms，然后发起 I/O（10ms），再执行 10ms
进程 B：到达时间=0，需要执行 20ms
进程 C：到达时间=0，需要执行 5ms，然后发起 I/O（5ms），再执行 5ms
```

**没有调度的情况**（先运行 A，再 B，再 C）：

```
A: [===计算 10ms===][IO 等待 10ms][===计算 10ms===]
B:                                    [========计算 20ms========]
C:                                                                [计算 5ms][IO 5ms][计算 5ms]
时间：0        10       20       30          50         55    60    65

CPU 空闲时间：0（A 等待 I/O 时 CPU 空闲）
总完成时间：65ms
CPU 利用率 = 40ms / 65ms = 61.5%
```

**有调度的情况**（A 等待 I/O 时切换到其他进程）：

```
A: [===计算 10ms===][IO 等待 10ms]          [===计算 10ms===]
B:          [===计算 10ms===][暂停][===计算 10ms===]
C:                    [计算 5ms][IO 5ms][计算 5ms]
时间：0     5    10   15   20   25   30   35   40

CPU 空闲时间：0
总完成时间：40ms
CPU 利用率 = 40ms / 40ms = 100%
```

调度将 CPU 利用率从 61.5% 提升到 100%，总完成时间从 65ms 降低到 40ms。这就是调度的价值——通过在进程等待 I/O 时切换到其他进程，让 CPU 始终忙碌。

### 10.1.2 调度时机

操作系统在以下时刻需要进行调度决策。理解这些时机对于理解调度器的设计至关重要。

**时机一：进程从运行态转为阻塞态**

这是最常见的调度时机。当进程发起 I/O 请求（如 `read()`）、等待信号量（如 `sem_wait()`）、或等待子进程（如 `wait()`）时，它无法继续执行，必须让出 CPU。此时调度器**必须**选择一个新进程运行。

这种调度是**非自愿**的——进程主动放弃了 CPU。在非抢占式调度系统中，这是唯一的调度时机。

```c
// 示例：进程发起 read() 系统调用
int data = read(fd, buffer, size);
// read() 可能阻塞数毫秒（磁盘 I/O）
// 在此期间，调度器选择其他进程运行
// 当数据就绪后，进程从阻塞态变为就绪态
```

**时机二：进程从运行态转为就绪态**

当定时器中断触发时，当前进程的时间片用完。此时调度器**可能**选择切换到其他进程（抢占式调度），也可能让当前进程继续运行（非抢占式调度）。

在抢占式调度系统中，这种时机允许操作系统强制暂停长时间运行的进程，确保所有进程都能获得 CPU 时间。

```
时间线：
Process A: [====执行 10ms ====]
                                ← 定时器中断（时间片用完）
Process B:                      [====执行 10ms ====]
                                ← 定时器中断
Process A:                                     [====执行 10ms ====]
```

**时机三：进程从阻塞态转为就绪态**

当 I/O 完成、信号量释放、或子进程终止时，一个阻塞的进程变为就绪。此时调度器**可能**决定立即运行这个新就绪的进程（抢占当前进程），也可能将其加入就绪队列等待。

```c
// 示例：I/O 完成中断
// 进程 A 正在运行，进程 B 等待磁盘 I/O
// 磁盘中断触发 → 进程 B 变为就绪态
// 调度器决定：是否抢占 A，立即运行 B？
// 这取决于 B 的优先级和调度策略
```

**时机四：进程终止**

当进程调用 `exit()` 或因异常终止时，当前 CPU 空闲，调度器必须选择新进程。这是最简单的调度情况——不需要抢占任何进程。

**时机五：进程创建**

当 `fork()` 创建新进程时，调度器可能决定立即运行新进程（抢占当前进程），也可能将其加入就绪队列。

### 10.1.3 抢占式 vs 非抢占式调度

**非抢占式调度（Non-preemptive Scheduling）**：

进程一直运行直到它**主动让出** CPU（阻塞或终止）。调度器不会强制暂停进程。

```c
// 非抢占式调度的调度点
void schedule(void) {
    // 只在以下情况调度：
    // 1. 当前进程阻塞（I/O、wait、sleep）
    // 2. 当前进程终止（exit）
    // 3. 当前进程主动让出（yield）
}
```

优点：
- 实现简单——不需要处理"在任意时刻被抢占"的复杂性
- 没有竞态条件问题——进程不会在修改共享数据时被暂停
- 上下文切换频率低——只在进程主动让出时切换

缺点：
- 响应性差——一个长时间运行的 CPU 密集型进程会阻塞所有其他进程
- 不适合交互式系统——用户可能等待很长时间才能获得响应
- 不适合实时系统——无法保证截止时间

**抢占式调度（Preemptive Scheduling）**：

操作系统可以在**任何时候**强制暂停当前进程，切换到其他进程。

```c
// 抢占式调度的调度点
void timer_interrupt(void) {
    current->remaining_time--;
    if (current->remaining_time == 0) {
        // 时间片用完，强制切换
        schedule();
    }
}

void io_complete_interrupt(Process *p) {
    p->state = READY;
    if (p->priority > current->priority) {
        // 高优先级进程就绪，抢占当前进程
        schedule();
    }
}
```

优点：
- 响应性好——高优先级进程可以立即获得 CPU
- 公平性好——所有进程都能获得 CPU 时间
- 适合交互式和实时系统

缺点：
- 实现复杂——需要处理进程在任意时刻被暂停的情况
- 竞态条件风险——进程可能在修改共享数据时被抢占
- 上下文切换开销——切换频率更高

现代通用操作系统（Linux、Windows、macOS）都使用**抢占式调度**。实时操作系统也使用抢占式调度。只有简单的嵌入式系统或教学操作系统（如 xv6 的早期版本）使用非抢占式调度。

### 10.1.4 调度器的位置

操作系统中有三个层次的调度器，它们在不同的时间尺度上工作：

**长期调度器（Long-term Scheduler / Job Scheduler）**：

长期调度器决定哪些进程从**外存**（磁盘）加载到**内存**中。它的调度频率很低（秒级或分钟级），目标是控制**多道程序度**（Degree of Multiprogramming）——内存中同时存在的进程数。

在现代系统中，长期调度器通常是 `fork()` 和 `exec()` 的隐含结果——进程被创建时就进入内存。但在批处理系统中，长期调度器会从作业队列中选择作业加载。

```
外存（磁盘）：
[作业1][作业2][作业3][作业4][作业5]
         ↓ 长期调度器选择
内存：
[进程1][进程2][进程3]
```

**中期调度器（Medium-term Scheduler）**：

中期调度器决定哪些进程被**换出（swap out）**到磁盘，以释放物理内存。当内存紧张时，中期调度器将不活跃的进程换出；当内存充足时，再将其换入。

```
内存紧张时：
[进程1][进程2][进程3][进程4] → 内存不足
         ↓ 中期调度器换出进程2
[进程1][进程3][进程4] + 空闲内存

需要进程2时：
         ↓ 中期调度器换入进程2
[进程1][进程2][进程3][进程4]
```

**短期调度器（Short-term Scheduler / CPU Scheduler）**：

短期调度器决定**下一个运行的进程**。这是最频繁的调度——每次上下文切换都需要短期调度器做出决策。调度频率为毫秒级（每 10-100ms 调度一次）。

本章和后续章节主要讨论短期调度器。

```
就绪队列：
[进程1][进程2][进程3][进程4]
         ↓ 短期调度器选择
CPU：运行进程2
```

三个调度器的关系：

```
外存 ──长期调度──→ 内存 ──中期调度──→ 内存（换出/换入）
                        └──短期调度──→ CPU
```

## 10.2 调度评价指标

评价一个调度算法的好坏，需要定义明确的性能指标。不同的指标反映了不同的优化目标，往往存在权衡。理解这些指标对于选择合适的调度算法至关重要。

### 10.2.1 CPU 利用率（CPU Utilization）

CPU 利用率是 CPU 忙碌时间占总时间的比例：

```
CPU 利用率 = CPU 忙碌时间 / 总时间 × 100%
```

这是最基本的指标——调度的首要目标是让 CPU 尽可能忙碌。空闲的 CPU 是浪费的资源。

在多道程序设计环境中，CPU 利用率通常在 40%-90% 之间：
- 低于 40%：系统负载太低，或调度效率差（大量时间浪费在 I/O 等待上）
- 40%-80%：正常的交互式工作负载
- 80%-95%：高负载服务器
- 高于 95%：系统接近满载，可能需要更多 CPU 资源

```bash
# 使用 top 命令查看 CPU 利用率
$ top
%Cpu(s): 10.5 us,  5.2 sy,  0.0 ni, 83.1 id,  1.2 wa,  0.0 hi,  0.0 si
#         ↑用户态  ↑内核态         ↑空闲    ↑I/O等待

# 使用 vmstat 查看 CPU 利用率
$ vmstat 1
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs  us sy id wa
 2  0      0 512000  64000 2048000    0    0     0     0  500 2000  10  5 85  0
                                                          ↑    ↑    ↑  ↑  ↑  ↑
                                                       中断  上下文  用户 内核 空闲 IO
```

### 10.2.2 吞吐量（Throughput）

吞吐量是单位时间内完成的进程数：

```
吞吐量 = 完成的进程数 / 总时间
```

例如，如果在一分钟内完成了 120 个进程，吞吐量为 2 个进程/秒。

吞吐量取决于多个因素：
- 进程的平均执行时间——执行时间越短，吞吐量越高
- 调度算法的效率——好的调度算法可以减少等待时间，提高吞吐量
- 上下文切换的频率和开销——过多的切换会降低有效吞吐量
- I/O 等待时间——I/O 等待越少，CPU 利用率越高，吞吐量越高

吞吐量与 CPU 利用率的关系：
```
吞吐量 ≈ CPU 利用率 × (1 / 平均执行时间)
```

### 10.2.3 周转时间（Turnaround Time）

周转时间是衡量进程"从提交到完成"的总时间：

```
周转时间 = 完成时间 - 到达时间
```

周转时间包括了进程在就绪队列中等待的时间、实际执行的时间、以及等待 I/O 的时间。它是用户最关心的指标——用户提交一个任务后，需要等多久才能得到结果。

**详细示例**：

```
进程 A：到达时间=0，执行时间=5，完成时间=5
  周转时间 = 5 - 0 = 5

进程 B：到达时间=1，执行时间=3，完成时间=8
  周转时间 = 8 - 1 = 7

进程 C：到达时间=2，执行时间=8，完成时间=16
  周转时间 = 16 - 2 = 14

平均周转时间 = (5 + 7 + 14) / 3 = 8.67
```

周转时间的组成：
```
周转时间 = 等待时间 + 执行时间 + I/O 等待时间
```

**带权周转时间**（Weighted Turnaround Time）= 周转时间 / 执行时间。这个指标衡量的是"相对于执行时间，进程等待了多久"。带权周转时间越接近 1 越好。

```
进程 A：带权周转时间 = 5 / 5 = 1.0（完美）
进程 B：带权周转时间 = 7 / 3 = 2.33（等待时间较长）
进程 C：带权周转时间 = 14 / 8 = 1.75
```

### 10.2.4 等待时间（Waiting Time）

等待时间是进程在**就绪队列**中等待的总时间，不包括执行时间和 I/O 等待时间：

```
等待时间 = 周转时间 - 执行时间
```

等待时间反映了调度算法的"公平性"——如果某个进程的等待时间远大于其他进程，说明调度器对它不公平。

**详细示例**：

```
FCFS 调度（到达时间均为 0）：
进程 A(执行=24)：等待时间 = 0
进程 B(执行=3)：等待时间 = 24
进程 C(执行=3)：等待时间 = 27
平均等待时间 = (0 + 24 + 27) / 3 = 17

SJF 调度：
进程 B(执行=3)：等待时间 = 0
进程 C(执行=3)：等待时间 = 3
进程 A(执行=24)：等待时间 = 6
平均等待时间 = (0 + 3 + 6) / 3 = 3  ← 大幅改善！
```

### 10.2.5 响应时间（Response Time）

响应时间是进程从**到达**到**首次执行**的时间：

```
响应时间 = 首次执行时间 - 到达时间
```

响应时间是交互式系统中最重要的指标。用户点击一个按钮后，期望系统在几毫秒内响应。即使任务的总完成时间（周转时间）很长，只要首次响应快，用户体验就好。

**示例**：

```
RR 调度（时间片=4ms，到达时间均为 0）：
进程 A(执行=24)：首次执行时间=0，响应时间=0
进程 B(执行=3)：首次执行时间=4，响应时间=4
进程 C(执行=3)：首次执行时间=7，响应时间=7

FCFS 调度：
进程 A(执行=24)：首次执行时间=0，响应时间=0
进程 B(执行=3)：首次执行时间=24，响应时间=24  ← 很差！
进程 C(执行=3)：首次执行时间=27，响应时间=27  ← 很差！
```

响应时间 vs 周转时间：
- 响应时间关注"首次响应"——交互式系统的用户体验
- 周转时间关注"总完成时间"——批处理系统的效率
- 优化响应时间通常会牺牲周转时间，反之亦然

### 10.2.6 公平性（Fairness）

公平性衡量 CPU 时间在进程间分配的均匀程度。一个公平的调度器应该让所有进程获得大致相等的 CPU 时间份额。

**Jain's Fairness Index**：

```
公平性指数 = (Σx_i)² / (n × Σx_i²)

其中 x_i = 进程 i 获得的 CPU 时间份额
```

值域为 [1/n, 1]：
- 1 = 完全公平（所有进程获得相等份额）
- 1/n = 完全不公平（一个进程获得所有 CPU 时间）

**示例**：

```
3 个进程，理想份额 = 1/3 ≈ 0.333

情况 1：实际份额 = [0.33, 0.33, 0.34]
公平性 = (0.33+0.33+0.34)² / (3 × (0.33²+0.33²+0.34²))
       = 1.0 / (3 × 0.3334) = 0.9998 ≈ 1（非常公平）

情况 2：实际份额 = [0.9, 0.05, 0.05]
公平性 = (0.9+0.05+0.05)² / (3 × (0.9²+0.05²+0.05²))
       = 1.0 / (3 × 0.815) = 0.409（很不公平）
```

### 10.2.7 指标间的权衡

不同的调度算法优化不同的指标，往往存在权衡：

| 算法 | 周转时间 | 等待时间 | 响应时间 | 公平性 | 适用场景 |
|------|---------|---------|---------|--------|---------|
| FCFS | 差 | 差 | 差 | 好 | 简单批处理 |
| SJF | 最优 | 最优 | 差 | 差 | 批处理 |
| SRTF | 最优 | 最优 | 中 | 差 | 批处理 |
| RR | 中 | 中 | 好 | 好 | 交互式 |
| 优先级 | 取决于 | 取决于 | 取决于 | 差 | 实时 |

> **关键洞察**：没有"最好"的调度算法。选择调度算法取决于应用场景——批处理系统优化周转时间，交互式系统优化响应时间，实时系统优化截止时间满足率。

## 10.3 工作负载假设

调度算法的设计和性能评估依赖于对工作负载的假设。不同的假设会导致不同的最优算法。在理论分析中，我们通常使用简化的假设；在实际系统中，这些假设被放松。

### 10.3.1 进程运行时间已知 vs 未知

**已知运行时间**：在理论分析中，我们通常假设知道每个进程的执行时间。这使得我们可以证明某些算法的最优性（如 SJF 最小化平均周转时间）。但在实际系统中，这是不可能的——操作系统不知道进程会运行多久。

**未知运行时间**：在实际系统中，调度器只能根据**历史信息**来估计进程的执行时间。常见的估计方法：

**指数加权移动平均（EWMA）**：

```
τ_{n+1} = α × t_n + (1 - α) × τ_n

其中：
  t_n = 第 n 次实际执行时间
  τ_n = 第 n 次预测值
  α = 权重因子（通常 0.5）
```

```c
// 预测进程执行时间
double predict_burst(Process *p) {
    return ALPHA * p->last_burst + (1 - ALPHA) * p->predicted_burst;
}

// 每次进程完成时更新预测
void update_prediction(Process *p, double actual_burst) {
    p->predicted_burst = ALPHA * actual_burst + (1 - ALPHA) * p->predicted_burst;
    p->last_burst = actual_burst;
}
```

**示例**：α=0.5, τ_0=10, 实际执行时间序列 = [6, 4, 8, 7]

```
τ_1 = 0.5 × 6  + 0.5 × 10 = 8
τ_2 = 0.5 × 4  + 0.5 × 8  = 6
τ_3 = 0.5 × 8  + 0.5 × 6  = 7
τ_4 = 0.5 × 7  + 0.5 × 7  = 7
```

EWMA 的优点是：最近的执行时间权重更大，能快速适应进程行为的变化。

### 10.3.2 进程同时到达 vs 不同时到达

**同时到达**：所有进程在同一时刻到达就绪队列。这简化了分析，但不现实。在同时到达的假设下，SJF 是最优的。

**不同时到达**：进程在不同时间到达。这是现实情况，调度算法需要处理动态到达的进程。在不同时到达的情况下，SJF 不一定是最优的（取决于到达时间的分布）。

### 10.3.3 CPU 密集型 vs I/O 密集型

**CPU 密集型（CPU-bound）进程**：大部分时间在执行计算，很少发起 I/O 请求。典型例子：图像渲染、科学计算、编译器。

```
CPU 密集型进程的时间分布：
[====计算====][IO][====计算====][IO][====计算====]
  95% 计算     5% I/O
```

**I/O 密集型（I/O-bound）进程**：大部分时间在等待 I/O，只有少量计算。典型例子：文本编辑器、Web 浏览器、数据库查询。

```
I/O 密集型进程的时间分布：
[计算][====IO等待====][计算][====IO等待====]
 10% 计算    90% I/O
```

一个好的调度算法应该区分这两种进程：
- I/O 密集型进程应该获得更高的优先级，因为它们只需要很少的 CPU 时间就能完成 I/O 并释放 CPU
- CPU 密集型进程应该在 I/O 密集型进程等待时运行

MLFQ 通过观察进程的行为来自动区分这两种进程——I/O 密集型进程在时间片内主动让出 CPU，保持高优先级；CPU 密集型进程用完时间片，优先级逐渐降低。

### 10.3.4 批处理 vs 交互式工作负载

**批处理工作负载**：进程不需要快速响应，用户关心的是总吞吐量和平均周转时间。典型例子：科学计算、数据处理、编译。

**交互式工作负载**：进程需要快速响应（如鼠标点击、键盘输入），用户关心的是响应时间。典型例子：文本编辑、Web 浏览、游戏。

**实时工作负载**：进程必须在截止时间前完成，否则后果严重。典型例子：飞行控制、心脏起搏器、音频处理。

不同的工作负载需要不同的调度策略：

| 工作负载 | 优化目标 | 适合的算法 |
|---------|---------|-----------|
| 批处理 | 周转时间、吞吐量 | SJF、SRTF |
| 交互式 | 响应时间、公平性 | RR、MLFQ、CFS |
| 实时 | 截止时间满足率 | RMS、EDF |

## 10.4 调度算法分类

调度算法可以按多个维度分类。理解这个分类框架有助于系统地学习和比较不同的算法。

### 10.4.1 按抢占性分类

**非抢占式算法**：FCFS、SJF（非抢占版本）
- 进程一直运行直到主动让出 CPU
- 实现简单，但响应性差

**抢占式算法**：SRTF、RR、MLFQ、CFS
- 操作系统可以在任何时候强制暂停进程
- 响应性好，但实现复杂

### 10.4.2 按优化目标分类

**批处理优化**：FCFS、SJF、SRTF
- 优化周转时间
- 适合不需要快速响应的场景

**交互式优化**：RR、MLFQ、CFS
- 优化响应时间
- 适合需要快速用户交互的场景

**实时优化**：RMS、EDF
- 优化截止时间满足率
- 适合有严格时间约束的场景

### 10.4.3 按决策方式分类

**静态优先级**：优先级在进程创建时确定，运行过程中不变。如 RMS。

**动态优先级**：优先级在运行过程中调整。如 MLFQ、CFS。

### 10.4.4 按处理器数量分类

**单处理器调度**：一个 CPU，一个就绪队列。

**多处理器调度**：多个 CPU，需要考虑负载均衡和处理器亲和性。

## 10.5 手算练习

### 练习 1：计算调度指标

给定以下进程，使用 FCFS 调度（到达时间均为 0），计算周转时间、等待时间、响应时间。

```
进程  执行时间
A     6
B     2
C     8
D     3
```

**解答**：

```
FCFS 调度顺序：A → B → C → D

Gantt 图：
A: [======6ms======]
B:                  [==2ms==]
C:                       [========8ms========]
D:                                        [==3ms==]
时间：0             6    8              16              19

周转时间：
  A = 6 - 0 = 6
  B = 8 - 0 = 8
  C = 16 - 0 = 16
  D = 19 - 0 = 19
  平均 = (6 + 8 + 16 + 19) / 4 = 12.25

等待时间：
  A = 6 - 6 = 0
  B = 8 - 2 = 6
  C = 16 - 8 = 8
  D = 19 - 3 = 16
  平均 = (0 + 6 + 8 + 16) / 4 = 7.5

响应时间（同时到达）：
  A = 0, B = 6, C = 8, D = 16
  平均 = (0 + 6 + 8 + 16) / 4 = 7.5
```

### 练习 2：SJF vs FCFS

使用练习 1 的进程，分别用 FCFS 和 SJF 调度，比较平均等待时间。

**解答**：

```
FCFS（A→B→C→D）：平均等待时间 = 7.5（如上）

SJF（B→D→A→C）：
B: [==2ms==]
D:        [==3ms==]
A:              [======6ms======]
C:                           [========8ms========]
时间：0  2    5          11              19

等待时间：
  B = 0, D = 2, A = 5, C = 11
  平均 = (0 + 2 + 5 + 11) / 4 = 4.5

SJF 的平均等待时间（4.5）比 FCFS（7.5）少了 40%！
```

## 10.6 面试高频考点

**Q1：周转时间和等待时间的区别？**

周转时间 = 完成时间 - 到达时间，是进程从提交到完成的总时间。等待时间 = 周转时间 - 执行时间，是进程在就绪队列中等待的总时间。周转时间包含了执行时间，等待时间不包含。

**Q2：响应时间和周转时间的区别？**

响应时间是进程从到达就绪队列到首次执行的时间，反映系统的交互响应速度。周转时间是进程从到达就绪队列到完成的总时间，反映任务的总完成速度。交互式系统优化响应时间，批处理系统优化周转时间。

**Q3：抢占式和非抢占式调度的区别？**

非抢占式调度中，进程一直运行直到主动让出 CPU（阻塞或终止）。抢占式调度中，操作系统可以强制暂停进程。抢占式响应性好但实现复杂，非抢占式简单但响应性差。现代通用系统都使用抢占式调度。

**Q4：CPU 密集型和 I/O 密集型进程对调度的影响？**

I/O 密集型进程只使用少量 CPU 时间就发起 I/O 并阻塞，应该获得较高优先级以快速响应用户。CPU 密集型进程长时间占用 CPU，应该在 I/O 密集型进程等待时运行。好的调度器应该区分这两种进程。

**Q5：为什么没有"最好"的调度算法？**

不同的应用场景有不同的优化目标：批处理优化周转时间，交互式优化响应时间，实时优化截止时间。这些目标往往冲突——优化响应时间（如 RR）会牺牲周转时间，优化周转时间（如 SJF）会导致长进程饥饿。因此需要根据场景选择合适的算法。
