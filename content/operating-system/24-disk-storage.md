## 24.1 硬盘驱动器（HDD）

### 24.1.1 HDD 物理结构

硬盘驱动器（Hard Disk Drive）是一种使用磁性存储技术的旋转式存储设备。

```
HDD 物理结构：

        磁头臂（Actuator Arm）
        ┌─────────────────────┐
        │                     │
        │   ┌─磁头(Head)      │
        │   │                 │
   ─────┼───┼─────────────────┼─────
   ║    │   │                 │    ║
   ║  ┌─┴───┴─────────────────┴─┐ ║
   ║  │      盘片（Platter）     │ ║  ← 磁道（Track）
   ║  │  ┌─────────────────┐   │ ║
   ║  │  │    ┌───────┐    │   │ ║
   ║  │  │    │ 中心轴 │    │   │ ║  ← 扇区（Sector）
   ║  │  │    └───────┘    │   │ ║
   ║  │  └─────────────────┘   │ ║
   ║  └────────────────────────┘ ║
   ║                              ║
   ║  ┌────────────────────────┐ ║
   ║  │      盘片 2            │ ║  ← 柱面（Cylinder）
   ║  └────────────────────────┘ ║
   ║                              ║
   ═══════════════════════════════
          主轴电机（Spindle Motor）
```

**关键术语**：

| 术语 | 定义 | 典型值 |
|------|------|--------|
| **盘片（Platter）** | 涂有磁性材料的圆盘 | 1-8 个 |
| **磁道（Track）** | 盘片上的同心圆 | 数万条 |
| **扇区（Sector）** | 磁道上的弧形段 | 512 字节或 4KB |
| **柱面（Cylinder）** | 所有盘片上同一位置的磁道集合 | - |
| **磁头（Head）** | 读写数据的传感器 | 每个盘面一个 |
| **转速（RPM）** | 盘片每分钟旋转次数 | 5400/7200/10000/15000 |

### 24.1.2 磁盘访问时间

磁盘访问时间由三部分组成：

```
磁盘访问时间 = 寻道时间 + 旋转延迟 + 传输时间

┌───────────────────────────────────────────────────┐
│ 寻道时间（Seek Time）                              │
│ 磁头移动到目标磁道的时间                            │
│ 典型值：平均 3-10ms（取决于磁道距离）               │
├───────────────────────────────────────────────────┤
│ 旋转延迟（Rotational Latency）                     │
│ 等待目标扇区旋转到磁头下方的时间                    │
│ 平均 = 0.5 × (60 / RPM)                          │
│ 7200 RPM → 平均 4.17ms                            │
├───────────────────────────────────────────────────┤
│ 传输时间（Transfer Time）                          │
│ 数据从盘片读取到缓存的时间                          │
│ 取决于数据大小和旋转速度                            │
│ 典型值：< 1ms（小块数据）                          │
└───────────────────────────────────────────────────┘
```

**计算示例**：

```
参数：
- 平均寻道时间：8ms
- 转速：7200 RPM
- 传输速率：150 MB/s
- 每磁道扇区数：500
- 每扇区：512 字节

计算读取 1 个扇区的访问时间：
1. 寻道时间 = 8ms
2. 旋转延迟 = 0.5 × (60/7200) × 1000 = 4.17ms
3. 传输时间 = (512 / (150×1024×1024)) × 1000 ≈ 0.003ms

总访问时间 ≈ 8 + 4.17 + 0.003 ≈ 12.17ms

吞吐量 = 512 / 12.17ms ≈ 42 KB/s（随机小块读取）
```

**关键洞察**：对于随机小块 I/O，寻道时间和旋转延迟占总访问时间的 99% 以上。这就是为什么 HDD 的随机 IOPS 通常只有 100-200。

## 24.2 磁盘调度算法

### 24.2.1 为什么需要磁盘调度？

当多个 I/O 请求在队列中等待时，调度顺序直接影响总寻道时间。好的调度算法可以大幅减少磁头移动距离。

```
示例请求队列：98, 183, 37, 122, 14, 124, 65, 67
当前磁头位置：53

不同调度顺序的磁头移动距离：
FCFS:     53→98→183→37→122→14→124→65→67  = 640 磁道
优化顺序: 53→37→14→65→67→98→122→124→183  = 236 磁道
```

### 24.2.2 FCFS（先来先服务）

最简单的调度算法：按请求到达顺序处理。

```
FCFS 调度示例：
请求队列：98, 183, 37, 122, 14, 124, 65, 67
当前位置：53

53 ──→ 98 ──→ 183 ──→ 37 ──→ 122 ──→ 14 ──→ 124 ──→ 65 ──→ 67
   45    85     146     85      108    110     59      2

总移动距离 = 45+85+146+85+108+110+59+2 = 640 磁道
```

**优缺点**：
- 优点：公平，无饥饿
- 缺点：性能差，磁头来回大幅移动

### 24.2.3 SSTF（最短寻道时间优先）

选择距离当前磁头位置最近的请求。

```
SSTF 调度示例：
请求队列：98, 183, 37, 122, 14, 124, 65, 67
当前位置：53

选择过程：
53 → 最近的是 37(距离16) 或 65(距离12) → 选 65
65 → 最近的是 67(距离2) → 选 67
67 → 最近的是 98(距离31) → 选 98
...以此类推

53 ──→ 65 ──→ 67 ──→ 98 ──→ 122 ──→ 124 ──→ 183 ──→ 37 ──→ 14
   12     2     31     24       2      59      146     23

总移动距离 = 12+2+31+24+2+59+146+23 = 299 磁道
```

**优缺点**：
- 优点：总寻道距离短
- 缺点：可能饥饿——远离磁头的请求长期得不到服务

### 24.2.4 SCAN（电梯算法）

磁头向一个方向移动，处理沿途所有请求，到达端点后反向。

```
SCAN 调度示例：
请求队列：98, 183, 37, 122, 14, 124, 65, 67
当前位置：53，向磁道号增加方向移动

53 ──→ 65 ──→ 67 ──→ 98 ──→ 122 ──→ 124 ──→ 183 ──→ [端点199]
                                                         │
[端点0] ←── 37 ←── 14 ←────────────────────────────────┘

总移动距离 = (183-53) + (183-14) = 130 + 169 = 299 磁道
（假设端点为 0-199）
```

**优缺点**：
- 优点：公平性好，无饥饿
- 缺点：到达端点的请求被延迟

### 24.2.5 C-SCAN（循环扫描）

只在一个方向上服务请求，到达端点后快速返回起点重新开始。

```
C-SCAN 调度示例：
请求队列：98, 183, 37, 122, 14, 124, 65, 67
当前位置：53，向磁道号增加方向移动

53 ──→ 65 ──→ 67 ──→ 98 ──→ 122 ──→ 124 ──→ 183 ──→ [端点199]
                                                         │
[端点0] ──→ 14 ──→ 37 ──────────────────────────────────┘

总移动距离 = (199-53) + (199-0) + (37-0) = 146+199+37 = 382 磁道
```

**优缺点**：
- 优点：更公平——所有方向的请求等待时间更均匀
- 缺点：回程不服务请求，效率略低

### 24.2.6 LOOK 与 C-LOOK

SCAN/C-SCAN 的优化版本：不到达端点，处理完最后一个请求就反向。

```
LOOK 调度示例：
53 ──→ 65 ──→ 67 ──→ 98 ──→ 122 ──→ 124 ──→ 183
                                              │（反向）
         37 ←── 14 ←─────────────────────────┘

总移动距离 = (183-53) + (183-14) = 130+169 = 299 磁道
```

### 24.2.7 调度算法性能对比

```
示例：请求队列 [98,183,37,122,14,124,65,67]，当前位置 53

算法      总寻道距离   平均寻道   饥饿？   公平性
FCFS          640      80.0       否      最公平
SSTF          299      37.4       是      差
SCAN          299      37.4       否      好
C-SCAN        382      47.8       否      最好
LOOK          299      37.4       否      好
C-LOOK        299      37.4       否      最好
```

## 24.3 固态硬盘（SSD）

### 24.3.1 SSD 物理结构

SSD（Solid State Drive）使用闪存（NAND Flash）芯片存储数据，没有机械运动部件。

```
SSD 内部结构：

┌─────────────────────────────────────────────────┐
│                  SSD 控制器                       │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐      │
│  │ 闪存接口  │  │  FTL     │  │  缓存    │      │
│  │ 控制器    │  │ 映射表   │  │ (DRAM)   │      │
│  └──────────┘  └──────────┘  └──────────┘      │
└──────┬──────────────┬──────────────┬────────────┘
       │              │              │
  ┌────┴────┐    ┌────┴────┐    ┌────┴────┐
  │ NAND    │    │ NAND    │    │ NAND    │
  │ Chip 0  │    │ Chip 1  │    │ Chip 2  │
  │         │    │         │    │         │
  │ ┌─────┐ │    │ ┌─────┐ │    │ ┌─────┐ │
  │ │Block│ │    │ │Block│ │    │ │Block│ │
  │ │  0  │ │    │ │  0  │ │    │ │  0  │ │
  │ │  1  │ │    │ │  1  │ │    │ │  1  │ │
  │ │ ... │ │    │ │ ... │ │    │ │ ... │ │
  │ │  N  │ │    │ │  N  │ │    │ │  N  │ │
  │ └─────┘ │    │ └─────┘ │    │ └─────┘ │
  └─────────┘    └─────────┘    └─────────┘
```

**NAND Flash 层次结构**：

| 层次 | 名称 | 大小 |
|------|------|------|
| Chip | 闪存芯片 | 数 GB |
| Block | 块（擦除单位） | 数 MB |
| Page | 页（读写单位） | 4-16 KB |

### 24.3.2 NAND Flash 操作特性

```
NAND Flash 操作延迟：

操作        延迟           特点
读取(Read)   ~25μs         以页为单位，可以随机读
编程(Program) ~200-500μs    以页为单位，只能写空页
擦除(Erase)  ~2-5ms        以块为单位，必须先擦后写

关键约束：
1. 不能原地覆盖写（必须先擦除）
2. 擦除以块为单位（远大于页）
3. 擦除次数有限（SLC: ~100K, MLC: ~10K, TLC: ~3K）
```

### 24.3.3 闪存转换层（FTL）

FTL（Flash Translation Layer）是 SSD 控制器中的核心软件，负责将逻辑地址映射到物理闪存地址。

```
FTL 映射表：

逻辑页号    物理页位置
LPA 0   →  Chip 0, Block 2, Page 5
LPA 1   →  Chip 1, Block 0, Page 3
LPA 2   →  Chip 0, Block 1, Page 0
...

写入流程（写入 LPA 0）：
1. 查找映射表：LPA 0 → 物理位置 (C0,B2,P5)
2. 将新数据写入空闲页 (C0,B3,P1)
3. 更新映射表：LPA 0 → (C0,B3,P1)
4. 标记旧页 (C0,B2,P5) 为无效
```

### 24.3.4 写放大（Write Amplification）

由于 NAND Flash 的擦写特性，实际写入闪存的数据量可能大于用户请求的数据量。

```
写放大示例：

用户请求写入 1 个页
但需要：
1. 读取包含旧页的整个块（128 页）
2. 在内存中修改 1 个页
3. 擦除整个块
4. 将 128 页（含新数据）写回

写放大因子 = 实际写入 / 用户写入 = 128 / 1 = 128

理想情况（块内大部分页已无效）：
实际写入 = 2 页（1 新 + 1 有效旧页）
写放大因子 = 2 / 1 = 2
```

### 24.3.5 磨损均衡（Wear Leveling）

由于擦除次数有限，SSD 控制器需要均匀分配擦除操作到所有块。

```
磨损均衡策略：

静态磨损均衡：
- 常驻数据（很少更新）放在磨损高的块
- 频繁更新的数据放在磨损低的块
- 周期性交换冷热数据的位置

动态磨损均衡：
- 新写入总是选择磨损最低的块
- 简单但可能冷数据块磨损不均
```

### 24.3.6 TRIM 命令

当文件系统删除文件时，通知 SSD 哪些页不再使用。

```bash
# Linux 启用 TRIM
$ sudo fstrim -v /mount-point

# /etc/fstab 中启用 discard 选项
/dev/sda1 / ext4 defaults,discard 0 0
```

**TRIM 的作用**：
- 没有 TRIM：SSD 不知道哪些页已失效，GC 效率低
- 有 TRIM：SSD 可以提前回收无效页，减少写放大

## 24.4 RAID（磁盘阵列）

### 24.4.1 RAID 概述

RAID（Redundant Array of Independent Disks）将多个磁盘组合成一个逻辑单元，以提高性能和/或可靠性。

### 24.4.2 RAID 0（条带化）

数据被分割成块，交替写入多个磁盘。

```
RAID 0 — 条带化（Striping）：

数据块：A0 A1 A2 A3 A4 A5 A6 A7

磁盘 0: [A0] [A2] [A4] [A6]
磁盘 1: [A1] [A3] [A5] [A7]

优点：读写性能翻倍（并行访问）
缺点：无冗余——一块盘故障，所有数据丢失
适用：临时数据、缓存、对性能要求高但不关心可靠性
```

### 24.4.3 RAID 1（镜像）

每个数据块被完整复制到两个磁盘。

```
RAID 1 — 镜像（Mirroring）：

数据块：A0 A1 A2 A3

磁盘 0: [A0] [A1] [A2] [A3]   ← 主盘
磁盘 1: [A0] [A1] [A2] [A3]   ← 镜像盘

优点：高可靠性——一块盘故障，另一块有完整数据
     读性能可提升（从两块盘并行读）
缺点：成本翻倍，写性能无提升
适用：关键数据（操作系统、数据库）
```

### 24.4.4 RAID 4（专用校验盘）

数据分布在多块盘上，校验信息集中在一块专用校验盘。

```
RAID 4 — 专用校验盘：

磁盘 0: [D0] [D4] [D8]
磁盘 1: [D1] [D5] [D9]
磁盘 2: [D2] [D6] [D10]
磁盘 3: [D3] [D7] [D11]
校验盘: [P0] [P1] [P2]   ← 专用校验盘

P0 = D0 XOR D1 XOR D2 XOR D3

问题：校验盘成为写入瓶颈（每次写都要更新校验盘）
```

### 24.4.5 RAID 5（分布式校验）

将校验信息分布到所有磁盘上，消除校验盘瓶颈。

```
RAID 5 — 分布式校验：

磁盘 0: [D0] [D5] [P2] [D9]
磁盘 1: [D1] [P1] [D6] [D10]
磁盘 2: [P0] [D2] [D7] [D11]
磁盘 3: [D3] [D4] [D8] [P3]

P0 = D0 XOR D1 XOR D2 XOR D3
（校验块在各盘上轮转分布）

优点：读性能好，空间利用率高（(N-1)/N）
缺点：写性能受校验更新影响
适用：通用存储
```

### 24.4.6 RAID 6（双校验）

使用两组校验信息，可以容忍两块盘同时故障。

```
RAID 6 — 双校验：

磁盘 0: [D0] [D5] [Q2] [D9]
磁盘 1: [D1] [P1] [D6] [Q3]
磁盘 2: [P0] [D2] [D7] [D11]
磁盘 3: [D3] [D4] [P3] [D10]
磁盘 4: [Q0] [Q1] [D8] [P4]

P = XOR 校验（同 RAID 5）
Q = Reed-Solomon 校验（独立于 P）

优点：可容忍任意两块盘故障
缺点：空间利用率 (N-2)/N，写性能更差
适用：关键数据、大容量阵列
```

### 24.4.7 RAID 10（1+0）

先镜像再条带化——将 RAID 1 和 RAID 0 结合。

```
RAID 10 — 镜像 + 条带化：

         镜像对 0        镜像对 1
磁盘 0: [A0] [A2]    磁盘 2: [A1] [A3]
磁盘 1: [A0] [A2]    磁盘 3: [A1] [A3]
   ↑ 镜像               ↑ 镜像

条带化：A0→盘0, A1→盘2, A2→盘0, A3→盘2

优点：读写性能好，可靠性高（每对可坏一块）
缺点：成本 2x，空间利用率 50%
适用：数据库、高性能应用
```

## 24.5 Linux I/O 栈

### 24.5.1 I/O 栈层次

```
Linux I/O 栈层次：

┌─────────────────────────────────────────┐
│           用户空间                       │
│  read() / write() / mmap() / io_uring() │
├─────────────────────────────────────────┤
│           系统调用层                     │
├─────────────────────────────────────────┤
│           VFS（虚拟文件系统）             │
│  统一文件接口，分发到具体文件系统          │
├─────────────────────────────────────────┤
│           文件系统                       │
│  ext4 / XFS / Btrfs / F2FS              │
│  管理 inode、目录、数据块                │
├─────────────────────────────────────────┤
│           Page Cache                     │
│  缓存最近访问的文件数据                   │
│  减少磁盘 I/O 次数                       │
├─────────────────────────────────────────┤
│           Block Layer                    │
│  I/O 请求合并、排序、调度                 │
│  bio 结构管理                            │
├─────────────────────────────────────────┤
│           I/O Scheduler                  │
│  mq-deadline / BFQ / Kyber / none       │
├─────────────────────────────────────────┤
│           Device Driver                  │
│  NVMe / SCSI / virtio                   │
├─────────────────────────────────────────┤
│           硬件                           │
│  NVMe SSD / SATA HDD / virtio-blk       │
└─────────────────────────────────────────┘
```

### 24.5.2 Page Cache

Page Cache 是 Linux 内核中最重要的缓存机制之一。

```
Page Cache 的作用：

读操作：
1. 用户调用 read()
2. VFS 检查 Page Cache 是否有数据
3. 命中：直接返回（无磁盘 I/O）
4. 未命中：从磁盘读取到 Page Cache，再返回

写操作（回写模式）：
1. 用户调用 write()
2. 数据写入 Page Cache
3. 标记页为脏（dirty）
4. 后台线程定期将脏页写入磁盘

查看 Page Cache 状态：
$ free -h
              total    used    free    shared  buff/cache   available
Mem:           15Gi    6.2Gi   1.5Gi    256Mi    7.8Gi       8.6Gi
                                            ↑
                                      Page Cache 使用量
```

### 24.5.3 Block Layer

Block Layer 管理所有块设备的 I/O 请求。

```c
// bio 结构：描述一个块 I/O 请求
struct bio {
    struct bio *bi_next;        // 请求链表
    struct block_device *bi_bdev;  // 目标设备
    unsigned int bi_opf;        // 操作类型（READ/WRITE）
    unsigned short bi_vcnt;     // 向量数量
    unsigned short bi_idx;      // 当前向量索引
    struct bvec_iter bi_iter;   // 迭代器
    bio_end_io_t *bi_end_io;    // 完成回调
    void *bi_private;           // 私有数据
    bio_vec bi_io_vec[];        // 数据向量数组
};
```

## 24.6 面试高频考点

**Q1：HDD 的访问时间由哪些部分组成？**

寻道时间（磁头移动到目标磁道）+ 旋转延迟（等待目标扇区转到磁头下方）+ 传输时间（实际读写数据）。其中寻道时间和旋转延迟各占约 50%，是性能瓶颈。

**Q2：比较 SCAN 和 C-SCAN 调度算法？**

SCAN（电梯算法）在两个方向上都服务请求，到达端点后反向。C-SCAN 只在一个方向服务，到达端点后快速返回起点。C-SCAN 更公平——所有请求的等待时间更均匀。

**Q3：SSD 相比 HDD 的优势？**

无机械部件、随机 IOPS 高 1000 倍、延迟低 100 倍、抗震、低功耗。缺点是价格高、写入次数有限。

**Q4：RAID 5 和 RAID 6 的区别？**

RAID 5 使用单校验，可容忍 1 块盘故障。RAID 6 使用双校验（XOR + Reed-Solomon），可容忍 2 块盘同时故障。RAID 6 空间利用率更低 (N-2)/N，但更安全。

**Q5：什么是写放大？如何减少？**

写放大 = 实际写入闪存量 / 用户请求写入量。原因是 Flash 必须先擦后写，擦除以块为单位。减少方法：TRIM、GC 优化、预留空间（Over-Provisioning）。
