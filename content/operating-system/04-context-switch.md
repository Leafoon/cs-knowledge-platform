## 4.1 上下文切换概述

### 4.1.1 什么是上下文？

在第三章中，我们学习了进程是操作系统虚拟化 CPU 的基本单位。每个进程拥有独立的地址空间、独立的执行状态，操作系统通过快速切换进程来实现"多个程序同时运行"的假象。但这里有一个关键问题：当操作系统决定暂停进程 A、切换到进程 B 时，**CPU 中所有属于进程 A 的状态必须被完整保存**，以便将来进程 A 重新获得 CPU 时能够从**完全相同的状态**继续执行——就好像什么都没发生过一样。

这个"完整状态"就是**上下文（Context）**。更精确地说，上下文是一个进程（或线程）在某一时刻的**完整执行状态**，包括 CPU 中所有与其相关的寄存器值、处理器状态、以及地址空间信息。

让我们从一个直观的类比开始。想象你在阅读一本很厚的书，读到第 200 页第 15 行时，有人要求你暂时把书让给别人。为了让下次你能准确地继续阅读，你需要记住：

1. **你在读哪一页哪一行**（对应程序计数器 PC）
2. **你的书签放在哪里**（对应栈指针 SP）
3. **你在页边写了什么笔记**（对应寄存器中的中间计算结果）
4. **你正在用哪本书**（对应页表基址，即地址空间）

如果其中任何一项信息丢失或记录错误，你下次继续阅读时就会从错误的位置开始，整个阅读过程就乱了。操作系统面临的挑战与此完全相同——只不过它管理的不是书签和笔记，而是 CPU 中的几十个寄存器和复杂的内存映射关系。

上下文的核心组成部分包括以下几个类别：

**程序计数器（Program Counter, PC）**：这是 CPU 中最重要的寄存器之一，它指向**下一条要执行的指令的内存地址**。在 RISC-V 架构中，这个寄存器被称为 `pc`（有时也通过 `ra`——返回地址寄存器——间接表达）。当进程被暂停时，PC 的值必须被精确保存，因为恢复执行时 CPU 需要知道从哪里继续取指。如果 PC 保存错误，哪怕只差一个字节，进程恢复后执行的可能是完全无关的指令，导致不可预测的行为甚至系统崩溃。

**栈指针（Stack Pointer, SP）**：栈是函数调用的核心数据结构，存储着局部变量、函数参数、返回地址等关键信息。栈指针指向当前栈帧的位置。在 RISC-V 中是 `sp` 寄存器（x2），在 x86-64 中是 `rsp` 寄存器。如果栈指针保存错误，进程恢复后函数调用将使用错误的栈帧，局部变量读取到的是垃圾值，返回地址也可能是错误的——这几乎必然导致程序崩溃。

**通用寄存器（General Purpose Registers）**：这是 CPU 进行运算的工作区。在 RISC-V 架构中有 32 个通用寄存器（x0-x31），在 x86-64 架构中有 16 个（rax, rbx, rcx, ..., r15）。这些寄存器中保存着进程当前正在计算的中间结果、函数调用的参数、循环计数器等。例如，一个正在执行 `for (int i = 0; i < 100; i++)` 循环的进程，循环变量 `i` 的值就保存在某个寄存器中。如果这个寄存器的值没有被保存，进程恢复后 `i` 的值可能是随机的，循环逻辑就被破坏了。

**浮点寄存器（Floating Point Registers）**：现代 CPU 通常有独立的浮点寄存器组（RISC-V 有 f0-f31，x86-64 有 xmm0-xmm15 等），用于浮点运算。如果进程正在进行科学计算或图形处理，这些寄存器中保存着关键的浮点中间结果。

**处理器状态字（Processor Status Word）**：这是一个特殊的寄存器，保存着 CPU 的各种状态标志。在 RISC-V 中是 `sstatus` 寄存器，在 x86 中是 `EFLAGS` 寄存器。它包含条件码（最近一次比较或算术运算的结果）、中断使能位（是否允许中断）、当前特权级等信息。特别是中断使能位——如果保存错误，进程恢复后可能无法响应中断，或者在不应该响应中断的时候响应了中断。

**页表基址寄存器（Page Table Base Register）**：这个寄存器指向当前进程的页表，决定了虚拟地址到物理地址的映射关系。在 RISC-V 中是 `satp` 寄存器，在 x86 中是 `CR3` 寄存器。每个进程有自己独立的页表，因此切换进程时必须更新这个寄存器。如果不更新，进程 A 恢复后使用的是进程 B 的页表，它的内存访问将映射到完全错误的物理地址——这会导致进程读取到错误的数据，或者触发页错误。

> **关键洞察**：上下文切换的本质是**状态的保存与恢复**。这是一个必须精确完成的操作——如果保存或恢复不完整，哪怕只有一个寄存器的值出错，进程恢复执行后都可能产生不可预测的行为。事实上，许多内核 Bug 的根源正是上下文切换过程中的状态保存遗漏或错误。这也解释了为什么操作系统设计者在实现上下文切换时需要极其谨慎。

### 4.1.2 上下文切换的必要性

操作系统需要进行上下文切换的场景可以分为四类。理解这些场景对于理解操作系统的整体设计至关重要。

**场景一：时间片耗尽（Timer Interrupt）**

这是最常见的上下文切换场景。现代分时系统的核心思想是：虽然 CPU 在某一时刻只能执行一个进程，但通过快速轮转，让每个进程轮流执行一小段时间（称为**时间片**，通常为 1-100ms），用户就会感觉多个程序在"同时"运行。

为了实现这个机制，操作系统在启动时会配置硬件定时器（Timer），让它每隔固定时间产生一次中断。当中断发生时，CPU 会暂停当前进程的执行，跳转到操作系统的中断处理程序。中断处理程序发现是定时器中断，就知道当前进程的时间片用完了，于是调用调度器选择下一个就绪进程。

```
时间轴（时间片 = 10ms）：

Process A: [====执行 10ms ====]
                                ← 定时器中断（时间片用完）
Process B:                      [====执行 10ms ====]
                                ← 定时器中断
Process C:                                      [====执行 10ms ====]
                                ← 定时器中断
Process A:                                             [====执行 10ms ====]
```

这个机制的关键在于**公平性**：每个进程获得相同的 CPU 时间份额。如果进程 A 不主动让出 CPU（比如它在执行一个很长的计算），定时器中断会强制暂停它，确保其他进程也有机会运行。这正是"抢占式调度"的核心思想。

**场景二：I/O 阻塞**

当进程发起 I/O 请求时——比如读取文件、等待网络数据、等待用户输入——它必须等待 I/O 操作完成才能继续执行。在等待期间，进程处于"阻塞"状态，无法使用 CPU。如果操作系统让 CPU 空等，那就是极大的浪费。因此，操作系统会将阻塞的进程从运行队列中移除，切换到其他可运行的进程。

```
Process A: [计算] → [read() 系统调用] → [阻塞：等待磁盘数据] → [磁盘数据到达，恢复执行]
Process B:                      [============执行============]
Process C:                                        [===执行===]
```

这个场景与定时器中断不同：定时器中断是**强制切换**（进程可能还想继续运行），而 I/O 阻塞是**主动切换**（进程自己请求等待，无法继续运行）。在操作系统的术语中，前者称为"非自愿上下文切换"（involuntary context switch），后者称为"自愿上下文切换"（voluntary context switch）。

**场景三：进程终止**

当进程调用 `exit()` 系统调用正常结束，或者因为访问非法内存（段错误）、收到致命信号（如 SIGKILL）等原因异常终止时，操作系统需要回收该进程的资源，并切换到其他进程继续运行。这是上下文切换的一个特殊 case——因为被切换出去的进程不会再回来了，所以不需要保存它的上下文（或者保存了也不会被恢复）。

**场景四：更高优先级进程就绪**

在抢占式调度系统中，当一个更高优先级的进程变为就绪状态时——比如一个高优先级的 I/O 密集型进程完成了 I/O 等待——操作系统可能会抢占当前正在运行的低优先级进程，立即切换到高优先级进程。这种机制确保了对时间敏感的任务（如用户界面响应、实时音频处理）能够及时获得 CPU。

### 4.1.3 上下文切换的开销

上下文切换并非免费的操作。事实上，它是操作系统中**开销最大的基本操作之一**。理解这个开销的来源，对于理解操作系统的设计决策（如为什么需要 ASID、为什么协程比线程更高效）至关重要。

上下文切换的开销可以分为两个层次：**直接成本**和**间接成本**。直接成本是操作系统执行切换操作本身所需的时间，这部分可以通过代码分析来估算。间接成本是由于切换导致的硬件状态污染（缓存、TLB、分支预测器）带来的性能损失，这部分更难以量化，但往往比直接成本更大。

#### **直接成本（Direct Costs）**

直接成本是操作系统内核中上下文切换代码的执行时间。让我们逐项分析：

**保存和恢复寄存器**：这是上下文切换最核心的操作。在 RISC-V 架构中，需要保存 14 个 callee-saved 寄存器（ra, sp, s0-s11），每个寄存器 8 字节，总共 112 字节的内存写入和读取。在现代处理器上，这大约需要 10-50 纳秒。看起来很快，但这只是整个切换过程的一小部分。

**切换地址空间**：这是开销最大的直接操作。每个进程有自己独立的虚拟地址空间，由页表定义。切换进程时，必须更新页表基址寄存器（RISC-V 的 `satp`，x86 的 `CR3`），使 CPU 使用新进程的页表。写入这个寄存器本身很快（~10ns），但紧随其后的 TLB 刷新才是真正的性能杀手。

**刷新 TLB**：TLB（Translation Lookaside Buffer）是 CPU 中缓存页表条目的高速缓存。当页表基址寄存器被更新后，TLB 中缓存的旧地址映射就失效了。如果不刷新 TLB，CPU 可能使用旧进程的地址映射来翻译新进程的虚拟地址，导致访问到错误的物理内存。`sfence_vma()` 指令（RISC-V）或 `invlpg` 指令（x86）会清空 TLB 中的所有条目。这个操作本身大约需要 100-1000 纳秒，但更重要的是它带来的间接成本——后续的每次内存访问都需要重新查页表。

**内核栈切换**：每个进程有自己独立的内核栈，用于在内核态执行系统调用和中断处理。切换进程时，内核栈指针也需要更新。这个操作很快（~10-30ns），因为它只是一个寄存器写入。

**调度器决策**：选择下一个要运行的进程需要调度器算法的执行。在简单的调度算法（如 Round Robin）中，这只是遍历就绪队列找下一个进程，开销约为 100-500ns。在复杂的调度算法（如 Linux 的 CFS）中，需要维护红黑树、计算虚拟运行时间等，开销可能达到 1-5μs。

#### **间接成本（Indirect Costs）**

间接成本是上下文切换中**最隐蔽但最昂贵**的部分。它们不是由切换代码本身引起的，而是由切换导致的硬件状态变化引起的。理解这些间接成本需要一些计算机体系结构的知识。

**L1/L2 缓存失效**：现代 CPU 有多级缓存（L1、L2、L3），用于加速内存访问。L1 缓存通常为 32-64KB，L2 缓存为 256KB-1MB。当进程 A 运行时，它频繁访问的数据和指令会被加载到缓存中。切换到进程 B 后，进程 B 需要的数据和指令完全不同，它们需要从主存加载到缓存中，这个过程称为"缓存预热"（cache warmup）。缓存未命中的代价很高：L1 未命中需要 ~5ns（从 L2 加载），L2 未命中需要 ~20ns（从 L3 加载），L3 未命中需要 ~100ns（从主存加载）。如果进程的工作集（频繁访问的数据集合）为 256KB，而 L1 缓存只有 64KB，那么切换后大约需要几十次到上百次缓存未命中才能完成预热。

**TLB 未命中**：前面提到，地址空间切换会刷新 TLB。刷新后，新进程的前几次内存访问都会触发 TLB 未命中，需要遍历页表进行地址翻译。一次页表遍历（在四级页表的 x86-64 上）需要 4 次内存访问，每次约 100ns，总共约 400ns。如果进程的工作集跨越多个页面，可能需要十几次甚至几十次 TLB 未命中。

**分支预测失效**：现代 CPU 使用分支预测器来猜测条件分支的方向，以保持指令流水线满载。分支预测器会学习程序的分支模式。切换到新进程后，分支预测器中的历史信息是旧进程的，对新进程的分支行为毫无参考价值。这会导致大量分支预测错误，每次错误大约浪费 10-20 个 CPU 周期。

**指令流水线刷新**：CPU 的指令流水线中可能还有旧进程的指令在处理中。切换到新进程后，这些指令必须被丢弃，流水线需要重新填充。这大约浪费 10-20 个 CPU 周期。

> **定量分析**：根据 lmbench 基准测试，在现代 x86-64 处理器上，一次上下文切换的总开销约为 **1-5 微秒**。其中直接成本（寄存器保存/恢复、地址空间切换代码）约占 10-20%，间接成本（缓存预热、TLB 未命中、分支预测失效）占 80-90%。这意味着，即使你把上下文切换的汇编代码优化到极致，也只能减少 10-20% 的总开销——真正的性能瓶颈在于硬件状态的恢复。

### 4.1.4 上下文切换频率与系统性能

上下文切换频率是衡量系统负载和调度效率的重要指标。通过监控这个指标，可以判断系统是否过载、是否存在调度问题、或者是否有进程在频繁地等待 I/O。

**过低的切换频率**（如每秒几十次）通常意味着进程长期占用 CPU 不让出，这可能是以下原因：
- 进程是 CPU 密集型的，没有 I/O 操作
- 调度器的时间片设置过大
- 系统中只有少数进程在运行

**过高的切换频率**（如每秒几十万次）通常意味着：
- 进程太多，每个进程分到的 CPU 时间太少
- 大量进程在频繁地进行 I/O 操作
- 存在"锁竞争"问题，进程在自旋等待锁

**典型值**：桌面系统通常为 1000-10000 次/秒，Web 服务器通常为 10000-100000 次/秒，高并发数据库可能达到 100000+ 次/秒。

```bash
# 使用 vmstat 查看系统级上下文切换率
$ vmstat 1
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs  us sy id wa
 2  0      0 512000  64000 2048000    0    0     0     0  500 2000  10  5 85  0
 1  1      0 500000  64000 2048000    0    0  1024     0  800 5000  15  8 72  5
                                                          ↑    ↑
                                                       中断  上下文切换
```

在上面的输出中，`cs` 列显示每秒上下文切换次数。第一行显示 2000 次/秒，这是正常的。第二行显示 5000 次/秒，同时 `bi`（块输入）增加到 1024，说明有进程在进行磁盘读取，导致了更多的自愿上下文切换。

```bash
# 使用 pidstat 查看进程级上下文切换统计
$ pidstat -w 1
# 输出示例：
#   UID   PID   cswch/s nvcswch/s  Command
#     0  1234    150.00      5.00  nginx
#     0  5678   1200.00    500.00  java
#     0  9012      0.00      0.00  sleep
```

这里有两个重要指标：
- **cswch/s**（自愿上下文切换/秒）：进程主动让出 CPU 的次数，通常是等待 I/O。nginx 的 150 次/秒说明它在频繁处理网络 I/O。
- **nvcswch/s**（非自愿上下文切换/秒）：进程被强制让出 CPU 的次数，通常是时间片耗尽。java 的 500 次/秒说明它竞争 CPU 很激烈，可能是线程太多或者时间片太短。

## 4.2 上下文切换机制

### 4.2.1 上下文切换的完整流程

现在我们已经理解了什么是上下文、为什么需要切换、以及切换的开销，接下来让我们深入到机制层面，看看一次完整的上下文切换是如何在操作系统内部执行的。

一次完整的上下文切换涉及八个步骤，每个步骤都有其特定的目的和实现细节。这些步骤不是孤立的，而是紧密配合的——前一步的输出是后一步的输入，任何一步出错都会导致整个切换失败。

```
┌─────────────────────────────────────────────────────────────────┐
│                    上下文切换完整流程                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ① 触发切换（定时器中断 / I/O 阻塞 / 系统调用）                    │
│       ↓                                                         │
│  ② 陷入内核态（硬件自动保存部分寄存器到内核栈）                      │
│       ↓                                                         │
│  ③ 保存旧进程的完整上下文（寄存器 → PCB/trapframe）                │
│       ↓                                                         │
│  ④ 调用调度器（scheduler），选择下一个就绪进程                      │
│       ↓                                                         │
│  ⑤ 切换地址空间（更新页表基址寄存器）                              │
│       ↓                                                         │
│  ⑥ 恢复新进程的上下文（PCB/trapframe → 寄存器）                    │
│       ↓                                                         │
│  ⑦ 返回用户态（硬件恢复部分寄存器，跳转到 PC）                      │
│       ↓                                                         │
│  ⑧ 新进程从上次暂停处继续执行                                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

让我们逐一详细分析每个步骤。

**步骤一：触发切换**

上下文切换不会无缘无故发生——它必须有一个触发原因。最常见的触发原因是定时器中断（时间片用完）、I/O 中断（设备数据就绪）、系统调用（进程主动请求等待）、以及异常（如页错误）。无论哪种触发原因，CPU 都会暂停当前进程的执行，开始切换流程。

**步骤二：陷入内核态**

当触发事件发生时，CPU 硬件会自动执行一系列操作：将当前程序计数器（PC）保存到特殊寄存器（RISC-V 的 `sepc`，x86 的中断栈帧），将当前处理器状态保存到特殊寄存器（RISC-V 的 `sstatus`），将触发原因记录到特殊寄存器（RISC-V 的 `scause`），然后将 CPU 切换到内核模式，并跳转到操作系统预先设置的中断处理程序入口地址（RISC-V 的 `stvec` 寄存器中保存的地址）。

这个过程是**硬件自动完成**的，操作系统代码还没有开始执行。硬件做的这些事情相当于"先把门打开，让操作系统进来"。

**步骤三：保存旧进程的上下文**

这是上下文切换的核心步骤之一。操作系统必须将 CPU 中所有属于旧进程的状态完整保存到内存中（通常是进程的 PCB 或 trapframe 中）。在 xv6 中，这个过程分为两个层次：

首先，在 trampoline.S 的 `uservec` 中，操作系统将所有用户态寄存器（x1-x31）保存到进程的 trapframe 页中。这个页在用户页表和内核页表中都有映射，所以即使切换了页表，仍然可以访问。然后，在 `sched()` 函数中，操作系统将内核态的 callee-saved 寄存器（ra, sp, s0-s11）保存到进程的 context 结构中。

为什么需要两个层次的保存？因为 trapframe 保存的是用户态的完整寄存器状态（用于将来返回用户态），而 context 保存的是内核态的寄存器状态（用于将来被调度器重新调度）。它们服务于不同的目的。

**步骤四：调度器决策**

保存完旧进程的上下文后，操作系统调用调度器来选择下一个要运行的进程。调度器是操作系统的"大脑"之一，它决定了 CPU 资源如何在多个进程之间分配。调度器的输入是就绪队列（所有处于 RUNNABLE 状态的进程），输出是下一个要运行的进程。

**步骤五：切换地址空间**

选定了新进程后，操作系统需要更新页表基址寄存器，使 CPU 使用新进程的页表。这一步的代价很高，因为它会导致 TLB 刷新（除非使用了 ASID 优化）。

**步骤六：恢复新进程的上下文**

这是步骤三的逆操作。操作系统从新进程的 context 中加载内核态寄存器，从 trapframe 中加载用户态寄存器。

**步骤七和八：返回用户态并继续执行**

最后，操作系统执行 `sret` 指令（RISC-V）或 `iret` 指令（x86），将 CPU 从内核模式切回用户模式，并跳转到新进程的程序计数器指向的地址。新进程从它上次被暂停的地方继续执行，完全不知道中间发生了什么。

### 4.2.2 步骤详解：保存旧进程上下文

让我们更深入地看看保存上下文的具体实现。在 xv6（RISC-V）中，保存过程涉及两个关键的数据结构和两段关键的代码。

**trapframe** 是一个内存页（4096 字节），专门用于保存用户态的寄存器。每个进程有自己的 trapframe 页，在进程创建时分配。trapframe 中存储着所有 32 个通用寄存器的值、`sepc`（程序计数器）、`sstatus`（处理器状态）等。当进程从用户态陷入内核态时，trampoline.S 中的 `uservec` 代码会将这些寄存器的值逐个写入 trapframe。

**context** 是一个 14 个 uint64 的小结构体（112 字节），存储在进程的 PCB 中。它只保存 callee-saved 寄存器（ra, sp, s0-s11），因为 caller-saved 寄存器已经由编译器在函数调用时自动保存到了栈上。context 用于内核态的上下文切换——当进程在内核态被切换出去时，`swtch()` 函数将这些寄存器保存到 context 中。

```c
// xv6: kernel/trap.c - usertrap()
// 这个函数在进程从用户态陷入内核态时被调用
void usertrap(void) {
    // 此时硬件已经自动完成了以下操作：
    //   - 将 PC 保存到 sepc 寄存器
    //   - 将处理器状态保存到 sstatus 寄存器
    //   - 将中断原因保存到 scause 寄存器
    //   - 切换到 S 模式（内核模式）
    //   - 跳转到 stvec 指向的地址（uservec）
    
    // uservec（trampoline.S）已经将所有用户态寄存器保存到 trapframe
    // 现在我们在进程 A 的内核栈上执行
    
    // 检查是否是系统调用
    if (r_scause() == 8) {
        // 系统调用处理...
    }
    
    // 检查是否是定时器中断
    if (which_dev == 2) {
        // 时间片用完，让出 CPU
        yield();  // → yield() → sched() → swtch()
    }
    
    // 准备返回用户态
    usertrapret();
}
```

### 4.2.3 步骤详解：切换地址空间

地址空间切换是上下文切换中**开销最大**的步骤之一。为什么切换地址空间如此昂贵？让我们从硬件层面理解。

每个进程有自己的虚拟地址空间，由页表定义。页表是一个存储在物理内存中的数据结构，它将虚拟地址映射到物理地址。CPU 中有一个特殊的寄存器——页表基址寄存器（RISC-V 的 `satp`，x86 的 `CR3`）——指向当前进程的页表的物理地址。

当 CPU 需要访问一个虚拟地址时，它首先查询 TLB（Translation Lookaside Buffer），这是一个高速缓存，存储着最近使用的地址映射。如果 TLB 中有对应的条目（TLB 命中），地址翻译在 ~1ns 内完成。如果没有（TLB 未命中），CPU 需要遍历页表，逐级查找，这需要 4 次内存访问（四级页表），每次约 100ns，总共约 400ns。

问题在于：TLB 中的条目是与特定页表关联的。进程 A 的虚拟地址 0x1000 可能映射到物理地址 0x5000，而进程 B 的虚拟地址 0x1000 可能映射到物理地址 0x8000。如果切换到进程 B 后不清空 TLB，CPU 可能使用进程 A 的映射（0x1000 → 0x5000）来翻译进程 B 的地址，这会导致进程 B 访问到错误的物理内存。

因此，切换地址空间时必须刷新 TLB。刷新 TLB 本身只需要执行一条指令（`sfence_vma`），但它的后果是严重的：新进程的前几次内存访问都会触发 TLB 未命中，需要从主存中遍历页表。如果新进程的工作集跨越多个页面，可能需要几十次 TLB 未命中才能完成"TLB 预热"。

```c
// xv6: kernel/proc.c - scheduler()
// 调度器在选择新进程后切换地址空间
void scheduler(void) {
    for (;;) {
        for (p = proc; p < &proc[NPROC]; p++) {
            acquire(&p->lock);
            if (p->state == RUNNABLE) {
                p->state = RUNNING;
                c->proc = p;
                
                // 切换到新进程的页表
                w_satp(MAKE_SATP(p->pagetable));
                sfence_vma_zero();  // 刷新 TLB！这是性能瓶颈
                
                // 切换上下文
                swtch(&c->context, &p->context);
                
                // 当进程 p 让出 CPU 后，回到这里
                // 注意：此时页表已经切回调度器的页表（在 swtch 中完成）
                c->proc = 0;
            }
            release(&p->lock);
        }
    }
}
```

> **TLB 刷新的代价**：`sfence_vma()` 会使 TLB 中的所有条目失效。这意味着新进程的前几次内存访问都会触发 TLB 未命中，需要遍历页表进行地址翻译。根据工作集大小的不同，这可能导致几十到几百纳秒的额外延迟。这是上下文切换间接成本的主要来源之一。

**ASID 优化**：为了解决 TLB 刷新的性能问题，现代处理器引入了 ASID（Address Space Identifier）机制。ASID 的核心思想是：为每个地址空间分配一个唯一的标识符，TLB 条目不仅存储虚拟地址 → 物理地址映射，还存储对应的 ASID。当 CPU 查找 TLB 时，不仅匹配虚拟地址，还匹配当前进程的 ASID。这样，即使 TLB 中同时存在多个进程的地址映射，也不会混淆。

使用 ASID 后，切换地址空间时只需要更新 ASID 寄存器（~10ns），而不需要刷新 TLB。新进程可以继续使用 TLB 中已有的条目（只要它们的 ASID 匹配），大大减少了 TLB 未命中的次数。这可以将上下文切换的总开销减少 30-50%。

### 4.2.4 步骤详解：恢复新进程上下文

恢复上下文是保存上下文的逆操作。在 xv6 中，`swtch()` 函数首先从新进程的 context 中加载 14 个 callee-saved 寄存器，然后执行 `ret` 指令跳转到新进程上次调用 `swtch()` 时的返回地址。

这里有一个微妙但重要的点：`swtch()` 不需要显式地加载 caller-saved 寄存器（t0-t6, a0-a7），因为这些寄存器是被编译器管理的。当新进程从 `swtch()` 返回后，它会继续执行 `sched()` 函数的后续代码，而 `sched()` 是由 C 编译器生成的，编译器会在需要时从栈上恢复 caller-saved 寄存器。

恢复用户态寄存器则是在 `userret`（trampoline.S）中完成的。`userret` 从 trapframe 中加载所有用户态寄存器，然后执行 `sret` 指令，将 CPU 从内核模式切回用户模式，并跳转到 `sepc` 指向的地址（即用户态程序计数器）。

## 4.3 xv6 上下文切换实现

xv6 的上下文切换是理解操作系统调度机制的最佳切入点。与 Linux 内核的数千行调度代码相比，xv6 的实现只有几十行汇编和几百行 C 代码，但它完整地展示了从一个用户进程切换到另一个用户进程的全部细节。正如 OSTEP 教材所说："简单的实现可以帮助你理解核心思想，而不被复杂的优化所淹没。"

### 4.3.1 struct context 数据结构

xv6 使用 `struct context` 保存进程在内核态的寄存器状态。这个结构体的设计体现了 RISC-V 调用约定（ABI）的精妙之处。

```c
// kernel/defs.h
struct context {
    uint64 ra;      // 返回地址 (x1) — 函数返回时跳转到的地址
    uint64 sp;      // 栈指针 (x2) — 当前内核栈的位置
    uint64 s0;      // 保存寄存器 s0/fp (x8) — 帧指针
    uint64 s1;      // 保存寄存器 s1 (x9)
    uint64 s2;      // 保存寄存器 s2 (x18)
    uint64 s3;      // 保存寄存器 s3 (x19)
    uint64 s4;      // 保存寄存器 s4 (x20)
    uint64 s5;      // 保存寄存器 s5 (x21)
    uint64 s6;      // 保存寄存器 s6 (x22)
    uint64 s7;      // 保存寄存器 s7 (x23)
    uint64 s8;      // 保存寄存器 s8 (x24)
    uint64 s9;      // 保存寄存器 s9 (x25)
    uint64 s10;     // 保存寄存器 s10 (x26)
    uint64 s11;     // 保存寄存器 s11 (x27)
};
```

你可能会问：RISC-V 有 32 个通用寄存器（x0-x31），为什么 `struct context` 只保存 14 个？答案在于 **RISC-V 的调用约定（Calling Convention）**。

在 RISC-V 的调用约定中，寄存器被分为两类：

**Callee-saved 寄存器**（被调用者保存）：包括 ra, sp, s0-s11。当一个函数（被调用者）要使用这些寄存器时，它必须先将原来的值保存到栈上，函数返回前再恢复。这意味着调用者不需要担心这些寄存器的值被被调用者修改——它们在函数调用前后保持不变。

**Caller-saved 寄存器**（调用者保存）：包括 t0-t6, a0-a7。当一个函数（调用者）要调用另一个函数时，如果它还想保留这些寄存器中的值，它必须在调用前自己将它们保存到栈上。因为被调用者可以随意修改这些寄存器。

`swtch()` 是一个普通的 C 函数调用。当 `sched()` 调用 `swtch()` 时，编译器已经按照调用约定处理好了一切：caller-saved 寄存器要么已经不再需要（中间计算结果），要么已经被编译器保存到了栈上。因此，`swtch()` 只需要保存 callee-saved 寄存器即可——这是调用约定保证的"最小保存集"。

这种设计的好处是显而易见的：只保存 14 个寄存器（112 字节）比保存 32 个寄存器（256 字节）快得多。而且，这种设计是**正确的**——调用约定保证了不会丢失任何状态。

### 4.3.2 swtch() 汇编代码详解

`swtch()` 是 xv6 中上下文切换的核心——它是唯一一段直接操作寄存器的代码，是连接"旧进程"和"新进程"的桥梁。让我们逐行分析这段汇编代码。

```asm
# kernel/swtch.S
# void swtch(struct context *old, struct context *new);
#
# 功能：保存当前 CPU 的 callee-saved 寄存器到 old，
#       从 new 加载 callee-saved 寄存器，然后返回。
#
# 调用约定：
#   a0 = old context 指针（当前进程的 context）
#   a1 = new context 指针（目标进程的 context）
#
# 这个函数在 C 语言中的原型是：
#   void swtch(struct context *old, struct context *new);
# 调用 swtch(&current->context, &next->context) 即可完成切换。

.globl swtch
swtch:
        # ========================================
        # 第一部分：保存旧进程的 callee-saved 寄存器
        # ========================================
        # 此时 a0 指向旧进程的 context 结构体
        # sd = Store Doubleword（64 位存储）
        
        sd ra, 0(a0)        # 保存 ra（返回地址）到 old->ra
                             # ra 是调用 swtch() 时的返回地址
                             # 恢复时，ret 指令会跳转到这个地址
        sd sp, 8(a0)        # 保存 sp（栈指针）到 old->sp
                             # 这是旧进程的内核栈指针
        sd s0, 16(a0)       # 保存 s0（帧指针）到 old->s0
        sd s1, 24(a0)       # 保存 s1 到 old->s1
        sd s2, 32(a0)       # 保存 s2 到 old->s2
        sd s3, 40(a0)       # 保存 s3 到 old->s3
        sd s4, 48(a0)       # 保存 s4 到 old->s4
        sd s5, 56(a0)       # 保存 s5 到 old->s5
        sd s6, 64(a0)       # 保存 s6 到 old->s6
        sd s7, 72(a0)       # 保存 s7 到 old->s7
        sd s8, 80(a0)       # 保存 s8 到 old->s8
        sd s9, 88(a0)       # 保存 s9 到 old->s9
        sd s10, 96(a0)      # 保存 s10 到 old->s10
        sd s11, 104(a0)     # 保存 s11 到 old->s11
                             # 至此，旧进程的所有 callee-saved 寄存器
                             # 都已保存到内存中（old->context）

        # ========================================
        # 第二部分：加载新进程的 callee-saved 寄存器
        # ========================================
        # 此时 a1 指向新进程的 context 结构体
        # ld = Load Doubleword（64 位加载）
        
        ld ra, 0(a1)        # 加载 new->ra 到 ra
                             # 这是新进程上次调用 swtch() 时的返回地址
        ld sp, 8(a1)        # 加载 new->sp 到 sp
                             # 现在栈指针指向新进程的内核栈！
        ld s0, 16(a1)       # 加载 new->s0 到 s0
        ld s1, 24(a1)       # 加载 new->s1 到 s1
        ld s2, 32(a1)       # 加载 new->s2 到 s2
        ld s3, 40(a1)       # 加载 new->s3 到 s3
        ld s4, 48(a1)       # 加载 new->s4 到 s4
        ld s5, 56(a1)       # 加载 new->s5 到 s5
        ld s6, 64(a1)       # 加载 new->s6 到 s6
        ld s7, 72(a1)       # 加载 new->s7 到 s7
        ld s8, 80(a1)       # 加载 new->s8 到 s8
        ld s9, 88(a1)       # 加载 new->s9 到 s9
        ld s10, 96(a1)      # 加载 new->s10 到 s10
        ld s11, 104(a1)     # 加载 new->s11 到 s11
                             # 至此，新进程的所有 callee-saved 寄存器
                             # 都已从内存加载到 CPU 寄存器中

        # ========================================
        # 第三部分：跳转到新进程
        # ========================================
        ret                  # ret = jalr x0, ra, 0
                             # 即跳转到 ra 寄存器指向的地址
                             # ra 现在是新进程的返回地址
                             # → CPU 开始在新进程的上下文中执行！
```

让我们用一个具体的例子来理解 `swtch()` 的执行过程。假设进程 A 正在运行，调度器决定切换到进程 B：

1. **调用 swtch()**：进程 A 在 `sched()` 中调用 `swtch(&A->context, &B->context)`。此时 `a0 = &A->context`，`a1 = &B->context`。

2. **保存 A 的寄存器**：`swtch()` 将当前 CPU 中的 ra, sp, s0-s11 保存到 `A->context` 中。注意，此时 ra 的值是 `sched()` 中调用 `swtch()` 之后的下一条指令的地址——这是 A 将来恢复执行时的"返回点"。

3. **加载 B 的寄存器**：`swtch()` 从 `B->context` 中加载 ra, sp, s0-s11 到 CPU 寄存器中。现在 CPU 的 ra 指向 B 上次调用 `swtch()` 的返回地址，sp 指向 B 的内核栈。

4. **执行 ret**：`ret` 指令跳转到 ra 指向的地址。这个地址是 B 上次在 `sched()` 中调用 `swtch()` 之后的下一条指令。CPU 开始在进程 B 的上下文中执行。

关键洞察：`swtch()` 是一个**对称操作**。它被调用两次——一次从进程 A 切换到调度器，一次从调度器切换到进程 B。每次调用都执行"保存旧 + 加载新 + ret"的模式。整个过程中，`swtch()` 被调用了两次，但每次只执行一次"保存 + 加载"。

### 4.3.3 sched() 与 scheduler() 的协作

xv6 的调度采用**两阶段**设计：`sched()` 负责从当前进程切出，`scheduler()` 负责选择下一个进程切入。这种设计将"切出"和"切入"两个逻辑清晰地分离，使得代码更易理解和维护。

#### **sched() — 从当前进程切出**

`sched()` 是进程主动让出 CPU 的入口。它在以下情况下被调用：进程的时间片用完（定时器中断 → yield() → sched()）、进程等待 I/O（sleep() → sched()）、进程主动让出 CPU（yield() → sched()）。

```c
// kernel/proc.c
void sched(void) {
    struct proc *p = myproc();
    
    // 安全性检查：确保调用 sched() 的前置条件满足
    if (!holding(&ptable.lock))
        panic("sched ptable.lock");    // 必须持有进程表锁
    if (mycpu()->noff != 1)
        panic("sched locks");          // 不能持有其他锁
    if (p->state == RUNNING)
        panic("sched running");        // 进程状态必须已改为非 RUNNING
    if (intr_get())
        panic("sched interruptible");  // 中断必须已禁用

    // 保存当前 CPU 的中断使能状态
    intena = mycpu()->intena;
    
    // 执行上下文切换！
    // 保存 p->context，加载 mycpu()->context（调度器的 context）
    swtch(&p->context, &mycpu()->context);
    
    // ↑↑↑ 这里是"切换点" ↑↑↑
    // 当 swtch() 返回时，说明进程 p 被重新调度了
    // 此时 CPU 已经在进程 p 的内核栈上执行
    
    // 恢复中断使能状态
    mycpu()->intena = intena;
}
```

`sched()` 中的四个 `panic()` 检查非常值得深入理解。它们不是多余的——每一个都防止了一类严重的 Bug：

- `holding(&ptable.lock)`：进程表锁必须被持有，否则在切换过程中其他 CPU 可能同时修改进程状态，导致数据竞争。
- `mycpu()->noff == 1`：只能持有一把锁（ptable.lock）。如果持有其他锁，切换后其他 CPU 可能永远等待这把锁，导致死锁。
- `p->state != RUNNING`：进程状态必须已经被修改为 RUNNABLE（或其他非 RUNNING 状态），否则调度器可能选择这个进程，导致两个 CPU 同时运行同一个进程。
- `!intr_get()`：中断必须被禁用。如果中断开启，切换过程中可能发生定时器中断，导致在已经切换的状态下再次尝试切换，造成混乱。

#### **scheduler() — 选择下一个进程**

`scheduler()` 是每个 CPU 核心的主循环。它永远不会返回——它不断地从就绪队列中选择进程，运行它们，然后在它们让出 CPU 后选择下一个。

```c
// kernel/proc.c
void scheduler(void) {
    struct proc *p;
    struct cpu *c = mycpu();
    c->proc = 0;  // 当前没有运行任何进程
    
    for (;;) {
        // 确保中断已启用
        // （避免所有 CPU 都在 scheduler 中自旋，无法响应中断）
        intr_on();
        
        // 遍历进程表，寻找 RUNNABLE 进程
        // 这是一个简单的线性扫描——xv6 的设计哲学是"简单优先"
        for (p = proc; p < &proc[NPROC]; p++) {
            acquire(&p->lock);
            
            if (p->state == RUNNABLE) {
                // 找到一个可运行进程！
                // 将其状态改为 RUNNING（告诉其他 CPU 这个进程已被占用）
                p->state = RUNNING;
                c->proc = p;  // 设置当前 CPU 正在运行的进程
                
                // 执行上下文切换
                // 保存 scheduler 的 context，加载 p 的 context
                swtch(&c->context, &p->context);
                
                // ↑↑↑ 这里是"切换点" ↑↑↑
                // 当 swtch() 返回时，说明进程 p 让出了 CPU
                // （可能是因为时间片用完、I/O 阻塞、或进程终止）
                
                // 清空当前 CPU 正在运行的进程
                c->proc = 0;
            }
            
            release(&p->lock);
        }
    }
}
```

`scheduler()` 的设计有几个值得注意的细节：

**无限循环**：`scheduler()` 使用 `for (;;)` 无限循环。这是因为它永远不会"完成"——即使所有进程都阻塞了，它也会继续循环，等待进程变为 RUNNABLE。在 xv6 中，如果没有任何进程可运行，CPU 会在这个循环中空转，直到有进程被唤醒。

**中断启用**：循环开始时调用 `intr_on()` 是关键的。如果中断被禁用，定时器中断就无法触发，其他 CPU 上的进程就无法唤醒当前 CPU 上的进程，可能导致所有 CPU 都在 scheduler 中空转，系统死锁。

**每进程锁**：`scheduler()` 使用每进程的锁（`p->lock`）而不是全局锁。这允许多个 CPU 同时在进程表中搜索可运行进程，提高了多核系统的并发性。

### 4.3.4 完整切换路径

让我们追踪一个完整的上下文切换路径：**进程 A（正在运行）→ 定时器中断 → 调度 → 进程 B（开始运行）**。这个追踪将把前面所有的知识点串联起来。


## 4.4 Linux 上下文切换

### 4.4.1 schedule() 函数调用链

Linux 内核的调度系统比 xv6 复杂得多——它需要支持数千个进程、多核处理器、实时调度、cgroup 资源限制等。但核心思想与 xv6 相同：保存旧进程状态 → 选择新进程 → 恢复新进程状态。

Linux 的调度入口是 `schedule()` 函数。它在以下情况下被调用：

1. **定时器中断**：当前进程的时间片用完，`scheduler_tick()` 设置 `TIF_NEED_RESCHED` 标志，中断返回时检查此标志并调用 `schedule()`。
2. **系统调用返回**：在从系统调用返回用户态之前，检查是否需要重新调度。
3. **进程主动让出**：进程调用 `schedule()` 或 `cond_resched()` 主动让出 CPU。
4. **进程阻塞**：进程等待锁、信号量、I/O 等时调用 `schedule()`。

```c
// kernel/sched/core.c（简化版）
asmlinkage __visible void __sched schedule(void) {
    struct task_struct *tsk = current;
    
    // 提交待处理的 I/O 工作
    sched_submit_work(tsk);
    
    // 调用核心调度函数
    __schedule(SM_NONE);
}

static void __sched __schedule(unsigned int sched_mode) {
    struct task_struct *prev, *next;
    struct rq *rq;
    int cpu;
    
    // 获取当前 CPU 的运行队列
    // 每个 CPU 有自己的运行队列，避免锁竞争
    cpu = smp_processor_id();
    rq = cpu_rq(cpu);
    prev = rq->curr;  // 当前正在运行的进程
    
    // ... 省略锁获取、信号处理等 ...
    
    // 选择下一个要运行的进程
    // 这是调度算法的核心——CFS 使用红黑树选择 vruntime 最小的进程
    next = pick_next_task(rq, prev, &rf);
    
    if (prev != next) {
        // 需要切换！
        rq->nr_switches++;  // 统计切换次数
        
        // 执行上下文切换
        rq = context_switch(rq, prev, next, &rf);
    }
}
```

### 4.4.2 context_switch() 实现

`context_switch()` 是 Linux 中执行实际切换的函数。它分为两步：切换地址空间和切换寄存器状态。

```c
// kernel/sched/core.c（简化版）
static __always_inline struct rq *
context_switch(struct rq *rq, struct task_struct *prev,
               struct task_struct *next, struct rq_flags *rf)
{
    // 第一步：切换地址空间
    if (!next->mm) {
        // next 是内核线程，没有自己的地址空间
        // 借用 prev 的地址空间（lazy TLB 模式）
        atomic_inc(&prev->mm->mm_users);
        enter_lazy_tlb(prev->mm, next);
    } else {
        // next 是普通进程，有自己的地址空间
        // 切换页表基址寄存器（x86: CR3, RISC-V: satp）
        switch_mm_irqs_off(prev->active_mm, next->mm, next);
    }
    
    // 第二步：切换寄存器状态
    // switch_to 是一个架构相关的宏，通常展开为汇编代码
    // 它保存 prev 的寄存器，加载 next 的寄存器
    switch_to(prev, next, prev);
    
    // ↑↑↑ 这里是切换点 ↑↑↑
    // 当 switch_to 返回时，当前进程已经变成了 next
    // 注意第三个参数 prev：它在返回时被修改为"切换前的 next"
    // 这是因为 switch_to 保存的是当前进程的寄存器到 prev->thread
    // 但返回时当前进程已经变成了 next
    
    return finish_task_switch(prev);
}
```

`switch_to` 的三参数设计是 Linux 内核中一个著名的技巧。`switch_to(prev, next, prev)` 的第三个参数在返回时会被修改为"切换前的 next"。这是因为：

1. 进程 A 调用 `switch_to(A, B, A)`
2. `switch_to` 保存 A 的寄存器，加载 B 的寄存器
3. B 从它上次调用 `switch_to` 的地方恢复执行
4. B 上次的调用是 `switch_to(B, A, B)`
5. 所以 B 看到的 `prev` 参数是 A（切换前的进程）

这个技巧避免了额外的全局变量来跟踪"前一个进程"，是 Linux 内核调度器的经典设计。

### 4.4.3 进程上下文 vs 中断上下文

Linux 内核中有两种截然不同的执行上下文，理解它们的区别对于编写正确的内核代码至关重要。

**进程上下文（Process Context）**：当进程通过系统调用进入内核态，或者在内核态中被调度执行时，它处于进程上下文。此时有一个明确的"当前进程"（`current` 宏可以访问），进程可以睡眠（调用 `schedule()` 让出 CPU），可以访问用户空间内存（通过 `copy_from_user()`）。

**中断上下文（Interrupt Context）**：当硬件中断发生时，CPU 进入中断上下文。此时没有"当前进程"的概念——中断处理程序是为硬件服务的，不是为某个特定进程服务的。中断处理程序**不能睡眠**（因为没有进程可以被调度），不能访问用户空间内存（因为不知道当前映射的是哪个进程的页表），执行时间必须尽可能短。

```c
// 判断当前执行上下文
in_interrupt()    // 非零表示在中断/软中断上下文
in_task()         // 非零表示在进程上下文

// 在中断上下文中睡眠会导致 panic
// 正确做法：使用工作队列（workqueue）将工作延迟到进程上下文
```

### 4.4.4 ASID（Address Space Identifier）

传统的上下文切换在切换地址空间时需要刷新整个 TLB，这是一个非常昂贵的操作。ASID（Address Space Identifier）是现代处理器提供的一种优化机制，可以显著降低这个开销。

**问题**：为什么切换地址空间需要刷新 TLB？

TLB 是一个缓存，存储着虚拟地址 → 物理地址的映射。不同进程的同一虚拟地址可能映射到不同的物理地址。例如，进程 A 的虚拟地址 0x400000 可能映射到物理地址 0x1000000（A 的代码段），而进程 B 的虚拟地址 0x400000 可能映射到物理地址 0x2000000（B 的代码段）。如果 TLB 中缓存了 A 的映射，切换到 B 后不清空 TLB，B 访问 0x400000 时会错误地映射到 0x1000000，读取到 A 的代码。

**解决方案**：ASID 为每个地址空间分配一个唯一标识符（通常 8-16 位）。TLB 条目不仅存储虚拟地址 → 物理地址映射，还存储对应的 ASID。当 CPU 查找 TLB 时，不仅匹配虚拟地址，还匹配当前进程的 ASID。这样，即使 TLB 中同时存在多个进程的地址映射，也不会混淆。

使用 ASID 后，切换地址空间时只需要更新 ASID 寄存器（~10ns），而不需要刷新 TLB。新进程可以继续使用 TLB 中已有的条目（只要它们的 ASID 匹配），大大减少了 TLB 未命中的次数。

| 架构 | ASID 位数 | 最大 ASID 数 | 实现方式 |
|------|----------|-------------|----------|
| ARMv8 | 8 或 16 位 | 256 或 65536 | TTBR 寄存器中的 ASID 字段 |
| x86-64 | 12 位 | 4096 | PCID（Process Context Identifier） |
| RISC-V | 通常 9 位 | 512 | satp 寄存器中的 ASID 字段 |

> **ASID 回绕问题**：当 ASID 空间用完时（例如 8 位 ASID 只能表示 256 个地址空间），操作系统必须刷新 TLB 并重新分配 ASID。这是一种罕见但必须处理的情况。Linux 内核使用一个全局 ASID 计数器，当计数器溢出时触发 TLB 刷新。

## 4.5 测量上下文切换

### 4.5.1 lmbench 微基准测试

`lmbench` 是一套由 Larry McVoy 开发的经典系统性能基准测试工具。其中 `lat_ctx` 专门测量上下文切换延迟。它的工作原理是：创建 N 个进程，通过管道在它们之间传递令牌（token），每次传递都涉及一次上下文切换。通过测量总时间和切换次数，可以计算出每次切换的平均延迟。

```bash
# 安装 lmbench
$ sudo apt install lmbench    # Ubuntu/Debian
$ brew install lmbench         # macOS

# 测量 2 个进程之间的上下文切换延迟（工作集 = 0KB）
$ lat_ctx -s 0 2
# 典型输出：0.85 microseconds

# 测量不同工作集大小的影响
$ lat_ctx -s 64 2    # 64KB 工作集 → L1 缓存可能失效
$ lat_ctx -s 256 2   # 256KB 工作集 → L2 缓存开始失效
$ lat_ctx -s 1024 2  # 1MB 工作集 → L2 缓存完全失效
$ lat_ctx -s 8192 2  # 8MB 工作集 → L3 缓存受影响

# 测量不同进程数的影响
$ lat_ctx -s 0 2     # 2 个进程
$ lat_ctx -s 0 4     # 4 个进程
$ lat_ctx -s 0 8     # 8 个进程
$ lat_ctx -s 0 16    # 16 个进程
```

典型结果（现代 x86-64 处理器，如 Intel Core i7）：

| 工作集大小 | 上下文切换延迟 | 说明 |
|-----------|---------------|------|
| 0 KB | ~1-2 μs | 纯寄存器保存/恢复 + 地址空间切换 |
| 64 KB | ~2-4 μs | L1 缓存（32-64KB）部分失效 |
| 256 KB | ~4-8 μs | L2 缓存（256KB-1MB）开始失效 |
| 1 MB | ~8-15 μs | L2 缓存完全失效 |
| 8 MB | ~15-30 μs | L3 缓存受影响 |

这些数据清楚地展示了**间接成本**的影响：当工作集从 0KB 增加到 8MB 时，切换延迟增加了 10-20 倍。这完全是由缓存未命中引起的，而不是切换代码本身变慢了。

### 4.5.3 vmstat 与 pidstat

```bash
# vmstat：系统级上下文切换统计
$ vmstat 1
# 关键列：
#   cs  — 每秒上下文切换次数
#   in  — 每秒中断次数
#   us  — 用户态 CPU 时间百分比

#   id  — 空闲 CPU 时间百分比
```

> **自愿 vs 非自愿上下文切换**：
> - **自愿（voluntary）**：进程主动让出 CPU，通常是因为它在等待 I/O 操作完成（如读取文件、等待网络数据）。高自愿切换率通常说明进程是 I/O 密集型的。
> - **非自愿（involuntary）**：进程被强制让出 CPU，通常是因为时间片耗尽。高非自愿切换率通常说明系统中进程太多、CPU 资源紧张，或者时间片设置太短。

## 4.6 面试高频考点与常见陷阱

### 高频面试题

**Q1：上下文切换和模式切换（Mode Switch）有什么区别？**

这是一个经典的面试题，考察你对操作系统基本概念的理解深度。

**模式切换**是指 CPU 在用户态和内核态之间的切换。当进程执行系统调用（如 `read()`）时，CPU 从用户态切换到内核态；当系统调用完成时，CPU 从内核态切回用户态。模式切换不涉及进程的更换——同一个进程在用户态和内核态之间切换。

**上下文切换**是指 CPU 从一个进程切换到另一个进程。这涉及保存旧进程的全部状态、加载新进程的全部状态、切换地址空间等操作。

它们的关系是：上下文切换**一定包含**两次模式切换（因为调度器运行在内核态，切换前必须进入内核态，切换后必须返回用户态），但模式切换**不一定**导致上下文切换（进程执行系统调用后可能继续运行，没有切换到其他进程）。

| 维度 | 模式切换 | 上下文切换 |
|------|---------|-----------|
| **定义** | 用户态 ↔ 内核态的切换 | 从一个进程切换到另一个进程 |
| **触发** | 系统调用、中断、异常 | 调度器决策 |
| **是否切换进程** | ❌ 不切换 | ✅ 切换 |
| **地址空间** | 不变 | 改变（切换页表） |
| **开销** | 较小（~100 ns） | 较大（~1-5 μs） |
| **是否包含** | — | 包含两次模式切换 |

**Q2：为什么 xv6 的 swtch() 只保存 callee-saved 寄存器？**

因为 `swtch()` 是一个普通的 C 函数调用。根据 RISC-V 调用约定（ABI），caller-saved 寄存器（t0-t6, a0-a7）由调用者负责保存。当 `swtch()` 的调用者（如 `sched()`）是 C 编译器生成的代码时，编译器已经在调用 `swtch()` 之前将这些寄存器保存到了栈上。因此，`swtch()` 只需保存 callee-saved 寄存器——这是调用约定保证的"最小保存集"。

**Q3：上下文切换为什么不保存所有 32 个通用寄存器？**

1. **性能**：保存/恢复的寄存器越少，切换越快。只保存 14 个（112 字节）比保存 32 个（256 字节）快一倍多。
2. **ABI 约定**：caller-saved 寄存器已经由编译器在函数调用点保存，不需要 `swtch()` 重复保存。
3. **正确性**：按 ABI 约定操作，不会丢失任何状态——这是调用约定的数学保证。

**Q4：ASID 如何解决 TLB 刷新问题？**

传统方式下，切换地址空间必须刷新 TLB（`sfence_vma`），因为不同进程的相同虚拟地址可能映射到不同物理地址。ASID 为每个地址空间分配唯一 ID，TLB 条目同时标记虚拟地址和 ASID，切换时只需更新当前 ASID，TLB 中的条目仍然有效。这可以将上下文切换开销减少 30-50%。

### 常见陷阱

1. **忘记 TLB 刷新**：修改页表后必须刷新 TLB，否则 CPU 可能使用过时的地址映射。这是一个非常隐蔽的 Bug——大多数时候 TLB 中的旧条目可能恰好是正确的，程序看起来正常工作，但偶尔会访问到错误的物理内存，导致难以复现的崩溃。

2. **在中断上下文中睡眠**：中断上下文不属于任何进程，无法调度。如果在中断处理程序中调用 `sleep()` 或任何可能睡眠的函数（如 `kmalloc(GFP_KERNEL)`），会导致内核 panic。正确做法是使用工作队列将工作延迟到进程上下文。

3. **自旋锁 + 上下文切换 = 死锁**：持有自旋锁时不能进行上下文切换。因为自旋锁的设计假设是"持有锁的时间很短"，其他 CPU 会在自旋等待。如果持有锁的 CPU 切换到另一个进程，而新进程也需要这把锁（或者新进程长期运行不释放锁），其他 CPU 就会永远自旋等待。xv6 在 `sched()` 中通过 `panic("sched locks")` 检查此条件。

4. **ASID 回绕**：当 ASID 空间用完时（如 8 位 ASID 只有 256 个），必须刷新 TLB 并重新分配 ASID。这是一个罕见但必须处理的情况，否则会导致不同进程的 TLB 条目混淆。
