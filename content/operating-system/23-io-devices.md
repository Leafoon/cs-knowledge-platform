## 23.1 I/O 设备基础

### 23.1.1 什么是 I/O 设备？

I/O（Input/Output）设备是计算机系统中负责与外部世界交互的硬件组件。CPU 和内存构成系统的"大脑"，而 I/O 设备则是系统的"感官"和"四肢"。

```
计算机系统中的 I/O 设备：

┌─────────────────────────────────────────────────┐
│                  CPU                            │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐        │
│  │ ALU     │  │ 寄存器   │  │ 缓存    │        │
│  └─────────┘  └─────────┘  └─────────┘        │
└────────────────────┬────────────────────────────┘
                     │ 总线（Bus）
┌────────────────────┼────────────────────────────┐
│                    │         内存（RAM）         │
│  ┌─────────────────┴─────────────────────┐     │
│  │           主存                         │     │
│  └───────────────────────────────────────┘     │
└─────────────────────────────────────────────────┘
                     │
        ┌────────────┼────────────┐
        │            │            │
   ┌────┴────┐  ┌────┴────┐  ┌────┴────┐
   │ 磁盘    │  │ 网卡    │  │ 键盘    │
   │ (块设备) │  │ (网络)  │  │(字符设备)│
   └─────────┘  └─────────┘  └─────────┘
```

I/O 设备的核心特征：
1. **异构性**：不同设备的接口、速度、数据格式差异巨大
2. **速度差异**：设备速度可能比 CPU 慢几个数量级
3. **控制复杂性**：每个设备有自己的控制协议和状态机

### 23.1.2 设备分类

**按数据传输单位分类**：

| 类型 | 特征 | 示例 | 访问方式 |
|------|------|------|----------|
| **块设备** | 以固定大小块为单位传输 | 磁盘、SSD、光盘 | 随机访问，按块寻址 |
| **字符设备** | 以字节流为单位传输 | 键盘、鼠标、串口 | 顺序访问，字节流 |
| **网络设备** | 以数据包为单位传输 | 网卡、Wi-Fi | 报文交换 |

**按传输方向分类**：
- **输入设备**：键盘、鼠标、传感器、摄像头
- **输出设备**：显示器、打印机、扬声器
- **双向设备**：磁盘、网卡、串口

**按速度分类**：

| 设备类型 | 典型速度 | 延迟 |
|----------|----------|------|
| CPU 寄存器 | ~1ns | - |
| L1 缓存 | ~1ns | - |
| 主存（DRAM） | ~100ns | - |
| NVMe SSD | ~10μs | ~10μs |
| SATA SSD | ~50μs | ~50μs |
| HDD | ~10ms | ~5-10ms |
| 网络（LAN） | ~100μs | ~100μs |
| 网络（WAN） | ~10-100ms | ~10-100ms |

**关键洞察**：HDD 的延迟是 CPU 寄存器的 **1000 万倍**。这就是为什么 I/O 优化如此重要——一个磁盘 I/O 的时间，CPU 可以执行数百万条指令。

### 23.1.3 设备控制器

每个 I/O 设备通过**设备控制器（Device Controller）**连接到系统总线。控制器是硬件层面的"翻译器"，将设备的特定协议转换为系统总线的标准协议。

```
设备控制器结构：

┌─────────────────────────────────────────┐
│              设备控制器                   │
│                                         │
│  ┌───────────┐    ┌──────────────────┐ │
│  │ 控制寄存器 │    │   数据寄存器      │ │
│  │ (命令/状态)│    │   (数据缓冲)     │ │
│  └─────┬─────┘    └────────┬─────────┘ │
│        │                   │           │
│  ┌─────┴───────────────────┴─────────┐ │
│  │         控制逻辑                    │ │
│  │    (协议转换、时序控制)             │ │
│  └─────────────────┬─────────────────┘ │
│                    │                   │
└────────────────────┼───────────────────┘
                     │
              ┌──────┴──────┐
              │   I/O 设备   │
              │  (磁盘/网卡) │
              └─────────────┘
```

控制器提供三类寄存器：
1. **状态寄存器（Status Register）**：设备当前状态（忙/就绪/错误）
2. **命令寄存器（Command Register）**：CPU 写入命令让设备执行
3. **数据寄存器（Data Register）**：CPU 与设备交换数据的缓冲区

### 23.1.4 I/O 端口与内存映射 I/O

CPU 如何访问设备控制器的寄存器？有两种方式：

**方式 1：I/O 端口（Port-Mapped I/O）**

x86 架构使用专用的 I/O 地址空间（0x0000 - 0xFFFF），通过专用指令 `in`/`out` 访问：

```c
// x86 I/O 端口访问示例
// 从端口 0x60 读取键盘扫描码
unsigned char scancode = inb(0x60);

// 向端口 0x3F8 写入字符（串口）
outb(0x3F8, 'A');
```

```
I/O 端口地址空间：
0x0000 ─────────────── 0xFFFF
  │                        │
  │  键盘: 0x60            │
  │  串口: 0x3F8           │
  │  磁盘: 0x1F0-0x1F7     │
  │  ...                   │
  └────────────────────────┘
独立于内存地址空间
```

**方式 2：内存映射 I/O（Memory-Mapped I/O / MMIO）**

将设备寄存器映射到内存地址空间，CPU 通过普通的 `load`/`store` 指令访问：

```c
// 内存映射 I/O 示例
// 假设 UART 控制器映射到地址 0x10000000
volatile uint8_t *uart_base = (volatile uint8_t *)0x10000000;

// 写入字符
uart_base[0] = 'A';  // 直接写内存地址

// 读取状态
uint8_t status = uart_base[1];  // 直接读内存地址
```

```
内存映射 I/O 地址空间：
0x00000000 ──────────────────── 0xFFFFFFFF
  │                                   │
  │  用户空间: 0x00000000 - 0x7FFFFFFF │
  │  内核空间: 0x80000000 - 0xBFFFFFFF │
  │  设备映射: 0xC0000000 - 0xFFFFFFFF │
  │    UART:    0x10000000             │
  │    VIRTIO:  0x10001000             │
  │    PLIC:    0x0C000000             │
  └───────────────────────────────────┘
```

**两种方式的对比**：

| 特性 | I/O 端口 | 内存映射 I/O |
|------|----------|-------------|
| 地址空间 | 独立的 I/O 地址空间 | 共享内存地址空间 |
| 访问指令 | 专用指令（in/out） | 普通 load/store |
| 性能 | 需要专用指令，较慢 | 可利用缓存，较快 |
| 保护 | 硬件级别保护 | 页表保护 |
| RISC-V | 不支持 | 统一使用 MMIO |
| x86 | 支持（遗留） | 也支持 |

**xv6 使用 MMIO**：RISC-V 架构统一使用内存映射 I/O，所有设备寄存器都映射到物理内存地址空间。

## 23.2 I/O 方式

CPU 与设备之间如何传输数据？有三种基本方式，每种方式在 CPU 利用率、实现复杂度和适用场景上各有不同。

### 23.2.1 方式一：轮询（Polling）

轮询是最简单的 I/O 方式。CPU 主动反复检查设备状态寄存器，等待设备就绪后进行数据传输。

**基本流程**：

```
轮询方式的数据读取：

CPU                              设备控制器
 │                                   │
 │  1. 写入命令（读取数据）           │
 │ ───────────────────────────────→ │
 │                                   │  设备开始工作
 │                                   │  （寻道、旋转...）
 │  2. 读取状态寄存器                 │
 │ ←─────────────────────────────── │  状态 = BUSY
 │  3. 读取状态寄存器                 │
 │ ←─────────────────────────────── │  状态 = BUSY
 │  ...（反复检查...）                │
 │  N. 读取状态寄存器                 │
 │ ←─────────────────────────────── │  状态 = READY
 │  N+1. 读取数据寄存器               │
 │ ←─────────────────────────────── │  数据
 │                                   │
```

**代码示例**：

```c
// 轮询方式从磁盘读取数据
void disk_read_polling(uint32_t blockno, void *buf) {
    // 1. 写入要读取的块号
    outb(DISK_LBA_REG, blockno & 0xFF);
    
    // 2. 发送读命令
    outb(DISK_CMD_REG, CMD_READ);
    
    // 3. 轮询等待设备就绪
    while ((inb(DISK_STATUS_REG) & STATUS_READY) == 0) {
        // CPU 在此循环中空转，不断检查状态
        // 期间无法执行任何有用的计算
    }
    
    // 4. 读取数据
    for (int i = 0; i < BLOCK_SIZE; i++) {
        ((char *)buf)[i] = inb(DISK_DATA_REG);
    }
}
```

**优缺点分析**：

| 维度 | 评价 |
|------|------|
| 实现复杂度 | 最简单 |
| CPU 利用率 | 极低——CPU 大量时间在空转 |
| 延迟 | 低——数据就绪后立即读取 |
| 适用场景 | 设备速度很快（如内存映射设备） |
| 不适用 | 慢速设备（如磁盘、网络） |

**轮询的浪费计算**：

假设磁盘延迟 10ms，CPU 频率 1GHz：
- 轮询期间 CPU 空转的周期数 = 10ms × 1GHz = **1000 万次循环**
- 这些周期可以执行数百万条有用指令

### 23.2.2 方式二：中断驱动（Interrupt-Driven）

中断驱动方式让 CPU 在设备工作期间去做其他事情。设备完成后通过中断通知 CPU。

**基本流程**：

```
中断驱动方式的数据读取：

CPU                              设备控制器
 │                                   │
 │  1. 写入命令（读取数据）           │
 │ ───────────────────────────────→ │
 │                                   │  设备开始工作
 │  2. CPU 去执行其他进程...          │
 │     (进程 A 运行中)               │
 │     (进程 B 运行中)               │
 │                                   │  设备完成！
 │  ←─────────────────────────────── │  发送中断
 │  3. CPU 响应中断                   │
 │     保存当前进程上下文              │
 │     跳转到中断处理程序              │
 │  4. 中断处理程序：                  │
 │     读取数据                       │
 │     唤醒等待的进程                  │
 │     恢复之前被中断的进程            │
 │                                   │
```

**代码示例**：

```c
// 中断驱动方式从磁盘读取数据

// 1. 发起读取请求
void disk_read_interrupt(uint32_t blockno, void *buf) {
    // 设置 DMA/设备参数
    outb(DISK_LBA_REG, blockno & 0xFF);
    
    // 注册完成回调
    disk_request.buf = buf;
    disk_request.done = 0;
    
    // 发送读命令
    outb(DISK_CMD_REG, CMD_READ);
    
    // CPU 可以去做其他事情
    // 当前进程可以 sleep 等待
    while (!disk_request.done) {
        sleep(&disk_request, &disk_lock);
    }
}

// 2. 中断处理程序（设备完成时调用）
void disk_interrupt_handler() {
    // 从设备读取数据到缓冲区
    for (int i = 0; i < BLOCK_SIZE; i++) {
        disk_request.buf[i] = inb(DISK_DATA_REG);
    }
    
    disk_request.done = 1;
    wakeup(&disk_request);  // 唤醒等待的进程
}
```

**中断处理的完整流程**：

```
中断处理流程：

1. 设备完成操作，发出中断信号
        │
        ▼
2. CPU 检测到中断（每条指令执行后检查）
        │
        ▼
3. 保存当前上下文（PC、寄存器）
        │
        ▼
4. 跳转到中断向量表对应的处理程序
        │
        ▼
5. 中断处理程序执行：
   ├── 读取设备数据
   ├── 更新设备状态
   └── 唤醒等待进程
        │
        ▼
6. 恢复被中断的上下文
        │
        ▼
7. 继续执行被中断的程序
```

**中断 vs 轮询的性能对比**：

假设磁盘延迟 10ms，CPU 频率 1GHz，上下文切换开销 5μs：

| 指标 | 轮询 | 中断驱动 |
|------|------|----------|
| CPU 浪费 | ~10ms（1000 万周期） | ~5μs（上下文切换） |
| CPU 利用率 | ~0%（等待期间） | ~99.95% |
| 响应延迟 | 低（立即检测到） | 略高（中断延迟） |
| 实现复杂度 | 简单 | 中等 |

**中断的问题**：

1. **中断风暴（Interrupt Storm）**：高频率中断导致 CPU 频繁切换上下文
2. **活锁（Livelock）**：CPU 一直在处理中断，无法执行用户进程
3. **中断合并（Interrupt Coalescing）**：批量处理中断以减少开销

### 23.2.3 方式三：直接内存访问（DMA）

对于大量数据传输，即使是中断驱动方式，CPU 仍然需要逐字节地从设备读取数据。DMA（Direct Memory Access）让设备直接与内存交换数据，CPU 只需发起传输请求和接收完成通知。

**DMA 控制器的工作原理**：

```
DMA 传输过程：

CPU                     DMA 控制器                设备         内存
 │                          │                      │            │
 │  1. 设置 DMA 参数：       │                      │            │
 │    源地址、目标地址        │                      │            │
 │    传输长度、传输方向      │                      │            │
 │ ──────────────────────→ │                      │            │
 │                          │                      │            │
 │  2. 启动 DMA 传输         │                      │            │
 │ ──────────────────────→ │                      │            │
 │                          │                      │            │
 │  3. CPU 去执行其他任务    │  4. DMA 控制器        │            │
 │     (无需参与传输)        │  从设备读取数据 ────→ │            │
 │                          │  写入内存 ────────────────────→ │
 │                          │  从设备读取数据 ────→ │            │
 │                          │  写入内存 ────────────────────→ │
 │                          │  ...重复直到完成     │            │
 │                          │                      │            │
 │  ←──────────────────────│  5. 传输完成，中断    │            │
 │  6. CPU 处理完成中断      │                      │            │
```

**DMA 的优势**：

```
三种方式的 CPU 参与度对比：

轮询（传输 1000 字节）：
CPU: ████████████████████████████████ 100% 参与
     [逐字节读取 + 状态检查]

中断驱动（传输 1000 字节）：
CPU: ██░░░░░░░░░░░░░░░░░░░░░░░░░░░░ ~10% 参与
     [发起 + 逐字节中断读取]

DMA（传输 1000 字节）：
CPU: █░░░░░░░░░░░░░░░░░░░░░░░░░░░░░ ~1% 参与
     [设置参数 + 完成中断]
```

**DMA 控制器的寄存器**：

| 寄存器 | 功能 |
|--------|------|
| 源地址寄存器 | 数据来源地址（设备寄存器或内存地址） |
| 目标地址寄存器 | 数据目标地址（内存地址） |
| 传输长度寄存器 | 要传输的字节数 |
| 控制寄存器 | 传输方向、传输模式、启动/停止 |

**DMA 的高级特性**：

1. **Scatter-Gather DMA**：支持不连续内存区域的传输
   ```
   Scatter-Gather DMA：
   描述符链：
   [描述符1: 地址A, 长度100] →
   [描述符2: 地址B, 长度200] →
   [描述符3: 地址C, 长度150] → NULL
   
   DMA 控制器自动按描述符链传输，无需 CPU 干预
   ```

2. **Bus Mastering**：DMA 控制器作为总线主设备，直接控制总线

3. **DMA 缓存一致性**：
   ```
   问题：DMA 写入内存后，CPU 缓存中的数据可能是旧的
   解决方案：
   1. Cache Flush：DMA 前刷新缓存
   2. Uncacheable 映射：DMA 缓冲区设为不可缓存
   3. 硬件一致性：IOMMU / Snoop 协议
   ```

### 23.2.4 三种方式的综合对比

```
┌──────────────┬──────────────┬──────────────┬──────────────┐
│     维度      │    轮询      │  中断驱动    │     DMA      │
├──────────────┼──────────────┼──────────────┼──────────────┤
│ CPU 利用率    │    极低      │     高       │    最高      │
│ 实现复杂度    │    最简      │     中       │     高       │
│ 传输单位      │   字节/字    │   字节/字    │   数据块     │
│ 适用数据量    │    小        │     中       │     大       │
│ 适用设备速度  │    快        │     中       │    慢/中     │
│ 延迟          │    最低      │    较低      │    略高      │
│ 中断开销      │    无        │     有       │     少       │
└──────────────┴──────────────┴──────────────┴──────────────┘
```

**实际系统中的选择**：

- **键盘/鼠标**：中断驱动（数据量小，延迟敏感）
- **磁盘大块传输**：DMA（数据量大，CPU 不应参与逐字节传输）
- **网络包**：DMA + 中断（大包用 DMA，完成用中断通知）
- **嵌入式快速设备**：轮询（设备延迟极低，中断开销不划算）

## 23.3 设备驱动程序

### 23.3.1 什么是设备驱动程序？

设备驱动程序是操作系统中**直接控制硬件设备**的软件模块。它在硬件和操作系统其他部分之间提供抽象层。

```
设备驱动在系统中的位置：

┌─────────────────────────────────────────────────┐
│              用户应用程序                         │
│         read() / write() / ioctl()              │
├─────────────────────────────────────────────────┤
│              系统调用接口                         │
├─────────────────────────────────────────────────┤
│              文件系统 / VFS                       │
├─────────────────────────────────────────────────┤
│              设备驱动程序                         │  ← 驱动在这里
│    disk_read() / disk_write() / disk_ioctl()    │
├─────────────────────────────────────────────────┤
│              设备控制器（硬件）                    │
├─────────────────────────────────────────────────┤
│              物理设备                              │
└─────────────────────────────────────────────────┘
```

**驱动程序的职责**：
1. **设备初始化**：探测设备、配置寄存器、分配资源
2. **数据传输**：实现 read/write 操作
3. **中断处理**：响应设备中断，完成数据传输
4. **错误处理**：检测并处理设备错误
5. **电源管理**：设备休眠/唤醒

### 23.3.2 驱动程序接口

操作系统为驱动程序定义了标准接口，使得上层软件可以通过统一的方式访问不同设备。

**xv6 的设备驱动接口**：


### 23.3.3 xv6 UART 驱动详解

xv6 使用 UART（Universal Asynchronous Receiver/Transmitter）作为控制台设备。让我们详细分析其实现。

**UART 硬件基础**：

```
UART 寄存器布局（以 16550 为例）：
基地址 + 0: 数据寄存器 (RBR/THR)  — 读：接收数据，写：发送数据
基地址 + 1: 中断使能寄存器 (IER)
基地址 + 2: 中断标识寄存器 (IIR) / FIFO 控制
基地址 + 3: 线路控制寄存器 (LCR)
基地址 + 4: 调制解调器控制 (MCR)
基地址 + 5: 线路状态寄存器 (LSR)   — bit 0: 数据就绪，bit 5: 发送缓冲区空
基地址 + 6: 调制解调器状态 (MSR)
```

**xv6 UART 驱动实现**：

```c
// kernel/uart.c

// UART 寄存器地址（内存映射 I/O）
#define UART0 0x10000000L    // UART 基地址
#define Reg(reg) ((volatile uint8 *)(UART0 + (reg)))

// 发送一个字节
void uartputc(int c) {
    // 等待发送缓冲区空
    // LSR bit 5 (THRE) = 1 表示可以发送
    while ((Reg(5) & (1 << 5)) == 0)
        ;  // 轮询等待
    
    // 写入数据到发送寄存器
    Reg(0) = c;
}

// 接收一个字节
int uartgetc(void) {
    // 检查是否有数据到达
    // LSR bit 0 (DR) = 1 表示数据就绪
    if ((Reg(5) & 1) == 0)
        return -1;  // 没有数据
    
    // 读取数据
    return Reg(0);
}
```

**中断驱动的 UART 接收**：

```c
// 中断驱动方式接收字符
void uartintr(void) {
    while (1) {
        int c = uartgetc();
        if (c == -1)
            break;  // 没有更多数据
        
        // 将字符送入控制台输入缓冲区
        consoleintr(c);
    }
}

// consoleintr() 处理特殊字符
void consoleintr(int c) {
    switch (c) {
    case C('U'):  // Ctrl+U: 清除当前行
        while (input.e != input.w &&
               input.buf[(input.e - 1) % INPUT_BUF] != '\n') {
            input.e--;
            consputc(BACKSPACE);
        }
        break;
    case C('H'):  // Backspace
    case '\x7f':
        if (input.e != input.w) {
            input.e--;
            consputc(BACKSPACE);
        }
        break;
    default:
        if (c != 0 && input.e - input.r < INPUT_BUF) {
            c = (c == '\r') ? '\n' : c;
            input.buf[input.e++ % INPUT_BUF] = c;
            consputc(c);
            if (c == '\n' || c == C('D') || input.e == input.r + INPUT_BUF) {
                input.w = input.e;
                wakeup(&input.r);  // 唤醒等待输入的进程
            }
        }
        break;
    }
}
```

### 23.3.4 xv6 磁盘驱动详解

xv6 使用 virtio 设备接口访问磁盘。virtio 是一种半虚拟化设备标准。

```c
// kernel/virtio_disk.c

// virtio 描述符：描述一个 I/O 请求
struct virtq_desc {
    uint64 addr;    // 数据缓冲区物理地址
    uint32 len;     // 数据长度
    uint16 flags;   // 标志位
    uint16 next;    // 链表下一个描述符
};

// 磁盘读写请求
void virtio_disk_rw(struct buf *b, int write) {
    // 获取自旋锁
    acquire(&disk.vdisk_lock);
    
    // 1. 准备描述符
    // 第一个描述符：设备请求头
    struct virtio_blk_req *buf0 = &disk.ops[desc[0]];
    buf0->type = write ? VIRTIO_BLK_T_OUT : VIRTIO_BLK_T_IN;
    buf0->sector = b->blockno * (BSIZE / 512);
    
    // 第二个描述符：数据
    disk.desc[1].addr = (uint64)b->data;
    disk.desc[1].len = BSIZE;
    disk.desc[1].flags = write ? 0 : VRING_DESC_F_WRITE;
    
    // 第三个描述符：状态
    disk.info[desc[0]].status = 0xff;
    
    // 2. 提交到可用环
    disk.avail->ring[disk.avail->idx % NUM] = desc[0];
    __sync_synchronize();
    disk.avail->idx++;
    
    // 3. 通知设备
    *R(VIRTIO_MMIO_QUEUE_NOTIFY) = 0;
    
    // 4. 等待完成（在中断中处理）
    while (b->disk == 1) {
        sleep(b, &disk.vdisk_lock);
    }
    
    release(&disk.vdisk_lock);
}

// 磁盘中断处理
void virtio_disk_intr() {
    acquire(&disk.vdisk_lock);
    
    // 检查完成环中的已完成请求
    while (disk.used_idx != disk.used->idx) {
        __sync_synchronize();
        int id = disk.used->ring[disk.used_idx % NUM].id;
        
        if (disk.info[id].status != 0)
            panic("virtio_disk_intr status");
        
        struct buf *b = disk.info[id].buf;
        b->disk = 0;  // 标记完成
        wakeup(b);     // 唤醒等待的进程
        
        disk.used_idx++;
    }
    
    release(&disk.vdisk_lock);
}
```

## 23.4 I/O 缓冲

### 23.4.1 为什么需要缓冲？

CPU 和设备之间的速度差异巨大。缓冲区作为中间层，解决速度不匹配问题。

```
无缓冲的问题：
CPU ──写入──→ 设备
  1GHz         10MB/s
  │            │
  │ 每写一个字节，CPU 都要等设备完成
  │ 浪费大量 CPU 时间

有缓冲的情况：
CPU ──写入──→ [缓冲区] ──批量传输──→ 设备
  1GHz         内存       10MB/s
  │
  │ CPU 写入缓冲区后立即返回
  │ 缓冲区满或刷新时才实际传输
```

### 23.4.2 单缓冲（Single Buffer）

最简单的缓冲策略：使用一个缓冲区在用户空间和内核空间之间传输数据。

```
单缓冲工作流程：

用户进程                    缓冲区                     设备
   │                          │                         │
   │  1. read(fd, buf, n)     │                         │
   │ ───────────────────────→ │                         │
   │                          │  2. 从设备读取到缓冲区    │
   │                          │ ←───────────────────────│
   │  3. 从缓冲区复制到用户buf │                         │
   │ ←─────────────────────── │                         │
   │                          │                         │
```

**问题**：用户处理数据时，设备无法传输新数据（生产者-消费者互斥）。

### 23.4.3 双缓冲（Double Buffer）

使用两个缓冲区交替工作：一个在被 CPU 处理时，另一个在从设备传输数据。

```
双缓冲工作流程：

用户进程           缓冲区 A          缓冲区 B           设备
   │                 │                 │                 │
   │ 处理 A 中数据    │ ← 设备填充 B   │ ←──────────────│
   │ ←────────────── │                 │                 │
   │                 │                 │                 │
   │ 处理 B 中数据    │ ← 设备填充 A   │                 │
   │                 │ ←──────────────│ ←──────────────│
   │ ←────────────── │                 │                 │
```

**优势**：CPU 处理和设备传输可以并行，提高吞吐量。

### 23.4.4 循环缓冲（Circular Buffer）

使用环形队列管理多个缓冲区，适用于持续的数据流。

```
循环缓冲区（环形队列）：

       ┌───┐
    ┌──┤ 3 ├──┐
    │  └───┘  │
  ┌─┴─┐     ┌─┴─┐
  │ 2 │     │ 4 │
  └─┬─┘     └─┬─┘
    │  ┌───┐  │
    └──┤ 1 ├──┘
       └───┘
       
  生产者指针 →  写入位置
  消费者指针 →  读取位置

  空条件：生产者 == 消费者
  满条件：(生产者 + 1) % N == 消费者
```

**Linux 内核的环形缓冲区**：

```c
// 简化的环形缓冲区实现
struct circular_buffer {
    char *buf;
    int size;
    int head;  // 读指针
    int tail;  // 写指针
};

// 写入数据
int cb_write(struct circular_buffer *cb, const char *data, int len) {
    for (int i = 0; i < len; i++) {
        int next = (cb->tail + 1) % cb->size;
        if (next == cb->head)
            return i;  // 缓冲区满
        cb->buf[cb->tail] = data[i];
        cb->tail = next;
    }
    return len;
}

// 读取数据
int cb_read(struct circular_buffer *cb, char *data, int len) {
    int i;
    for (i = 0; i < len && cb->head != cb->tail; i++) {
        data[i] = cb->buf[cb->head];
        cb->head = (cb->head + 1) % cb->size;
    }
    return i;
}
```

### 23.4.5 缓冲策略

| 策略 | 描述 | 适用场景 |
|------|------|----------|
| **直写（Write-Through）** | 每次写入立即传到设备 | 数据安全性要求高 |
| **回写（Write-Back）** | 积攒到一定量再写入设备 | 追求性能 |
| **预读（Read-Ahead）** | 预测性地提前读取数据 | 顺序访问模式 |
| **绕过（Write-Around）** | 跳过缓存直接写入 | 一次性写入 |

## 23.5 I/O 调度

### 23.5.1 为什么需要 I/O 调度？

当多个进程同时发起 I/O 请求时，操作系统需要决定请求的执行顺序。对于磁盘设备，请求顺序直接影响性能（因为寻道时间占大头）。

```
I/O 请求队列示例：

进程 A: 读取块 100
进程 B: 读取块 5000
进程 C: 读取块 200
进程 D: 读取块 4900

磁盘磁头当前位置：块 150

执行顺序对性能的影响：
  FCFS 顺序: 100 → 5000 → 200 → 4900  (大量来回寻道)
  优化顺序:  100 → 200 → 4900 → 5000  (顺序扫描)
```

### 23.5.2 I/O 调度目标

1. **吞吐量**：单位时间完成的 I/O 请求数
2. **延迟**：请求从发起到完成的时间
3. **公平性**：所有进程都能公平地获得 I/O 服务
4. **优先级**：高优先级请求优先处理

### 23.5.3 Linux I/O 调度器

Linux 内核历史上使用过多种 I/O 调度器：

| 调度器 | 特点 | 适用场景 |
|--------|------|----------|
| **Noop** | 不调度，FIFO | SSD（无寻道开销） |
| **CFQ** | 完全公平队列 | HDD 通用场景 |
| **Deadline** | 截止时间保证 | 数据库等延迟敏感应用 |
| **BFQ** | 预算公平队列 | 桌面/交互式系统 |
| **mq-deadline** | 多队列 Deadline | 现代默认调度器 |
| **Kyber** | 基于目标延迟 | 快速设备（NVMe） |

```bash
# 查看当前 I/O 调度器
$ cat /sys/block/sda/queue/scheduler
[mq-deadline] kyber bfq none

# 切换 I/O 调度器
$ echo bfq > /sys/block/sda/queue/scheduler
```

## 23.6 面试高频考点

**Q1：轮询和中断的区别？**

轮询是 CPU 主动检查设备状态，中断是设备主动通知 CPU。轮询实现简单但浪费 CPU，中断利用率高但有上下文切换开销。快速设备用轮询，慢速设备用中断。

**Q2：DMA 的工作原理？**

CPU 设置 DMA 控制器的源地址、目标地址、传输长度，然后启动传输。DMA 控制器直接在设备和内存之间传输数据，完成后通过中断通知 CPU。CPU 在传输期间可以执行其他任务。

**Q3：设备驱动程序的作用？**

设备驱动是操作系统中直接控制硬件的软件模块。它在硬件和 OS 其他部分之间提供抽象层，使得上层软件可以通过统一接口访问不同设备，而不需要了解硬件细节。

**Q4：为什么需要 I/O 缓冲？**

CPU 和设备之间速度差异巨大（CPU 1GHz vs 磁盘 10MB/s）。缓冲区作为中间层，让 CPU 和设备可以并行工作，提高系统整体吞吐量。

**Q5：用户态驱动 vs 内核态驱动的优缺点？**

内核态驱动：可以直接访问硬件和中断，性能好，但 bug 可能导致内核崩溃。用户态驱动：安全性好（bug 不影响内核），但需要系统调用访问硬件，性能较差。DPDK/SPDK 使用用户态驱动获得极致性能。
