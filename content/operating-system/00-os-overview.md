## 0.1 操作系统的定义与作用

### 0.1.1 什么是操作系统？资源管理器 vs 虚拟机

操作系统（Operating System, OS）是现代计算机系统中最重要的系统软件，但对其精确定义却存在多种视角。我们可以从两个互补的角度来理解操作系统的本质：

#### **视角一：资源管理器（Resource Manager）**

从这个视角看，操作系统是计算机硬件资源的**管理者与仲裁者**。现代计算机系统包含多种硬件资源：

- **CPU**：执行指令的处理器
- **内存**：存储程序和数据的 RAM
- **I/O 设备**：磁盘、网络、键盘、显示器等外设

在没有操作系统的裸机环境中，程序需要直接操作这些硬件，面临诸多挑战：

```c
// 裸机编程示例：直接操作串口硬件
#define UART_BASE 0x10000000
#define UART_THR  (UART_BASE + 0)  // Transmitter Holding Register

void putchar_bare(char c) {
    // 直接写入硬件寄存器
    volatile char *uart_thr = (char *)UART_THR;
    *uart_thr = c;
}
```

这种直接硬件访问模式存在严重问题：

1. **资源冲突**：多个程序同时访问同一设备时无法协调
2. **安全隐患**：任何程序都能访问所有硬件，可能破坏其他程序
3. **效率低下**：程序员需要了解所有硬件细节，开发成本极高
4. **可移植性差**：硬件更换后代码需要重写

**操作系统作为资源管理器的核心职责**包括：

- **分配（Allocation）**：决定哪个程序可以使用哪些资源
- **调度（Scheduling）**：在多个请求之间协调资源的使用顺序
- **保护（Protection）**：防止程序非法访问或破坏资源
- **回收（Reclamation）**：在程序结束后回收资源供其他程序使用

#### **视角二：虚拟机/扩展机器（Virtual Machine / Extended Machine）**

从这个视角看，操作系统是对硬件的**抽象与扩展**，它隐藏了底层硬件的复杂性，向上层应用提供简洁、统一、易用的接口。

**硬件提供的原始接口**往往晦涩难用：
- 直接的内存地址操作
- 复杂的设备寄存器读写
- 中断处理机制
- DMA 控制器配置

**操作系统提供的抽象接口**则优雅且强大：
- **进程（Process）**：抽象 CPU，让程序以为独占处理器
- **虚拟内存（Virtual Memory）**：抽象物理内存，提供连续的地址空间
- **文件（File）**：抽象磁盘，提供持久化的字节序列
- **套接字（Socket）**：抽象网络，提供通信端点

对比以下两种编程范式：

```c
// 裸机编程：直接操作磁盘控制器
void read_sector_bare(int sector, char *buffer) {
    // 1. 等待磁盘就绪
    while (inb(DISK_STATUS) & DISK_BUSY);
    
    // 2. 设置扇区号
    outb(DISK_SECTOR_LOW, sector & 0xFF);
    outb(DISK_SECTOR_MID, (sector >> 8) & 0xFF);
    outb(DISK_SECTOR_HIGH, (sector >> 16) & 0xFF);
    
    // 3. 发送读命令
    outb(DISK_COMMAND, CMD_READ);
    
    // 4. 等待数据就绪
    while (!(inb(DISK_STATUS) & DISK_DATA_READY));
    
    // 5. 读取 512 字节数据
    for (int i = 0; i < 512; i++) {
        buffer[i] = inb(DISK_DATA);
    }
}

// 操作系统抽象：使用文件接口
void read_data(const char *filename, char *buffer, size_t size) {
    int fd = open(filename, O_RDONLY);  // 打开文件
    read(fd, buffer, size);              // 读取数据
    close(fd);                            // 关闭文件
}
```

第二种方式显然更加简洁、安全、可移植。操作系统将复杂的硬件细节封装起来，程序员只需关注**做什么（what）**，而无需关心**怎么做（how）**。

<div data-component="AbstractionLayersVisualization"></div>

#### **两种视角的统一**

这两种视角并不矛盾，而是一体两面：

- **向下**（对硬件）：操作系统是资源管理器，控制和协调硬件资源
- **向上**（对应用）：操作系统是虚拟机，提供简洁的抽象接口

**Tanenbaum 的经典定义**：
> "操作系统是在裸机之上的第一层软件，它管理系统的所有资源，并向用户和应用程序提供接口。"

**OSTEP 的实用定义**：
> "操作系统负责三件事：虚拟化（Virtualization）、并发（Concurrency）、持久化（Persistence）。"

### 0.1.2 操作系统的历史演进

操作系统的发展是一部**问题驱动的演化史**，每一代操作系统都是为了解决当时计算环境面临的核心挑战。

#### **第一代（1945-1955）：无操作系统 / 直接编程**

**硬件特征**：
- 真空管计算机（ENIAC、UNIVAC）
- 极其昂贵（数百万美元）
- 体积庞大（占据整个房间）
- 可靠性差（真空管频繁损坏）

**编程方式**：
- 直接用机器语言或汇编语言编程
- 通过**插线板**或**打孔卡片**输入程序
- 一次只运行一个程序（串行处理）

**核心问题**：
- **CPU 利用率极低**：程序员手动操作机器（装卡、启动、取结果），大部分时间 CPU 空闲
- **调试困难**：程序出错需要重新打孔卡片
- **成本高昂**：机器时间按分钟计费

**示例场景**：
```
1. 程序员编写程序并打孔（耗时数小时）
2. 预约机器时间（可能需要等待数天）
3. 操作员装载打孔卡片（耗时数分钟）
4. 运行程序（可能只需数秒）
5. 取出结果卡片（耗时数分钟）
6. 程序员分析结果，发现 bug（耗时数小时）
7. 重复步骤 1-6
```

**结果**：昂贵的 CPU 大部分时间在等待人工操作，利用率不足 10%。

#### **第二代（1955-1965）：批处理系统（Batch Processing）**

**设计动机**：提高 CPU 利用率，减少人工操作时间。

**核心思想**：
- 引入**操作员**角色，专门负责机器操作
- 将多个程序**打包成批（batch）**，连续执行
- 引入**监控程序（Monitor）**，自动加载下一个作业

**工作流程**：
```
1. 程序员提交打孔卡片给操作员
2. 操作员将一批（例如 50 个）作业装载到磁带
3. 监控程序从磁带读取第一个作业并执行
4. 作业完成后，监控程序自动加载下一个作业
5. 所有作业完成后，操作员取出结果磁带
6. 程序员第二天来取结果
```

**关键技术**：
- **顺序读卡器**：自动读取打孔卡片
- **磁带存储**：批量存储作业和结果
- **简单的作业控制语言（JCL）**：描述作业需求

```assembly
// IBM 7094 监控程序伪代码
MONITOR:
    LOAD_NEXT_JOB:
        IF no_more_jobs GOTO HALT
        READ job_from_tape
        LOAD job_into_memory
        TRANSFER_CONTROL to job
        
    JOB_RETURN:  // 作业完成后跳转到这里
        WRITE results_to_tape
        GOTO LOAD_NEXT_JOB
        
    HALT:
        STOP
```

**改进效果**：
- CPU 利用率提升到 40%-50%
- 周转时间从分钟级降低到秒级（对于连续作业）

**遗留问题**：
- **无交互性**：程序员无法在运行时调试
- **周转时间长**：提交到拿到结果可能需要数小时甚至数天
- **I/O 瓶颈**：CPU 在等待 I/O 时仍然空闲

#### **第三代（1965-1980）：多道程序设计与分时系统**

**多道程序设计（Multiprogramming）**

**设计动机**：即使在批处理系统中，CPU 仍有大量时间在等待 I/O 操作完成（如读取磁带、打印输出）。

**核心思想**：
- 内存中同时驻留**多个程序**
- 当一个程序等待 I/O 时，操作系统切换到另一个程序执行
- 实现 **CPU 与 I/O 的重叠（overlap）**

```
程序 A: ████[I/O]████[I/O]████
程序 B:     ████    ████    
程序 C:         ████    ████

时间轴: ──────────────────────>
CPU:    AAABBBAAACCCAAABBBAAACCC
```

**关键技术突破**：
- **内存保护**：防止程序互相干扰
- **CPU 调度**：选择下一个运行的程序
- **中断机制**：I/O 完成时通知 CPU

**分时系统（Time-Sharing System）**

**设计动机**：批处理系统缺乏交互性，程序员无法实时调试。

**核心思想**：
- 给每个用户一个**虚拟终端**
- CPU 在多个用户之间快速切换（例如每 100ms）
- 每个用户感觉**独占**整个系统

**经典系统**：
- **CTSS（Compatible Time-Sharing System）**：MIT 1961 年，第一个真正的分时系统
- **Multics**：MIT、Bell Labs、GE 联合开发，影响深远但过于复杂
- **Unix**：Bell Labs 1969 年，简化 Multics 设计，成为后续操作系统的基石

**Unix 哲学的诞生**：
- 简单性（Simplicity）
- 模块化（Modularity）
- 一切皆文件（Everything is a file）

#### **第四代（1980-至今）：个人计算与网络时代**

**个人计算时代（1980s-1990s）**：
- **微处理器革命**：Intel 8086、Motorola 68000
- **图形用户界面（GUI）**：Xerox Alto → Apple Macintosh → Windows
- **桌面操作系统**：MS-DOS、Windows、Mac OS

**网络与互联网时代（1990s-2000s）**：
- **网络操作系统**：支持 TCP/IP、Web 服务器
- **Linux 的崛起**：开源操作系统成为服务器主流
- **移动操作系统**：iOS、Android

**云计算与虚拟化时代（2010s-现在）**：
- **虚拟化技术**：VMware、KVM、Xen
- **容器化**：Docker、Kubernetes
- **云原生操作系统**：CoreOS、Flatcar

### 0.1.3 操作系统在计算机系统中的位置

计算机系统可以看作一个**层次化的结构**，操作系统位于硬件与应用软件之间，起到承上启下的关键作用。

#### **经典五层模型**

```
┌─────────────────────────────────────┐
│     应用程序（Applications）         │  ← 用户直接交互
│  Web 浏览器、文本编辑器、游戏等     │
├─────────────────────────────────────┤
│    系统工具（System Utilities）      │  ← 系统管理工具
│    编译器、调试器、Shell 等          │
├─────────────────────────────────────┤
│      操作系统（Operating System）    │  ← 本课程重点
│  内核、设备驱动、系统调用接口        │
├─────────────────────────────────────┤
│      硬件（Hardware）                │  ← 物理设备
│  CPU、内存、磁盘、网络设备等         │
└─────────────────────────────────────┘
```

#### **操作系统的核心组成部分**

操作系统本身也是分层的，包含多个子系统：

```
操作系统
├── 内核（Kernel）                    ← 核心部分，运行在特权模式
│   ├── 进程管理（Process Management）
│   ├── 内存管理（Memory Management）
│   ├── 文件系统（File System）
│   ├── I/O 管理（I/O Management）
│   └── 进程间通信（IPC）
├── 设备驱动（Device Drivers）        ← 硬件接口层
│   ├── 磁盘驱动
│   ├── 网卡驱动
│   └── 显卡驱动
└── 系统调用接口（System Call Interface） ← 用户态与内核态的桥梁
    ├── 文件操作：open(), read(), write()
    ├── 进程控制：fork(), exec(), wait()
    └── 内存管理：mmap(), brk()
```

#### **用户态 vs 内核态**

现代处理器提供**特权级别**机制，将指令分为两类：

| 特性 | 用户态（User Mode） | 内核态（Kernel Mode） |
|------|-------------------|---------------------|
| **权限** | 受限 | 完全控制 |
| **可访问内存** | 仅用户空间 | 全部内存 |
| **可执行指令** | 普通指令 | 所有指令（包括特权指令） |
| **硬件访问** | 禁止 | 允许 |
| **运行内容** | 应用程序 | 操作系统内核 |

**为什么需要两种模式？**

1. **安全性**：防止应用程序破坏系统或其他程序
2. **稳定性**：应用程序崩溃不会导致整个系统崩溃
3. **资源管理**：操作系统可以控制资源分配

**模式切换的时机**：
- **用户态 → 内核态**：
  - 系统调用（System Call）
  - 异常（Exception）：如除零、非法内存访问
  - 中断（Interrupt）：如时钟中断、I/O 完成
  
- **内核态 → 用户态**：
  - 系统调用返回
  - 进程调度切换到用户进程

```c
// 用户程序示例
#include <unistd.h>

int main() {
    char buffer[100];
    
    // 运行在用户态
    int x = 1 + 2;  // 普通计算
    
    // 系统调用：切换到内核态
    ssize_t n = read(0, buffer, 100);  // 读取标准输入
    // read() 内部会触发软件中断，陷入内核
    
    // 返回用户态
    return 0;
}
```

### 0.1.4 为什么需要操作系统？硬件抽象与资源复用

通过前面的讨论，我们可以总结出操作系统存在的**核心价值**：

#### **1. 硬件抽象（Hardware Abstraction）**

**问题**：硬件种类繁多且接口各异
- 不同厂商的硬盘有不同的命令集
- 网卡型号数以千计，每种都有独特的寄存器布局
- 显卡编程极其复杂（数千页的技术手册）

**解决方案**：操作系统提供统一的抽象接口
- **文件系统抽象**：无论是机械硬盘、SSD 还是 U 盘，都通过 `open()`、`read()`、`write()` 访问
- **网络抽象**：socket API 屏蔽了不同网络协议和硬件的差异
- **进程抽象**：让程序以为运行在独占的 CPU 上

**收益**：
- 应用程序无需修改即可在不同硬件上运行（可移植性）
- 程序员无需学习硬件细节（生产力提升）
- 硬件升级不影响已有软件（向后兼容）

#### **2. 资源复用（Resource Multiplexing）**

**问题**：硬件资源有限但需求众多
- 一台计算机可能只有 4 个 CPU 核心，但需要运行数百个进程
- 物理内存只有 16GB，但所有程序的总需求可能达到数百 GB
- 网络带宽有限，多个应用同时传输数据

**解决方案**：操作系统通过复用技术实现资源共享

**时间复用（Time Multiplexing）**：
- **CPU 时间片轮转**：每个进程轮流使用 CPU（例如每 10ms 切换一次）
- **网络带宽分时**：数据包交错发送，形成虚拟的独占连接

**空间复用（Space Multiplexing）**：
- **内存分区**：每个进程分配独立的内存区域
- **磁盘分区**：不同用户拥有独立的文件空间

**示例：CPU 时间片轮转**

```
实际物理 CPU（单核）：
─────────────────────────────────────────>
AAABBBCCCAAABBBCCCAAABBBCCC...

虚拟 CPU 视角：
进程 A 认为：AAAAAAAAAAAAAAA...  （独占 CPU）
进程 B 认为：BBBBBBBBBBBBBBB...  （独占 CPU）
进程 C 认为：CCCCCCCCCCCCCCC...  （独占 CPU）
```

**示例：虚拟内存**

```
物理内存（16GB）：
┌─────────────────────────┐
│  进程 A 的部分页面       │  4GB
│  进程 B 的部分页面       │  4GB
│  进程 C 的部分页面       │  4GB
│  操作系统内核            │  4GB
└─────────────────────────┘

虚拟内存视角：
进程 A 认为拥有：0 - 256TB  （独占地址空间）
进程 B 认为拥有：0 - 256TB  （独占地址空间）
进程 C 认为拥有：0 - 256TB  （独占地址空间）
```

#### **3. 安全与保护（Security & Protection）**

**问题**：多个程序共享硬件时存在安全风险
- 恶意程序可能读取其他程序的内存
- 程序错误可能破坏文件系统
- 一个程序崩溃可能导致整个系统宕机

**解决方案**：操作系统强制实施隔离与保护

**内存保护**：
- 每个进程只能访问自己的地址空间
- 访问非法地址触发**段错误（Segmentation Fault）**

```c
int main() {
    int *p = NULL;
    *p = 42;  // 访问空指针，操作系统会终止程序
    return 0;
}
```

**文件权限**：
- Unix 风格的读/写/执行权限（rwx）
- 所有者、组、其他用户的权限分离

```bash
-rw-r--r--  1 alice  staff  1024  Jan  1 12:00  secret.txt
# alice 可读写，其他人只读
```

**进程隔离**：
- 进程 A 无法直接访问进程 B 的内存
- 需要通过操作系统提供的 IPC 机制通信

#### **4. 并发控制（Concurrency Control）**

**问题**：多个程序同时访问共享资源时可能出现冲突
- 两个进程同时写同一个文件
- 多个线程修改同一个变量
- 设备驱动与应用程序竞争硬件访问

**解决方案**：操作系统提供同步原语

**文件锁（File Locking）**：
```c
int fd = open("counter.txt", O_RDWR);
flock(fd, LOCK_EX);  // 获取排他锁

// 临界区：只有一个进程可以执行
int count = read_count(fd);
count++;
write_count(fd, count);

flock(fd, LOCK_UN);  // 释放锁
```

**互斥锁（Mutex）**：
```c
pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;

void *thread_func(void *arg) {
    pthread_mutex_lock(&lock);
    
    // 临界区
    shared_counter++;
    
    pthread_mutex_unlock(&lock);
    return NULL;
}
```

## 0.2 操作系统的核心功能

**OSTEP（Operating Systems: Three Easy Pieces）** 将操作系统的核心功能总结为三个关键词：

1. **虚拟化（Virtualization）**
2. **并发（Concurrency）**
3. **持久化（Persistence）**

这三个概念贯穿整个操作系统设计，是理解现代操作系统的钥匙。

---

### 0.2.1 虚拟化（Virtualization）：CPU、内存

**虚拟化的核心思想**：将**物理资源**转换为更通用、更易用的**虚拟形式**。

#### **CPU 虚拟化：进程抽象**

**物理现实**：
- 一台计算机可能只有 4 个 CPU 核心
- 同一时刻最多 4 个指令流并行执行

**虚拟假象**：
- 系统中可以运行数百个进程
- 每个进程都**认为**自己独占 CPU

**实现机制**：
- **时间片轮转（Time Slicing）**：操作系统快速在进程间切换（例如每 10ms）
- **上下文切换（Context Switch）**：保存旧进程状态，加载新进程状态

```c
// 用户视角：程序认为自己在连续执行
int main() {
    for (int i = 0; i < 1000000; i++) {
        printf("%d\n", i);  // 看起来连续输出
    }
    return 0;
}

// 实际执行：可能被打断数千次
// 时刻 0-10ms:   进程 A 执行
// 时刻 10-20ms:  进程 B 执行（A 被挂起）
// 时刻 20-30ms:  进程 C 执行
// 时刻 30-40ms:  进程 A 恢复执行
// ...
```

**虚拟化的好处**：
- **简化编程**：程序员无需考虑其他程序的存在
- **公平性**：每个程序都能获得 CPU 时间
- **隔离性**：一个程序崩溃不影响其他程序

#### **内存虚拟化：虚拟地址空间**

**物理现实**：
- 物理内存（DRAM）是有限的连续字节数组
- 地址从 0 开始，到物理内存大小结束（例如 0 - 16GB）

**虚拟假象**：
- 每个进程都**认为**自己拥有巨大的、私有的、连续的地址空间
- 地址从 0 开始，到虚拟地址空间上限（例如 0 - 256TB）

**实现机制**：
- **地址翻译（Address Translation）**：虚拟地址 → 物理地址
- **页表（Page Table）**：记录映射关系
- **内存管理单元（MMU）**：硬件辅助地址翻译

```c
// 两个进程的代码完全相同
int main() {
    int *p = malloc(sizeof(int));  // 分配内存
    printf("地址: %p\n", p);        // 打印地址
    *p = 42;
    return 0;
}

// 进程 A 输出：地址: 0x7f8a1c0000
// 进程 B 输出：地址: 0x7f8a1c0000  （相同的虚拟地址）

// 但它们映射到不同的物理内存：
// 进程 A: 虚拟地址 0x7f8a1c0000 → 物理地址 0x12340000
// 进程 B: 虚拟地址 0x7f8a1c0000 → 物理地址 0x56780000
```

**虚拟化的好处**：
- **隔离性**：进程无法访问其他进程的内存
- **简化编程**：每个程序都使用相同的地址布局（例如堆栈总是从高地址开始）
- **内存超载**：虚拟内存总和可以超过物理内存（通过磁盘交换）

### 0.2.2 并发（Concurrency）：多线程与同步

**并发的定义**：多个执行流（线程）**同时**执行或**交错**执行。

**为什么需要并发？**
1. **性能**：多核 CPU 可以真正并行执行多个线程
2. **响应性**：GUI 程序需要一个线程处理界面，另一个线程处理耗时任务
3. **资源利用**：一个线程等待 I/O 时，其他线程可以使用 CPU

**并发的挑战：竞态条件（Race Condition）**

```c
// 两个线程同时执行以下代码
int counter = 0;  // 共享变量

void *thread_func(void *arg) {
    for (int i = 0; i < 1000000; i++) {
        counter++;  // 看似简单，实际包含三步：
                    // 1. 读取 counter 到寄存器
                    // 2. 寄存器值 +1
                    // 3. 写回 counter
    }
    return NULL;
}

int main() {
    pthread_t t1, t2;
    pthread_create(&t1, NULL, thread_func, NULL);
    pthread_create(&t2, NULL, thread_func, NULL);
    pthread_join(t1, NULL);
    pthread_join(t2, NULL);
    
    printf("counter = %d\n", counter);
    // 期望：2000000
    // 实际：可能是 1234567（小于 2000000）
}
```

**为什么结果错误？**

```
时刻 1: 线程 1 读取 counter = 0 到寄存器 R1
时刻 2: 线程 2 读取 counter = 0 到寄存器 R2
时刻 3: 线程 1 计算 R1 + 1 = 1
时刻 4: 线程 2 计算 R2 + 1 = 1
时刻 5: 线程 1 写回 counter = 1
时刻 6: 线程 2 写回 counter = 1  （覆盖了线程 1 的写入）

结果：两次递增操作，counter 只增加了 1！
```

**解决方案：同步原语**

```c
pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;

void *thread_func_safe(void *arg) {
    for (int i = 0; i < 1000000; i++) {
        pthread_mutex_lock(&lock);    // 获取锁
        counter++;                     // 临界区
        pthread_mutex_unlock(&lock);  // 释放锁
    }
    return NULL;
}

// 现在结果总是 2000000
```

**操作系统提供的同步机制**：
- **互斥锁（Mutex）**：保证同一时刻只有一个线程访问共享资源
- **信号量（Semaphore）**：控制资源访问的数量
- **条件变量（Condition Variable）**：线程间的等待/通知机制
- **读写锁（RWLock）**：允许多个读者或单个写者

### 0.2.3 持久化（Persistence）：文件系统与 I/O

**持久化的需求**：程序结束后数据仍然存在。

**挑战**：
- **DRAM 是易失性的**：断电后数据消失
- **磁盘/SSD 是持久性的**：但接口复杂、访问慢

**操作系统的解决方案：文件系统**

#### **文件抽象**

文件是**命名的字节序列**，提供简洁的持久化接口：

```c
// 写入数据到文件
int fd = open("data.txt", O_WRONLY | O_CREAT, 0644);
const char *msg = "Hello, Persistence!\n";
write(fd, msg, strlen(msg));
close(fd);

// 程序结束，甚至系统重启后，数据仍然存在

// 读取数据
int fd = open("data.txt", O_RDONLY);
char buffer[100];
ssize_t n = read(fd, buffer, sizeof(buffer));
close(fd);
// buffer 中包含之前写入的数据
```

#### **文件系统的核心功能**

**1. 数据组织**：
- **目录层次结构**：`/home/alice/documents/report.txt`
- **元数据管理**：文件大小、创建时间、权限等

**2. 空间管理**：
- **磁盘块分配**：将文件数据存储到磁盘扇区
- **碎片整理**：优化文件存储布局

**3. 崩溃一致性**：
- **日志（Journaling）**：保证系统崩溃后文件系统仍然一致
- **写屏障（Write Barriers）**：控制写入顺序

**示例：文件系统如何存储文件**

```
文件 "/home/alice/report.txt" (1000 字节)

磁盘布局：
┌─────────────────────────────────────┐
│ 超级块（文件系统元数据）             │  块 0
├─────────────────────────────────────┤
│ inode 表                            │  块 1-10
│   inode 42:                         │
│     类型: 普通文件                   │
│     大小: 1000 字节                  │
│     数据块: [1024, 1025]             │
├─────────────────────────────────────┤
│ 数据块 1024: "Hello, this is..."   │
│ 数据块 1025: "...the content"       │
├─────────────────────────────────────┤
│ 目录 /home/alice:                   │
│   report.txt → inode 42             │
└─────────────────────────────────────┘
```

**崩溃一致性问题示例**：

```
操作：创建文件 "foo.txt" 并写入数据

需要的磁盘写入：
1. 分配 inode（修改 inode 位图）
2. 分配数据块（修改数据块位图）
3. 写入 inode 元数据
4. 写入数据块内容
5. 更新目录（添加 foo.txt → inode 映射）

如果在步骤 3 和 4 之间系统崩溃：
- inode 指向的数据块包含垃圾数据
- 文件系统不一致

解决方案：日志文件系统
- 先将所有操作写入日志
- 日志提交后，再更新实际数据结构
- 崩溃后，重放日志恢复一致性
```

### 0.2.4 安全与保护（Security & Protection）

**安全（Security）**与**保护（Protection）**是操作系统的基础职责，但常被初学者忽视。

#### **保护（Protection）：内部威胁**

**目标**：防止系统内部的程序互相干扰或误操作。

**机制**：

**1. 内存保护**：
```c
// 进程 A
int main() {
    int *p = (int *)0x12340000;  // 尝试访问其他进程的地址
    *p = 42;  // 触发 Segmentation Fault，进程被终止
}
```

**2. 文件权限**：
```bash
$ ls -l secret.txt
-rw-------  1 alice  staff  1024  Jan  1 12:00  secret.txt

$ cat secret.txt  # 如果当前用户不是 alice
cat: secret.txt: Permission denied
```

**3. CPU 特权级别**：
```c
// 用户程序尝试执行特权指令
asm("hlt");  // 停止 CPU 指令（特权指令）
// 结果：触发异常，进程被终止
```

#### **安全（Security）：外部威胁**

**目标**：防止恶意攻击者破坏系统。

**常见威胁**：
- **缓冲区溢出**：覆盖返回地址，执行恶意代码
- **权限提升**：普通用户获取 root 权限
- **信息泄露**：读取其他用户的敏感数据

**防护机制**：

**1. ASLR（地址空间布局随机化）**：
```c
// 每次运行，栈地址都不同
int main() {
    int x;
    printf("栈地址: %p\n", &x);
}

// 运行 1: 栈地址: 0x7ffc12345678
// 运行 2: 栈地址: 0x7ffe98765432
// 攻击者无法预测地址，难以利用溢出
```

**2. DEP/NX（数据执行保护）**：
- 将数据页标记为不可执行
- 即使攻击者注入代码到栈，也无法执行

**3. 沙箱（Sandboxing）**：
- 限制程序的权限和资源访问
- 例如浏览器沙箱、容器技术

## 0.3 操作系统的设计目标

设计一个操作系统需要在多个相互冲突的目标之间权衡。

---

### 0.3.1 性能（Performance）：吞吐量、响应时间、利用率

#### **吞吐量（Throughput）**

**定义**：单位时间内完成的工作量。

**示例**：
- 批处理系统：每小时处理的作业数
- Web 服务器：每秒处理的请求数（QPS）
- 数据库：每秒执行的事务数（TPS）

**优化策略**：
- **批处理**：将多个小任务合并为大任务
- **预取（Prefetching）**：提前加载数据
- **并行化**：利用多核 CPU

#### **响应时间（Response Time / Latency）**

**定义**：从请求提交到收到第一个响应的时间。

**示例**：
- 交互式系统：按键后到字符显示的时间
- 实时系统：传感器数据到控制信号的时间
- 云服务：API 调用的延迟

**优化策略**：
- **优先级调度**：重要任务优先执行
- **缓存**：减少磁盘/网络访问
- **异步 I/O**：避免阻塞等待

#### **利用率（Utilization）**

**定义**：资源被有效使用的时间比例。

**示例**：
- CPU 利用率：`(CPU 执行指令时间) / (总时间)`
- 内存利用率：`(已使用内存) / (总内存)`
- 磁盘利用率：`(磁盘 I/O 时间) / (总时间)`

**权衡**：
- 高利用率可能导致响应时间增加（队列排队）
- 低利用率浪费资源但响应快

```
排队论：
平均响应时间 = 平均服务时间 / (1 - 利用率)

利用率 50%: 响应时间 = 2 × 服务时间
利用率 90%: 响应时间 = 10 × 服务时间
利用率 99%: 响应时间 = 100 × 服务时间
```

### 0.3.2 可靠性（Reliability）：容错、恢复、一致性

#### **容错（Fault Tolerance）**

**定义**：系统在部分组件故障时仍能正常工作。

**故障类型**：
- **瞬时故障**：宇宙射线导致的内存位翻转
- **间歇性故障**：网络抖动
- **永久性故障**：硬盘损坏

**容错机制**：
- **冗余（Redundancy）**：
  - 数据冗余：RAID、备份
  - 时间冗余：重试
  - 空间冗余：多副本
  
- **错误检测**：
  - 校验和（Checksum）
  - ECC（Error-Correcting Code）
  
- **错误恢复**：
  - 回滚（Rollback）
  - 重启（Restart）

#### **恢复（Recovery）**

**定义**：系统从故障中恢复到一致状态的能力。

**示例：文件系统崩溃恢复**

```
操作：rm -rf /tmp/old_data

正常流程：
1. 删除目录项
2. 释放 inode
3. 释放数据块
4. 更新位图

崩溃场景：
- 如果在步骤 2 和 3 之间崩溃：
  - inode 已释放，但数据块未释放
  - 数据块泄漏（无法被使用，也无法被访问）

恢复机制（fsck）：
1. 扫描所有 inode
2. 标记被引用的数据块
3. 释放未被引用的数据块
```

**日志恢复**：

```
日志记录：
[BEGIN]
操作 A: 写 inode
操作 B: 写数据块
[COMMIT]

恢复时：
- 如果日志包含完整的 [BEGIN]...[COMMIT]：重放操作
- 如果日志不完整：丢弃该事务
```

#### **一致性（Consistency）**

**定义**：系统状态满足预定义的约束条件。

**示例：银行转账**

```c
// 一致性要求：总金额不变
int account_A = 1000;
int account_B = 500;

// 转账 $100
account_A -= 100;  // 步骤 1
// 如果这里系统崩溃，钱消失了！
account_B += 100;  // 步骤 2

// 解决方案：事务（Transaction）
BEGIN_TRANSACTION
    account_A -= 100;
    account_B += 100;
COMMIT_TRANSACTION
// 要么两个操作都完成，要么都不完成
```

### 0.3.3 安全性（Security）：隔离、权限、审计

#### **隔离（Isolation）**

**定义**：确保不同实体（用户、进程）互不干扰。

**层次**：

**1. 进程隔离**：
```c
// 进程 A 无法访问进程 B 的内存
int main() {
    // 尝试读取其他进程的地址
    int *p = (int *)0x400000;  // 进程 B 的代码段地址
    printf("%d\n", *p);  // Segmentation Fault
}
```

**2. 用户隔离**：
```bash
# 用户 alice 无法读取用户 bob 的文件
$ cat /home/bob/private.txt
cat: /home/bob/private.txt: Permission denied
```

**3. 内核-用户隔离**：
```c
// 用户程序无法直接修改内核数据结构
extern int *kernel_data;  // 即使知道地址
*kernel_data = 42;  // 触发保护异常
```

#### **权限（Privilege）**

**最小权限原则（Principle of Least Privilege）**：
> 每个程序和用户只应拥有完成其任务所需的最小权限。

**实现机制**：

**1. 用户权限**：
```bash
# Unix 权限模型
-rwxr-xr--  1 alice  staff  1024  file.txt
# |||||||||
# |||||||++- 其他用户权限：r--（只读）
# ||||+++--- 组权限：r-x（读+执行）
# |||+------ 所有者权限：rwx（读+写+执行）
# ||+------- 类型：-（普通文件）
```

**2. 能力（Capabilities）**：
```c
// Linux capabilities：细粒度权限
// 而不是简单的 root vs 非 root

CAP_NET_BIND_SERVICE  // 绑定 < 1024 端口
CAP_SYS_ADMIN         // 系统管理
CAP_KILL              // 发送信号给任意进程
```

#### **审计（Auditing）**

**定义**：记录系统活动，用于事后分析和追责。

**审计日志示例**：
```
[2026-01-01 10:15:32] USER alice LOGIN from 192.168.1.100
[2026-01-01 10:16:45] USER alice OPEN /etc/passwd READ
[2026-01-01 10:17:01] USER alice FAILED OPEN /etc/shadow PERMISSION_DENIED
[2026-01-01 10:20:15] USER alice LOGOUT
```

### 0.3.4 可扩展性（Scalability）：多核、分布式

#### **纵向扩展（Scale Up）**

**定义**：增加单机资源（更多 CPU、内存、磁盘）。

**挑战**：
- **Amdahl 定律**：串行部分限制加速比

```
加速比 = 1 / (S + P/N)

S: 串行部分比例
P: 并行部分比例 = 1 - S
N: 核心数

示例：
- 如果 10% 代码是串行的（S = 0.1）
- 无限核心时，最大加速比 = 1 / 0.1 = 10 倍
```

**多核扩展性问题**：

```c
// 简单的计数器
int counter = 0;
pthread_mutex_t lock;

void increment() {
    pthread_mutex_lock(&lock);
    counter++;
    pthread_mutex_unlock(&lock);
}

// 问题：锁成为瓶颈
// 1 核：100 万次/秒
// 2 核：120 万次/秒（不是 200 万）
// 4 核：150 万次/秒（不是 400 万）
// 锁竞争导致性能无法线性扩展
```

**解决方案**：
- **细粒度锁**：减少临界区大小
- **无锁数据结构**：使用原子操作
- **分区（Partitioning）**：每个核心独立的数据结构

```c
// 每核心计数器（避免共享）
int counter[NUM_CPUS];

void increment() {
    int cpu = get_cpu_id();
    counter[cpu]++;  // 无需加锁
}

int get_total() {
    int sum = 0;
    for (int i = 0; i < NUM_CPUS; i++) {
        sum += counter[i];
    }
    return sum;
}
```

#### **横向扩展（Scale Out）**

**定义**：增加机器数量（分布式系统）。

**挑战**：
- **网络延迟**：毫秒级 vs 纳秒级（本地内存）
- **部分失败**：某些节点故障，其他节点正常
- **一致性**：分布式状态同步

**示例：分布式文件系统（GFS / HDFS）**

```
Master 节点：
- 维护元数据（文件名 → Chunk 列表）
- 协调 Chunk 放置

Chunk 服务器（多个）：
- 存储 64MB Chunk
- 每个 Chunk 有 3 个副本

读取流程：
1. 客户端向 Master 查询文件位置
2. Master 返回 Chunk 服务器列表
3. 客户端直接从 Chunk 服务器读取数据
```

### 0.3.5 易用性（Usability）：API 设计、兼容性

#### **API 设计原则**

**1. 简洁性（Simplicity）**

```c
// 好的设计：POSIX 文件 API
int fd = open("file.txt", O_RDONLY);
read(fd, buffer, size);
close(fd);

// 糟糕的设计：需要记住复杂的参数
int result = complex_function(
    param1, param2, param3, param4, 
    param5, param6, param7, param8,
    NULL, 0, FLAG_A | FLAG_B, ...
);
```

**2. 一致性（Consistency）**

```c
// Unix 哲学：一切皆文件
read(file_fd, buffer, size);     // 读文件
read(socket_fd, buffer, size);   // 读网络
read(pipe_fd, buffer, size);     // 读管道
// 相同的接口，不同的设备
```

**3. 正交性（Orthogonality）**

**定义**：功能独立，无冗余，可组合。

```bash
# Unix 工具的正交性
cat file.txt | grep "error" | sort | uniq -c
# cat：读文件
# grep：过滤
# sort：排序
# uniq：去重
# 每个工具职责单一，可灵活组合
```

#### **向后兼容性（Backward Compatibility）**

**定义**：新版本系统能运行旧版本程序。

**重要性**：
- 保护用户投资
- 避免生态系统分裂
- 降低升级成本

**示例：Linux 系统调用兼容性**

```c
// 1991 年的 Linux 0.01 程序
int fd = open("/etc/passwd", O_RDONLY);
// 在 2026 年的 Linux 6.x 上仍然可以运行

// 即使底层实现完全重写（VFS、ext4 文件系统、RCU 等）
// API 保持不变
```

**兼容性挑战**：

```c
// 旧 API（32 位文件偏移）
off_t lseek(int fd, off_t offset, int whence);
// 问题：off_t 是 32 位，最大文件 2GB

// 新 API（64 位文件偏移）
off64_t lseek64(int fd, off64_t offset, int whence);
// 或者在 64 位系统上，off_t 自动变成 64 位

// 兼容性解决方案：
// - 提供两个版本的 API
// - 或者通过编译选项自动选择
```

## 0.4 操作系统类型与架构

操作系统的**架构**决定了其性能、安全性、可维护性等关键特性。

---

### 0.4.1 单体内核（Monolithic Kernel）：Linux、Unix

#### **核心思想**

**所有操作系统服务运行在内核态，处于同一地址空间。**

```
┌──────────────────────────────────────┐
│          用户态（User Mode）          │
│  应用程序 A   应用程序 B   应用程序 C │
├──────────────────────────────────────┤
│        内核态（Kernel Mode）          │
│  ┌────────────────────────────────┐  │
│  │   进程管理   │   内存管理      │  │
│  ├────────────────────────────────┤  │
│  │   文件系统   │   网络栈        │  │
│  ├────────────────────────────────┤  │
│  │   设备驱动   │   IPC          │  │
│  └────────────────────────────────┘  │
└──────────────────────────────────────┘
```

#### **优点**

**1. 性能高**：
- 内核组件之间直接函数调用，无需上下文切换
- 数据共享无需复制

```c
// 文件系统调用网络栈（内核内部调用）
void fs_send_data(struct socket *sock, struct page *page) {
    // 直接调用，无开销
    tcp_sendpage(sock, page, 0, PAGE_SIZE, 0);
}
```

**2. 实现简单**：
- 无需复杂的进程间通信（IPC）
- 调试相对容易（所有代码在同一地址空间）

#### **缺点**

**1. 安全性差**：
- 任何内核模块的 bug 都可能导致整个系统崩溃

```c
// 驱动程序中的空指针解引用
void buggy_driver_function() {
    int *p = NULL;
    *p = 42;  // 整个系统崩溃（Kernel Panic）
}
```

**2. 难以维护**：
- 代码耦合度高（数百万行代码）
- 修改一个模块可能影响其他模块

**3. 可扩展性受限**：
- 添加新功能需要修改内核
- 内核越来越庞大（Linux 5.x 约 3000 万行代码）

#### **代表系统**

- **Linux**：最成功的单体内核
- **Unix**：历史悠久，影响深远
- **FreeBSD**：高性能，常用于服务器

---

### 0.4.2 微内核（Microkernel）：Minix、L4、seL4

#### **核心思想**

**将操作系统服务移到用户态，内核只保留最基本的机制。**

```
┌──────────────────────────────────────┐
│          用户态（User Mode）          │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌─────┐│
│  │ 文件 │ │ 网络 │ │ 设备 │ │ 应用││
│  │ 服务器│ │ 服务器│ │ 驱动 │ │程序 ││
│  └──────┘ └──────┘ └──────┘ └─────┘│
├──────────────────────────────────────┤
│        内核态（Kernel Mode）          │
│  ┌────────────────────────────────┐  │
│  │  IPC   │ 进程调度  │ 地址空间  │  │
│  └────────────────────────────────┘  │
│          极简的微内核（< 10000 行）   │
└──────────────────────────────────────┘
```

#### **优点**

**1. 安全性高**：
- 驱动程序运行在用户态，崩溃不影响内核

```c
// 设备驱动崩溃
void device_driver_bug() {
    int *p = NULL;
    *p = 42;  // 只有驱动进程崩溃，系统继续运行
}
```

**2. 可靠性高**：
- 服务隔离，故障不传播
- 可以重启单个服务而无需重启系统

**3. 易于验证**：
- 内核代码少（seL4 只有 10000 行）
- 可以进行形式化验证

#### **缺点**

**1. 性能开销**：
- 服务间通信需要 IPC（用户态 → 内核态 → 用户态）
- 频繁的上下文切换

```
单体内核：
应用 → 文件系统（内核态，直接调用）
开销：~10 ns

微内核：
应用 → IPC → 内核 → IPC → 文件服务器 → IPC → 内核 → IPC → 应用
开销：~1000 ns（100 倍）
```

**2. 实现复杂**：
- 需要高效的 IPC 机制
- 服务间依赖复杂

#### **代表系统**

- **Minix**：教学用，启发了 Linux 的诞生
- **L4**：高性能微内核家族
- **seL4**：世界上第一个经过形式化验证的微内核

---

### 0.4.3 混合内核（Hybrid Kernel）：Windows NT、macOS

#### **核心思想**

**结合单体内核的性能与微内核的模块化。**

```
┌──────────────────────────────────────┐
│          用户态（User Mode）          │
│  应用程序  │  部分设备驱动  │  服务   │
├──────────────────────────────────────┤
│        内核态（Kernel Mode）          │
│  ┌────────────────────────────────┐  │
│  │  核心服务（类似单体内核）       │  │
│  │  进程、内存、基础文件系统        │  │
│  ├────────────────────────────────┤  │
│  │  可加载模块（类似微内核）       │  │
│  │  部分驱动、文件系统模块         │  │
│  └────────────────────────────────┘  │
└──────────────────────────────────────┘
```

#### **特点**

- **性能关键部分在内核态**（如调度、内存管理）
- **非关键部分可模块化**（如部分驱动、文件系统）
- **灵活性**：可以根据需要调整架构

#### **代表系统**

- **Windows NT**：从 3.1 到 Windows 11
- **macOS**：基于 XNU 内核（Mach 微内核 + BSD 单体内核）

---

### 0.4.4 外核（Exokernel）与 Unikernel

#### **外核（Exokernel）**

**核心思想**：
> 操作系统不提供抽象，只提供安全的资源复用。应用自己实现抽象。

```
传统操作系统：
硬件 → 内核抽象（文件、进程、内存） → 应用

外核：
硬件 → 安全机制（确保隔离） → 应用库操作系统 → 应用
```

**优点**：
- 应用可以自定义抽象（例如数据库有自己的文件系统）
- 性能极致优化

**缺点**：
- 应用开发复杂
- 没有广泛采用

#### **Unikernel**

**核心思想**：
> 将应用与操作系统库编译为单一镜像，直接运行在虚拟机监控器上。

```
传统虚拟机：
硬件 → Hypervisor → 客户操作系统（完整 Linux） → 应用

Unikernel：
硬件 → Hypervisor → Unikernel（应用 + 最小 OS 库）
```

**优点**：
- 启动快（毫秒级）
- 内存占用小（几 MB vs 几百 MB）
- 攻击面小（安全）

**缺点**：
- 缺乏标准工具（无 shell、调试器）
- 生态不成熟

## 本章小结

通过本章学习，我们建立了对操作系统的宏观认识：

**核心定义**：
- 操作系统是资源管理器与虚拟机的统一体
- 向下管理硬件，向上提供抽象

**三大支柱**：
- **虚拟化**：进程与虚拟内存，让程序以为独占资源
- **并发**：线程与同步，充分利用多核 CPU
- **持久化**：文件系统，数据超越程序生命周期

**设计权衡**：
- 性能 vs 安全性
- 简洁性 vs 功能完整性
- 兼容性 vs 创新

**架构选择**：
- 单体内核：高性能，广泛采用
- 微内核：高安全，形式化验证
- 混合内核：折中方案
- 新兴架构：外核、Unikernel

在接下来的章节中，我们将深入探讨这些概念的实现细节，从硬件基础开始，逐步构建对操作系统的完整理解。
