## 5.1 线程的概念

### 5.1.1 为什么需要线程？进程的局限性

在前面的章节中，我们学习了进程是操作系统进行 CPU 虚拟化的基本单位。每个进程拥有独立的地址空间、独立的文件描述符表、独立的执行状态。这种隔离性带来了安全性和简洁性，但也引入了显著的**开销**。在许多应用场景中，这些开销是不可接受的。

让我们从一个具体的场景开始理解为什么需要线程。考虑一个 Web 服务器（如 Nginx、Apache）的设计：服务器需要同时处理来自成百上千个客户端的请求。每个请求的处理逻辑大致相同——读取 HTTP 请求、解析、查找文件、发送响应——但每个请求需要独立的执行流。

如果使用进程模型，每来一个客户端请求，服务器就 `fork()` 一个子进程来处理：

```c
// 简单的进程模型 Web 服务器
while (1) {
    int client_fd = accept(listen_fd, ...);  // 等待客户端连接
    if (fork() == 0) {                        // 创建子进程
        handle_request(client_fd);            // 子进程处理请求
        exit(0);
    }
    close(client_fd);  // 父进程关闭客户端 fd
}
```

这个模型存在三个严重问题：

**问题一：创建开销大**。`fork()` 系统调用需要复制父进程的页表（即使使用写时复制，页表本身也需要复制）、文件描述符表、信号处理表等。在现代 Linux 系统上，`fork()` 的开销约为 1-10 毫秒。对于一个每秒处理 10000 个请求的 Web 服务器来说，仅 `fork()` 的开销就可能占满 CPU。

**问题二：切换开销大**。如第四章所述，进程间的上下文切换需要切换地址空间、刷新 TLB，开销约为 1-5 微秒。如果系统中有 1000 个并发连接，每秒可能有数万次上下文切换，这些切换的累积开销非常可观。

**问题三：不共享内存**。这是最根本的问题。Web 服务器的所有子进程需要共享同一份代码（HTTP 处理逻辑）、同一份数据（缓存、连接池、配置信息）。但进程模型默认不共享内存，子进程之间要通信必须使用 IPC 机制（管道、共享内存、消息队列等），这些机制都有额外的开销和编程复杂度。

```c
// 进程间共享数据需要显式 IPC
int shared_fd = shm_open("/my_shm", O_CREAT | O_RDWR, 0666);
ftruncate(shared_fd, sizeof(SharedData));
SharedData *shared = mmap(NULL, sizeof(SharedData),
                          PROT_READ | PROT_WRITE, MAP_SHARED, shared_fd, 0);
// 还需要信号量来同步访问...
```

**线程的出现**正是为了解决这些问题。线程的思想是：在一个进程内创建多个执行流，它们**共享**进程的地址空间，但各自拥有**独立的执行状态**。这样：

1. 创建线程不需要复制页表、文件描述符等，开销约为 10-100 微秒（比 `fork()` 快 100 倍）
2. 同一进程内的线程切换不需要切换地址空间，开销约为 0.1-1 微秒（比进程切换快 10 倍）
3. 线程间直接共享内存，不需要 IPC 机制

```c
// 线程模型 Web 服务器
void *handle_request_thread(void *arg) {
    int client_fd = *(int *)arg;
    handle_request(client_fd);
    return NULL;
}

while (1) {
    int client_fd = accept(listen_fd, ...);
    pthread_t tid;
    pthread_create(&tid, NULL, handle_request_thread, &client_fd);
    // 线程直接共享所有全局变量（如缓存、连接池）
}
```

### 5.1.2 线程的定义：轻量级进程

**线程（Thread）** 是进程中的一个执行单元，是 CPU 调度的基本单位。更精确地说，线程是进程中的一个**执行上下文**——它有自己的程序计数器（指向当前执行的指令）、自己的栈指针（指向当前的栈帧）、自己的寄存器状态（保存着当前的计算结果），但它与其他线程共享进程的地址空间、文件描述符、信号处理器等资源。

一个进程至少包含一个线程，称为主线程（Main Thread）。主线程在进程创建时自动产生，它执行 `main()` 函数。进程可以通过 `pthread_create()` 创建额外的线程，这些线程与主线程并发执行。

```
┌─────────────────── 进程 ───────────────────┐
│                                             │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐    │
│  │ Thread 1 │  │ Thread 2 │  │ Thread 3 │    │
│  │ (主线程) │  │          │  │          │    │
│  │ PC=main  │  │ PC=func  │  │ PC=work  │    │
│  │ SP=0x... │  │ SP=0x... │  │ SP=0x... │    │
│  │ 栈:8MB  │  │ 栈:2MB  │  │ 栈:2MB  │    │
│  └────┬────┘  └────┬────┘  └────┬────┘    │
│       │            │            │          │
│       └────────────┼────────────┘          │
│                    │                        │
│    ┌───────────────┼───────────────┐       │
│    │           共享资源            │       │
│    │  • 代码段 (Text)              │       │
│    │  • 数据段 (Data/BSS)          │       │
│    │  • 堆 (Heap)                  │       │
│    │  • 文件描述符表               │       │
│    │  • 信号处理器                 │       │
│    │  • 工作目录                   │       │
│    │  • 用户/组 ID                │       │
│    └───────────────────────────────┘       │
└─────────────────────────────────────────────┘
```

### 5.1.3 线程 vs 进程：共享 vs 独立

理解线程与进程的区别是操作系统学习的核心内容之一。让我们从多个维度进行详细对比。

**共享的部分**：同一进程内的所有线程共享以下资源——代码段（所有线程执行同一份程序代码）、数据段和 BSS 段（所有线程共享全局变量和静态变量）、堆（所有线程共享动态分配的内存，`malloc()` 分配的内存可以被任何线程访问）、文件描述符表（一个线程打开的文件可以被其他线程读写）、信号处理表（信号处理器函数是进程级的）、工作目录、用户/组 ID。

**独立的部分**：每个线程拥有以下私有资源——程序计数器（每个线程执行不同的代码路径）、栈（每个线程有自己的栈空间，存储局部变量和函数调用帧）、寄存器状态（每个线程有自己的中间计算结果）、线程局部存储（TLS，`__thread` 变量）、信号掩码（每个线程可以屏蔽不同的信号）、`errno`（每个线程有自己的错误码）。

这种"共享大部分、独立小部分"的设计带来了两个后果：

**好处**：线程间通信非常高效。两个线程可以通过共享内存直接交换数据，不需要系统调用。例如，一个线程计算出一个结果，写入全局变量，另一个线程直接读取即可。这比进程间的 IPC（需要系统调用、数据复制）快几个数量级。

**代价**：共享带来复杂性。多个线程同时读写共享数据时，如果不加同步，就会产生**数据竞争**——程序的结果取决于线程执行的相对顺序，变得不可预测。这是多线程编程中最常见、最难调试的 Bug。

| 维度 | 进程 | 线程 |
|------|------|------|
| **地址空间** | 独立 | 共享（同进程内） |
| **代码段** | 独立（可通过共享库共享） | 共享 |
| **数据段** | 独立 | 共享 |
| **堆** | 独立 | 共享 |
| **栈** | 独立 | **独立**（每个线程有自己的栈） |
| **寄存器** | 独立 | **独立**（PC、SP 等） |
| **文件描述符** | 独立 | 共享 |
| **信号处理** | 独立 | 共享 |
| **PID** | 独立 | 共享（同 TGID） |
| **TID** | — | 独立 |
| **创建开销** | 大（~1-10 ms） | 小（~10-100 μs） |
| **切换开销** | 大（~1-5 μs） | 小（~0.1-1 μs） |
| **通信方式** | IPC（管道、共享内存等） | 直接读写共享变量 |
| **隔离性** | 强（一个崩溃不影响其他） | 弱（一个崩溃影响整个进程） |

### 5.1.4 线程的优势与代价

线程的优势已经在前面讨论过，这里总结并补充一些实际应用场景。

**优势一：快速创建**。线程创建不需要复制页表、文件描述符等进程级资源。在 Linux 中，`pthread_create()` 底层调用 `clone()` 系统调用，只需要分配一个新的栈（默认 8MB，可通过 `ulimit -s` 或 `pthread_attr` 修改）和一个 `task_struct`。创建时间约为 10-100 微秒，比 `fork()` 的 1-10 毫秒快 100 倍。

**优势二：低开销切换**。同一进程内的线程切换不需要切换地址空间（因为它们共享同一地址空间），因此不需要更新页表基址寄存器、不需要刷新 TLB。这使得线程切换的开销约为 0.1-1 微秒，比进程切换的 1-5 微秒快 10 倍。

**优势三：共享数据**。线程间可以直接通过共享内存通信，不需要系统调用。这对于需要频繁交换数据的场景（如生产者-消费者模型、并行计算中的数据共享）尤为重要。

**优势四：并行执行**。在多核处理器上，多个线程可以真正并行运行在不同的 CPU 核心上，充分利用硬件资源。这是单线程程序无法做到的。

**优势五：响应性**。在 GUI 应用程序中，主线程负责响应用户输入（如鼠标点击、键盘输入），后台线程负责执行耗时操作（如文件下载、图像处理）。如果使用单线程，耗时操作会阻塞 UI，导致界面"卡死"。

**代价一：共享带来的复杂性**。多个线程同时访问共享数据时，如果不加同步，就会产生数据竞争。这个问题在进程模型中不存在（因为进程不共享内存），但在线程模型中是核心挑战。

**代价二：调试困难**。多线程 Bug（死锁、竞态条件）通常难以复现——它们取决于线程执行的精确时序，而这个时序每次运行都可能不同。一个 Bug 可能运行 1000 次只出现 1 次，使得调试极其困难。

**代价三：一个线程崩溃影响全部**。与进程不同，线程没有隔离性。如果一个线程访问了非法内存（如空指针解引用），整个进程（包括所有线程）都会被终止。

## 5.2 线程模型

操作系统和用户空间库对线程的管理方式不同，形成了三种主要的线程模型。理解这三种模型对于理解线程的实现原理和性能特征至关重要。

### 5.2.1 用户级线程（User-Level Threads）

用户级线程是最早出现的线程实现方式。它的核心思想是：线程的创建、调度、同步等操作完全在**用户空间**中由一个线程库（Thread Library）完成，操作系统内核完全不知道线程的存在——对内核而言，它看到的只是一个普通的进程。

```
用户空间:
┌──────────────────────────────────────────┐
│  Thread Library (如 GNU Pth, Green       │
│  Threads, Ruby 的 Thread)                │
│                                          │
│  ┌──────┐  ┌──────┐  ┌──────┐          │
│  │ ULT1 │  │ ULT2 │  │ ULT3 │          │
│  │ PC   │  │ PC   │  │ PC   │          │
│  │ SP   │  │ SP   │  │ SP   │          │
│  │ 栈   │  │ 栈   │  │ 栈   │          │
│  └──┬───┘  └──┬───┘  └──┬───┘          │
│     └─────────┼─────────┘               │
│               │                          │
│         线程调度器（用户态）              │
│         维护就绪队列、执行调度            │
│         切换时不涉及内核                  │
└───────────────┬──────────────────────────┘
                │
                │ 系统调用（如 I/O）
                ↓
内核空间:        
┌───────────────▼──────────────────────────┐
│  KLT（内核看到的单个执行流）              │
│  对内核而言，这就是一个普通进程           │
└──────────────────────────────────────────┘
```

用户级线程的工作原理如下：

**线程创建**：当程序调用线程库的创建函数（如 `thread_create()`）时，线程库在用户空间分配一个新的栈，初始化一组寄存器值（PC 指向线程函数，SP 指向新栈），将新线程加入就绪队列。这个过程完全在用户空间完成，不需要系统调用。

**线程调度**：线程库维护自己的就绪队列和调度算法。当一个线程需要让出 CPU（如等待 I/O、主动 yield）时，线程库保存当前线程的寄存器到其 TCB 中，从就绪队列中选择下一个线程，加载其寄存器，跳转到其 PC。这个过程类似于操作系统的上下文切换，但完全在用户空间完成，不需要陷入内核。

**线程切换**：用户级线程的切换只需要保存/恢复寄存器和切换栈，不需要切换地址空间、刷新 TLB、或切换内核栈。因此，切换速度极快——通常只需 1-10 微秒。

然而，用户级线程有一个致命的缺陷：**一个线程阻塞会导致整个进程阻塞**。当一个用户级线程执行阻塞系统调用（如 `read()`）时，内核不知道还有其他线程在等待运行，它会将整个进程置于阻塞状态。所有线程——包括那些本来可以运行的线程——都被挂起了。

```
ULT2 调用 read() → 内核阻塞整个进程
                   ↓
ULT1 和 ULT3 也被挂起（尽管它们不需要等待 I/O）
                   ↓
CPU 空闲（内核以为进程在等待 I/O，不会调度它）
```

另一个问题是**无法利用多核 CPU**。因为内核只看到一个执行流，它只会将进程调度到一个 CPU 核心上。即使系统有 8 个核心，用户级线程也只能使用其中一个。

**典型实现**：GNU Portable Threads、早期 Java 的 Green Threads（已废弃）、Ruby 的 Thread（受 GIL 限制）。

### 5.2.2 内核级线程（Kernel-Level Threads）

内核级线程是现代操作系统中最常见的线程实现方式。它的核心思想是：线程的管理（创建、调度、同步）由**操作系统内核**直接完成，每个线程都是内核调度的独立单位。

```
用户空间:
┌──────────────────────────────────────────┐
│  ┌──────┐  ┌──────┐  ┌──────┐          │
│  │ KLT1 │  │ KLT2 │  │ KLT3 │          │
│  │ PC   │  │ PC   │  │ PC   │          │
│  │ SP   │  │ SP   │  │ SP   │          │
│  │ 栈   │  │ 栈   │  │ 栈   │          │
│  └──┬───┘  └──┬───┘  └──┬───┘          │
└─────┼─────────┼─────────┼───────────────┘
      │         │         │
      │  系统调用 │  系统调用 │  系统调用
      ↓         ↓         ↓
内核空间:       
┌─────▼─────────▼─────────▼───────────────┐
│  ┌──────┐  ┌──────┐  ┌──────┐          │
│  │ TCB1 │  │ TCB2 │  │ TCB3 │          │
│  └──┬───┘  └──┬───┘  └──┬───┘          │
│     └─────────┼─────────┘               │
│         内核调度器                       │
│   每个线程独立调度                       │
│   一个线程阻塞不影响其他线程             │
└──────────────────────────────────────────┘
```

内核级线程的工作原理如下：

**线程创建**：当程序调用 `pthread_create()` 时，库函数最终会调用内核的 `clone()` 系统调用（Linux）或 `CreateThread()` API（Windows）。内核为新线程分配一个 `task_struct`（Linux）或内核线程对象，分配内核栈，将线程加入调度器的就绪队列。

**线程调度**：内核调度器将每个线程视为独立的调度单位。线程有自己的调度优先级、时间片、状态（运行、就绪、阻塞等）。调度器可以将不同的线程调度到不同的 CPU 核心上，实现真正的并行。

**线程阻塞**：当一个内核级线程执行阻塞系统调用时，只有该线程被阻塞，其他线程继续运行。这是内核级线程相对于用户级线程的最大优势。

**多核利用**：因为内核可以看到所有线程，它可以将线程分配到不同的 CPU 核心上，充分利用多核硬件。

然而，内核级线程也有代价：

**切换开销大**：线程切换需要陷入内核、保存/恢复内核栈、可能需要切换地址空间（如果线程属于不同进程）。这使得切换开销约为 1-5 微秒，比用户级线程的切换慢 10-100 倍。

**创建开销大**：每次创建线程都需要系统调用，内核需要分配 `task_struct`、内核栈等资源。创建时间约为 10-100 微秒。

**典型实现**：Windows 线程、Linux 通过 NPTL 实现的 pthread、macOS 的 pthread。

### 5.2.3 混合线程模型（M:N 模型）

混合模型试图结合用户级和内核级线程的优点：将 M 个用户级线程映射到 N 个内核级线程上（通常 M >> N）。用户级线程的创建和大部分切换在用户空间完成（低开销），但当一个用户级线程阻塞时，线程库可以将它的内核线程让给其他用户级线程使用（避免整个进程阻塞）。

```
用户空间:
┌──────────────────────────────────────────┐
│  Thread Library (如 Go runtime)          │
│  ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐   │
│  │ULT1│ │ULT2│ │ULT3│ │ULT4│ │ULT5│   │
│  └─┬──┘ └─┬──┘ └─┬──┘ └─┬──┘ └─┬──┘   │
│    │      │      │      │      │        │
│    └──────┼──────┘      └──────┼──┘     │
│           │ M:N 调度器         │         │
│           │ 将 ULT 映射到 KLT  │         │
└───────────┼────────────────────┼─────────┘
            │                    │
内核空间:    │                    │
┌───────────▼────────────────────▼─────────┐
│  ┌──────┐  ┌──────┐  ┌──────┐           │
│  │ KLT1 │  │ KLT2 │  │ KLT3 │           │
│  └──────┘  └──────┘  └──────┘           │
│         内核调度器                        │
└──────────────────────────────────────────┘

5 个 ULT 映射到 3 个 KLT → 可利用 3 个 CPU 核心
```

混合模型的关键组件是用户空间的**M:N 调度器**。它负责：
- 将用户级线程分配到内核级线程上
- 当一个 ULT 阻塞时，将对应的 KLT 重新分配给其他 ULT
- 当 ULT 数量超过 KLT 数量时，决定哪个 ULT 优先使用 KLT

**Go 语言的 goroutine** 是混合模型最成功的实现之一。Go 运行时维护一个 M:N 调度器，将大量的 goroutine（轻量级线程）映射到少量的操作系统线程上。goroutine 的创建开销极低（约 2KB 栈空间，微秒级创建时间），可以轻松创建数十万个并发 goroutine。

**代价**：混合模型的实现非常复杂。用户空间调度器需要与内核协调，处理各种边界情况（如信号投递、CPU 亲和性）。某些操作（如阻塞系统调用）仍然可能导致问题，需要特殊的处理（如 Go 的 netpoller 将阻塞 I/O 转换为非阻塞 I/O + epoll）。

### 5.2.4 三种模型的性能对比与权衡

| 维度 | 用户级线程 | 内核级线程 | 混合模型 |
|------|-----------|-----------|----------|
| **创建开销** | ~1 μs | ~10-100 μs | ~1-10 μs |
| **切换开销** | ~1 μs | ~1-5 μs | ~1-2 μs |
| **需要系统调用** | ❌ | ✅ | 部分 |
| **阻塞影响** | 整个进程 | 仅该线程 | 取决于实现 |
| **多核利用** | ❌ | ✅ | ✅ |
| **实现复杂度** | 低 | 中 | 高 |
| **调度灵活性** | 高（可自定义） | 低（内核决定） | 高 |
| **典型代表** | Green Threads | Windows 线程 | Go goroutine |

选择哪种模型取决于应用场景：
- 如果需要跨平台且不需要多核并行，用户级线程足够
- 如果需要充分利用多核 CPU 且线程数量不多，内核级线程是最佳选择
- 如果需要大规模并发（如 Web 服务器、微服务），混合模型（Go goroutine）是最优解

## 5.3 线程控制块（TCB）

### 5.3.1 TCB vs PCB：共享与独立部分

线程控制块（Thread Control Block, TCB）是操作系统管理线程的核心数据结构。它存储着线程的私有状态信息。理解 TCB 与 PCB（Process Control Block）的关系，是理解线程实现的关键。

在 Linux 中，进程和线程在内核中都是 `task_struct` 结构体——它们没有本质区别。区别在于 `task_struct` 中某些字段的值：属于同一进程的线程共享某些字段（如页表、文件描述符表），而各自拥有其他字段（如栈指针、寄存器状态）。

```
┌─────────────────────────────────────────────────────────────────┐
│                     PCB / task_struct (进程级)                   │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 进程级信息（所有线程共享）                                │   │
│  │  • PID / TGID（线程组 ID）                               │   │
│  │  • 页表基址（mm_struct → 页表）                          │   │
│  │  • 文件描述符表（files_struct）                          │   │
│  │  • 信号处理表（sighand_struct）                          │   │
│  │  • 工作目录（fs_struct）                                 │   │
│  │  • 资源限制（rlimit）                                    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐                        │
│  │  TCB 1  │  │  TCB 2  │  │  TCB 3  │                        │
│  │  TID    │  │  TID    │  │  TID    │                        │
│  │  PC     │  │  PC     │  │  PC     │  ← 各自执行不同的代码   │
│  │  SP     │  │  SP     │  │  SP     │  ← 各自的栈指针         │
│  │  寄存器 │  │  寄存器 │  │  寄存器 │  ← 各自的计算状态       │
│  │  栈     │  │  栈     │  │  栈     │  ← 各自的栈空间         │
│  │  线程状态│  │  线程状态│  │  线程状态│                        │
│  │  信号掩码│  │  信号掩码│  │  信号掩码│                        │
│  └─────────┘  └─────────┘  └─────────┘                        │
└─────────────────────────────────────────────────────────────────┘
```

### 5.3.2 线程私有数据

每个线程拥有以下**私有数据**，其他线程无法直接访问：

**程序计数器（PC）**：线程当前执行的指令地址。每个线程执行不同的代码路径（或同一代码的不同位置），因此需要独立的 PC。例如，线程 1 可能在执行 `for (int i = 0; i < 100; i++)` 的第 50 次迭代，而线程 2 可能在执行同一条循环的第 10 次迭代。

**栈指针（SP）**：线程的栈帧位置。每个线程有自己的栈空间（默认 8MB），栈中存储着函数调用帧、局部变量、返回地址等。栈指针指向当前栈帧的位置。如果两个线程共享栈，它们的局部变量会互相覆盖，函数调用会混乱——这会导致立即崩溃。

**通用寄存器**：线程的中间计算结果。CPU 的寄存器是物理资源，在某一时刻只能保存一个线程的数据。当线程被切换出去时，其寄存器值被保存到 TCB 中；当线程被切换回来时，从 TCB 恢复到 CPU 寄存器。

**栈（Stack）**：每个线程有自己的栈空间，存储函数调用帧和局部变量。栈的大小默认为 8MB（可通过 `pthread_attr_setstacksize()` 修改或 `ulimit -s` 查看）。栈空间在进程的虚拟地址空间中分配，每个线程的栈位于不同的虚拟地址范围。

**线程局部存储（TLS）**：TLS 是一种让每个线程拥有自己私有全局变量的机制。在 C 中使用 `__thread` 关键字声明，在 C++ 中使用 `thread_local` 关键字。TLS 变量在每个线程中有独立的副本，互不影响。

```c
// 线程局部存储示例
__thread int thread_local_counter = 0;  // 每个线程有自己的副本

void *thread_func(void *arg) {
    thread_local_counter++;  // 只修改本线程的副本
    printf("Thread %lu: counter = %d\n", pthread_self(), thread_local_counter);
    return NULL;
}
```

TLS 的实现依赖于架构：
- **x86-64**：使用 `FS` 段寄存器指向 TLS 区域
- **RISC-V**：使用 `tp`（x4）寄存器指向 TLS 区域
- **ARM**：使用专用寄存器或通过系统调用获取

**信号掩码**：每个线程可以独立地屏蔽（block）某些信号。`pthread_sigmask()` 函数用于设置线程级的信号掩码。这允许不同的线程处理不同的信号。

**`errno`**：在多线程程序中，`errno` 是线程局部的。每个线程有自己的 `errno` 副本，一个线程设置 `errno` 不会影响其他线程。这是通过 TLS 实现的。

### 5.3.3 线程共享数据

同一进程内的所有线程**共享**以下资源。理解哪些资源是共享的，对于编写正确的多线程程序至关重要。

**代码段（Text Segment）**：所有线程执行同一份程序代码。这是显然的——线程是进程中的执行流，它们运行的是同一个程序。代码段是只读的，因此共享不会有同步问题。

**数据段（Data/BSS Segment）**：所有线程共享全局变量和静态变量。这是多线程编程中**最容易出问题**的地方。如果两个线程同时读写同一个全局变量，就会产生数据竞争。

```c
int shared_counter = 0;  // 所有线程共享

// 线程 1 和线程 2 同时执行：
shared_counter++;  // ⚠️ 这不是原子操作！
// 可能的结果：两个线程都读到 0，都写入 1，最终 counter = 1（期望是 2）
```

**堆（Heap）**：所有线程共享动态分配的内存。`malloc()` 和 `free()` 是线程安全的（C 标准库保证），但通过指针共享堆内存时需要同步。

```c
int *data = malloc(sizeof(int) * 100);  // 堆上分配
// 线程 1: data[0] = 42;
// 线程 2: data[0] = 100;  // ⚠️ 如果不加同步，结果不确定
```

**文件描述符表**：一个线程打开的文件可以被其他线程读写。文件偏移量是共享的——如果两个线程同时读取同一个文件，它们会交替地读取数据（因为偏移量在每次 `read()` 后自动增加）。

**信号处理器**：信号处理函数是进程级的。当信号到达时，内核选择一个没有屏蔽该信号的线程来处理。

## 5.4 POSIX 线程（Pthreads）

POSIX 线程（Pthreads）是 IEEE POSIX 1003.1c 标准定义的线程 API，是 Unix/Linux 系统上最广泛使用的线程接口。它提供了一套完整的线程管理函数，包括线程创建、终止、同步等。

### 5.4.1 线程创建：pthread_create()

`pthread_create()` 是创建新线程的核心函数。它的四个参数分别指定了新线程的标识、属性、执行函数和参数。

```c
#include <pthread.h>

int pthread_create(
    pthread_t *thread,              // 输出：新线程的 ID
    const pthread_attr_t *attr,     // 线程属性（NULL = 默认属性）
    void *(*start_routine)(void *), // 线程函数：新线程从这里开始执行
    void *arg                       // 传递给线程函数的参数
);
// 返回值：成功返回 0，失败返回错误码
// 常见错误码：EAGAIN（资源不足）、EINVAL（属性无效）
```

让我们通过一个完整的例子来理解 `pthread_create()` 的使用：

```c
#include <stdio.h>
#include <stdlib.h>
#include <pthread.h>
#include <unistd.h>

// 线程函数：每个线程执行这个函数
void *thread_func(void *arg) {
    int id = *(int *)arg;  // 获取传入的参数
    
    printf("Thread %d: started (TID = %lu)\n", id, pthread_self());
    
    // 模拟一些工作
    for (int i = 0; i < 3; i++) {
        printf("Thread %d: working... step %d\n", id, i + 1);
        usleep(100000);  // 休眠 100ms
    }
    
    printf("Thread %d: finished\n", id);
    
    // 返回结果给 pthread_join
    int *result = malloc(sizeof(int));
    *result = id * 10;
    return result;
}

int main() {
    pthread_t threads[3];
    int ids[3] = {1, 2, 3};

    printf("Main: creating 3 threads\n");
    
    // 创建 3 个线程
    for (int i = 0; i < 3; i++) {
        int ret = pthread_create(&threads[i], NULL, thread_func, &ids[i]);
        if (ret != 0) {
            fprintf(stderr, "pthread_create failed: %d\n", ret);
            exit(1);
        }
        printf("Main: created thread %d (TID = %lu)\n", ids[i], threads[i]);
    }

    // 等待所有线程完成
    printf("Main: waiting for threads to finish\n");
    for (int i = 0; i < 3; i++) {
        void *result;
        pthread_join(threads[i], &result);
        printf("Main: thread %d returned %d\n", ids[i], *(int *)result);
        free(result);
    }

    printf("Main: all threads finished\n");
    return 0;
}
```

> **关于参数传递的陷阱**：`pthread_create()` 的第四个参数是 `void *` 类型，可以传递任意数据的指针。但必须确保指针指向的内存在线程访问时仍然有效。上面的 `&ids[i]` 是安全的，因为 `ids` 是 `main` 中的局部数组，在 `pthread_join` 之前不会被释放。但如果你传递的是循环变量 `&i`，就会有问题——因为 `i` 的值在循环中不断变化，线程读取到的可能是最终值而不是创建时的值。

### 5.4.2 线程终止与等待

#### **pthread_exit() — 线程主动退出**

```c
void pthread_exit(void *retval);
// retval：线程的返回值，可被 pthread_join 获取
```

在线程函数中调用 `return` 等价于调用 `pthread_exit(return_value)`。两者都会终止当前线程，并将返回值传递给等待的 `pthread_join()`。

> **重要警告**：不要在线程中调用 `exit()`！`exit()` 会终止**整个进程**（包括所有线程）。如果只想终止当前线程，应该使用 `pthread_exit()` 或 `return`。

#### **pthread_join() — 等待线程完成**

```c
int pthread_join(pthread_t thread, void **retval);
// thread：要等待的线程 ID
// retval：输出参数，获取线程的返回值（可以为 NULL 表示不关心返回值）
// 返回值：成功返回 0，失败返回错误码
```

`pthread_join()` 的行为类似于进程中的 `waitpid()`：
- **阻塞调用者**：调用 `pthread_join()` 的线程会阻塞，直到指定的线程终止。
- **回收资源**：终止线程的资源（如栈空间）会被回收。如果不调用 `pthread_join()`，已终止线程的资源不会被回收（类似于僵尸进程）。
- **获取返回值**：通过 `retval` 参数获取线程函数的返回值。

### 5.4.3 线程分离：pthread_detach()

```c
int pthread_detach(pthread_t thread);
// 将线程标记为"分离"状态
// 分离后的线程终止时自动回收资源，不需要 pthread_join
```

线程有两种状态：**可汇合（joinable）**和**分离（detached）**。

**可汇合线程**（默认）：线程终止后，其资源被保留，等待 `pthread_join()` 回收。如果不调用 `pthread_join()`，线程的资源会一直占用（类似僵尸进程）。这是为了允许其他线程获取其返回值。

**分离线程**：线程终止后，其资源立即自动回收。其他线程无法通过 `pthread_join()` 获取其返回值。适用于不需要返回值的"发后即忘"（fire-and-forget）场景。

```c
// 创建分离线程的两种方式：

// 方式 1：创建后分离
pthread_t tid;
pthread_create(&tid, NULL, thread_func, NULL);
pthread_detach(tid);  // 分离后不能再 pthread_join

// 方式 2：通过属性创建时就设置为分离
pthread_attr_t attr;
pthread_attr_init(&attr);
pthread_attr_setdetachstate(&attr, PTHREAD_CREATE_DETACHED);
pthread_create(&tid, &attr, thread_func, NULL);
pthread_attr_destroy(&attr);
```

> **注意**：一个线程必须是 joinable 或 detached 之一。对已经 detached 的线程调用 `pthread_join()` 是未定义行为。对已经 joinable 的线程再次调用 `pthread_detach()` 也是未定义行为。

### 5.4.4 线程取消：pthread_cancel()

```c
int pthread_cancel(pthread_t thread);
// 请求取消指定线程（非阻塞，只是发送取消请求）
```

`pthread_cancel()` 向目标线程发送一个取消请求。但线程不会立即被终止——它的行为取决于两个设置：

**取消状态**：`pthread_setcancelstate()` 控制线程是否接受取消请求。
- `PTHREAD_CANCEL_ENABLE`（默认）：接受取消请求
- `PTHREAD_CANCEL_DISABLE`：忽略取消请求

**取消类型**：`pthread_setcanceltype()` 控制何时执行取消。
- `PTHREAD_CANCEL_DEFERRED`（默认）：在线程到达**取消点**时才执行取消。取消点包括 `read()`、`write()`、`sleep()`、`pthread_join()`、`printf()` 等阻塞函数。
- `PTHREAD_CANCEL_ASYNCHRONOUS`：随时可能被取消（危险，可能导致资源泄漏）。

## 5.5 多线程问题

### 5.5.1 竞态条件（Race Condition）

竞态条件是多线程编程中最常见、最危险的问题。当多个线程的执行顺序影响程序结果时，就存在竞态条件。

让我们通过一个经典的例子来理解竞态条件。假设我们有一个共享计数器，两个线程分别对其执行 100000 次递增操作：

```c
#include <pthread.h>
#include <stdio.h>

int counter = 0;  // 共享变量

void *increment(void *arg) {
    for (int i = 0; i < 100000; i++) {
        counter++;  // ⚠️ 这不是原子操作！
    }
    return NULL;
}

int main() {
    pthread_t t1, t2;
    pthread_create(&t1, NULL, increment, NULL);
    pthread_create(&t2, NULL, increment, NULL);
    pthread_join(t1, NULL);
    pthread_join(t2, NULL);
    printf("Counter = %d (expected: 200000)\n", counter);
    return 0;
}
```

运行这个程序，你可能得到 200000，也可能得到 183456、197231 或其他值。每次运行结果都不同。这就是竞态条件的表现。

**为什么 `counter++` 不是原子操作？**

`counter++` 在 C 语言中看起来是一个简单的操作，但在机器指令层面，它被分解为三步：

```
1. LOAD  counter → 寄存器    // 从内存读取 counter 的当前值到寄存器
2. ADD   寄存器, 1           // 将寄存器中的值加 1
3. STORE 寄存器 → counter    // 将寄存器中的新值写回内存
```

如果两个线程**交错执行**这三步，就会出现问题：

```
时间    线程 1                      线程 2                      counter 的值
──────────────────────────────────────────────────────────────────────────
t1      LOAD counter → R1 (R1=0)
t2      ADD R1, 1 (R1=1)
t3                                  LOAD counter → R2 (R2=0)    ← 读到旧值！
t4      STORE R1 → counter          ADD R2, 1 (R2=1)
t5                                  STORE R2 → counter          ← 覆盖了线程 1 的结果！

最终结果：counter = 1（期望值：2）
```

在 t3 时刻，线程 2 读取 counter 的值时，线程 1 还没有将新值写回内存，所以线程 2 读到了旧值（0）。然后线程 2 也计算出 1 并写回。最终 counter = 1，而不是期望的 2。

如果这种交错在 200000 次递增中发生了 16544 次，最终结果就是 200000 - 16544 = 183456。

### 5.5.2 临界区（Critical Section）

**临界区**是访问共享资源的代码段。在上面的例子中，`counter++` 就是临界区——它访问了共享变量 `counter`。

临界区问题的核心要求是：同一时刻最多只有一个线程可以执行临界区代码。这称为**互斥（Mutual Exclusion）**。

除了互斥，一个好的临界区解决方案还应该满足：

**进步（Progress）**：如果临界区空闲（没有线程在执行临界区代码），那么等待进入临界区的线程应该能在有限时间内进入。不能出现"临界区空闲但所有线程都在等待"的死锁情况。

**有限等待（Bounded Waiting）**：每个线程等待进入临界区的时间应该有上界。不能出现"某个线程永远无法进入临界区"的饥饿情况。

使用互斥锁保护临界区是最常见的解决方案：

```c
#include <pthread.h>

pthread_mutex_t mutex = PTHREAD_MUTEX_INITIALIZER;
int counter = 0;

void *safe_increment(void *arg) {
    for (int i = 0; i < 100000; i++) {
        pthread_mutex_lock(&mutex);    // 获取锁（进入临界区）
        counter++;                      // 临界区：受保护的共享变量访问
        pthread_mutex_unlock(&mutex);  // 释放锁（离开临界区）
    }
    return NULL;
}
```

`pthread_mutex_lock()` 保证了同一时刻只有一个线程可以持有锁。当一个线程持有锁时，其他线程调用 `pthread_mutex_lock()` 会阻塞等待，直到锁被释放。这确保了 `counter++` 的三步操作不会被其他线程打断。

### 5.5.3 数据竞争（Data Race）vs 竞态条件

这两个概念经常被混淆，但它们有严格的定义和区别。

**数据竞争（Data Race）**的定义是：两个线程同时访问同一内存位置，**至少一个是写操作**，且**没有使用同步原语**保护。数据竞争是 C/C++ 标准中的概念——存在数据竞争的程序是**未定义行为**（Undefined Behavior），编译器可以做出任何假设，包括优化掉你的代码。

**竞态条件（Race Condition）**的定义是：程序的正确性依赖于线程执行的相对顺序。竞态条件是逻辑层面的问题——即使没有数据竞争，程序也可能有竞态条件。

```c
// 数据竞争 + 竞态条件
int counter = 0;
// 线程 1: counter++
// 线程 2: counter++
// 数据竞争：两个线程同时读写 counter，没有同步
// 竞态条件：结果依赖于两个 ++ 的执行顺序

// 竞态条件但无数据竞争
int x = 0, y = 0;
// 线程 1: x = 1; y = 2;
// 线程 2: if (y == 2) assert(x == 1);
// 没有数据竞争（访问不同的变量）
// 但有竞态条件（assert 可能失败，取决于执行顺序）
```

### 5.5.4 线程安全（Thread Safety）与可重入

**线程安全函数**：可以被多个线程同时调用而不会产生错误结果的函数。线程安全的关键是：函数不能依赖共享的可变状态，或者对共享状态的访问必须有同步保护。

**线程不安全的常见模式**：

```c
// 模式 1：使用全局/静态变量
int global_state = 0;
void unsafe_func() {
    global_state++;  // ⚠️ 多线程同时修改，数据竞争
}

// 模式 2：返回指向静态缓冲区的指针
char *unsafe_ctime(const time_t *timer) {
    static char buf[26];  // ⚠️ 所有调用共享同一缓冲区
    // ... 填充 buf ...
    return buf;
    // 如果线程 A 调用 ctime()，然后线程 B 也调用 ctime()
    // 线程 A 读取到的 buf 可能是线程 B 写入的结果
}

// 模式 3：调用线程不安全的函数
void unsafe_wrapper() {
    unsafe_ctime(&t);  // ⚠️ 间接调用了线程不安全函数
}
```

**线程安全的替代方案**：

```c
// 替代 ctime：使用线程安全版本
char *ctime_r(const time_t *timer, char *buf);
// 调用者提供缓冲区，避免共享

// 替代全局变量：使用线程局部存储
__thread int local_state = 0;  // 每个线程有自己的副本
```

**可重入函数（Reentrant Function）**比线程安全更严格。可重入函数不使用任何静态/全局数据，不调用任何不可重入函数，可以在执行过程中被中断（如信号处理程序）并重新进入而不产生错误。

| 特性 | 线程安全 | 可重入 |
|------|---------|--------|
| 使用全局变量 | 可以（加锁保护） | 不可以 |
| 使用静态变量 | 可以（加锁保护） | 不可以 |
| 可被信号中断重入 | 不保证 | 保证 |
| 性能 | 可能有锁开销 | 通常更快 |
| 关系 | — | 可重入 ⇒ 线程安全 |

## 5.6 Linux 线程实现

### 5.6.1 clone() 系统调用：统一进程与线程

在 Linux 中，进程和线程的创建最终都通过 `clone()` 系统调用实现。这是 Linux 线程实现的核心——它揭示了一个深刻的洞察：**进程和线程在 Linux 内核中没有本质区别，它们都是执行上下文（task_struct），区别仅在于共享了多少资源**。

`clone()` 的关键特点是可以通过**标志位**控制子任务与父任务共享哪些资源：

```c
// Linux 系统调用（简化）
int clone(int (*fn)(void *), void *stack, int flags, void *arg);
```

关键标志位及其含义：

- `CLONE_VM`：父子共享地址空间（同一页表）。不设置此标志时，子任务获得地址空间的副本（类似 `fork()`）。
- `CLONE_FS`：父子共享文件系统信息（根目录、当前工作目录）。
- `CLONE_FILES`：父子共享文件描述符表。不设置时，子任务获得文件描述符表的副本。
- `CLONE_SIGHAND`：父子共享信号处理器。不设置时，子任务获得信号处理器的副本。
- `CLONE_THREAD`：将子任务放入父任务的线程组（共享 TGID）。这是区分"线程"和"进程"的关键标志。

```c
// fork() 的内部实现（简化）
pid_t fork(void) {
    return clone(NULL, 0, SIGCHLD, NULL);
    // 不设置任何 CLONE_* 标志
    // → 完全独立：独立地址空间、独立文件描述符、独立信号处理
    // → 这就是"进程"
}

// pthread_create() 的内部实现（简化）
int pthread_create(...) {
    return clone(start_routine, stack,
                 CLONE_VM | CLONE_FS | CLONE_FILES |
                 CLONE_SIGHAND | CLONE_THREAD | CLONE_SETTLS |
                 CLONE_PARENT_SETTID | CLONE_CHILD_CLEARTID,
                 arg);
    // 设置了大量 CLONE_* 标志
    // → 共享地址空间、共享文件描述符、共享信号处理
    // → 这就是"线程"
}
```

> **核心洞察**：在 Linux 中，`fork()` 和 `pthread_create()` 底层使用同一个系统调用 `clone()`，区别仅在于标志位。`fork()` 不设置 `CLONE_*` 标志（完全独立），`pthread_create()` 设置 `CLONE_VM | CLONE_FILES | CLONE_THREAD` 等标志（共享资源）。从内核的角度看，进程和线程都是 `task_struct`，只是资源共享的程度不同。

### 5.6.2 NPTL（Native POSIX Thread Library）

NPTL 是 Linux 上标准的 Pthreads 实现，由 Ulrich Drepper 和 Ingo Molnár 设计，于 2003 年随 Linux 2.6 引入，取代了早期的 LinuxThreads 实现。

NPTL 的设计目标是提供一个高性能、可扩展的 1:1 线程模型实现。它的关键特性包括：

**1:1 模型**：每个用户线程对应一个内核线程（`task_struct`）。这意味着内核可以独立调度每个线程，线程可以并行运行在多个 CPU 核心上。

**futex 同步**：NPTL 使用 futex（Fast Userspace Mutex）实现互斥锁和条件变量。futex 的核心思想是：在无竞争的情况下，锁操作完全在用户空间完成（只需一个原子操作，约 10ns）；只有在有竞争的情况下才需要陷入内核（约 1μs）。这使得大多数锁操作的开销极低。

**线程栈管理**：NPTL 使用 `mmap()` 分配线程栈（默认 8MB，其中 4KB 的 guard page 用于检测栈溢出）。栈空间使用延迟分配（demand paging），只有实际使用的页面才会分配物理内存。

**TLS 实现**：NPTL 使用架构特定的寄存器实现 TLS。在 x86-64 上使用 `FS` 段寄存器，在 RISC-V 上使用 `tp`（x4）寄存器。每个线程的 TLS 区域通过 `arch_prctl()` 系统调用设置。

### 5.6.3 线程组与 TGID

Linux 使用**线程组（Thread Group）**概念来管理属于同一进程的线程。线程组中的所有线程共享同一个 TGID（Thread Group ID），TGID 等于主线程的 PID。

```c
// 查看进程的线程
$ ps -T -p <PID>
// 输出示例：
//   PID   SPID  CMD
//  1234   1234  ./my_server    ← 主线程（PID = TGID）
//  1234   1235  ./my_server    ← 线程 2（PID ≠ TGID）
//  1234   1236  ./my_server    ← 线程 3

// 或者查看 /proc 文件系统
$ ls /proc/<PID>/task/
// 1234  1235  1236  ← 每个子目录对应一个线程
```

- `pid`：每个 `task_struct` 的唯一标识（内核视角的线程 ID）
- `tgid`：线程组 ID，等于主线程的 `pid`（用户视角的进程 PID）
- `getpid()` 系统调用返回 `tgid`（用户看到的 PID）
- `gettid()` 系统调用返回 `pid`（内核线程 ID）
- `pthread_self()` 返回 `tid`（POSIX 线程 ID，可能与内核 TID 不同）

### 5.6.4 线程调度

Linux 调度器对线程和进程一视同仁——它们都是 `task_struct`，使用相同的调度策略。CFS（Completely Fair Scheduler）对线程的处理如下：

- 每个线程（`task_struct`）独立计算 vruntime
- 线程间公平竞争 CPU 时间
- 同一进程内的线程通过**组调度**（Group Scheduling）共享 CPU 配额（可选）

```c
// 设置线程的调度策略
struct sched_param param;
param.sched_priority = 50;
pthread_setschedparam(tid, SCHED_FIFO, &param);

// 调度策略：
// SCHED_OTHER — 普通 CFS 调度（默认）
// SCHED_FIFO  — 实时先进先出（高优先级）
// SCHED_RR    — 实时轮转
// SCHED_BATCH — 批处理优化
// SCHED_IDLE  — 最低优先级
```

## 5.7 面试高频考点

**Q1：进程和线程的区别？**

这是操作系统面试中最基本的问题，回答时应该从多个维度对比：资源分配（进程是资源分配单位，线程是调度单位）、地址空间（进程独立，线程共享）、创建开销（进程大，线程小）、切换开销（进程大因为要切换页表，线程小因为共享页表）、通信方式（进程用 IPC，线程直接共享内存）、隔离性（进程强，线程弱）。

**Q2：线程共享哪些资源，不共享哪些资源？**

共享：代码段、数据段、堆、文件描述符、信号处理器、工作目录。不共享：栈、程序计数器、寄存器、TLS、信号掩码、errno。记忆技巧：**执行状态独立，资源共享**。

**Q3：如何避免竞态条件？**

互斥锁（最常用）、原子操作（最高效）、读写锁（多读者场景）、条件变量（等待条件）、无锁数据结构（高级技巧）。选择哪种方式取决于场景和性能要求。

**Q4：用户级线程和内核级线程的区别？**

用户级线程由用户态库管理，内核不可见，切换快但一个阻塞全部阻塞，无法利用多核。内核级线程由内核管理，可以利用多核，一个阻塞不影响其他，但切换需要系统调用。现代系统通常使用 1:1 模型（Linux NPTL）或 M:N 模型（Go goroutine）。

**Q5：Linux 中 fork() 和 pthread_create() 的实现有什么共同点？**

两者底层都使用 `clone()` 系统调用，区别在于标志位。`fork()` 不设置 `CLONE_*` 标志（完全独立），`pthread_create()` 设置 `CLONE_VM | CLONE_FILES | CLONE_THREAD`（共享资源）。在 Linux 中，进程和线程本质上都是 `task_struct`，只是资源共享的程度不同。
