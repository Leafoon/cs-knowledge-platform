## 31.1 安全威胁模型

### 31.1.1 什么是安全威胁模型

**安全威胁模型（Threat Model）** 是系统化分析系统可能面临的攻击向量、攻击者能力和潜在损害的方法。在操作系统层面，安全威胁模型帮助我们理解：

1. **攻击者是谁？** — 内部用户、外部黑客、恶意软件、物理攻击者
2. **攻击者能做什么？** — 读取/修改内存、执行代码、拦截网络流量
3. **我们要保护什么？** — 数据机密性、完整性、可用性（CIA 三要素）

```
安全威胁模型的核心框架：

┌─────────────────────────────────────────────────────┐
│                   安全威胁模型                        │
├─────────────────────────────────────────────────────┤
│                                                     │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐          │
│  │ 资产     │  │ 威胁     │  │ 攻击者   │          │
│  │ Assets   │  │ Threats  │  │ Attacker │          │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘          │
│       │             │             │                 │
│       └─────────────┼─────────────┘                 │
│                     │                               │
│              ┌──────┴──────┐                        │
│              │  风险评估    │                        │
│              │ Risk = I × P │                        │
│              │ 影响 × 概率  │                        │
│              └──────┬──────┘                        │
│                     │                               │
│              ┌──────┴──────┐                        │
│              │  防护措施    │                        │
│              │ Mitigations │                        │
│              └─────────────┘                        │
└─────────────────────────────────────────────────────┘
```

### 31.1.2 威胁类型分类

操作系统面临的安全威胁可以分为以下几类：

| 威胁类型 | 说明 | 典型例子 |
|---------|------|---------|
| **机密性威胁** | 未经授权的信息读取 | 内存泄露、侧信道攻击 |
| **完整性威胁** | 未经授权的信息修改 | 缓冲区溢出、恶意软件 |
| **可用性威胁** | 系统资源被耗尽或不可用 | DoS 攻击、fork 炸弹 |
| **认证威胁** | 身份冒充或绕过认证 | 暴力破解、会话劫持 |
| **授权威胁** | 越权访问受保护资源 | 权限提升、TOCTTOU |

#### 典型攻击示例

**1. 缓冲区溢出（Buffer Overflow）**

```c
// 经典的栈溢出漏洞
void vulnerable_function(char *input) {
    char buffer[64];
    strcpy(buffer, input);  // 没有边界检查！
    // 如果 input 超过 64 字节，会覆盖返回地址
}
```

攻击者通过精心构造输入数据，覆盖函数返回地址，使程序跳转到攻击者注入的恶意代码（shellcode）。

**2. 权限提升（Privilege Escalation）**

```
权限提升攻击流程：

普通用户进程 ──→ 利用内核漏洞 ──→ 获得 root 权限
     │                              │
     │  uid=1000                    │  uid=0
     │  权限有限                     │  完全控制
     └──────────────────────────────┘
```

**3. 侧信道攻击（Side-Channel Attack）**

通过观察系统的物理特征（时间、功耗、电磁辐射）推断敏感信息，而非直接破解加密算法。

### 31.1.3 攻击面分析

**攻击面（Attack Surface）** 是系统中所有可被攻击者利用的入口点的总和。

```

│  ┌─────────────────────────────────────────────┐    │

│  │  • read/write/open/ioctl                     │    │
│  │  • mmap/brk                                  │    │
│  │  • socket/connect/accept                     │    │
│  │  • execve/fork/clone                         │    │
│  └─────────────────────────────────────────────┘    │
│                                                     │
│  ┌─────────────────────────────────────────────┐    │
│  │  设备驱动 (Device Drivers)                    │    │
│  │  • 网卡驱动                                  │    │
│  │  • USB 驱动                                  │    │
│  │  • GPU 驱动                                  │    │
│  │  • 文件系统驱动                               │    │
│  └─────────────────────────────────────────────┘    │
│                                                     │
│  ┌─────────────────────────────────────────────┐    │

│  ┌─────────────────────────────────────────────┐    │
│  │  进程间通信 (IPC)                             │    │
│  │  • 共享内存                                  │    │
│  │  • 管道/消息队列                              │    │

│  └─────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────┘
```

**减少攻击面的策略**：

1. **最小化接口** — 只暴露必要的系统调用和 API
2. **输入验证** — 对所有外部输入进行严格检查
3. **代码审计** — 定期审查关键路径的代码
4. **权限限制** — 以最低权限运行服务

### 31.1.4 最小权限原则

**最小权限原则（Principle of Least Privilege, POLP）** 是安全设计的核心原则：每个程序、每个用户、每个进程只应拥有完成其任务所必需的最小权限。

```
最小权限原则示例：

场景：Web 服务器处理用户请求

❌ 违反最小权限原则：
┌─────────────────────────────────────┐
│  Web Server (以 root 运行)           │
│  • 可以读写所有文件                   │
│  • 可以修改系统配置                   │
│  • 可以安装软件                       │
│  • 可以创建用户                       │
│  如果被攻破 → 完全控制系统             │
└─────────────────────────────────────┘

✅ 遵循最小权限原则：
┌─────────────────────────────────────┐
│  Web Server (以 www-data 运行)       │
│  • 只能读取 /var/www/               │
│  • 只能写入 /var/log/               │
│  • 只能绑定 80/443 端口              │
│  • 无法访问其他用户文件               │
│  如果被攻破 → 损害有限               │
└─────────────────────────────────────┘
```

**最小权限的实现方式**：

1. **用户/组权限** — 使用专用的服务账户
2. **Capabilities** — 将 root 权限拆分为细粒度的能力
3. **seccomp** — 限制进程可用的系统调用
4. **SELinux/AppArmor** — 强制访问控制策略
5. **容器** — 使用 namespace 和 cgroup 限制资源

## 31.2 访问控制

访问控制是操作系统安全的基础，它决定了 **谁（主体）** 可以对 **什么（客体）** 执行 **什么操作（权限）**。

```
访问控制的基本模型：

┌──────────┐    请求    ┌──────────┐    检查    ┌──────────┐
│  主体    │ ────────→ │  引用    │ ────────→ │  客体    │
│ Subject  │           │  监控器  │           │ Object   │
│ (进程/   │           │ Reference│           │ (文件/   │
│  用户)   │           │ Monitor  │           │  设备)   │
└──────────┘           └────┬─────┘           └──────────┘
                            │
                     ┌──────┴──────┐
                     │  访问控制    │
                     │  数据库     │
                     │ (策略规则)  │
                     └─────────────┘
```

### 31.2.1 DAC：自主访问控制

**DAC（Discretionary Access Control，自主访问控制）** 是最传统的访问控制模型。资源的所有者可以自主决定谁可以访问该资源。

#### Unix 文件权限

Unix/Linux 使用经典的 DAC 模型，通过 **用户（owner）、组（group）、其他（others）** 三类主体和 **读（r）、写（w）、执行（x）** 三种权限来控制访问。

```
Unix 文件权限结构：

          owner    group    others
          ┌─┬─┬─┐ ┌─┬─┬─┐ ┌─┬─┬─┐
          │r│w│x│ │r│w│x│ │r│w│x│
          └─┴─┴─┘ └─┴─┴─┘ └─┴─┴─┘
           1 1 1   1 0 1   1 0 0
           = 7     = 5     = 4
           
权限表示：754 (八进制)
- owner: rwx (7) — 读、写、执行
- group: r-x (5) — 读、执行
- others: r-- (4) — 只读
```

**权限检查算法**：

```c
// 简化的 DAC 权限检查逻辑
int check_access(struct inode *file, struct task_struct *process, int requested) {
    // 1. 如果是 root 用户，允许所有操作
    if (process->uid == 0)
        return ALLOW;
    
    // 2. 如果是文件所有者
    if (process->uid == file->i_uid) {
        if ((file->i_mode & S_IRWXU) & requested)
            return ALLOW;
    }
    // 3. 如果是文件所属组成员
    else if (in_group(file->i_gid)) {
        if ((file->i_mode & S_IRWXG) & requested)
            return ALLOW;
    }
    // 4. 其他用户
    else {
        if ((file->i_mode & S_IRWXO) & requested)
            return ALLOW;
    }
    
    return DENY;
}
```

**DAC 的优缺点**：

| 优点 | 缺点 |
|------|------|
| 灵活，用户可以自主管理权限 | 权限传播难以控制 |
| 实现简单，开销小 | 容易受到提权攻击 |
| 符合直觉，易于理解 | 无法强制执行组织级安全策略 |

### 31.2.2 MAC：强制访问控制

**MAC（Mandatory Access Control，强制访问控制）** 由系统管理员定义全局安全策略，用户无法自行修改。每个主体和客体都被赋予一个安全标签，系统根据标签之间的关系决定是否允许访问。

#### Bell-LaPadula 模型（机密性）

```
Bell-LaPadula 安全级别：

        最高 ───── 高 (Top Secret)
        │
        │   "不上读" (No Read Up)
        │   → 低级别主体不能读高级别客体
        │
        │   "不下写" (No Write Down)
        │   → 高级别主体不能写低级别客体
        │
        中 ───── (Secret)
        │
        低 ───── (Confidential)
        │
        最低 ───── (Unclassified)

示例：
- 普通员工 (Confidential) 不能读取机密文件 (Secret)
- 高级官员 (Secret) 不能将机密信息写入普通文件 (Confidential)
```

#### Biba 模型（完整性）

```
Biba 完整性级别：

        高 ───── (Very Trusted)
        │
        │   "不上写" (No Write Up)
        │   → 低完整性主体不能修改高完整性客体
        │
        │   "不下读" (No Read Down)
        │   → 高完整性主体不能读取低完整性数据
        │
        中 ───── (Trusted)
        │
        低 ───── (Untrusted)
```

#### SELinux

**SELinux（Security-Enhanced Linux）** 是 Linux 内核中实现的 MAC 系统，由 NSA 开发。

```
SELinux 架构：

┌─────────────────────────────────────────────────────┐
│                    用户空间                           │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐          │
│  │ 进程 A   │  │ 进程 B   │  │ 进程 C   │          │
│  │ context: │  │ context: │  │ context: │          │
│  │ httpd_t  │  │ sshd_t   │  │ user_t   │          │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘          │
│       │             │             │                 │
├───────┼─────────────┼─────────────┼─────────────────┤
│       │      Linux 内核           │                 │
│       │             │             │                 │
│  ┌────┴─────────────┴─────────────┴─────────────┐   │
│  │           SELinux 安全服务器                   │   │
│  │  ┌─────────────┐  ┌─────────────────────┐    │   │
│  │  │  AVC 缓存   │  │  安全策略数据库      │    │   │
│  │  │ (Access     │  │  • type_enforce     │    │   │
│  │  │  Vector     │  │  • role_based       │    │   │
│  │  │  Cache)     │  │  • MLS/MCS          │    │   │
│  │  └─────────────┘  └─────────────────────┘    │   │
│  └──────────────────────────────────────────────┘   │
│                                                     │
│  ┌──────────────────────────────────────────────┐   │
│  │              文件系统 (带安全标签)              │   │
│  │  /var/www/html  context: httpd_sys_content_t  │   │
│  │  /etc/shadow    context: shadow_t             │   │
│  └──────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘
```

**SELinux 策略示例**：

```
# 允许 httpd 进程读取网页内容
allow httpd_t httpd_sys_content_t:file { read getattr open };

# 拒绝 httpd 进程读取 shadow 文件（默认拒绝）
# 无需显式拒绝规则，未允许即拒绝
```

#### AppArmor

**AppArmor** 是另一种 Linux MAC 实现，使用 **路径名** 而非安全标签来标识资源，配置更简单。

```
AppArmor vs SELinux 对比：

┌──────────────┬──────────────────┬──────────────────┐
│   特性       │    AppArmor      │    SELinux       │
├──────────────┼──────────────────┼──────────────────┤
│ 标识方式     │ 路径名           │ 安全上下文标签    │
│ 配置复杂度   │ 简单             │ 复杂             │
│ 粒度         │ 程序级           │ 系统级           │
│ 默认发行版   │ Ubuntu, SUSE     │ RHEL, Fedora     │
│ 学习曲线     │ 平缓             │ 陡峭             │
│ 灵活性       │ 中等             │ 非常高           │
└──────────────┴──────────────────┴──────────────────┘
```

### 31.2.3 RBAC：基于角色的访问控制

**RBAC（Role-Based Access Control）** 将权限分配给 **角色（Role）**，再将角色分配给用户。用户通过角色间接获得权限，简化了权限管理。

```
RBAC 模型结构：

┌──────────┐    分配    ┌──────────┐    授权    ┌──────────┐
│  用户    │ ────────→ │  角色    │ ────────→ │  权限    │
│ Users    │           │ Roles    │           │Perms     │
└──────────┘           └──────────┘           └──────────┘
  Alice                  Admin                  读写所有文件
  Bob                    Developer              读写代码文件
  Charlie                Viewer                 只读文件
```

**RBAC 的层次结构**：

```
RBAC 层次模型：

        ┌─────────────────────────────────────┐
        │           RBAC0 (基础)               │
        │  用户 → 角色 → 权限                  │
        └──────────────┬──────────────────────┘
                       │
        ┌──────────────┴──────────────────────┐
        │           RBAC1 (层次)               │
        │  角色可以继承：Admin > Developer     │
        └──────────────┬──────────────────────┘
                       │
        ┌──────────────┴──────────────────────┐
        │           RBAC2 (约束)               │
        │  • 互斥角色：审计员 ≠ 开发者         │
        │  • 基数约束：最多 N 个用户有某角色    │
        │  • 先决条件：必须先有 A 角色才能有 B  │
        └──────────────┬──────────────────────┘
                       │
        ┌──────────────┴──────────────────────┐
        │           RBAC3 (综合)               │
        │  RBAC1 + RBAC2                       │
        └─────────────────────────────────────┘
```

**RBAC 在 Linux 中的实现**：Linux 本身没有原生 RBAC，但通过 SELinux 的角色（role）机制、PAM 模块或第三方工具（如 AppArmor）可以实现类似功能。

### 31.2.4 Capabilities：细粒度权限

传统的 Unix 权限模型将 root（uid=0）视为"超级用户"，拥有所有权限。**Linux Capabilities** 将 root 的权限拆分为多个独立的能力，实现细粒度的权限控制。

```
Linux Capabilities 机制：

传统模型：
┌─────────────────────────────────────┐
│  root (uid=0)                       │
│  拥有所有权限（全能）                │
└─────────────────────────────────────┘

Capabilities 模型：
┌─────────────────────────────────────┐
│  root 权限被拆分为 ~40 种能力        │
├─────────────────────────────────────┤
│  CAP_CHOWN       — 改变文件所有者    │
│  CAP_DAC_OVERRIDE — 覆盖 DAC 检查   │
│  CAP_KILL        — 发送信号          │
│  CAP_NET_BIND_SERVICE — 绑定低端口   │
│  CAP_NET_ADMIN   — 网络管理          │
│  CAP_SYS_ADMIN   — 系统管理          │
│  CAP_SYS_PTRACE  — 跟踪进程          │
│  CAP_SYS_MODULE  — 加载内核模块      │
│  CAP_SYS_RAWIO   — 原始 I/O         │
│  CAP_SETUID      — 设置 UID         │
│  CAP_SETGID      — 设置 GID         │
│  ...                                │
└─────────────────────────────────────┘

示例：只给 Web 服务器绑定低端口的能力
$ setcap cap_net_bind_service=+ep /usr/sbin/nginx
# nginx 不需要 root 就能绑定 80 端口
```

**Capabilities 的三个集合**：

| 集合 | 说明 |
|------|------|
| **Permitted (P)** | 进程可以使用的能力集合 |
| **Inheritable (I)** | 进程在 execve 后可以继承的能力 |
| **Effective (E)** | 当前生效的能力（内核实际检查的） |

**文件也有三个集合**：Permitted、Inheritable、Effective，用于在 execve 时与进程的集合进行运算。

## 31.3 隔离机制

隔离是操作系统安全的基石。通过不同层次的隔离机制，确保进程之间、用户之间、系统之间的资源不被非法访问。

### 31.3.1 进程隔离：地址空间

每个进程拥有独立的 **虚拟地址空间**，通过 **页表（Page Table）** 将虚拟地址映射到物理地址。一个进程无法直接访问另一个进程的物理内存。

```
进程地址空间隔离：

┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│   进程 A     │  │   进程 B     │  │   进程 C     │
│  虚拟地址空间 │  │  虚拟地址空间 │  │  虚拟地址空间 │
│              │  │              │  │              │
│  0xFFFF...   │  │  0xFFFF...   │  │  0xFFFF...   │
│  ┌────────┐  │  │  ┌────────┐  │  │  ┌────────┐  │
│  │ 内核   │  │  │  │ 内核   │  │  │  │ 内核   │  │
│  ├────────┤  │  │  ├────────┤  │  │  ├────────┤  │

  └─────────────────────────────────────────────┘
```

**隔离的关键机制**：

1. **页表隔离** — 每个进程有独立的页表（CR3 寄存器），虚拟地址到物理地址的映射互不干扰
2. **内核/用户态分离** — 高地址空间（如 0xFFFF800000000000 以上）映射到内核，用户态代码无法直接访问
3. **页权限位** — R/W（读写）、U/S（用户/超级用户）、NX（不可执行）位控制访问

### 31.3.2 用户隔离：UID/GID

Unix/Linux 使用 **UID（用户标识符）** 和 **GID（组标识符）** 来隔离不同用户的资源。

```
用户隔离机制：

┌─────────────────────────────────────────────────────┐
│                用户隔离层次                           │
├─────────────────────────────────────────────────────┤
│                                                     │

│  ┌─────────────────────────────────────────────┐    │

│  文件层：                                           │
│  ┌─────────────────────────────────────────────┐    │
│  │  每个文件有：                                │    │

└─────────────────────────────────────────────────────┘
```

**setuid 机制详解**：

```c
// setuid 程序示例

// 设置: chmod u+s myapp

int main() {
    uid_t real_uid = getuid();      // 真实用户 ID
    uid_t effective_uid = geteuid(); // 有效用户 ID (可能是 root)
```

### 31.3.4 虚拟机隔离：硬件级别

虚拟机通过 **Hypervisor** 实现最强的隔离级别，每个 VM 拥有独立的操作系统内核和虚拟硬件。

```
隔离强度对比：

最弱 ──────────────────────────────────────────── 最强

进程隔离    容器隔离        虚拟机隔离
(页表)     (NS + cgroup)   (Hypervisor)
  │           │               │
  │  同一内核  │  共享内核     │  独立内核
  │  线程切换  │  系统调用     │  VM Exit
  │  微秒级   │  毫秒级       │  毫秒级
  │           │               │
  ▼           ▼               ▼
共享内核    部分隔离         完全隔离
```

**虚拟机隔离的安全优势**：

1. **内核隔离** — 每个 VM 有独立的内核，内核漏洞只影响单个 VM
2. **硬件隔离** — 通过 EPT/NPT 实现独立的物理地址映射
3. **I/O 隔离** — 虚拟设备或设备直通实现独立的 I/O 通道
4. **快照恢复** — 可以快速回滚到已知安全状态

## 31.4 常见漏洞与防护

### 31.4.1 缓冲区溢出

**缓冲区溢出（Buffer Overflow）** 是最经典的安全漏洞之一。当程序向缓冲区写入的数据超过其容量时，溢出的数据会覆盖相邻的内存区域。

#### 栈溢出（Stack Buffer Overflow）

```
栈溢出攻击原理：

正常函数调用时的栈布局（高地址 → 低地址）：

高地址
┌─────────────────────────┐
│    调用者的栈帧           │
├─────────────────────────┤
│    返回地址 (RET)         │ ← 函数返回时跳转到此处
├─────────────────────────┤
│    保存的 EBP/RBP        │ ← 栈帧指针
├─────────────────────────┤
│    buffer[60-63]         │
│    buffer[56-59]         │
│    ...                   │ ← char buffer[64]
│    buffer[4-7]           │
│    buffer[0-3]           │
└─────────────────────────┘
低地址

攻击后（输入超过 64 字节）：

高地址
┌─────────────────────────┐
│    调用者的栈帧           │
├─────────────────────────┤
│  攻击者控制的地址 0x90... │ ← 返回地址被覆盖！
├─────────────────────────┤
│    shellcode (NOP sled)  │ ← 攻击者注入的代码
│    /bin/sh 的机器码       │
├─────────────────────────┤
│    ... (填充数据)         │ ← 64 字节 + padding
│    buffer[0-3]           │
└─────────────────────────┘
低地址
```

**经典栈溢出攻击代码**：

```c
// 漏洞程序
#include <stdio.h>
#include <string.h>

void vulnerable() {
    char buffer[64];
    gets(buffer);  // 危险！没有长度限制
}

void secret_function() {
    system("/bin/sh");  // 攻击者想要执行的代码
}

int main() {
    vulnerable();
    return 0;
}

// 攻击脚本 (Python)
// payload = b"A" * 72                    # 填充 buffer + saved RBP
// payload += p64(secret_function_addr)    # 覆盖返回地址
// process.send(payload)
```

#### 堆溢出（Heap Overflow）

```
堆溢出攻击原理：

堆内存布局：

低地址
┌─────────────────────────────┐
│  chunk A (用户数据)          │
│  ┌───────────────────────┐  │
│  │  prev_size | size     │  │ ← chunk 头部
│  │  user data ...        │  │
│  └───────────────────────┘  │
├─────────────────────────────┤
│  chunk B (用户数据)          │
│  ┌───────────────────────┐  │
│  │  prev_size | size     │  │ ← 攻击目标
│  │  user data ...        │  │
│  └───────────────────────┘  │
├─────────────────────────────┤
│  空闲空间                    │
└─────────────────────────────┘
高地址

堆溢出：向 chunk A 写入过多数据 → 覆盖 chunk B 的头部
→ 篡改 malloc/free 的元数据
→ 实现任意地址写入
```

### 31.4.2 缓冲区溢出防护机制

#### 1. ASLR（地址空间布局随机化）

```
ASLR 防护原理：

没有 ASLR：
┌─────────────────────────┐
│  栈: 0x7fff12340000     │ ← 每次相同
│  堆: 0x00601000         │ ← 每次相同
│  库: 0x7f1234500000     │ ← 每次相同
│  代码: 0x400000         │ ← 每次相同
└─────────────────────────┘
攻击者可以预先计算地址

有 ASLR：
┌─────────────────────────┐
│  栈: 0x7fff????0000     │ ← 随机偏移
│  堆: 0x0060????         │ ← 随机偏移
│  库: 0x7f????????0000   │ ← 随机偏移
│  代码: 0x????0000       │ ← 随机偏移 (PIE)
└─────────────────────────┘
攻击者无法预测地址
```

#### 2. Stack Canary（栈保护）

```
Stack Canary 机制：

函数入口：
push rbp
mov rbp, rsp
sub rsp, 64
mov [rbp-8], <canary_value>  ← 放置 canary

函数出口：
mov rax, [rbp-8]              ← 读取 canary
xor rax, fs:0x28              ← 与 TLS 中的原始值比较
jz .ok
call __stack_chk_fail         ← 不匹配 → 检测到溢出！
.ok:
leave
ret

栈布局：
高地址
┌─────────────────────────┐
│  返回地址                 │
├─────────────────────────┤
│  保存的 RBP              │
├─────────────────────────┤
│  Canary (0x????????)     │ ← 保护返回地址
├─────────────────────────┤
│  buffer[64]              │ ← 攻击者溢出这里
└─────────────────────────┘
低地址
```

#### 3. DEP/NX（数据执行保护）

```
DEP/NX 防护原理：

没有 DEP：
┌─────────────────────────┐
│  栈 (RWX)               │ ← 可读、可写、可执行
│  攻击者可以：            │
│  1. 注入 shellcode 到栈  │
│  2. 覆盖返回地址         │
│  3. 跳转执行 shellcode   │
└─────────────────────────┘

有 DEP/NX：
┌─────────────────────────┐
│  栈 (RW-)               │ ← 可读、可写、不可执行
│  堆 (RW-)               │ ← 可读、可写、不可执行
│  代码 (R-X)             │ ← 可读、不可写、可执行
│                        │
│  跳转到栈执行 → 段错误！  │
└─────────────────────────┘

应对 DEP 的攻击技术：
• ROP (Return-Oriented Programming)
  — 复用已有的代码片段 (gadgets)
  — 不需要注入新代码
```

### 31.4.3 竞态条件（Race Condition）

**竞态条件** 发生在多个进程/线程访问共享资源时，结果取决于执行的时序。

#### TOCTTOU（Time of Check to Time of Use）

```
TOCTTOU 攻击：

场景：程序检查用户是否有权访问文件，然后打开文件

时间线：
    用户进程                    攻击者进程
        │                           │
    ①  │ access("file", W_OK)      │   ← 检查权限
        │                           │
        │                      ②   │ symlink("file", "/etc/passwd")
        │                           │   ← 切换文件指向
    ③  │ open("file", O_WRONLY)    │   ← 使用时已改变！
        │   → 实际打开 /etc/passwd   │
        │                           │

漏洞代码：
if (access(filename, W_OK) == 0) {    // ① 检查
    // 时间窗口！攻击者可以在这里修改文件
    fd = open(filename, O_WRONLY);     // ③ 使用
    write(fd, data, len);
}

安全修复：
fd = open(filename, O_WRONLY | O_NOFOLLOW);  // 原子操作
if (faccessat(fd, filename, W_OK, AT_EACCESS) == 0) {
    write(fd, data, len);
}
```

### 31.4.4 特权提升

**特权提升（Privilege Escalation）** 攻击使低权限用户获得高权限。

```
特权提升类型：

垂直提权：
┌─────────────────────────────────────────┐
│  普通用户 (uid=1000)                    │
│     │                                   │
│     │ 利用漏洞                          │
│     ▼                                   │
│  root (uid=0)                           │
└─────────────────────────────────────────┘

水平提权：
┌─────────────────────────────────────────┐
│  用户 A (uid=1000)                      │
│     │                                   │
│     │ 冒充身份                          │
│     ▼                                   │
│  用户 B (uid=1001)                      │
└─────────────────────────────────────────┘

常见提权方式：
1. SUID 程序漏洞
2. 内核漏洞 (Dirty COW, Dirty Pipe)
3. sudo 配置错误
4. 容器逃逸
```

### 31.4.5 侧信道攻击

**侧信道攻击（Side-Channel Attack）** 通过观察系统的物理特征（时间、缓存状态、功耗）推断敏感信息。

#### Spectre 攻击

```
Spectre 攻击原理（利用推测执行）：

正常执行：
if (x < array1_size) {        // 条件检查
    y = array2[array1[x]];     // 内存访问
}

攻击场景（x 是越界值）：

1. 攻击者训练分支预测器，使其预测 x < size 为真
2. CPU 推测执行，使用越界 x 访问 array1[x]
3. 推测执行加载 array1[x] 的值（包含秘密数据）
4. 用这个值索引 array2，造成缓存加载
5. 虽然最终分支被回滚，但缓存状态已改变
6. 通过测量 array2 各元素的访问时间，推断秘密值

时间线：
├── 分支预测 ──→ 推测执行 ──→ 缓存加载 ──→ 分支纠正
│                                    │
│   回滚寄存器状态 ✓                  │
│   回滚缓存状态 ✗ ← 信息泄露！      │
```

#### Meltdown 攻击

```
Meltdown 攻击原理：

1. 用户态代码读取内核地址
2. CPU 检测到权限错误，但推测执行继续
3. 推测执行期间，秘密数据被加载到缓存
4. 异常处理回滚执行，但缓存状态保留
5. 通过缓存时序分析恢复秘密数据

┌─────────────────────────────────────────┐
│  用户态进程                              │
│  mov rax, [kernel_address]  // 读内核    │
│  mov rbx, [array + rax*4096] // 缓存加载 │
│                                         │
│  → 推测执行：秘密值 rax 影响缓存         │
│  → 异常：权限错误                        │
│  → 通过时序测量恢复 rax 的值             │
└─────────────────────────────────────────┘

防护：KPTI (Kernel Page Table Isolation)
— 内核页表在用户态不可见
— 推测执行无法访问内核数据
```

## 31.5 安全增强技术

### 31.5.1 ASLR 详解

**ASLR（Address Space Layout Randomization）** 随机化进程的内存布局，使攻击者无法预测代码、数据、栈、堆的地址。

```
ASLR 随机化的区域：

┌──────────────────────────────────────┐
│  每次程序启动时随机化的地址：         │
│                                      │
│  1. 栈地址     — 随机偏移 ~28 位     │
│  2. 堆地址     — 随机偏移 ~28 位     │
│  3. mmap 区域  — 共享库加载地址      │
│  4. 可执行基址 — PIE 启用时          │
│                                      │
│  随机化熵：                           │
│  • 32 位系统：栈 ~8 bits, 库 ~8 bits │
│  • 64 位系统：栈 ~28 bits, 库 ~28 bits│
└──────────────────────────────────────┘

查看 ASLR 状态：
$ cat /proc/sys/kernel/randomize_va_space
2 — 完全随机化（默认）
1 — 部分随机化
0 — 关闭

关闭 ASLR（调试用）：
$ echo 0 | sudo tee /proc/sys/kernel/randomize_va_space
```

### 31.5.2 KASLR

**KASLR（Kernel Address Space Layout Randomization）** 将 ASLR 应用到内核本身，随机化内核代码和数据的加载地址。

```
KASLR 原理：

没有 KASLR：
┌──────────────────────────────────────┐
│  内核代码基址固定: 0xffffffff81000000│
│  攻击者知道内核符号地址              │
│  可以直接构造 ROP chain              │
└──────────────────────────────────────┘

有 KASLR：
┌──────────────────────────────────────┐
│  内核代码基址随机: 0xffffffff????0000│
│  攻击者不知道内核符号地址            │
│  需要先泄露内核地址才能构造攻击      │
└──────────────────────────────────────┘

绕过 KASLR 的技术：
• 内核信息泄露漏洞
• 时序侧信道
• /proc/kallsyms 泄露（未禁用时）
```

### 31.5.3 Secure Boot

**Secure Boot** 是 UEFI 固件的安全机制，确保只有经过签名验证的代码才能在启动过程中执行。

```
Secure Boot 启动链：

┌─────────────────────────────────────────────────────┐
│                Secure Boot 验证链                    │
├─────────────────────────────────────────────────────┤
│                                                     │
│  ① UEFI 固件                                        │
│  │  内置平台密钥 (PK) 和密钥交换密钥 (KEK)           │
│  │                                                  │
│  ▼                                                  │
│  ② Bootloader (GRUB/shim)                           │
│  │  使用 KEK 签名验证                               │
│  │  签名有效 → 继续启动                             │
│  │  签名无效 → 拒绝启动                             │
│  │                                                  │
│  ▼                                                  │
│  ③ OS Kernel                                        │
│  │  使用发行版密钥签名验证                           │
│  │  验证通过 → 加载内核                             │
│  │                                                  │
│  ▼                                                  │
│  ④ 内核模块                                         │
│     加载时验证模块签名                               │
└─────────────────────────────────────────────────────┘

信任链：
PK → KEK → OS Vendor Key → Bootloader → Kernel → Modules

防护目标：
• 防止 Bootkit (启动时恶意软件)
• 防止 Rootkit (内核级恶意软件)
• 确保系统完整性
```

### 31.5.4 TEE（可信执行环境）

**TEE（Trusted Execution Environment）** 在主处理器上创建一个隔离的安全区域，保护代码和数据的机密性与完整性。

```
TEE 架构：

┌─────────────────────────────────────────────────────┐
│                    主处理器                          │
│                                                     │
│  ┌──────────────────────┐  ┌─────────────────────┐ │
│  │    REE (普通世界)     │  │    TEE (安全世界)    │ │
│  │  Rich OS (Linux)     │  │  Trusted OS         │ │
│  │  ┌────────────────┐  │  │  ┌───────────────┐  │ │
│  │  │  普通应用       │  │  │  │  Trusted App  │  │ │
│  │  │  (Client App)  │  │  │  │  (TA)         │  │ │
│  │  └────────┬───────┘  │  │  └───────┬───────┘  │ │
│  │           │          │  │          │          │ │
│  │  ┌────────┴───────┐  │  │  ┌───────┴───────┐  │ │
│  │  │   TEE Client   │──┼──┼──│  TEE Internal │  │ │
│  │  │   API          │  │  │  │  API          │  │ │
│  │  └────────────────┘  │  │  └───────────────┘  │ │
│  └──────────────────────┘  └─────────────────────┘ │
│                                                     │
│  硬件隔离：内存加密 + 总线保护                       │
└─────────────────────────────────────────────────────┘

TEE 应用场景：
• 指纹/面部识别数据处理
• 数字版权管理 (DRM)
• 移动支付安全
• 区块链密钥管理
```

## 31.6 面试高频考点

### 考点 1：DAC vs MAC vs RBAC

**Q：解释 DAC、MAC、RBAC 的区别和适用场景。**

**A：**
- **DAC**：资源所有者自主决定权限（如 Unix 文件权限），灵活但安全性低
- **MAC**：系统强制执行全局策略（如 SELinux），安全性高但配置复杂
- **RBAC**：通过角色管理权限（如企业系统），适合组织管理
- 选择取决于安全需求：个人系统用 DAC，军事/政府用 MAC，企业用 RBAC

### 考点 2：缓冲区溢出防护

**Q：列举至少 4 种缓冲区溢出防护机制。**

**A：**
1. **ASLR** — 随机化内存布局，增加攻击难度
2. **Stack Canary** — 在返回地址前放置哨兵值，检测溢出
3. **DEP/NX** — 标记数据区域不可执行，阻止 shellcode 执行
4. **PIE** — 可执行文件地址随机化
5. **编译器保护** — `-fstack-protector`、`-D_FORTIFY_SOURCE=2`
6. **Safe language** — 使用 Rust、Go 等内存安全语言

### 考点 3：容器 vs 虚拟机

**Q：容器和虚拟机在安全隔离上有什么区别？**

**A：**
- **容器**：共享内核，通过 namespace + cgroup 隔离，轻量但攻击面大（内核漏洞可逃逸）
- **虚拟机**：独立内核，通过 hypervisor 隔离，安全但开销大
- 容器适合可信环境，虚拟机适合多租户/不可信代码

### 考点 4：Spectre/Meltdown

**Q：解释 Spectre 和 Meltdown 攻击的原理及防护。**

**A：**
- **Spectre**：利用分支预测器的推测执行，诱导 CPU 访问越界内存并留下缓存痕迹
- **Meltdown**：利用推测执行在异常处理前访问内核内存，通过缓存侧信道泄露数据
- **防护**：KPTI（内核页表隔离）、retpoline（间接分支限制）、微码更新、缓存刷新

### 考点 5：TOCTTOU 竞态条件

**Q：什么是 TOCTTOU 漏洞？如何防护？**

**A：**
- **TOCTTOU**：检查（Check）和使用（Use）之间存在时间窗口，攻击者可在此期间修改资源状态
- **防护**：使用原子操作（`openat` + `faccessat`）、避免路径名操作、使用文件描述符而非路径名

### 考点 6：Linux Capabilities

**Q：Linux Capabilities 如何实现细粒度权限控制？**

**A：**
- 将 root 的全能权限拆分为 ~40 种独立能力（如 `CAP_NET_BIND_SERVICE`、`CAP_SYS_PTRACE`）
- 进程只需获得必要的能力，而非完整的 root 权限
- 使用 `setcap` 命令为可执行文件设置能力
- 三个集合：Permitted（可使用）、Inheritable（可继承）、Effective（生效）
