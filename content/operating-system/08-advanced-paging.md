## 8.1 多级页表

### 8.1.1 线性页表的空间问题

在上一章中，我们看到 48 位虚拟地址空间、4KB 页的线性页表需要 512GB 空间。即使对于 32 位系统，4MB 的页表在进程数量很多时也是巨大的开销——1000 个进程需要 4GB 仅用于存储页表。

问题的根源在于：线性页表必须为**整个虚拟地址空间**分配页表项，即使大部分虚拟地址从未被使用。一个典型的进程只使用了虚拟地址空间的一小部分——代码段在低地址，栈在高地址，中间大部分是未映射的空洞。

**多级页表（Multi-Level Page Table）** 的核心思想是：只为**使用的虚拟地址区域**分配页表空间，未使用的区域不分配。

### 8.1.2 二级页表结构

二级页表将线性页表拆分为两层：**页目录（Page Directory）** 和**页表（Page Table）**。

```
虚拟地址（32 位，二级页表）：
┌─────────────┬─────────────┬────────────┐
│ PD Index(10)│ PT Index(10)│ Offset(12) │
│ 页目录索引  │ 页表索引    │ 页内偏移   │
└─────────────┴─────────────┴────────────┘
```

**地址翻译过程**：

```
步骤 1：用 PD Index 索引页目录
  PDE = 页目录[PD Index]
  如果 PDE 无效 → 页错误
  如果 PDE 有效 → 获取页表的物理地址

步骤 2：用 PT Index 索引页表
  PTE = 页表[PT Index]
  如果 PTE 无效 → 页错误
  如果 PTE 有效 → 获取 PFN

步骤 3：组合物理地址
  物理地址 = (PFN << 12) | Offset
```

```
页目录（PD）                    页表（PT）                    物理内存
┌──────────┐                   ┌──────────┐
│ PDE[0]   │ ──→ [页表0]      │ PTE[0]   │ ──→ PFN
│ PDE[1]   │ ──→ [页表1]      │ PTE[1]   │ ──→ PFN
│ PDE[2]   │ 无效（未使用）    │ ...      │
│ PDE[3]   │ ──→ [页表3]      │ PTE[1023]│ ──→ PFN
│ ...      │                   └──────────┘
│ PDE[1023]│
└──────────┘
```

**空间节省**：如果一个进程只使用了 4 个页目录项（每个指向一个 4KB 页表），那么页表总空间 = 4KB（页目录）+ 4 × 4KB（4 个页表）= 20KB。而线性页表需要 4MB。节省了 99.5%！

### 8.1.3 四级页表：x86-64

x86-64 使用 48 位虚拟地址、4KB 页的**四级页表**：

```
虚拟地址（48 位，四级页表）：
┌─────────┬─────────┬─────────┬─────────┬────────────┐
│ PML4(9) │ PDPT(9) │ PD(9)   │ PT(9)   │ Offset(12) │
│ 4级索引  │ 3级索引 │ 2级索引 │ 1级索引 │ 页内偏移   │
└─────────┴─────────┴─────────┴─────────┴────────────┘
  位 47-39  位 38-30  位 29-21  位 20-12  位 11-0
```

**翻译过程**：

```
CR3 → PML4 表
  PML4[PML4索引] → PDPT 表地址
    PDPT[PDPT索引] → PD 表地址
      PD[PD索引] → PT 表地址
        PT[PT索引] → PFN
          物理地址 = (PFN << 12) | Offset
```

每次地址翻译需要 4 次内存访问（查 4 级页表）。这就是 TLB 如此重要的原因——没有 TLB，每次内存访问都需要 4 次额外的内存访问。

### 8.1.4 多级页表的时间-空间权衡

多级页表用**更多的时间**换取**更少的空间**：

| 维度 | 线性页表 | 二级页表 | 四级页表 |
|------|---------|---------|---------|
| 空间（典型进程） | 4MB | 20KB | 4KB-16KB |
| 翻译时间（无 TLB） | 1 次内存访问 | 2 次 | 4 次 |
| TLB 命中时 | 1ns | 1ns | 1ns |
| TLB 未命中时 | ~100ns | ~200ns | ~400ns |

关键洞察：由于 TLB 命中率极高（99%+），多级页表的额外翻译时间几乎不影响性能。但空间节省是巨大的。

### 8.1.5 手算示例：x86-64 四级页表翻译

**虚拟地址**：0x00007F3A_1234_5678

```
步骤 1：提取各级索引
  PML4 索引 = (0x00007F3A12345678 >> 39) & 0x1FF = 0x0FE
  PDPT 索引 = (0x00007F3A12345678 >> 30) & 0x1FF = 0x1C8
  PD  索引  = (0x00007F3A12345678 >> 21) & 0x1FF = 0x091
  PT  索引  = (0x00007F3A12345678 >> 12) & 0x1FF = 0x1AB
  Offset    = 0x00007F3A12345678 & 0xFFF        = 0x678

步骤 2：查 PML4 表（CR3 指向）
  PML4[0x0FE] = {PFN=0x1000, V=1}
  → PDPT 物理地址 = 0x1000000

步骤 3：查 PDPT 表
  PDPT[0x1C8] = {PFN=0x2000, V=1}
  → PD 物理地址 = 0x2000000

步骤 4：查 PD 表
  PD[0x091] = {PFN=0x3000, V=1}
  → PT 物理地址 = 0x3000000

步骤 5：查 PT 表
  PT[0x1AB] = {PFN=0x4567, V=1}
  → 物理页框 = 0x4567000

步骤 6：组合物理地址
  物理地址 = 0x4567000 | 0x678 = 0x4567678
```

### 8.1.6 大页支持

x86-64 支持通过跳过页表的最后几级来创建大页：

**2MB 大页**：PD 条目设置 PS（Page Size）位，跳过 PT 级。
```
PML4 → PDPT → PD（PS=1）→ 2MB 页
PFN 从 PD 条目获取，Offset 为 21 位
```

**1GB 大页**：PDPT 条目设置 PS 位，跳过 PD 和 PT 级。
```
PML4 → PDPT（PS=1）→ 1GB 页
PFN 从 PDPT 条目获取，Offset 为 30 位
```

大页减少了页表级数，降低了 TLB 未命中的代价。

## 8.2 反向页表（Inverted Page Table）

### 8.2.1 设计动机

在 64 位系统中，虚拟地址空间非常大（2^64 字节），即使使用多级页表，页表的开销仍然可观。反向页表通过一个不同的思路来减少页表空间：**按物理页框号索引，而不是按虚拟页号索引**。

```
传统页表：索引 = 虚拟页号，值 = 物理页框号
  页表大小 = 虚拟页数 × PTE 大小

反向页表：索引 = 物理页框号，值 = 虚拟页号
  页表大小 = 物理页数 × PTE 大小
```

反向页表的大小只与**物理内存大小**有关，与虚拟地址空间大小无关。对于 64GB 物理内存、4KB 页，反向页表只有 64GB/4KB = 16M 个条目。

### 8.2.2 查找过程

反向页表的查找需要搜索整个表（或使用哈希表加速）：

```
给定虚拟地址 (VPN, Offset)：
1. 在反向页表中搜索 VPN
2. 如果找到第 i 个条目的 VPN 匹配 → PFN = i
3. 物理地址 = (i << 12) | Offset

问题：线性搜索 O(N) 太慢
解决：使用哈希表加速，O(1) 平均查找
```

### 8.2.3 优缺点

| 优点 | 缺点 |
|------|------|
| 空间与虚拟地址大小无关 | 查找需要哈希表 |
| 适合 64 位系统 | 共享页面实现困难 |
| 物理内存小时页表小 | TLB 未命中代价更高 |

反向页表在 IBM PowerPC、UltraSPARC 等处理器中使用。x86-64 和 RISC-V 使用多级页表。

## 8.3 ARM 分页机制

### 8.3.1 ARMv8 地址翻译

ARMv8（AArch64）支持 48 位或 52 位虚拟地址，使用四级页表（4KB 页）或三级页表（64KB 页）。

ARMv8 的独特之处在于**双页表基址寄存器**：
- **TTBR0_EL1**：用户空间页表基址（虚拟地址 0x0000_0000_0000_0000 到 0x0000_FFFF_FFFF_FFFF）
- **TTBR1_EL1**：内核空间页表基址（虚拟地址 0xFFFF_0000_0000_0000 到 0xFFFF_FFFF_FFFF_FFFF）

```
ARMv8 虚拟地址空间：
0x0000_0000_0000_0000 ┌──────────────┐
                      │ 用户空间     │ ← TTBR0
                      │ ...          │
0x0000_FFFF_FFFF_FFFF ├──────────────┤
                      │ 未映射       │
0xFFFF_0000_0000_0000 ├──────────────┤
                      │ 内核空间     │ ← TTBR1
                      │ ...          │
0xFFFF_FFFF_FFFF_FFFF └──────────────┘
```

这种设计的好处是：用户空间和内核空间使用独立的页表，切换进程时只需更新 TTBR0（用户页表），TTBR1（内核页表）保持不变。这减少了上下文切换的开销。

## 8.4 面试高频考点

**Q1：为什么需要多级页表？**

线性页表必须为整个虚拟地址空间分配页表项，48 位地址空间需要 512GB 页表。多级页表只为使用的区域分配页表空间，典型进程只需几 KB。代价是地址翻译需要更多次内存访问（4 次 vs 1 次），但 TLB 命中率 99%+ 使这个代价几乎不可见。

**Q2：x86-64 的四级页表是哪四级？**

PML4（Page Map Level 4）→ PDPT（Page Directory Pointer Table）→ PD（Page Directory）→ PT（Page Table）。每级 9 位索引，512 个条目。CR3 寄存器指向 PML4 表的物理地址。

**Q3：大页（Huge Pages）的好处是什么？**

大页（2MB/1GB）减少了页表级数，降低了 TLB 未命中的代价。一个 2MB 大页 TLB 条目覆盖 2MB 内存（vs 4KB），TLB 覆盖范围增大 512 倍。适合大内存工作负载（数据库、科学计算）。

**Q4：反向页表和多级页表的区别？**

多级页表按虚拟页号索引，大小与虚拟地址空间相关。反向页表按物理页框号索引，大小与物理内存相关。反向页表在 64 位系统中更节省空间，但查找需要哈希表，共享页面实现困难。
