## 1.1 冯·诺依曼架构回顾

操作系统的设计深深植根于底层硬件架构。理解硬件特性是理解操作系统机制的前提。

### 1.1.1 CPU、内存、I/O 设备的基本结构

#### **冯·诺依曼架构的核心思想**

1945 年，约翰·冯·诺依曼提出了**存储程序**（Stored Program）的概念，奠定了现代计算机的基础。

**核心原理**：
> 程序（指令）和数据以相同形式存储在内存中，CPU 按顺序从内存读取指令并执行。

**与之前架构的对比**：

```
早期计算机（如 ENIAC）：
- 程序通过插线板"硬编码"
- 修改程序 = 重新布线（数小时）
- 程序与数据分离

冯·诺依曼架构：
- 程序存储在内存中
- 修改程序 = 修改内存内容（瞬间）
- 程序与数据统一（都是二进制）
```

#### **1. 中央处理器（CPU / Central Processing Unit）**

CPU 是计算机的"大脑"，负责执行指令。

**核心组件**：

**控制单元（Control Unit, CU）**：
- **功能**：解析指令，生成控制信号
- **工作流程**：
  1. 从内存读取指令（通过 PC 寄存器）
  2. 解码指令（确定操作类型、操作数位置）
  3. 生成控制信号（激活 ALU、内存、寄存器等）

**算术逻辑单元（Arithmetic Logic Unit, ALU）**：
- **功能**：执行算术和逻辑运算
- **支持的操作**：
  - 算术：加、减、乘、除
  - 逻辑：AND、OR、NOT、XOR
  - 比较：等于、大于、小于
  - 移位：左移、右移

**寄存器组（Registers）**：
- **功能**：CPU 内部的高速存储
- **典型寄存器**：

| 寄存器 | 功能 | 示例（x86-64） |
|-------|------|---------------|
| **通用寄存器** | 存储临时数据 | RAX, RBX, RCX, RDX |
| **程序计数器（PC）** | 指向下一条指令地址 | RIP (Instruction Pointer) |
| **栈指针（SP）** | 指向栈顶 | RSP (Stack Pointer) |
| **基址指针（BP）** | 指向栈帧基址 | RBP (Base Pointer) |
| **状态寄存器（FLAGS）** | 保存运算结果标志 | RFLAGS (Zero, Carry, Overflow...) |

**性能参数**：
- **时钟频率**：现代 CPU 通常 2-5 GHz（每秒 20-50 亿个时钟周期）
- **核心数**：从单核到 64+ 核（服务器 CPU）
- **缓存**：
  - L1 Cache：32-64 KB/核心，1-2 个时钟周期
  - L2 Cache：256-512 KB/核心，3-10 个时钟周期
  - L3 Cache：8-64 MB（共享），20-50 个时钟周期

#### **2. 内存（Memory）**

内存是程序和数据的临时存储场所。

**类型**：

**DRAM（Dynamic RAM）**：
- **特点**：易失性（断电丢失）、需要定期刷新
- **容量**：现代计算机通常 8-128 GB
- **访问延迟**：约 50-100 ns（纳秒）
- **带宽**：DDR4-3200 约 25 GB/s

**ROM（Read-Only Memory）**：
- **特点**：非易失性、只读（或难以写入）
- **用途**：BIOS/UEFI 固件、嵌入式系统程序

**内存层次结构（Memory Hierarchy）**：

```
速度   容量    成本
 ↑      ↓      ↑
 
寄存器    < 1 KB      ~0.3 ns     极高
L1 Cache  64 KB       ~1 ns       高
L2 Cache  512 KB      ~5 ns       中高
L3 Cache  16 MB       ~20 ns      中
DRAM      16 GB       ~100 ns     低
SSD       512 GB      ~100 μs     很低
HDD       2 TB        ~10 ms      极低
```

**设计原则**：
- **局部性原理（Locality）**：
  - **时间局部性**：最近访问的数据很可能再次被访问（循环变量）
  - **空间局部性**：访问某地址后，其附近地址也可能被访问（数组遍历）

```c
// 示例：利用局部性原理
int sum_array(int arr[], int n) {
    int sum = 0;  // sum 变量：时间局部性（反复访问）
    for (int i = 0; i < n; i++) {
        sum += arr[i];  // 数组元素：空间局部性（连续访问）
    }
    return sum;
}

// CPU 会将 arr 的多个元素预取到缓存中
// 后续访问直接命中缓存，速度提升 100 倍
```

#### **3. I/O 设备（Input/Output Devices）**

I/O 设备连接计算机与外部世界。

**分类**：

**块设备（Block Devices）**：
- **特点**：以固定大小的块（通常 512 字节或 4KB）为单位传输
- **支持随机访问**：可以读取任意块
- **示例**：硬盘、SSD、U 盘、光盘

**字符设备（Character Devices）**：
- **特点**：以字节流形式传输
- **通常顺序访问**
- **示例**：键盘、鼠标、串口、打印机

**网络设备**：
- **特点**：以数据包（Packet）为单位传输
- **示例**：网卡、Wi-Fi 适配器

**性能差异巨大**：

| 设备 | 延迟 | 带宽 |
|-----|------|------|
| DRAM | 100 ns | 25 GB/s |
| NVMe SSD | 10 μs | 3 GB/s |
| SATA SSD | 50 μs | 500 MB/s |
| HDD | 10 ms | 100 MB/s |
| 千兆网卡 | 1 ms | 125 MB/s |

**注意**：HDD 延迟是 DRAM 的 **100,000 倍**！这是操作系统需要缓存、异步 I/O 等机制的根本原因。

#### **4. 总线（Bus）**

总线是连接各组件的通信通道。

**主要总线**：

**系统总线（System Bus）**：
- **数据总线**：传输数据（64 位 CPU 通常是 64 位宽）
- **地址总线**：指定内存地址（64 位地址总线可访问 16 EB 内存）
- **控制总线**：传输控制信号（读/写、中断等）

**I/O 总线**：
- **PCI Express（PCIe）**：高速设备（显卡、NVMe SSD）
  - PCIe 4.0 x16：约 32 GB/s
- **USB**：通用外设
  - USB 3.2：约 2.5 GB/s
- **SATA**：传统存储设备
  - SATA 3.0：约 600 MB/s

### 1.1.2 指令执行周期（Fetch-Decode-Execute）

CPU 执行程序的核心过程是一个不断重复的循环。

#### **经典三阶段模型**

```
┌─────────────────────────────────────────┐
│                                         │
│   ┌──────┐   ┌──────┐   ┌──────┐      │
│   │ 取指 │ → │ 解码 │ → │ 执行 │ ──┐  │
│   │Fetch │   │Decode│   │Execute│   │  │
│   └──────┘   └──────┘   └──────┘   │  │
│                                     │  │
│                                     │  │
└─────────────────────────────────────┼──┘
                                      │
                                      └─ 重复
```

#### **阶段详解**

**1. 取指（Fetch）**

**目标**：从内存读取下一条指令。

**步骤**：
1. CPU 将 **PC（程序计数器）** 的值发送到地址总线
2. 控制单元发送"读内存"信号
3. 内存将指令通过数据总线返回给 CPU
4. 指令存储到 **指令寄存器（IR, Instruction Register）**
5. PC 自动递增，指向下一条指令

```
示例：
PC = 0x1000  （假设当前 PC 值）
内存[0x1000] = 0x8B4508  （一条 x86 指令：mov eax, [ebp+8]）

取指后：
IR = 0x8B4508
PC = 0x1003  （x86 指令长度可变，此指令 3 字节）
```

**2. 解码（Decode）**

**目标**：解析指令，确定操作类型和操作数。

**步骤**：
1. 控制单元分析 IR 中的指令
2. 识别操作码（Opcode）：确定是哪种指令（ADD、MOV、JMP 等）
3. 识别操作数（Operands）：从哪里读取数据，结果写到哪里
4. 生成控制信号

```
示例：
IR = 0x8B4508  （mov eax, [ebp+8]）

解码结果：
- 操作码：MOV（数据传送）
- 目标：EAX 寄存器
- 源：内存地址 [EBP + 8]
- 控制信号：
  * 激活 ALU（计算 EBP + 8）
  * 激活内存读取
  * 激活 EAX 写入
```

**3. 执行（Execute）**

**目标**：完成指令指定的操作。

**步骤**：
1. 根据指令类型执行相应操作
2. 算术/逻辑指令：ALU 计算
3. 访存指令：读/写内存
4. 控制指令：修改 PC（跳转、调用）
5. 更新状态寄存器（FLAGS）

```
示例：
执行 mov eax, [ebp+8]

步骤：
1. ALU 计算 EBP + 8 = 0x7FFFFFFC（假设 EBP = 0x7FFFFFF4）
2. 内存读取地址 0x7FFFFFFC 的内容 = 0x12345678
3. 将 0x12345678 写入 EAX 寄存器

结果：
EAX = 0x12345678
```

#### **现代 CPU 的优化：流水线（Pipeline）**

经典三阶段模型效率低下：

```
串行执行：
指令 1: [Fetch][Decode][Execute]
指令 2:                 [Fetch][Decode][Execute]
指令 3:                                 [Fetch][Decode][Execute]

总时间：9 个时钟周期（每阶段 1 周期）
```

**流水线技术**：同时处理多条指令的不同阶段。

```
流水线执行：
        时钟周期: 1  2  3  4  5  6  7  8  9
指令 1:           F  D  E
指令 2:              F  D  E
指令 3:                 F  D  E
指令 4:                    F  D  E
...

总时间：5 个时钟周期（3 条指令）
吞吐量提升：接近 3 倍
```

**现代 CPU 流水线阶段更多**：
- **Intel Core**：14-19 级流水线
- **ARM Cortex-A**：8-15 级流水线

**流水线阶段示例（RISC-V）**：

```
1. IF (Instruction Fetch):      取指
2. ID (Instruction Decode):     解码 + 读寄存器
3. EX (Execute):                ALU 运算
4. MEM (Memory Access):         内存访问
5. WB (Write Back):             写回寄存器
```

**流水线冒险（Hazards）**：

**1. 数据冒险（Data Hazard）**：
```assembly
add x1, x2, x3   # x1 = x2 + x3
sub x4, x1, x5   # x4 = x1 - x5  （需要 x1，但 add 还未写回）

解决方案：
- 数据前递（Forwarding）：从 EX 阶段直接转发结果
- 停顿（Stall）：等待前一条指令完成
```

**2. 控制冒险（Control Hazard）**：
```assembly
beq x1, x2, label  # 如果 x1 == x2，跳转到 label
add x3, x4, x5     # 这条指令是否执行取决于跳转结果

解决方案：
- 分支预测（Branch Prediction）：猜测跳转方向
- 延迟槽（Delay Slot）：跳转指令后总是执行一条指令
```

**3. 结构冒险（Structural Hazard）**：
```
两条指令同时需要访问内存

解决方案：
- 分离指令缓存和数据缓存（Harvard 架构）
- 增加内存端口
```

### 1.1.3 寄存器、缓存、主存层次结构

为什么需要多层次的存储？**因为速度与容量的矛盾无法调和。**

#### **存储器性能差距**

```
寄存器访问：     0.3 ns    (1 个时钟周期 @ 3 GHz)
L1 缓存访问：     1 ns      (3 个时钟周期)
L2 缓存访问：     5 ns      (15 个时钟周期)
L3 缓存访问：    20 ns      (60 个时钟周期)
主存访问：      100 ns      (300 个时钟周期)
SSD 访问：   100,000 ns      (30 万个时钟周期)
HDD 访问：10,000,000 ns      (3000 万个时钟周期)

如果 CPU 等待 HDD，相当于一个人等待 3 个月！
```

#### **层次结构设计**

**核心思想**：
- **小而快**的存储靠近 CPU（寄存器、缓存）
- **大而慢**的存储远离 CPU（主存、磁盘）
- 通过**缓存技术**弥补速度差距

**寄存器（Registers）**

- **数量**：16-32 个（x86-64 有 16 个通用寄存器）
- **容量**：每个 64 位 = 8 字节
- **总容量**：< 1 KB
- **访问速度**：0.3 ns（1 个时钟周期）
- **管理方式**：程序员/编译器显式管理

```c
int add(int a, int b) {
    return a + b;  // a, b 通常存储在寄存器中（如 RAX, RBX）
}
```

**缓存（Cache）**

**L1 缓存**：
- **容量**：32-64 KB/核心
- **分离**：指令缓存（I-Cache）+ 数据缓存（D-Cache）
- **延迟**：1-2 个时钟周期
- **命中率**：95-99%

**L2 缓存**：
- **容量**：256-512 KB/核心
- **统一**：指令和数据共享
- **延迟**：5-10 个时钟周期
- **命中率**：70-80%（L1 未命中时）

**L3 缓存**：
- **容量**：8-64 MB（所有核心共享）
- **延迟**：20-50 个时钟周期
- **命中率**：50-60%（L2 未命中时）

**缓存结构**：

```
┌─────────────────────────────────────────────┐
│            缓存行（Cache Line）              │
│  ┌──────────┬──────────┬─────────────────┐ │
│  │  标签     │  索引    │      数据        │ │
│  │  Tag      │  Index   │      Data        │ │
│  └──────────┴──────────┴─────────────────┘ │
│   (用于比对) (定位缓存行)  (64 字节数据)    │
└─────────────────────────────────────────────┘

典型缓存行大小：64 字节
```

**缓存映射方式**：

**1. 直接映射（Direct-Mapped）**：
```
内存地址 % 缓存行数 = 缓存行位置

优点：简单、快速
缺点：冲突频繁（多个地址映射到同一行）
```

**2. 全相联（Fully Associative）**：
```
数据可以放在任意缓存行

优点：冲突少
缺点：查找慢（需要比较所有标签）
```

**3. 组相联（Set-Associative）**：
```
折中方案：8 路组相联（每组 8 个缓存行）

地址 % 组数 = 组位置
组内 8 个位置任选

Intel CPU 通常使用 8 路或 12 路组相联
```

**缓存替换策略**：

- **LRU（Least Recently Used）**：替换最久未使用的
- **Random**：随机替换（简单但有效）
- **PLRU（Pseudo-LRU）**：近似 LRU，硬件实现更简单

**缓存写策略**：

**写直达（Write-Through）**：
```
写操作同时更新缓存和主存

优点：数据一致性好
缺点：写性能差
```

**写回（Write-Back）**：
```
写操作只更新缓存，稍后再写回主存

优点：写性能好
缺点：需要脏位（Dirty Bit）标记
```

**主存（DRAM）**

- **容量**：8-128 GB
- **延迟**：100 ns
- **带宽**：25-50 GB/s（DDR4/DDR5）
- **管理方式**：操作系统通过页表管理

#### **程序访问内存的实际过程**

```c
int sum = 0;
for (int i = 0; i < n; i++) {
    sum += array[i];  // 访问 array[i]
}
```

**访问 array[i] 的过程**：

1. **CPU 计算地址**：`&array + i * sizeof(int)`
2. **查找 L1 缓存**：
   - 命中 → 1 ns，直接返回数据
   - 未命中 → 继续查找 L2
3. **查找 L2 缓存**：
   - 命中 → 5 ns
   - 未命中 → 继续查找 L3
4. **查找 L3 缓存**：
   - 命中 → 20 ns
   - 未命中 → 访问主存
5. **访问主存**：
   - 从 DRAM 读取数据 → 100 ns
   - 同时将数据加载到所有缓存层次
   - 预取相邻数据（缓存行 64 字节）

**为什么数组遍历快？**

```c
// 连续访问（缓存友好）
for (int i = 0; i < n; i++) {
    sum += array[i];  // 每 64 字节只有 1 次缓存未命中
}

// 随机访问（缓存不友好）
for (int i = 0; i < n; i++) {
    int index = rand() % n;
    sum += array[index];  // 几乎每次都缓存未命中
}

性能差异：可能 10-100 倍！
```

### 1.1.4 总线与设备互连

#### **总线的作用**

总线是共享的通信通道，连接 CPU、内存、I/O 设备。

**传统总线架构**：

```
┌──────┐
│ CPU  │
└──┬───┘
   │
┌──┴──────────────────────────────┐
│       系统总线（FSB / QPI）       │  ← 高速
└──┬─────────┬─────────┬──────────┘
   │         │         │
```

**DMA（Direct Memory Access）**：

**问题**：CPU 参与 I/O 数据传输效率低。

```c
// CPU 参与的 I/O（编程 I/O，PIO）
for (int i = 0; i < block_size; i++) {
    char byte = inb(DISK_DATA_PORT);  // CPU 从设备读一个字节
    buffer[i] = byte;                  // CPU 写入内存
}
// CPU 100% 忙碌，无法执行其他任务
```

**解决方案**：DMA 控制器直接在内存和设备间传输数据。

```c
// DMA 传输
dma_setup(DISK, buffer, block_size);  // 配置 DMA
dma_start();                          // 启动传输
// CPU 可以执行其他任务
// DMA 完成后发送中断通知 CPU
```

**DMA 传输流程**：

```
1. CPU 配置 DMA 控制器：
   - 源地址（设备寄存器）
   - 目标地址（内存地址）
   - 传输长度

2. DMA 控制器接管总线：
   - 从设备读取数据
   - 写入内存
   - 更新地址计数器

3. 传输完成：
   - DMA 发送中断给 CPU
   - CPU 处理数据

优势：CPU 利用率大幅提升
```

## 1.2 处理器模式与特权级

现代操作系统的安全性和稳定性依赖于硬件提供的**特权级别机制**。

---

### 1.2.1 用户态（User Mode）vs 内核态（Kernel Mode）

#### **为什么需要两种模式？**

**假设没有特权级别**：

```c
// 恶意程序可以直接破坏系统
int main() {
    // 1. 访问其他进程的内存
    int *p = (int *)0x12345678;  // 其他进程的地址
    *p = 0;  // 破坏其他进程数据
    
    // 2. 停止整个系统
    asm("cli");  // 禁用中断
    while (1);   // 死循环，系统死锁
    
    // 3. 直接操作硬件
    outb(0x64, 0xFE);  // 重启计算机
}

// 结果：任何程序都能破坏系统
```

**解决方案：特权级别**

CPU 提供两种（或更多）运行模式，限制程序的权限。

#### **用户态（User Mode）**

**特点**：
- **受限权限**：无法执行特权指令
- **内存隔离**：只能访问自己的地址空间
- **需要系统调用**：访问硬件或操作系统服务

**允许的操作**：
- 普通算术/逻辑运算
- 访问自己的内存
- 执行普通指令

**禁止的操作**：
- 修改控制寄存器（如页表基址寄存器）
- 执行 I/O 指令（访问硬件）
- 禁用中断
- 停止 CPU

```c
// 用户态程序
int main() {
    int x = 1 + 2;  // ✓ 允许
    printf("hello");  // ✓ 允许（通过系统调用）
    
    asm("hlt");  // ✗ 禁止（特权指令，尝试执行会触发异常）
}
```

#### **内核态（Kernel Mode / Supervisor Mode）**

**特点**：
- **完全权限**：可以执行所有指令
- **访问所有内存**：包括用户进程和内核内存
- **直接操作硬件**

**典型操作**：
- 处理系统调用
- 中断处理
- 进程调度
- 内存管理
- 设备驱动

```c
// 内核态代码（操作系统）
void syscall_handler() {
    // ✓ 可以访问硬件
    outb(PORT, value);
    
    // ✓ 可以修改控制寄存器
    write_cr3(new_page_table);
    
    // ✓ 可以访问任意内存
    user_memory[0x12345678] = 0;
}
```

#### **模式标识**

**x86/x86-64**：
- **CPL（Current Privilege Level）**：当前特权级别字段，在 CS 寄存器中
  - CPL = 0：内核态（Ring 0）
  - CPL = 3：用户态（Ring 3）

**ARM**：
- **Mode bits** 在 CPSR（Current Program Status Register）中
  - User mode：用户态
  - Supervisor mode：内核态
  - 其他模式：FIQ、IRQ、Abort 等

**RISC-V**：
- **特权模式**：
  - M-mode（Machine）：最高权限
  - S-mode（Supervisor）：内核态
  - U-mode（User）：用户态

### 1.2.2 特权指令与系统调用

#### **特权指令（Privileged Instructions）**

只能在内核态执行的指令。

**x86-64 特权指令示例**：

| 指令 | 功能 | 为什么是特权指令 |
|------|------|---------------|
| `hlt` | 停止 CPU | 防止恶意程序让系统停机 |
| `cli` / `sti` | 禁用/启用中断 | 防止程序关闭中断导致系统无响应 |
| `mov cr3, reg` | 修改页表基址 | 防止程序破坏内存隔离 |
| `lgdt` / `lidt` | 加载 GDT/IDT | 防止程序篡改系统表 |
| `in` / `out` | I/O 端口访问 | 防止程序直接控制硬件 |

**尝试在用户态执行特权指令**：

```assembly
; 用户态程序
mov eax, 0x12345678
mov cr3, eax  ; 尝试修改页表

; 结果：
; CPU 检测到 CPL = 3（用户态）
; 触发 General Protection Fault（#GP 异常）
; 操作系统终止进程
```

#### **系统调用（System Call）**

**定义**：用户态程序请求内核服务的机制。

**为什么需要系统调用？**

用户程序无法直接：
- 读写文件（需要操作文件系统）
- 创建进程（需要操作进程表）
- 分配内存（需要操作页表）
- 发送网络数据（需要操作网络栈）

**系统调用机制**：

```
用户态程序           系统调用接口              内核
   │                    │                      │
   │  open("file.txt")  │                      │
   │ ──────────────────>│                      │
   │                    │  软件中断/syscall     │
   │                    │ ────────────────────>│
   │                    │                      │ sys_open()
   │                    │                      │ 执行实际操作
   │                    │                      │
   │                    │<──────────────────── │
   │<────────────────── │  返回文件描述符       │
   │  fd = 3            │                      │
```

**系统调用的硬件实现**：

**方法 1：软件中断（x86）**

```assembly
; Linux x86-32 系统调用
mov eax, 5         ; 系统调用号（5 = open）
mov ebx, filename  ; 参数 1：文件名
mov ecx, flags     ; 参数 2：标志
mov edx, mode      ; 参数 3：模式
int 0x80           ; 触发中断，陷入内核

; CPU 自动执行：
; 1. 保存当前 CS, EIP, EFLAGS
; 2. 切换到内核态（CPL = 0）
; 3. 跳转到中断处理程序
```

**方法 2：专用指令（x86-64）**

```assembly
; Linux x86-64 系统调用
mov rax, 2         ; 系统调用号（2 = open）
mov rdi, filename  ; 参数 1
mov rsi, flags     ; 参数 2
mov rdx, mode      ; 参数 3
syscall            ; 专用系统调用指令

; CPU 执行：
; 1. 从 MSR 寄存器读取内核入口地址
; 2. 切换到内核态
; 3. 跳转到系统调用处理程序
```

**方法 3：Supervisor Call（ARM）**

```assembly
; ARM 系统调用
mov r7, #5         ; 系统调用号
mov r0, filename   ; 参数 1
mov r1, flags      ; 参数 2
mov r2, mode       ; 参数 3
svc #0             ; Supervisor Call 指令
```

#### **系统调用的性能开销**

```c
// 测试：直接函数调用 vs 系统调用

// 1. 直接函数调用
int sum(int a, int b) {
    return a + b;
}
sum(1, 2);  // 开销：~1 ns

// 2. 系统调用
getpid();  // 获取进程 ID（最简单的系统调用）
// 开销：~100 ns（100 倍）

// 原因：
// - 模式切换（用户态 → 内核态 → 用户态）
// - 寄存器保存/恢复
// - 缓存失效
// - 参数复制
```

**优化：vDSO（Virtual Dynamic Shared Object）**

对于某些简单的系统调用，Linux 将代码映射到用户空间，避免模式切换。

```c
// gettimeofday() 通过 vDSO 优化
struct timeval tv;
gettimeofday(&tv, NULL);  // 无需陷入内核，直接读取共享内存

// 性能提升：100 ns → 20 ns
```

### 1.2.3 模式切换的硬件机制

#### **用户态 → 内核态**

**触发条件**：
1. **系统调用**：程序主动请求（`int 0x80`、`syscall`、`svc`）
2. **异常**：程序错误（除零、非法指令、缺页）
3. **中断**：硬件事件（时钟中断、I/O 完成）

**CPU 自动执行的操作**：

```
1. 保存当前状态：
   - 保存用户态的 PC（程序计数器）
   - 保存用户态的 EFLAGS/CPSR（状态寄存器）
   - 保存用户态的栈指针

2. 切换到内核态：
   - 设置 CPL = 0（或设置特权位）
   - 切换到内核栈（每个进程有独立的内核栈）

3. 跳转到处理程序：
   - 从 IDT（中断描述符表）读取处理程序地址
   - 设置 PC 为处理程序入口

4. 开始执行内核代码
```

**x86-64 示例**：

```assembly
; 用户态代码
mov rax, 1      ; 系统调用号 1 = write
mov rdi, 1      ; 文件描述符 1 = stdout
mov rsi, msg    ; 消息地址
mov rdx, len    ; 消息长度
syscall         ; ← 这里触发模式切换

; ────────── 模式切换 ──────────

; CPU 自动保存到内核栈：
; [RSP + 0]: 用户态 SS
; [RSP + 8]: 用户态 RSP
; [RSP + 16]: 用户态 RFLAGS
; [RSP + 24]: 用户态 CS
; [RSP + 32]: 用户态 RIP

; CPU 从 MSR_LSTAR 读取入口地址并跳转
; 内核态代码（entry_SYSCALL_64）
entry_SYSCALL_64:
    swapgs              ; 切换 GS 寄存器（指向内核数据）
    mov [gs:rsp_scratch], rsp  ; 保存用户栈
    mov rsp, [gs:cpu_current_top_of_stack]  ; 切换到内核栈
    
    push rax            ; 保存寄存器
    push rcx
    push rdx
    ...
    
    call do_syscall_64  ; 调用系统调用处理函数
    
    pop rdx             ; 恢复寄存器
    pop rcx
    pop rax
    ...
    
    mov rsp, [gs:rsp_scratch]  ; 恢复用户栈
    swapgs              ; 恢复用户 GS
    sysretq             ; 返回用户态
```

#### **内核态 → 用户态**

**触发条件**：
1. 系统调用完成
2. 中断/异常处理完成
3. 进程调度切换回用户进程

**CPU 执行的操作**：

```
1. 恢复用户态状态：
   - 从内核栈恢复 PC
   - 恢复 EFLAGS/CPSR
   - 恢复栈指针

2. 切换到用户态：
   - 设置 CPL = 3

3. 跳转回用户代码：
   - PC 设置为保存的返回地址
```

**专用指令**：
- x86：`iret`（Interrupt Return）或 `sysret`（System Call Return）
- ARM：`eret`（Exception Return）
- RISC-V：`sret`（Supervisor Return）

#### **性能影响**

```
模式切换的开销来源：

1. 直接开销（~30-50 个时钟周期）：
   - 保存/恢复寄存器
   - 切换栈
   - 切换页表（TLB 刷新）

2. 间接开销：
   - L1/L2 缓存失效（内核代码替换用户代码）
   - 分支预测器状态丢失
   - 流水线刷新

总开销：约 50-100 ns

频繁的模式切换会严重影响性能！
```

### 1.2.4 保护环（Protection Rings）：x86 的 Ring 0-3

x86 架构提供**四个特权级别**（Ring 0-3），而不仅仅是两个。

#### **四环模型**

```
     Ring 0 (内核态)
        ↑  最高权限
     Ring 1 
        ↑  设备驱动（很少使用）
     Ring 2
        ↑  设备驱动（很少使用）
     Ring 3 (用户态)
        ↓  最低权限
```

**实际使用**：

```
现代操作系统通常只使用两个环：
- Ring 0：内核
- Ring 3：用户程序

Ring 1 和 Ring 2 几乎不用：
- Linux、Windows、macOS 都是 Ring 0 + Ring 3
- 最初设计用于分层保护（如设备驱动在 Ring 1），但实践中太复杂
```

#### **权限检查**

每个段选择子包含 RPL（Requested Privilege Level）和 CPL（Current Privilege Level）。

```
访问规则：
- 代码段：CPL 必须 ≤ DPL（Descriptor Privilege Level）
- 数据段：max(CPL, RPL) 必须 ≤ DPL

示例：
- Ring 3 代码可以读取 Ring 3 数据 ✓
- Ring 3 代码不能读取 Ring 0 数据 ✗
- Ring 0 代码可以读取 Ring 3 数据 ✓（内核可以访问用户内存）
```

#### **调用门（Call Gate）**

允许受控的跨特权级调用。

```
用户程序（Ring 3）
    │
    │ 通过调用门
    ▼
系统服务（Ring 0）
```

**但现代系统更常用系统调用指令（syscall / sysenter），而不是调用门。**

#### **虚拟化的挑战**

**问题**：客户操作系统期望运行在 Ring 0，但不能真的给它 Ring 0 权限。

**解决方案**：

**1. 全虚拟化（Full Virtualization）**：
- 二进制翻译：将客户 Ring 0 代码翻译为 Ring 3 代码
- 模拟特权指令

**2. 半虚拟化（Paravirtualization）**：
- 修改客户操作系统，使其知道自己在虚拟化环境中
- 用 Hypercall 替代特权指令

**3. 硬件辅助虚拟化（Hardware-Assisted Virtualization）**：
- Intel VT-x / AMD-V
- 引入 Ring -1（VMX root mode）给 Hypervisor
- 客户操作系统运行在 Ring 0（VMX non-root mode），但受限

```
     Ring -1 (Hypervisor)
        ↑  最高权限
     Ring 0 (Guest OS)
        ↑  受限的"内核态"
     Ring 3 (Guest 用户程序)
        ↓  最低权限
```

## 1.3 中断与异常机制

中断和异常是操作系统响应硬件事件和程序错误的核心机制。

---

### 1.3.1 中断的分类：硬件中断、软件中断、异常

#### **术语澄清**

不同架构和文献对"中断"和"异常"的定义略有差异，但通常：

**中断（Interrupt）**：
- 外部硬件设备发出的**异步**事件
- 与当前执行的指令**无关**
- 可以在指令执行完成后的任意时刻发生

**异常（Exception）**：
- CPU 执行指令时检测到的**同步**事件
- 由当前执行的指令**直接引起**
- 发生在特定指令执行时

**陷阱（Trap）**：
- 一种特殊的异常，**预期且有意**触发
- 用于系统调用、调试断点

#### **中断的详细分类**

**1. 硬件中断（Hardware Interrupt）**

**定义**：外部设备通过中断控制器通知 CPU 的信号。

**特点**：
- 异步：可以在任意时刻发生
- 与当前指令无关
- 可屏蔽（Maskable）或不可屏蔽（Non-Maskable）

**示例**：
```
时钟中断（Timer Interrupt）:
- 每隔固定时间（如 1ms）触发一次
- 用于：进程调度、时间更新、超时检测

键盘中断（Keyboard Interrupt）:
- 用户按下按键时触发
- 用于：读取输入字符

网卡中断（Network Interrupt）:
- 网络数据包到达时触发
- 用于：接收网络数据

磁盘中断（Disk Interrupt）:
- 磁盘 I/O 完成时触发
- 用于：通知数据已准备好
```

**中断控制器**：

```
旧架构：PIC（Programmable Interrupt Controller）
- 8259 芯片，最多 15 个中断源
- 级联两个 PIC 支持更多中断

现代架构：APIC（Advanced PIC）/ IOAPIC
- 支持更多中断源（224 个）
- 支持多核 CPU
- 支持中断路由（将中断发送到特定 CPU）

更现代：MSI（Message Signaled Interrupts）
- 设备直接向 CPU 发送消息，无需中断控制器
- 更高性能、更低延迟
```

**2. 软件中断（Software Interrupt）**

**定义**：通过执行特殊指令主动触发的中断。

**指令**：
- x86：`int n`（例如 `int 0x80` 用于 Linux 系统调用）
- ARM：`svc #imm`（Supervisor Call）
- RISC-V：`ecall`（Environment Call）

**用途**：
- 系统调用（最常见）
- 软件主动触发的中断处理

```assembly
; x86 系统调用（旧方法）
mov eax, 1      ; 系统调用号 1 = exit
mov ebx, 0      ; 退出码 0
int 0x80        ; 触发软件中断
```

**3. 异常（Exception）**

**定义**：CPU 执行指令时检测到的错误或特殊情况。

**分类**：

**故障（Fault）**：
- 可以被修正并重新执行引发异常的指令
- 示例：
  - **缺页异常（Page Fault）**：访问未映射的内存
  - **除零错误（Division by Zero）**

```c
int x = 1 / 0;  // 触发 Divide-by-Zero Exception

// 处理流程：
// 1. CPU 检测到除零
// 2. 触发异常（#DE，Division Error）
// 3. 跳转到异常处理程序
// 4. 操作系统终止进程（发送 SIGFPE 信号）
```

**陷阱（Trap）**：
- 有意触发，用于调试或系统调用
- 异常处理后，继续执行**下一条**指令
- 示例：
  - **断点异常（Breakpoint）**：`int 3` 指令
  - **系统调用**：`syscall` 指令

```c
// 调试器设置断点
int main() {
    int x = 1;
    asm("int3");  // 触发断点异常
    int y = 2;    // 调试器暂停在这里
}
```

**终止（Abort）**：
- 严重错误，无法恢复
- 示例：
  - **双重故障（Double Fault）**：处理异常时又发生异常
  - **机器检查异常（Machine Check）**：硬件故障

```
处理缺页异常时又发生缺页 → 双重故障 → 系统崩溃（Kernel Panic）
```

#### **完整分类表**

| 类型 | 来源 | 时机 | 返回位置 | 示例 |
|------|------|------|---------|------|
| **硬件中断** | 外部设备 | 异步 | 下一条指令 | 时钟中断、键盘中断 |
| **软件中断** | 指令触发 | 同步 | 下一条指令 | `int 0x80` |
| **故障（Fault）** | 指令错误 | 同步 | 当前指令 | 缺页异常、除零 |
| **陷阱（Trap）** | 预期事件 | 同步 | 下一条指令 | 断点、系统调用 |
| **终止（Abort）** | 严重错误 | 同步 | 不返回 | 双重故障、硬件错误 |

### 1.3.2 中断描述符表（IDT）与中断向量

#### **中断向量（Interrupt Vector）**

**定义**：中断/异常的编号，用于索引处理程序。

**x86-64 中断向量表**：

| 向量号 | 类型 | 描述 | 来源 |
|-------|------|------|------|
| 0 | 故障 | 除零错误 | #DE |
| 1 | 故障/陷阱 | 调试异常 | #DB |
| 2 | 中断 | 非屏蔽中断（NMI） | 硬件 |
| 3 | 陷阱 | 断点 | #BP |
| 4 | 陷阱 | 溢出 | #OF |
| 5 | 故障 | 边界检查 | #BR |
| 6 | 故障 | 无效操作码 | #UD |
| 7 | 故障 | 设备不可用 | #NM |
| 8 | 终止 | 双重故障 | #DF |
| ... | | | |
| 13 | 故障 | 通用保护故障 | #GP |
| 14 | 故障 | 缺页异常 | #PF |
| ... | | | |
| 32-255 | 中断 | 用户定义 | 硬件/软件 |

#### **中断描述符表（IDT, Interrupt Descriptor Table）**

**结构**：256 个条目的数组，每个条目描述一个中断处理程序。

**IDT 条目格式（x86-64）**：

```
┌──────────────────────────────────────┐
│  Offset 63:32 (处理程序地址高 32 位) │
├──────────────────────────────────────┤
│  P  │DPL│  Type  │  Reserved         │
│ (1) │(2)│  (4)   │                   │
├──────────────────────────────────────┤
│  Segment Selector (代码段选择子)     │
├──────────────────────────────────────┤
│  Offset 31:16 (处理程序地址中 16 位) │
├──────────────────────────────────────┤
│  Offset 15:0 (处理程序地址低 16 位)  │
└──────────────────────────────────────┘

字段说明：
- Offset: 处理程序入口地址（64 位）
- Segment Selector: 代码段选择子（指定内核代码段）
- P (Present): 1 = 有效，0 = 无效
- DPL (Descriptor Privilege Level): 允许的最低权限级别
- Type: 中断门、陷阱门、任务门
```

**中断门 vs 陷阱门**：

```
中断门（Interrupt Gate）:
- 进入处理程序时自动禁用中断（清除 IF 标志）
- 防止中断处理程序被其他中断打断
- 用于：硬件中断处理

陷阱门（Trap Gate）:
- 进入处理程序时不禁用中断
- 允许中断嵌套
- 用于：系统调用、调试
```

#### **IDT 的加载与使用**

```assembly
; 定义 IDT
idt_start:
    ; 条目 0: 除零异常
    dw divide_error_handler & 0xFFFF      ; 偏移 15:0
    dw 0x0008                              ; 内核代码段选择子
    db 0                                   ; 保留
    db 0x8E                                ; P=1, DPL=0, Type=Interrupt Gate
    dw (divide_error_handler >> 16) & 0xFFFF  ; 偏移 31:16
    dd divide_error_handler >> 32         ; 偏移 63:32
    dd 0                                   ; 保留
    
    ; 条目 1: 调试异常
    ...
    
    ; 条目 14: 缺页异常
    ...
    
    ; 256 个条目
idt_end:

idt_descriptor:
    dw idt_end - idt_start - 1  ; IDT 大小 - 1
    dq idt_start                 ; IDT 基址

; 加载 IDT
lidt [idt_descriptor]
```

#### **中断处理流程（硬件自动执行）**

```
1. 确定中断向量号 n

2. 从 IDTR（IDT 寄存器）读取 IDT 基址

3. 读取 IDT[n]（第 n 个条目）

4. 检查权限：
   - 检查 Present 位
   - 检查 DPL（如果是软件中断）

5. 保存当前状态到栈：
   - SS（栈段选择子）
   - RSP（栈指针）
   - RFLAGS（标志寄存器）
   - CS（代码段选择子）
   - RIP（返回地址）
   - 错误码（某些异常会压入）

6. 切换到内核态（CPL = 0）

7. 如果是中断门，清除 IF 标志（禁用中断）

8. 从 IDT[n] 读取处理程序地址

9. 跳转到处理程序
```

**栈布局（异常发生后）**：

```
高地址
┌──────────────┐
│  用户态 SS   │
├──────────────┤
│  用户态 RSP  │
├──────────────┤
│  RFLAGS      │
├──────────────┤
│  用户态 CS   │
├──────────────┤
│  用户态 RIP  │  ← 返回地址
├──────────────┤
│  错误码      │  ← 仅部分异常有
└──────────────┘ ← RSP（当前栈顶）
低地址
```

### 1.3.3 中断处理程序的编写与返回

#### **中断处理程序的基本结构**

中断处理程序（Interrupt Service Routine, ISR）必须遵循严格的约定。

**基本要求**：
1. **保存现场**：保存所有可能被修改的寄存器
2. **执行处理逻辑**：完成中断/异常的实际处理
3. **恢复现场**：恢复保存的寄存器
4. **返回**：使用特殊指令返回到被中断的代码

#### **x86-64 中断处理程序示例**

```assembly
; 时钟中断处理程序（汇编层）
timer_interrupt_handler:
    ; 1. 保存寄存器（硬件已保存 SS, RSP, RFLAGS, CS, RIP）
    push rax
    push rbx
    push rcx
    push rdx
    push rsi
    push rdi
    push rbp
    push r8
    push r9
    push r10
    push r11
    push r12
    push r13
    push r14
    push r15
    
    ; 2. 保存段寄存器（如果需要切换）
    mov rax, ds
    push rax
    mov rax, es
    push rax
    
    ; 3. 切换到内核数据段
    mov rax, 0x10  ; 内核数据段选择子
    mov ds, rax
    mov es, rax
    
    ; 4. 调用高级语言处理函数
    call do_timer_interrupt
    
    ; 5. 发送 EOI（End Of Interrupt）信号给中断控制器
    mov al, 0x20
    out 0x20, al   ; PIC 主控制器
    
    ; 6. 恢复段寄存器
    pop rax
    mov es, rax
    pop rax
    mov ds, rax
    
    ; 7. 恢复通用寄存器
    pop r15
    pop r14
    pop r13
    pop r12
    pop r11
    pop r10
    pop r9
    pop r8
    pop rbp
    pop rdi
    pop rsi
    pop rdx
    pop rcx
    pop rbx
    pop rax
    
    ; 8. 返回（硬件自动恢复 RIP, CS, RFLAGS, RSP, SS）
    iretq  ; Interrupt Return (64-bit)
```

**高级语言处理函数**：

```c
// C 语言中断处理函数
void do_timer_interrupt(void) {
    // 1. 更新系统时钟
    jiffies++;  // 全局时钟计数器
    
    // 2. 更新当前进程的时间片
    current_process->time_slice--;
    
    // 3. 检查是否需要调度
    if (current_process->time_slice == 0) {
        schedule();  // 触发进程调度
    }
    
    // 4. 处理定时器队列
    run_timer_list();
}
```

#### **xv6 中断处理示例**

**xv6 的 trap 处理框架**（kernel/trap.c）：

```c
// xv6: 通用 trap 入口
void usertrap(void) {
    int which_dev = 0;
    
    // 确保来自用户态
    if ((r_sstatus() & SSTATUS_SPP) != 0)
        panic("usertrap: not from user mode");
    
    // 设置内核 trap 向量
    w_stvec((uint64)kernelvec);
    
    struct proc *p = myproc();
    
    // 保存用户 PC
    p->trapframe->epc = r_sepc();
    
    // 判断 trap 类型
    if (r_scause() == 8) {
        // 系统调用
        if (p->killed)
            exit(-1);
        
        // sepc 指向 ecall 指令，需要跳过
        p->trapframe->epc += 4;
        
        // 启用中断（允许时钟中断抢占系统调用）
        intr_on();
        
        syscall();  // 处理系统调用
    } 
    else if ((which_dev = devintr()) != 0) {
        // 设备中断（时钟、UART 等）
        // 已在 devintr() 中处理
    } 
    else {
        // 未知异常
        printf("usertrap(): unexpected scause %p pid=%d\n", r_scause(), p->pid);
        printf("            sepc=%p stval=%p\n", r_sepc(), r_stval());
        p->killed = 1;
    }
    
    if (p->killed)
        exit(-1);
    
    // 检查是否需要调度
    if (which_dev == 2)  // 时钟中断
        yield();  // 让出 CPU
    
    usertrapret();  // 返回用户态
}

// 设备中断处理
int devintr() {
    uint64 scause = r_scause();
    
    if ((scause & 0x8000000000000000L) &&
        (scause & 0xff) == 9) {
        // 这是一个外部中断（Supervisor External Interrupt）
        int irq = plic_claim();  // 从 PLIC 获取中断号
        
        if (irq == UART0_IRQ) {
            uartintr();  // UART 中断
        } 
        else if (irq == VIRTIO0_IRQ) {
            virtio_disk_intr();  // 磁盘中断
        } 
        else if (irq) {
            printf("unexpected interrupt irq=%d\n", irq);
        }
        
        if (irq)
            plic_complete(irq);  // 通知 PLIC 中断已处理
        
        return 1;
    } 
    else if (scause == 0x8000000000000001L) {
        // 软件中断（用于时钟）
        if (cpuid() == 0) {
            clockintr();  // 时钟中断处理
        }
        
        // 清除软件中断位
        w_sip(r_sip() & ~2);
        
        return 2;  // 返回 2 表示时钟中断
    } 
    else {
        return 0;  // 不是设备中断
    }
}
```

#### **中断返回指令**

不同架构使用不同的返回指令：

| 架构 | 指令 | 功能 |
|------|------|------|
| x86 | `iret` | 16 位中断返回 |
| x86-32 | `iretd` | 32 位中断返回 |
| x86-64 | `iretq` | 64 位中断返回 |
| ARM | `eret` | 异常返回 |
| RISC-V | `sret` | Supervisor 模式返回 |
| RISC-V | `mret` | Machine 模式返回 |

**iretq 执行的操作**：

```
1. 从栈弹出 RIP  → 恢复程序计数器
2. 从栈弹出 CS   → 恢复代码段
3. 从栈弹出 RFLAGS → 恢复标志寄存器
4. 如果发生特权级变化：
   - 从栈弹出 RSP → 恢复栈指针
   - 从栈弹出 SS  → 恢复栈段
5. 恢复 CPL（当前特权级别）
6. 继续执行被中断的代码
```

#### **中断嵌套**

**问题**：如果在处理中断时又发生中断怎么办？

**方案 1：禁用中断**（中断门）

```c
// 进入中断处理程序时自动禁用中断（IF = 0）
void interrupt_handler() {
    // 此时不会被其他中断打断
    // 处理逻辑...
}
// 返回时自动恢复 IF 标志
```

**优点**：简单、安全
**缺点**：可能错过重要中断（如高优先级中断）

**方案 2：允许中断嵌套**

```c
void interrupt_handler() {
    // 1. 快速保存必要状态
    save_minimal_context();
    
    // 2. 重新启用中断（允许更高优先级中断）
    sti();  // Set Interrupt Flag
    
    // 3. 执行实际处理逻辑
    do_work();
    
    // 4. 禁用中断
    cli();  // Clear Interrupt Flag
    
    // 5. 恢复状态并返回
    restore_context();
    iretq();
}
```

**优点**：响应延迟低
**缺点**：复杂、容易出错（栈溢出、死锁）

**Linux 的方案：中断上下文 + 软中断**

```c
// 硬中断处理：快速、禁用中断
irqreturn_t timer_interrupt(int irq, void *dev_id) {
    // 1. 读取硬件（清除中断源）
    ack_timer();
    
    // 2. 标记软中断
    raise_softirq(TIMER_SOFTIRQ);
    
    return IRQ_HANDLED;
}

// 软中断处理：延迟、允许中断
void run_timer_softirq(struct softirq_action *h) {
    // 实际的复杂处理逻辑
    update_process_times();
    run_local_timers();
    scheduler_tick();
}
```

**中断栈布局（嵌套时）**：

```
高地址
┌────────────────┐
│ 用户态上下文    │ ← 最初保存的用户状态
├────────────────┤
│ 中断 1 处理栈   │
├────────────────┤
│ 中断 1 上下文   │ ← 中断 1 发生时保存的状态
├────────────────┤
│ 中断 2 处理栈   │
├────────────────┤
│ 中断 2 上下文   │ ← 中断 2 发生时保存的状态
└────────────────┘ ← 当前 RSP
低地址

返回顺序：中断 2 返回 → 中断 1 继续 → 中断 1 返回 → 用户代码继续
```

### 1.3.4 中断的性能影响与优化

#### **中断的性能开销**

**直接开销**：
```
保存/恢复寄存器：  ~50 个时钟周期
缓存失效：        ~200 个时钟周期
TLB 失效：        ~100 个时钟周期
分支预测失效：    ~20 个时钟周期
--------------------------------------
总计：            ~370 个时钟周期 ≈ 100-150 ns @ 3 GHz
```

**间接开销**：
- 处理程序代码执行时间（取决于复杂度）
- 后续指令的缓存未命中（因为缓存被中断处理程序污染）

**高频中断的影响**：

```c
// 假设时钟中断频率 1000 Hz（每秒 1000 次）
// 每次中断开销 5 μs

// CPU 开销：
1000 次/秒 × 5 μs/次 = 5000 μs/秒 = 0.5%

// 看起来不多，但实际影响更大（缓存污染、流水线中断）
// 实际开销可能达到 2-5%
```

#### **优化技术**

**1. 中断合并（Interrupt Coalescing）**

```
传统方式：每个数据包到达都触发中断
优化方式：累积多个数据包后再触发一次中断

// 网卡驱动配置
ethtool -C eth0 rx-usecs 50  # 等待 50 微秒或足够数据包

优势：减少中断次数（10000 次/秒 → 1000 次/秒）
劣势：增加延迟（适合吞吐优先的场景）
```

**2. 中断亲和性（Interrupt Affinity）**

```bash
# 将网卡中断绑定到特定 CPU 核心
echo 2 > /proc/irq/45/smp_affinity_list  # 绑定到 CPU 2

# 优势：
# - 减少缓存失效（中断总是在同一核心处理）
# - 避免跨核心通信开销
```

**3. NAPI（New API，Linux）**

```c
// 传统方式：每个数据包都触发中断
irqreturn_t net_interrupt(int irq, void *dev_id) {
    receive_packet();  // 处理一个数据包
    return IRQ_HANDLED;
}

// NAPI 方式：中断 + 轮询混合
irqreturn_t net_interrupt_napi(int irq, void *dev_id) {
    disable_irq(irq);        // 禁用中断
    schedule_poll();         // 调度轮询
    return IRQ_HANDLED;
}

void poll_packets(void) {
    while (has_packets() && budget > 0) {
        receive_packet();    // 轮询接收数据包
        budget--;
    }
    
    if (!has_packets())
        enable_irq(irq);     // 重新启用中断
}

// 高负载时：轮询（避免中断风暴）
// 低负载时：中断（低延迟）
```

**4. 消息信号中断（MSI / MSI-X）**

```
传统 APIC：
设备 → 中断线 → 中断控制器 → CPU

MSI：
设备 → 直接写入内存地址 → CPU

优势：
- 无需中断控制器仲裁
- 支持更多中断源（每个设备可有多个 MSI）
- 更低延迟
```

**5. 用户态驱动与轮询（DPDK）**

```c
// 完全绕过中断，用户态轮询
while (1) {
    if (has_packet()) {
        process_packet();
    }
}

// 优势：
// - 零中断开销
// - 极低延迟（~1 μs）

// 劣势：
// - 100% CPU 占用
// - 不适合低流量场景
```

## 1.4 内存管理单元（MMU）与地址翻译

内存管理单元是实现虚拟内存的硬件基础。

---

### 1.4.1 物理地址 vs 虚拟地址

#### **为什么需要虚拟地址？**

**假设直接使用物理地址**：

```c
// 进程 A
int *p = (int *)0x1000;
*p = 123;

// 进程 B（同时运行）
int *q = (int *)0x1000;
*q = 456;

// 问题：进程 A 和 B 访问同一物理地址，互相破坏数据！
```

**其他问题**：
- 内存碎片化（无法利用不连续的空闲内存）
- 无法实现内存保护（进程可访问任意物理地址）
- 程序必须知道自己的加载地址（链接时确定）

#### **虚拟地址的优势**

**定义**：
- **虚拟地址（Virtual Address, VA）**：程序看到的地址
- **物理地址（Physical Address, PA）**：实际 DRAM 地址

**核心思想**：每个进程有独立的虚拟地址空间。

```
进程 A 的视图：          进程 B 的视图：          物理内存：
┌────────────┐          ┌────────────┐          ┌────────────┐
│ 0x0000...  │          │ 0x0000...  │          │ 0x0000...  │
│            │          │            │          │  内核      │
│ 代码段      │─┐        │ 代码段      │─┐        ├────────────┤
│ 数据段      │ │        │ 数据段      │ │        │  进程 A    │
│ 堆         │ │        │ 堆         │ │        │  物理页    │
│ 栈         │ │        │ 栈         │ │        ├────────────┤
│            │ │        │            │ │        │  进程 B    │
│ 0xFFFF...  │ │        │ 0xFFFF...  │ │        │  物理页    │
└────────────┘ │        └────────────┘ │        ├────────────┤
               │                       │        │  空闲      │
               └───────── MMU ─────────┴──────> │  内存      │
                        翻译                    └────────────┘

进程 A 和 B 的虚拟地址可以相同，但映射到不同的物理地址！
```

**优势**：

**1. 内存隔离**：
```c
// 进程 A
int *p = (int *)0x1000;  // 虚拟地址 0x1000 → 物理地址 0x50000
*p = 123;

// 进程 B
int *q = (int *)0x1000;  // 虚拟地址 0x1000 → 物理地址 0x80000
*q = 456;

// 映射到不同物理地址，不会冲突
```

**2. 简化编程**：
```c
// 所有程序都可以使用相同的地址布局
// 无需关心实际加载到物理内存的哪里

int main() {
    // 代码总是从 0x400000 开始（x86-64 默认）
    // 栈总是从 0x7FFFFFFFFFFF 向下增长
}
```

**3. 内存保护**：
```c
// 页表可以标记权限：只读、读写、可执行

char *code = (char *)0x400000;  // 代码段：只读 + 可执行
*code = 0x90;  // ✗ 写入代码段 → 触发 Segmentation Fault

int *rodata = (int *)0x600000;  // 只读数据段
*rodata = 123;  // ✗ 写入只读段 → Segmentation Fault
```

**4. 共享内存**：
```c
// 多个进程的虚拟地址可以映射到同一物理地址

进程 A：VA 0x1000 ──┐
                   ├──> PA 0x50000（共享库代码）
进程 B：VA 0x2000 ──┘

// 节省内存（libc.so 只需一份物理拷贝）
```

**5. 延迟分配**：
```c
// 分配虚拟内存不需要立即分配物理内存
char *p = malloc(1GB);  // 只分配虚拟地址空间（极快）
// 物理内存在实际访问时才分配（缺页异常）
```

### 1.4.2 分页机制：页表结构与多级页表

#### **分页（Paging）基本概念**

**定义**：
- 将虚拟地址空间和物理内存都划分为固定大小的**页（Page）**
- 典型页大小：4 KB（x86-64）、8 KB、16 KB（ARM）

**术语**：
- **页（Page）**：虚拟内存中的单位
- **页框（Page Frame / Physical Frame）**：物理内存中的单位
- **页表（Page Table）**：虚拟页到物理页框的映射表

#### **地址组成**

```
虚拟地址（VA）= 虚拟页号（VPN）+ 页内偏移（Offset）

示例（4 KB 页，32 位地址）：
┌─────────────────────┬──────────────┐
│  虚拟页号 VPN        │  页内偏移     │
│  (20 bits)          │  (12 bits)   │
└─────────────────────┴──────────────┘
   指定哪一页            页内的具体位置

物理地址（PA）= 物理页框号（PFN）+ 页内偏移（Offset）
┌─────────────────────┬──────────────┐
│  物理页框号 PFN      │  页内偏移     │
│  (20 bits)          │  (12 bits)   │
└─────────────────────┴──────────────┘

注意：页内偏移在虚拟地址和物理地址中相同（页对齐）
```

**地址翻译示例**：

```
虚拟地址：0x00401234
二进制：  00000000 01000000 00010010 00110100

分解：
VPN    = 0x00401234 >> 12 = 0x00401（虚拟页 1025）
Offset = 0x00401234 & 0xFFF = 0x234（页内偏移 564 字节）

查页表：
PageTable[0x00401] = 0x00050（物理页框 80）

物理地址：
PA = (PFN << 12) | Offset
   = (0x00050 << 12) | 0x234
   = 0x00050234
```

#### **页表项（Page Table Entry, PTE）结构**

**x86-64 页表项（64 位）**：

```
┌─┬─┬───┬─┬─┬─┬─┬─┬─┬─┬────────────────────────────────┐
│X│P│...│G│S│D│A│W│U│P│    物理页框号（PFN）            │
│D│K│   │ │ │ │ │ │/│ │    (52-12 = 40 bits)           │
│ │E│   │ │ │ │ │T│S│ │                                 │
└─┴─┴───┴─┴─┴─┴─┴─┴─┴─┴────────────────────────────────┘
 63 62   12 11 8 7 6 5 4 3 2 1 0

字段说明：
P   (Present):          1 = 页在内存中，0 = 不在内存（触发缺页）
U/S (User/Supervisor):  1 = 用户可访问，0 = 仅内核可访问
W/R (Write/Read):       1 = 可写，0 = 只读
A   (Accessed):         1 = 页被访问过（由硬件设置）
D   (Dirty):            1 = 页被写过（由硬件设置）
G   (Global):           1 = 全局页（切换进程时不刷新 TLB）
PAT (Page Attribute Table): 缓存策略
XD  (Execute Disable):  1 = 禁止执行（NX 位，防止代码注入）
PFN (Physical Frame Number): 物理页框号
```

**权限检查**：

```c
// 用户态访问内核页
VA = 0x12345678  // 虚拟地址
PTE = PageTable[VA >> 12]

if (!(PTE & P)) {
    // Present 位为 0 → 缺页异常
    trigger_page_fault();
}

if (CPL == 3 && !(PTE & U/S)) {
    // 用户态访问内核页 → 保护异常
    trigger_protection_fault();
}

if (write_operation && !(PTE & W/R)) {
    // 写只读页 → 保护异常
    trigger_protection_fault();
}

// 权限检查通过
PA = (PTE & ~0xFFF) | (VA & 0xFFF);
```

#### **单级页表的问题**

**示例：32 位地址空间，4 KB 页**

```
虚拟地址空间：2^32 字节 = 4 GB
页大小：      2^12 字节 = 4 KB
页数：        4 GB / 4 KB = 2^20 = 1,048,576 页

每个进程的页表大小：
1,048,576 个页表项 × 4 字节/项 = 4 MB

问题：
- 每个进程需要 4 MB 连续物理内存存储页表
- 100 个进程 = 400 MB 仅用于页表！
- 大部分虚拟地址空间未使用，但页表仍占空间
```

**64 位地址空间更严重**：

```
如果使用单级页表（64 位地址，4 KB 页）：
页数：2^64 / 2^12 = 2^52 个页
页表大小：2^52 × 8 字节 = 32 PB（Petabytes）

完全不可行！
```

#### **多级页表（Multi-Level Page Table）**

**核心思想**：将页表本身也分页。

**x86-64 四级页表**：

```
虚拟地址（48 位有效）：
┌─────┬─────┬─────┬─────┬──────────┐
│ PML4 │ PDPT│ PD  │ PT  │  Offset  │
│ (9b) │ (9b)│ (9b)│ (9b)│  (12b)   │
└─────┴─────┴─────┴─────┴──────────┘
  L4     L3    L2    L1      页内偏移

PML4: Page Map Level 4（第 4 级页表）
PDPT: Page Directory Pointer Table（第 3 级）
PD:   Page Directory（第 2 级）
PT:   Page Table（第 1 级）

每一级：512 个条目（2^9）
```

**查找过程**：

```
1. 从 CR3 寄存器读取 PML4 基址

2. 使用 PML4 索引（VA[47:39]）查找 PML4 表
   PML4_entry = PML4_base[VA[47:39]]
   
3. 从 PML4_entry 获取 PDPT 基址

4. 使用 PDPT 索引（VA[38:30]）查找 PDPT 表
   PDPT_entry = PDPT_base[VA[38:30]]
   
5. 从 PDPT_entry 获取 PD 基址

6. 使用 PD 索引（VA[29:21]）查找 PD 表
   PD_entry = PD_base[VA[29:21]]
   
7. 从 PD_entry 获取 PT 基址

8. 使用 PT 索引（VA[20:12]）查找 PT 表
   PT_entry = PT_base[VA[20:12]]
   
9. 从 PT_entry 获取物理页框号（PFN）

10. 组合物理地址：
    PA = (PFN << 12) | VA[11:0]
```

**示例代码（xv6 RISC-V 三级页表）**：

```c
// xv6: kernel/vm.c
// 遍历页表，查找虚拟地址对应的 PTE
pte_t *walk(pagetable_t pagetable, uint64 va, int alloc) {
    if (va >= MAXVA)
        panic("walk");
    
    // 遍历三级页表
    for (int level = 2; level > 0; level--) {
        pte_t *pte = &pagetable[PX(level, va)];  // 获取当前级别的 PTE
        
        if (*pte & PTE_V) {
            // PTE 有效，获取下一级页表
            pagetable = (pagetable_t)PTE2PA(*pte);
        } else {
            // PTE 无效
            if (!alloc || (pagetable = (pde_t*)kalloc()) == 0)
                return 0;  // 不分配或分配失败
            
            memset(pagetable, 0, PGSIZE);  // 清零新页表
            *pte = PA2PTE(pagetable) | PTE_V;  // 设置 PTE
        }
    }
    
    // 返回最后一级的 PTE
    return &pagetable[PX(0, va)];
}
```

**多级页表的优势**：

```
假设进程只使用 1 GB 虚拟地址（常见情况）

单级页表：
1 GB / 4 KB = 262,144 个页
页表大小：262,144 × 8 字节 = 2 MB（必须连续）

四级页表：
L4: 1 个页表（必须存在）          = 4 KB
L3: 1 个页表（只需 1 个）          = 4 KB
L2: 1 个页表（只需 1 个）          = 4 KB
L1: 256 个页表（1 GB / 4 MB）     = 1 MB
--------------------------------------
总计：                             ≈ 1 MB（可以不连续）

节省空间：2 MB → 1 MB
更重要的是：页表可以分散存储，不需要连续物理内存
```

### 1.4.3 TLB（Translation Lookaside Buffer）

#### **为什么需要 TLB？**

**问题**：多级页表虽然节省空间，但查找慢。

```
四级页表的内存访问次数：
1. 读取 PML4 表  → 访问内存 1 次
2. 读取 PDPT 表  → 访问内存 1 次
3. 读取 PD 表    → 访问内存 1 次
4. 读取 PT 表    → 访问内存 1 次
5. 读取实际数据  → 访问内存 1 次
--------------------------------------
总计：5 次内存访问（每次 ~100 ns）

一次简单的 load 指令：500 ns！
如果没有优化，性能下降 100 倍！
```

#### **TLB：地址翻译缓存**

**定义**：Translation Lookaside Buffer，缓存最近使用的虚拟地址到物理地址的映射。

**结构**：

```
┌─────────────────────────────────────────────┐
│  TLB（小型、全相联或组相联缓存）              │
│  ┌───────────┬────────────┬──────────────┐  │
│  │ 虚拟页号  │ 物理页框号  │  权限/标志    │  │
│  │  (VPN)    │  (PFN)     │  (Flags)     │  │
│  ├───────────┼────────────┼──────────────┤  │
│  │ 0x00401   │ 0x00050    │ U, W, X      │  │
│  ├───────────┼────────────┼──────────────┤  │
│  │ 0x00402   │ 0x00051    │ U, W, -      │  │
│  ├───────────┼────────────┼──────────────┤  │
│  │ 0x00400   │ 0x00048    │ U, R, X      │  │
│  ├───────────┼────────────┼──────────────┤  │
│  │   ...     │   ...      │    ...       │  │
│  └───────────┴────────────┴──────────────┘  │
│  典型大小：64-512 个条目                     │
└─────────────────────────────────────────────┘
```

**查找过程**：

```c
uint64 translate_address(uint64 va) {
    uint64 vpn = va >> 12;
    uint64 offset = va & 0xFFF;
    
    // 1. 查找 TLB
    TLB_Entry *entry = tlb_lookup(vpn);
    
    if (entry && entry->valid) {
        // TLB 命中！
        if (!check_permissions(entry, access_type))
            trigger_protection_fault();
        
        return (entry->pfn << 12) | offset;  // ~1 ns
    }
    
    // 2. TLB 未命中，查找页表
    PTE pte = walk_page_table(va);  // ~400 ns（4 次内存访问）
    
    if (!(pte & PTE_P))
        trigger_page_fault();
    
    // 3. 更新 TLB
    tlb_insert(vpn, pte);
    
    return (PTE_PFN(pte) << 12) | offset;
}
```

**性能提升**：

```
TLB 命中率：95-99%（典型程序）

平均访问时间：
= TLB 命中时间 × 命中率 + TLB 未命中时间 × 未命中率
= 1 ns × 0.98 + 400 ns × 0.02
= 0.98 ns + 8 ns
= 8.98 ns

相比无 TLB（400 ns），性能提升 40 倍！
```

#### **TLB 的类型**

**指令 TLB（ITLB）**：
- 缓存代码页的翻译
- 通常 32-64 个条目

**数据 TLB（DTLB）**：
- 缓存数据页的翻译
- 通常 64-512 个条目

**统一 TLB（Unified TLB）**：
- 指令和数据共享
- 通常作为二级 TLB（L2 TLB）

```
Intel CPU 的 TLB 层次结构：
L1 ITLB:  128 个条目（4 KB 页）
L1 DTLB:  64 个条目（4 KB 页）
L2 TLB:   1536 个条目（统一）
```

#### **TLB 条目结构**

```
┌────────────┬─────────┬───────────────────┬──────┐
│  VPN       │  ASID   │  PFN              │ Flags│
│(虚拟页号)  │(地址空间)│(物理页框号)        │      │
└────────────┴─────────┴───────────────────┴──────┘
   20-40 位    8-16 位      20-40 位        多位

Flags:
- V (Valid):     条目有效
- G (Global):    全局页（所有进程共享）
- D (Dirty):     脏页
- R/W/X:         权限位
```

**ASID（Address Space Identifier）**：

```
问题：进程切换时，新进程的虚拟地址可能与旧进程冲突

旧方案：进程切换时刷新整个 TLB
mov cr3, new_page_table  ; 切换页表基址（自动刷新 TLB）

缺点：下一个进程开始时 TLB 完全冷启动

新方案：使用 ASID 区分不同进程的 TLB 条目
┌──────────┬──────┬────────┬───────┐
│  VPN     │ ASID │  PFN   │ Flags │
├──────────┼──────┼────────┼───────┤
│ 0x00401  │  1   │0x00050 │  ...  │  ← 进程 1 的映射
├──────────┼──────┼────────┼───────┤
│ 0x00401  │  2   │0x00080 │  ...  │  ← 进程 2 的映射（相同 VPN）
└──────────┴──────┴────────┴───────┘

进程切换时只需更新当前 ASID，TLB 条目保留！
```

**全局页（Global Pages）**：

```
内核代码和数据在所有进程中地址相同

设置 G 位：
- 进程切换时不刷新这些 TLB 条目
- 减少 TLB 未命中

示例：
mov cr4, cr4 | CR4_PGE  ; 启用全局页支持
```

#### **TLB 管理**

**硬件管理（x86）**：

```
CPU 自动：
- 查找 TLB
- TLB 未命中时遍历页表
- 自动填充 TLB

操作系统只需：
- 修改页表时刷新 TLB
```

**软件管理（MIPS、早期 RISC）**：

```
TLB 未命中触发异常，由操作系统处理：

void tlb_miss_handler(uint64 va) {
    PTE pte = walk_page_table(va);
    
    if (!(pte & PTE_V))
        trigger_page_fault();
    
    // 选择一个 TLB 条目替换（LRU、Random 等）
    int index = choose_tlb_entry();
    
    // 写入 TLB
    write_tlb_entry(index, va >> 12, PTE_PFN(pte), PTE_FLAGS(pte));
}
```

**TLB 刷新指令**：

```assembly
; x86-64
invlpg [address]  ; 刷新单个页的 TLB 条目
mov cr3, cr3      ; 刷新所有 TLB（除了全局页）

; ARM
tlbi vale1, x0    ; 刷新指定虚拟地址的 TLB

; RISC-V
sfence.vma        ; 刷新所有 TLB
sfence.vma x5, x6 ; 刷新指定 ASID 和虚拟地址
```

### 1.4.4 大页（Huge Pages）与性能优化

#### **大页的动机**

**问题**：TLB 覆盖范围有限。

```
假设：
- TLB 有 64 个条目
- 每个条目对应 4 KB 页

TLB 覆盖的虚拟地址空间：
64 × 4 KB = 256 KB

如果程序访问 10 GB 内存：
10 GB / 256 KB = 40,960 次 TLB 刷新（频繁缺失）
```

**解决方案**：使用更大的页。

```
使用 2 MB 大页：
64 × 2 MB = 128 MB（覆盖范围扩大 512 倍）

使用 1 GB 巨页：
64 × 1 GB = 64 GB（覆盖范围扩大 262,144 倍）
```

#### **大页的大小**

| 架构 | 标准页 | 大页 | 巨页 |
|------|-------|------|------|
| x86-64 | 4 KB | 2 MB | 1 GB |
| ARM64 | 4 KB | 2 MB | 1 GB |
| RISC-V | 4 KB | 2 MB | 1 GB |

**x86-64 页表中的大页**：

```
四级页表：
PML4 → PDPT → PD → PT → 4 KB 页

大页（2 MB）：
PML4 → PDPT → PD → 2 MB 页（跳过 PT）
         在 PD 条目中设置 PS (Page Size) 位

巨页（1 GB）：
PML4 → PDPT → 1 GB 页（跳过 PD 和 PT）
         在 PDPT 条目中设置 PS 位
```

#### **大页的使用**

**1. 显式分配（Explicit Huge Pages）**：

```c
// Linux: 使用 mmap 分配大页
#include <sys/mman.h>

#define HUGE_PAGE_SIZE (2 * 1024 * 1024)  // 2 MB

void *ptr = mmap(NULL, 100 * HUGE_PAGE_SIZE,
                 PROT_READ | PROT_WRITE,
                 MAP_PRIVATE | MAP_ANONYMOUS | MAP_HUGETLB,
                 -1, 0);

if (ptr == MAP_FAILED) {
    perror("mmap huge pages");
    exit(1);
}

// 使用大页内存
memset(ptr, 0, 100 * HUGE_PAGE_SIZE);
```

**配置大页池**：

```bash
# 查看大页配置
cat /proc/meminfo | grep Huge

# 预留 1000 个 2 MB 大页
echo 1000 > /proc/sys/vm/nr_hugepages

# 挂载 hugetlbfs
mount -t hugetlbfs none /mnt/huge
```

**2. 透明大页（Transparent Huge Pages, THP）**：

```bash
# 启用 THP（内核自动管理）
echo always > /sys/kernel/mm/transparent_hugepage/enabled

# 内核会在后台自动：
# - 检测连续的 4 KB 页
# - 合并为 2 MB 大页
# - 在内存压力下拆分回 4 KB 页
```

**THP 的工作原理**：

```
1. 进程分配大量内存：
   malloc(100 MB)  // 返回虚拟地址

2. 进程首次访问：
   *(char *)ptr = 0;  // 缺页异常

3. 内核检查：
   - 是否对齐到 2 MB 边界？
   - 是否有连续的 2 MB 物理内存？
   - 是否启用 THP？

4. 如果满足条件：
   - 分配 2 MB 物理页
   - 设置页表（使用 2 MB 大页）

5. 如果不满足：
   - 分配 4 KB 页（正常流程）
   - 后台线程（khugepaged）尝试合并
```

#### **性能对比**

**示例：访问 10 GB 内存（顺序读取）**

```c
#include <stdio.h>
#include <stdlib.h>
#include <time.h>

void benchmark(size_t size) {
    char *data = malloc(size);
    
    clock_t start = clock();
    
    // 顺序访问所有内存
    for (size_t i = 0; i < size; i += 4096) {
        data[i] = 0;
    }
    
    clock_t end = clock();
    double time = (double)(end - start) / CLOCKS_PER_SEC;
    
    printf("Size: %zu GB, Time: %.2f s\n", size / (1024*1024*1024), time);
    
    free(data);
}

int main() {
    benchmark(10ULL * 1024 * 1024 * 1024);  // 10 GB
}
```

**测试结果**（示例）：

```
标准 4 KB 页：     15.2 s
2 MB 大页：        8.5 s    （性能提升 79%）
1 GB 巨页：        7.8 s    （性能提升 95%）

原因：
- 4 KB 页：约 260 万次 TLB 未命中
- 2 MB 页：约 5000 次 TLB 未命中
- 1 GB 页：约 10 次 TLB 未命中
```

#### **大页的缺点**

**1. 内存浪费**：

```
分配 100 KB 数据：
- 4 KB 页：需要 25 个页 = 100 KB（无浪费）
- 2 MB 页：需要 1 个页 = 2 MB（浪费 1.9 MB）

内部碎片严重！
```

**2. 分配延迟**：

```
找到连续的 2 MB 物理内存比找 4 KB 难得多
可能需要：
- 页面迁移（将现有页移动到其他位置）
- 内存压缩
- 等待内存回收

首次分配可能延迟数百毫秒！
```

**3. 不适合小对象**：

```
数据库缓冲池：       适合（GB 级别，连续访问）
Web 服务器：         不适合（大量小对象）
嵌入式系统：         不适合（内存紧张）
```

#### **最佳实践**

```bash
# 数据库（适合显式大页）
echo 5000 > /proc/sys/vm/nr_hugepages
# 在数据库配置中启用大页支持

# 通用服务器（适合 THP）
echo madvise > /sys/kernel/mm/transparent_hugepage/enabled
# 只在程序显式请求时使用大页

# 延迟敏感应用（禁用 THP）
echo never > /sys/kernel/mm/transparent_hugepage/enabled
# 避免后台大页合并导致的延迟抖动
```

## 1.5 系统启动流程：从加电到操作系统

了解计算机从按下电源按钮到操作系统运行的完整过程。

---

### 1.5.1 BIOS/UEFI 固件初始化

#### **启动流程概览**

```
┌──────────────┐
│ 1. 加电       │  电源供电，CPU 复位
└──────┬───────┘
       ▼
┌──────────────┐
│ 2. BIOS/UEFI │  硬件自检，初始化设备
└──────┬───────┘
       ▼
┌──────────────┐
│ 3. 启动加载器│  GRUB / Windows Boot Manager
└──────┬───────┘
       ▼
┌──────────────┐
│ 4. 操作系统   │  内核加载，初始化
└──────┬───────┘
       ▼
┌──────────────┐
│ 5. 用户空间   │  init / systemd，登录
└──────────────┘
```

#### **CPU 复位状态**

当按下电源按钮时，CPU 进入已知的初始状态。

**x86-64 CPU 复位后**：

```
CS (代码段):      0xF000
IP (指令指针):     0xFFF0
物理地址:         0xFFFF0  (实模式地址计算：CS << 4 + IP)

这个地址通常映射到 BIOS ROM

EFLAGS:          0x00000002
CR0:             0x60000010  (实模式，分页禁用)
所有通用寄存器:    未定义（通常为 0）
```

**为什么是 0xFFFF0？**

```
0xFFFF0 位于 1 MB 地址空间的顶部（实模式只能访问 1 MB）
这个地址通常存储一条跳转指令：

jmp far 0xF000:0xE05B  ; 跳转到 BIOS 真正的启动代码
```

#### **BIOS（Basic Input/Output System）**

**BIOS 的主要任务**：

**1. POST（Power-On Self-Test，加电自检）**：

```
1. 检查 CPU 寄存器
2. 验证 BIOS 代码校验和
3. 检测内存（逐页测试）
4. 初始化计时器
5. 初始化 DMA 控制器
6. 检查显卡（显示 POST 信息）
7. 检测键盘
8. 检测硬盘、光驱、USB 设备

如果发现错误：
- 屏幕显示错误代码
- 或发出蜂鸣声（Beep 代码）
  示例：3 短 1 长 = 内存错误
```

**2. 硬件初始化**：

```
- 初始化中断控制器（PIC）
- 设置中断向量表（IVT）
- 初始化实时时钟（RTC）
- 设置内存映射
```

**BIOS 中断服务（实模式下）**：

```assembly
; BIOS 提供的中断服务
int 0x10  ; 视频服务
int 0x13  ; 磁盘服务
int 0x16  ; 键盘服务

; 示例：在屏幕上显示字符 'A'
mov ah, 0x0E  ; 功能：写字符
mov al, 'A'   ; 字符
int 0x10      ; 调用 BIOS
```

**3. 启动设备选择**：

```
BIOS 按预定顺序查找可启动设备：
1. 软盘
2. CD-ROM
3. 硬盘
4. USB 设备
5. 网络（PXE 启动）

可在 BIOS 设置中调整顺序
```

**4. 加载引导扇区**：

```
找到可启动设备后：

1. 读取第一个扇区（512 字节）
   - 位置：磁盘的 LBA 0（逻辑块地址 0）
   - 也称为 MBR（Master Boot Record，主引导记录）

2. 检查引导签名：
   - 最后两字节必须是 0x55 0xAA
   - 如果不是 → 尝试下一个设备

3. 将 MBR 加载到内存 0x7C00

4. 跳转到 0x7C00 开始执行
   jmp 0x0000:0x7C00
```

**MBR 结构**：

```
┌──────────────────────────────────────┐  0x0000
│                                      │
│    引导代码（Boot Code）              │  446 字节
│                                      │
├──────────────────────────────────────┤  0x01BE
│    分区表（Partition Table）          │  64 字节
│    (4 个分区条目，每个 16 字节)       │
├──────────────────────────────────────┤  0x01FE
│    引导签名（Boot Signature）         │  2 字节
│    0x55 0xAA                         │
└──────────────────────────────────────┘  0x0200 (512 字节)
```

#### **UEFI（Unified Extensible Firmware Interface）**

UEFI 是 BIOS 的现代替代品。

**UEFI 的优势**：

```
BIOS:
- 16 位实模式代码
- 1 MB 内存限制
- 只能启动 MBR 磁盘（最大 2 TB）
- 文本界面
- 无网络支持

UEFI:
- 32/64 位保护模式
- 无内存限制（可访问全部 RAM）
- 支持 GPT 磁盘（远超 2 TB）
- 图形界面
- 内置网络栈（支持 PXE、HTTP 启动）
- 安全启动（Secure Boot）
```

**UEFI 启动流程**：

```
1. SEC (Security Phase):
   - CPU 初始化
   - 临时内存初始化

2. PEI (Pre-EFI Initialization):
   - 内存初始化
   - CPU、芯片组初始化

3. DXE (Driver Execution Environment):
   - 加载设备驱动
   - 构建 UEFI 服务表

4. BDS (Boot Device Selection):
   - 读取启动选项（Boot Variables）
   - 选择启动设备

5. TSL (Transient System Load):
   - 加载启动加载器（如 GRUB）
   - 启动加载器继续引导操作系统

6. RT (Runtime):
   - 进入操作系统
   - UEFI 提供运行时服务（读写 NVRAM、时间服务等）
```

**GPT（GUID Partition Table）**：

```
MBR 磁盘布局：
┌─────────────┐
│    MBR      │  1 个扇区
├─────────────┤
│  分区 1     │
├─────────────┤
│  分区 2     │
└─────────────┘

GPT 磁盘布局：
┌─────────────┐
│ 保护性 MBR   │  1 个扇区（兼容性）
├─────────────┤
│ GPT 头       │  1 个扇区
├─────────────┤
│ 分区表项     │  通常 32 个扇区（最多 128 个分区）
├─────────────┤
│ ESP 分区     │  EFI System Partition（存储启动加载器）
├─────────────┤
│ 分区 1       │
├─────────────┤
│ ...          │
└─────────────┘

ESP 分区：
- FAT32 文件系统
- 存储路径：/EFI/BOOT/BOOTX64.EFI（默认启动文件）
- 或：/EFI/ubuntu/grubx64.efi（Ubuntu 的 GRUB）
```

### 1.5.2 引导加载器（Boot Loader）

#### **引导加载器的作用**

**问题**：BIOS/UEFI 只能加载很小的代码（512 字节 MBR 或小型 EFI 应用），无法直接加载复杂的操作系统。

**解决方案**：引导加载器（Boot Loader）作为中间层。

**典型引导加载器**：
- **GRUB（Grand Unified Bootloader）**：Linux 主流
- **Windows Boot Manager**：Windows 10/11
- **rEFInd**：UEFI 多系统引导
- **U-Boot**：嵌入式系统

#### **GRUB 的工作原理**

**多阶段引导**：

```
阶段 1：MBR 中的引导代码（446 字节）
  ↓
阶段 1.5：核心映像（存储在 MBR 和第一个分区之间的间隙）
  ↓
阶段 2：GRUB 主程序（存储在 /boot/grub/）
  ↓
内核加载
```

**GRUB Stage 1（boot.img）**：

```assembly
; GRUB Stage 1（简化版本）
.code16                     ; 16 位实模式代码
.section .text

_start:
    cli                     ; 禁用中断
    xor ax, ax              ; AX = 0
    mov ds, ax              ; DS = 0
    mov es, ax              ; ES = 0
    mov ss, ax              ; SS = 0
    mov sp, 0x7C00          ; 设置栈指针
    
    ; 加载 Stage 1.5 或 Stage 2
    mov ah, 0x02            ; BIOS 读扇区功能
    mov al, num_sectors     ; 读取扇区数
    mov ch, 0               ; 柱面号
    mov cl, 2               ; 扇区号（从第 2 扇区开始）
    mov dh, 0               ; 磁头号
    mov dl, boot_drive      ; 驱动器号
    mov bx, 0x7E00          ; 目标地址
    int 0x13                ; BIOS 磁盘服务
    
    jc error                ; 如果出错，跳转
    
    jmp 0x0000:0x7E00       ; 跳转到 Stage 1.5
    
error:
    mov si, error_msg
    call print_string
    hlt
    
print_string:
    lodsb                   ; AL = [SI++]
    or al, al               ; AL == 0?
    jz done
    mov ah, 0x0E            ; BIOS 写字符
    int 0x10
    jmp print_string
done:
    ret
    
error_msg: .asciz "Boot error!\r\n"
boot_drive: .byte 0x80      ; 第一块硬盘

.org 510
.word 0xAA55                ; 引导签名
```

**GRUB Stage 2（核心功能）**：

```
1. 切换到保护模式（32 位）
2. 加载文件系统驱动（ext4、FAT、NTFS 等）
3. 读取配置文件（/boot/grub/grub.cfg）
4. 显示启动菜单
5. 加载内核和 initramfs
6. 跳转到内核入口
```

**grub.cfg 示例**：

```bash
# /boot/grub/grub.cfg

# 默认启动项
set default=0
set timeout=5

# 菜单项 1：Ubuntu
menuentry 'Ubuntu 22.04' {
    set root='hd0,gpt2'           # 设置根设备（第 1 块硬盘的第 2 个 GPT 分区）
    linux /boot/vmlinuz-5.15.0-76-generic root=UUID=xxxx-xxxx ro quiet splash
    initrd /boot/initrd.img-5.15.0-76-generic
}

# 菜单项 2：恢复模式
menuentry 'Ubuntu 22.04 (recovery mode)' {
    set root='hd0,gpt2'
    linux /boot/vmlinuz-5.15.0-76-generic root=UUID=xxxx-xxxx ro recovery nomodeset
    initrd /boot/initrd.img-5.15.0-76-generic
}

# 菜单项 3：Windows
menuentry 'Windows 11' {
    set root='hd0,gpt1'           # EFI System Partition
    chainloader /EFI/Microsoft/Boot/bootmgfw.efi
}
```

**GRUB 加载内核的过程**：

```c
// GRUB 内部逻辑（简化）

void load_kernel(void) {
    // 1. 读取内核文件（vmlinuz）
    void *kernel = read_file("/boot/vmlinuz-5.15.0-76-generic");
    
    // 2. 解析内核格式
    //    - bzImage（x86-64 压缩内核）
    //    - ELF（某些架构）
    
    // 3. 将内核加载到内存
    //    通常加载到物理地址 0x100000（1 MB）
    memcpy((void *)0x100000, kernel, kernel_size);
    
    // 4. 读取 initramfs（初始 RAM 文件系统）
    void *initramfs = read_file("/boot/initrd.img-5.15.0-76-generic");
    memcpy((void *)initramfs_addr, initramfs, initramfs_size);
    
    // 5. 设置内核启动参数（存储在约定的内存位置）
    struct boot_params *params = (struct boot_params *)0x10000;
    params->hdr.type_of_loader = 0x71;  // GRUB
    params->hdr.cmd_line_ptr = (uint32_t)cmdline;
    params->hdr.ramdisk_image = initramfs_addr;
    params->hdr.ramdisk_size = initramfs_size;
    
    // 6. 切换到保护模式（如果还未切换）
    enable_protected_mode();
    
    // 7. 跳转到内核入口点
    //    x86-64 内核入口：0x100000 + 偏移
    void (*kernel_entry)(void) = (void (*)(void))0x100000;
    kernel_entry();  // 进入内核，不再返回
}
```

#### **UEFI 启动加载器**

**UEFI 应用程序格式**：PE/COFF（与 Windows EXE 相同）

```
EFI System Partition（ESP）目录结构：
/EFI/
  ├── BOOT/
  │   └── BOOTX64.EFI          ← 默认启动文件
  ├── ubuntu/
  │   ├── grubx64.efi          ← GRUB for UEFI
  │   └── grub.cfg
  ├── Microsoft/

  └── refind/
      └── refind_x64.efi       ← rEFInd 引导管理器
```

**UEFI GRUB 的优势**：

```
1. 直接运行在 64 位模式（无需切换）
2. 可以使用 UEFI 提供的服务：
   - 文件系统访问（无需自己实现驱动）
   - 图形输出（GOP，Graphics Output Protocol）
   - 网络栈（PXE 启动）
3. 支持安全启动（Secure Boot）
```

### 1.5.3 内核加载与初始化

#### **Linux 内核启动过程**

**内核入口点（x86-64）**：

```assembly
; arch/x86/boot/compressed/head_64.S
; 内核启动入口

.code32                         ; 开始仍然是 32 位模式
startup_32:
    cli                         ; 禁用中断
    
    ; 1. 设置段寄存器
    movl $__BOOT_DS, %eax
    movl %eax, %ds
    movl %eax, %es
    movl %eax, %ss
    
    ; 2. 建立临时页表（启用长模式需要分页）
    call make_boot_page_tables
    
    ; 3. 启用 PAE（Physical Address Extension）
    movl %cr4, %eax
    orl $X86_CR4_PAE, %eax
    movl %eax, %cr4
    
    ; 4. 加载页表基址
    leal pgtable(%ebx), %eax
    movl %eax, %cr3
    
    ; 5. 启用长模式（IA-32e）
    movl $MSR_EFER, %ecx
    rdmsr                       ; 读取 EFER 寄存器
    orl $EFER_LME, %eax        ; 设置 Long Mode Enable 位
    wrmsr
    
    ; 6. 启用分页（这会激活长模式）
    movl %cr0, %eax
    orl $X86_CR0_PG, %eax      ; 设置 PG 位
    movl %eax, %cr0
    
    ; 7. 跳转到 64 位代码
    ljmp $__KERNEL_CS, $startup_64
    
.code64                         ; 现在是 64 位模式
startup_64:
    ; 8. 设置 64 位段寄存器
    xorl %eax, %eax
    movl %eax, %ds
    movl %eax, %es
    movl %eax, %ss
    
    ; 9. 解压内核（bzImage 是压缩的）
    call extract_kernel         ; 解压到内存
    
    ; 10. 跳转到解压后的内核
    jmp *%rax                   ; RAX 保存解压后内核的入口地址
```

**内核 C 代码入口（start_kernel）**：

```c
// init/main.c
// 内核初始化的主函数

asmlinkage __visible void __init start_kernel(void) {
    char *command_line;
    char *after_dashes;
    
    // 1. 设置初始任务（idle 进程）
    set_task_stack_end_magic(&init_task);
    
    // 2. 初始化 CPU
    smp_setup_processor_id();
    cgroup_init_early();
    local_irq_disable();        // 禁用中断
    boot_cpu_init();
    
    // 3. 打印内核版本信息
    pr_notice("%s", linux_banner);  // "Linux version 5.15.0..."
    
    // 4. 初始化内存管理
    setup_arch(&command_line);      // 架构相关初始化
    mm_init_cpumask(&init_mm);
    setup_command_line(command_line);
    
    // 5. 初始化调度器
    sched_init();
    
    // 6. 初始化中断系统
    trap_init();                    // 设置异常处理程序
    init_IRQ();                     // 初始化中断控制器
    
    // 7. 初始化时钟
    tick_init();
    init_timers();
    hrtimers_init();
    
    // 8. 初始化内存分配器
    page_alloc_init();
    kmem_cache_init();
    
    // 9. 初始化 RCU（Read-Copy-Update）

    // 10. 初始化进程 0（idle 进程）

    proc_caches_init();
    
    // 11. 初始化缓冲区和缓存

    vfs_caches_init();

}
```

**rest_init() 创建用户空间 init**：

```c
// init/main.c
static noinline void __init rest_init(void) {
    int pid;
    
    // 创建 kernel_init 内核线程（将成为 init 进程）
    pid = kernel_thread(kernel_init, NULL, CLONE_FS);
    if (pid < 0)
        panic("Failed to create init process");
    
    // 创建 kthreadd 内核线程（管理其他内核线程）
    pid = kernel_thread(kthreadd, NULL, CLONE_FS | CLONE_FILES);
    if (pid < 0)
        panic("Failed to create kthreadd");
    
    // 当前线程变为 idle 进程（PID 0）
    init_idle_preempt_count(current, smp_processor_id());
    schedule_preempt_disabled();
    
    // idle 进程的主循环（永远运行）
    cpu_startup_entry(CPUHP_ONLINE);
}
```

**kernel_init 转变为 init 进程**：

```c
// init/main.c
static int __ref kernel_init(void *unused) {
    kernel_init_freeable();  // 完成各种初始化
    
    // 尝试运行用户空间的 init 程序
    if (ramdisk_execute_command) {
        run_init_process(ramdisk_execute_command);
    }
    
    // 尝试标准路径
    if (!try_to_run_init_process("/sbin/init") ||
        !try_to_run_init_process("/etc/init") ||
        !try_to_run_init_process("/bin/init") ||
        !try_to_run_init_process("/bin/sh"))
        return 0;
    
    panic("No working init found.");
}

static int run_init_process(const char *init_filename) {
    argv_init[0] = init_filename;
    return do_execve(getname_kernel(init_filename),
                     (const char __user *const __user *)argv_init,
                     (const char __user *const __user *)envp_init);
}
```

#### **initramfs（初始 RAM 文件系统）**

**为什么需要 initramfs？**

```
问题：内核需要挂载根文件系统，但根文件系统可能在：
- 需要驱动程序的设备上（RAID、LVM、加密磁盘）
- 网络文件系统（NFS）
- 内核编译时不包含的文件系统类型

解决方案：initramfs 提供一个临时的根文件系统
```

**initramfs 的内容**：

```
initramfs/
├── bin/                    # 基本工具（busybox）
│   ├── sh → busybox
│   ├── mount → busybox
│   └── ...
├── sbin/
│   └── init                # init 脚本
├── dev/                    # 设备节点
├── proc/                   # proc 挂载点
├── sys/                    # sysfs 挂载点
├── lib/                    # 共享库和内核模块
│   └── modules/
│       └── 5.15.0/
│           ├── kernel/
│           │   ├── drivers/
│           │   │   ├── scsi/
│           │   │   └── md/  # RAID 驱动
│           │   └── fs/
│           │       └── ext4/
│           └── modules.dep
└── init                    # 启动脚本（可执行）
```

**initramfs 的 init 脚本**：

```bash
#!/bin/sh
# initramfs 的 /init 脚本

# 1. 挂载虚拟文件系统
mount -t proc none /proc
mount -t sysfs none /sys
mount -t devtmpfs none /dev

# 2. 加载必要的内核模块
modprobe ext4
modprobe ahci        # SATA 驱动

# 3. 等待设备就绪
udevd --daemon
udevadm settle

# 4. 挂载真正的根文件系统
#    通过内核参数 root=UUID=xxxx 或 root=/dev/sda1 指定
mount -o ro ${rootfs} /mnt/root

# 5. 切换到真正的根文件系统
cd /mnt/root
pivot_root . old_root

# 6. 执行真正的 init（systemd 或 sysvinit）
exec /sbin/init
```

**pivot_root vs chroot**：

```
chroot:
- 改变当前进程的根目录
- 旧的根文件系统仍然可以访问（通过 /proc/1/root）
- 主要用于沙箱隔离

pivot_root:
- 交换根文件系统和当前目录
- 旧的根文件系统移动到新位置
- 可以完全卸载旧的根文件系统
- 用于系统启动
```

### 1.5.4 用户空间初始化：init / systemd

#### **init 进程（PID 1）的职责**

```
1. 作为所有用户进程的祖先
   - 所有进程（除了内核线程）都由 init 直接或间接创建

2. 回收孤儿进程
   - 当进程的父进程退出时，进程会被 init 收养
   - init 负责调用 wait() 回收僵尸进程

3. 系统服务管理
   - 启动、停止、重启系统服务
   - 管理服务依赖关系

4. 运行级别管理（传统 SysV init）
   - 单用户模式、多用户模式、图形界面等
```

#### **传统 SysV Init**

**运行级别**：

```
运行级别 0: 关机（halt）
运行级别 1: 单用户模式（救援模式）
运行级别 2: 多用户模式（无网络）
运行级别 3: 多用户模式（命令行）
运行级别 4: 未使用（自定义）
运行级别 5: 图形界面（X11）
运行级别 6: 重启（reboot）
```

**/etc/inittab 配置**：

```
# /etc/inittab
id:3:initdefault:               # 默认运行级别 3

# 系统初始化脚本
si::sysinit:/etc/rc.d/rc.sysinit

# 各运行级别的脚本
l0:0:wait:/etc/rc.d/rc 0        # 运行级别 0
l1:1:wait:/etc/rc.d/rc 1        # 运行级别 1
l3:3:wait:/etc/rc.d/rc 3        # 运行级别 3
l5:5:wait:/etc/rc.d/rc 5        # 运行级别 5
l6:6:wait:/etc/rc.d/rc 6        # 运行级别 6

# Getty（登录提示符）
1:2345:respawn:/sbin/mingetty tty1
2:2345:respawn:/sbin/mingetty tty2
```

**启动脚本结构**：

```
/etc/rc.d/
├── rc.sysinit              # 系统初始化
├── rc                      # 运行级别切换脚本
├── rc0.d/                  # 运行级别 0 的脚本（关机）
│   ├── K01network → ../init.d/network
│   └── S00halt → ../init.d/halt
├── rc3.d/                  # 运行级别 3 的脚本
│   ├── S10network → ../init.d/network
│   ├── S55sshd → ../init.d/sshd
│   └── S80httpd → ../init.d/httpd
└── init.d/                 # 实际的服务脚本
    ├── network
    ├── sshd
    └── httpd

K = Kill（停止服务）
S = Start（启动服务）
数字 = 启动顺序
```

**SysV Init 的问题**：

```
1. 启动缓慢
   - 所有服务串行启动（一个接一个）
   - 无法利用多核 CPU

2. 脚本复杂
   - 每个服务需要编写启动/停止脚本
   - 依赖关系难以管理

3. 难以监控
   - 服务崩溃后无法自动重启
   - 缺乏日志管理
```

#### **现代 systemd**

**systemd 的优势**：

```
1. 并行启动
   - 利用多核 CPU 同时启动多个服务
   - 通过依赖关系智能调度

2. Socket 激活
   - 服务不需要立即启动
   - 首次访问时才启动（按需激活）

3. 资源控制
   - 通过 cgroup 限制 CPU、内存、I/O
   - 防止单个服务占用过多资源

4. 统一日志
   - journald 收集所有日志
   - 支持结构化日志查询
```

**systemd 单元（Unit）类型**：

```
service:     系统服务（如 sshd.service）
socket:      套接字（用于激活服务）
device:      设备文件（如 /dev/sda）
mount:       挂载点（如 /home）
automount:   自动挂载点
target:      运行级别（如 multi-user.target）
timer:       定时任务（替代 cron）
```

**服务单元示例**（/etc/systemd/system/myapp.service）：

```ini
[Unit]
Description=My Application
After=network.target         # 依赖：在网络启动后
Requires=postgresql.service  # 强依赖：PostgreSQL 必须运行

[Service]
Type=simple                  # 服务类型
User=myapp                   # 运行用户
WorkingDirectory=/opt/myapp
ExecStart=/opt/myapp/bin/server  # 启动命令
ExecReload=/bin/kill -HUP $MAINPID  # 重载命令
Restart=on-failure           # 崩溃时自动重启
RestartSec=5s                # 重启间隔

# 资源限制
CPUQuota=50%                 # 最多使用 50% CPU
MemoryLimit=1G               # 最多使用 1 GB 内存

[Install]
WantedBy=multi-user.target   # 默认运行级别
```

**systemd 启动流程**：

```
1. 内核启动 systemd（PID 1）

2. systemd 读取默认 target：
   /etc/systemd/system/default.target → multi-user.target

3. multi-user.target 依赖的所有单元：
   ┌─ basic.target
   │  ├─ sysinit.target
   │  │  ├─ local-fs.target
   │  │  ├─ swap.target
   │  │  └─ ...
   │  └─ sockets.target
   ├─ network.target
   ├─ sshd.service
   └─ ...

4. systemd 并行启动所有单元（尊重依赖关系）

5. 所有服务就绪后，进入 multi-user.target
```

**常用 systemd 命令**：

```bash
# 服务管理
systemctl start myapp.service      # 启动服务
systemctl stop myapp.service       # 停止服务
systemctl restart myapp.service    # 重启服务
systemctl status myapp.service     # 查看状态

# 开机自启动
systemctl enable myapp.service     # 启用开机自启动
systemctl disable myapp.service    # 禁用开机自启动

# 查看日志
journalctl -u myapp.service        # 查看服务日志
journalctl -f                      # 实时查看所有日志
journalctl --since "1 hour ago"    # 查看最近 1 小时的日志

# 系统管理
systemctl poweroff                 # 关机
systemctl reboot                   # 重启
systemctl rescue                   # 进入救援模式
```

**systemd 依赖关系可视化**：

```bash
# 查看服务的依赖树
systemctl list-dependencies sshd.service

# 输出示例：
sshd.service
├─ system.slice
├─ sshd-keygen.target
│ ├─ sshd-keygen@ecdsa.service
│ ├─ sshd-keygen@ed25519.service
│ └─ sshd-keygen@rsa.service
└─ sysinit.target
  ├─ dev-hugepages.mount
  ├─ local-fs.target
  └─ ...
```

## 1.6 本章总结

### 核心概念回顾

**1. 硬件基础**
- 冯·诺依曼架构：CPU、内存、I/O、总线
- 指令执行周期：取指、解码、执行
- 存储层次结构：寄存器、缓存、主存、磁盘
- DMA：直接内存访问，减轻 CPU 负担

**2. 处理器特权级**
- 用户态 vs 内核态：权限隔离的基础
- 特权指令：只能在内核态执行
- 系统调用：用户态请求内核服务的机制
- 保护环：x86 的 Ring 0-3 模型

**3. 中断与异常**
- 硬件中断：外部设备异步通知
- 软件中断：主动触发（系统调用）
- 异常：指令执行错误（缺页、除零）
- IDT：中断描述符表，映射中断到处理程序
- 中断处理：保存现场、执行处理程序、恢复现场

**4. 内存管理单元（MMU）**
- 虚拟地址 vs 物理地址：内存抽象与隔离
- 分页机制：固定大小的页（4 KB）
- 多级页表：节省空间，支持稀疏地址空间
- TLB：缓存地址翻译，提升性能
- 大页：减少 TLB 未命中，适合大内存应用

**5. 系统启动**
- BIOS/UEFI：硬件初始化与自检
- 引导加载器：GRUB 加载内核
- 内核初始化：从实模式到保护模式，再到长模式
- init/systemd：用户空间初始化，服务管理

### 关键数据对比

| 组件 | 容量/速度 | 访问延迟 |

| L1 缓存 | 64 KB | 1 ns |

| 操作 | 开销 |
|------|------|
